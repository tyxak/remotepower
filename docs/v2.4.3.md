# v2.4.3 release notes

Released: 2026-05-17.

A lightweight mailbox monitor — see an unread-message count on the
dashboard without any IMAP/SMTP integration.

## Added

### Mailbox-count monitor

RemotePower can now count the messages in a mailbox and show the
number on the dashboard.

It works by counting files, not by talking email protocols. You
give a device one or more **directory paths**; the agent counts
the regular files directly inside each one and reports the
numbers in its heartbeat. For a Maildir mailbox the `new/` folder
holds one file per unread message, so a file count *is* the
unread count:

    /var/mail/vhosts/tvipper.com/jmo/new   →   N unread

This is the simple alternative to full IMAP/SMTP integration that
the feature request asked for. There is no email protocol client,
no credentials to store, and no message content is ever read —
only counted.

**Configuration.** Open a device's detail view → "Mailbox
monitor" section. Enter one absolute directory path per line and
Save. The agent picks up the list on its next heartbeat and
starts reporting counts a heartbeat later.

**Dashboard promotion.** The same section has a "Show on the
dashboard" checkbox. Tick it to *promote* the device — its
mailbox counts then appear in a "Mailbox monitor" widget on the
Home dashboard. Devices that aren't promoted are still monitored;
they just don't take up dashboard space. If nothing is promoted
the widget is hidden entirely.

**How the counting works.** The agent counts with `os.scandir()`,
non-recursively (depth 1, regular files only) — equivalent to
`find <path> -maxdepth 1 -type f | wc -l` but without invoking a
shell. No shell means no quoting or injection surface. A missing
or unreadable path is reported as such (the widget shows the
reason) rather than silently dropped.

## Tests

`tests/test_v243.py` (14 tests): the agent counter (regular files
only, subdirectories excluded, empty dir = 0, missing path,
file-not-directory, multiple paths, the path cap); server ingest
(stores a clean snapshot, rejects a nonsense negative count);
the config endpoint (stores + de-duplicates + drops relative
paths, the empty-list-clears-state case, rejects a non-list); the
overview endpoint (only configured devices, the dashboard flag,
the counts); and the frontend assets.

**Total: 996 tests, all passing.**

## Upgrading from 2.4.2

Drop-in. The agent gains the mailbox-count behaviour; deploy the
updated agent to any host you want to monitor mailboxes on.
Server-side there's no migration — the mailbox config is new
optional fields on the device record.

## Honest caveats

1. **The agent change is unit-tested for logic, not verified
   end-to-end.** `count_mailbox_paths` is covered by tests against
   real temp directories, and the heartbeat config-pickup mirrors
   the existing drift/services pattern exactly — but I have not
   run a real updated agent against a real server and watched a
   count flow through to the dashboard. The agent half of this
   feature should be smoke-tested on one host before you rely on
   it fleet-wide: configure a path, wait two heartbeats, confirm
   the number appears.

2. **Counts update on the agent's schedule, not live.** The agent
   reports mailbox counts every ~5 polls (about 5 minutes at the
   default 60s interval). A mailbox count on the dashboard can be
   up to a few minutes stale. This is deliberate — counting files
   every single heartbeat would be wasteful.

3. **It counts files, not messages.** For a Maildir `new/` folder
   that's the same thing. For other mailbox layouts it may not be
   — an mbox file is a single file containing many messages, so
   pointing this at an mbox directory counts mailboxes, not
   messages. This monitor fits Maildir; it is not a general email
   integration.

4. **The agent needs read access to the path.** If the mailbox
   directory isn't readable by the user the agent runs as, the
   count comes back as `permission_denied` (shown in the widget).
   That's surfaced, not hidden — but it means the agent's user may
   need to be in the right group.

5. **No alerting yet.** This release shows the count; it does not
   fire a webhook when the count crosses a threshold. Threshold
   alerting would be a natural follow-up but is not in 2.4.3.

6. **Browser-untested**, as usual — the dashboard widget and the
   device-detail config section are covered by asset tests, not a
   real browser load.
