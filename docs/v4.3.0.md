# RemotePower v4.3.0 — "ImprovementMatters"

A refinement release. No new headline subsystems and **no breaking changes** —
this round sharpens what's already there: faster reads on larger fleets, a few
self-observability touches, UX polish, and guardrails that stop whole classes
of regression at commit time. Everything degrades cleanly and every install
keeps working exactly as before.

After upgrading, hard-reload the dashboard once so the new front-end loads
(service-worker cache `remotepower-shell-v4.3.0`).

## Faster on bigger fleets

- **Single-row device reads.** On the SQLite / PostgreSQL backends, the
  endpoints that only need one device — per-host Checks, per-device CVE detail,
  the heartbeat's host-config and watched-file lookups, and the
  firewall/compose/RouterOS/OPNsense device actions — now read **one row**
  instead of reconstructing the whole fleet on every call. On a 200-device
  fleet that turns an O(fleet) `json.loads` per request into O(1). The flat-JSON
  backend is unchanged (it already holds the whole store in one file), and the
  data returned is identical — purely a read-path optimisation.

## A faster dashboard — especially in Firefox

A focused performance pass on the web UI, driven by a "feels sluggish, mostly
in Firefox" report. The headline causes were paint/compositor costs Firefox
handles worse than Chromium, plus needless re-rendering:

- **No more backdrop blur.** The sticky top bar and the modal overlay used
  `backdrop-filter: blur(…)` — Firefox re-blurs everything underneath on every
  frame (Chromium caches the layer). Both now use opaque/dimmed backgrounds
  that look near-identical and cost nothing. A regression test keeps
  `backdrop-filter` out for good.
- **Compositor-only animations.** The skeleton-loader shimmer animated
  `background-position` (a per-frame gradient re-raster); it now slides a
  transform-animated overlay. The AI-glow and sparkle loops get explicit layer
  promotion. A guardrail test now fails any `@keyframes` that animates a paint
  or layout property — the bug class that has shipped twice.
- **Quiet refreshes are (near-)free.** The 60s dashboard tick now skips
  re-rendering: the whole pass is skipped when the `/api/home` payload is
  byte-identical, and each of the 40+ add-on widgets independently skips its
  DOM write when its markup didn't change. Repaints (and, in non-English
  sessions, re-translation walks) only happen for widgets that actually moved.
- **Windowed device grid.** The card grid renders the first 60 devices and
  grows via "Show more" instead of building up to 300 cards per refresh.
- **Snappier tables.** Every live table filter is debounced (~180 ms typing
  pause) instead of rebuilding the table per keystroke, the live log viewer
  appends new lines instead of rebuilding thousands of rows every 30 s, and
  table scroll containers get `contain: paint` so their repaints stay inside
  the box. Hover transitions list explicit properties instead of
  `transition: all`. English sessions skip the i18n tree-walk entirely.
- **Fleet checks are cached server-side.** The Checks matrix (and the
  checks-rollup widget, which now shares it) is computed at most once per 15 s
  instead of per request; muting/toggling a check busts the cache instantly,
  and RBAC scope filtering is applied per request *after* the cache so nothing
  leaks across scopes.
- **Big API payloads are compressed.** The bulk read-only endpoints
  (`/api/home`, `/api/devices`, attention, alerts, fleet checks) are now
  gzipped when the browser supports it — typically 5–10× smaller on the wire.
  Token-bearing and input-reflecting endpoints stay uncompressed (the BREACH
  consideration that originally disabled JSON gzip in nginx still stands for
  them).
- **Optional: a persistent API worker.** Classic CGI forks a fresh Python
  interpreter — which re-parses the whole 2 MB `api.py` — on *every* request,
  before any handler runs. The new `api_worker.py` (SCGI prefork) imports the
  API once at service start and forks per request, keeping CGI's
  process-isolation semantics while removing the startup tax from every
  dashboard poll and agent heartbeat. Opt-in: install
  `server/conf/remotepower-api.service` and switch the nginx `/api/`
  location to the ready-made `scgi_pass` block commented in
  `server/conf/remotepower.conf`. The fcgiwrap path keeps working unchanged
  if you do nothing.

## Self-observability

- **Download the archived audit log.** The audit log already rolls evicted
  entries into a gzipped archive for compliance dives; there's now a button on
  the **Audit** page (and `GET /api/audit-log/archive`) to download it, so the
  full retained history is reachable without shell access to the server.
- **Staleness at a glance.** Cadence-driven jobs (monitors, the KEV/EPSS
  refresh, scheduled scans) surface a *last-ran* timestamp and a staleness
  badge on the **Server status** page, so a job that has quietly stopped is
  visible instead of silently absent.
- **Diagnostics bundle.** A button on **Server status** (and
  `GET /api/diagnostics`) downloads one JSON support bundle — versions, storage
  backend, fleet counts, recurring-job staleness, audit-log + hash-chain status,
  and which optional dependencies are installed — with all secrets scrubbed.
  One attachment instead of a back-and-forth when you ask for help.
- **Slow-request visibility.** Any handler slower than the threshold
  (`slow_handler_ms`, default 1500 ms) is recorded to a capped ring and shown
  on the **Server status** page — which endpoint is slow on *your* fleet,
  without external profiling. Only slow requests write a record; the heartbeat
  path is excluded so it can't dominate the ring.

## AI / RAG

- **More fleet awareness.** The RAG corpus the AI assistant retrieves from now
  indexes each device's **open alerts** and **local TLS cert expiry**, plus a
  fleet-wide open-alerts rollup — so questions like "what's alerting?" or
  "what's wrong with web01?" are answered from real, current state instead of
  the model punting to a tool call.

## Hardening

- **Rate-limit coverage.** The unauthenticated auth callbacks that verify
  cryptographic material and mint sessions — **SAML ACS**, the **OIDC
  callback**, and **WebAuthn login-complete** — now enforce a per-IP rate
  limit, matching the existing protection on password login and the passkey
  begin step. (Inbound webhooks and the public health/info endpoints are
  deliberately left unthrottled — they're legitimately high-volume.)

## Alert tuning

- **Per-device offline alert delay.** Each device's **Settings** tab (in the
  drawer) gains an *Offline alert delay* — extra minutes of silence before
  *that host* raises a `device_offline` alert. Handy for a box on a flaky link
  or a deliberately slow poller you don't want paging you on every blip. It only
  extends the per-device offline threshold the flap-debounce already uses; 0
  (the default) leaves alerting exactly as before, and it's capped at 24h.
- **Monitor flap dampening.** Each monitor (ping / TCP / HTTP / DNS / ICMP / DB)
  gains an *Alert after N consecutive failures* setting — a flaky target that
  blips for one check no longer raises a `monitor_down` alert until it has
  failed N checks in a row. A recovery before N resets the streak. 1 (the
  default) is unchanged — alert on the first failure.
- **Metric & SNMP dampening, the same way.** *Settings → Monitoring* gains two
  matching knobs: **Metric alert dampening** (consecutive over-threshold checks
  before a CPU / memory / disk / swap alert fires — 1 = default, first breach)
  and **SNMP alert dampening** (consecutive failed polls before *SNMP
  unreachable* — 2 = default). Escalation to critical, de-escalation and
  recovery are never delayed, so a genuine critical is never held back.

## UX polish

- **Clickable posture fixes.** Each warning row in the **security-posture
  self-check** now links straight to the Settings section that fixes it —
  Account guardrails, Audit forwarding, Secrets scanning — instead of naming a
  config key you then have to hunt for.
- **Consistent loading states.** Tables that previously flashed empty or showed
  a bare "Loading…" now use the same skeleton-row treatment as the rest of the
  app while their data fetches.

## Under the hood — regression guardrails

These ship as tests, not features, but they're the most valuable thing in the
release: they make three recurring bug-classes fail in CI instead of in a
later sweep.

- **Unbound-data guard.** A test diffs every `sysinfo` field the Checks engine
  and device drawer *read* against every field the heartbeat sanitizer
  *persists*. The `proc_names`, `mailq` and `pkg_scan_ts` bugs — checks that
  read a field the sanitizer silently dropped — were each found and fixed by
  hand in separate releases; this catches the next one at commit.
- **Webhook-registry guard.** A test walks every event in `WEBHOOK_EVENTS`
  against the alert-severity, channel-kind, friendly-title and front-end
  registries, so an event that half-registers (fires a webhook but never lands
  in the inbox — the phantom `service_recover` class) can't ship.
- **Performance-regression guard.** A test (forced onto the SQLite backend)
  counts how many times a single-device endpoint reconstructs the whole devices
  store and asserts it stays at zero — so a future edit that reintroduces
  `devices = load(...); dev = devices.get(id)` can't silently undo the
  single-row optimisation.
- **Agent parser golden tests.** The agent's fragile text parsers
  (apt/dnf/pacman upgrade lists, systemd timestamps, openssl cert dates) fail
  *silently* on format drift — they're now locked against representative
  fixtures so a parser regression is caught at commit, not when a fleet quietly
  stops reporting upgrades or cert expiry.
