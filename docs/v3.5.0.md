# RemotePower v3.5.0 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.5.0 is a four-feature batch that pushes RemotePower toward being a complete
Linux RMM:

1. **SBOM export** — generate a standards-compliant Software Bill of Materials
   per host or for the whole fleet, enriched with CVE findings (VEX-style).
2. **Lifecycle expiry tracking** — record warranty / license / support-contract
   end dates in the CMDB and get dashboard attention items before they lapse.
3. **Graphical remote access (VNC over SSH)** — a full desktop in the browser,
   tunnelled over the same SSH path the web terminal already uses.
4. **Sites & teams** — a first-class fleet grouping above device `group`, for
   organising hosts by location, team, or customer.

---

## 1. SBOM export (CVE → Findings, and the device drawer)

RemotePower already collects each host's installed-package inventory and scans
it against OSV. v3.5.0 turns that data into a portable SBOM:

- **Formats:** CycloneDX 1.5 JSON (primary) and SPDX 2.3 JSON.
- **purls:** each component carries a package URL (`pkg:deb/…`, `pkg:rpm/…`,
  `pkg:apk/…`, `pkg:alpm/…`) so downstream tools (Dependency-Track, Grype, …)
  can re-resolve it.
- **VEX:** CycloneDX output includes a `vulnerabilities[]` section built from the
  host's current CVE findings (ignore-list applied), with `affects` pointing at
  the affected component and the fixed version noted — so one document is both
  an inventory and a vulnerability report.
- **Where:** "SBOM (CycloneDX)" / "SBOM (SPDX)" buttons on a device's CVE detail,
  and "Fleet SBOM" buttons on the CVE Findings page (a ZIP of per-host docs).

**API:**
- `GET /api/devices/{id}/sbom?format=cyclonedx|spdx`
- `GET /api/sbom?format=cyclonedx|spdx[&group=&device_id=]` — ZIP of the fleet.

Output is deterministic (same host + same inventory snapshot → same document
serial), so re-exports are reproducible.

## 2. Lifecycle expiry tracking (CMDB → attention)

Three optional date fields were added to each CMDB asset: **warranty expiry**,
**license expiry**, and **support-contract expiry** (date pickers in the CMDB
asset modal). They mirror the existing end-of-life-OS pattern:

- **Thresholds:** expired or within 30 days → **warning**; within 90 days →
  **info**.
- These surface as needs-attention items on the dashboard, flow into the fleet
  health score, and are routable/silenceable as the NA kinds `warranty_expiry`,
  `license_expiry`, and `support_expiry` (Settings → channel routing).

No new webhook events — like `os_eol`, these are state-derived attention items,
not fired events.

**API:** `PUT /api/cmdb/{id}` accepts `warranty_expiry`, `license_expiry`,
`support_contract_expiry` (ISO `YYYY-MM-DD`, or `''` to clear).

## 3. Graphical remote access — VNC over SSH

A new **"Remote desktop"** action on the device drawer opens a graphical session
in the browser, rendered with [noVNC](https://novnc.com).

- **Transport:** reuses the web-terminal daemon. After the same admin-password
  re-auth + single-use ticket, the daemon opens a direct-TCP channel through the
  host's SSH connection to its **loopback** VNC port (default `127.0.0.1:5900`)
  and bridges raw RFB to the browser. The VNC server is never exposed to the
  network — the SSH tunnel is the only path, so the session inherits SSH's auth
  and encryption.
- **No agent change, no inbound firewall rules.** It rides the existing
  outbound-from-daemon SSH path.
- **Host requirement:** a VNC server bound to localhost (e.g.
  `x11vnc -localhost` or a `tigervnc` `:1` display). RDP is not supported in this
  release (a protocol gateway would be required); Linux VNC is the supported path.

## 4. Sites & teams

A **site** groups devices by location, team, or customer — one level above the
existing device `group`. This is a *soft* boundary for organisational clarity
(super-admins always see every site), not a hardened tenancy wall.

- **Admin → Sites:** create / rename / delete sites; each row shows its device
  count.
- **Assign:** a "Assign site" action on the device drawer moves a host into a
  site (or unassigns it). Deleting a site unassigns its devices.
- **Filter:** the Devices roster gains an "All sites" filter (including an
  "Unassigned" bucket).

**API:** `GET/POST /api/sites`, `PUT/DELETE /api/sites/{id}`,
`PATCH /api/devices/{id}/site`. Mutations are admin-only and audited.

---

## Upgrade notes

- The agent version is bumped to 3.5.0 for lockstep, but **no agent change is
  required** — all four features are server/UI-side and build on data the agent
  already reports (or on the SSH path for VNC).
- noVNC is vendored under `server/html/static/vendor/novnc/` (MPL-2.0) and loaded
  as native ES modules; no build step.
