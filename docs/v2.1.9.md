# v2.1.9 release notes

Released: 2026-05-15 (same-day hotfix after 2.1.8).

The runbook generator was hallucinating wildly on smaller local
models. Three compounding root causes, all fixed here. Plus a demo
URL correction.

## Fixed

### Runbooks were inventing services, ports, firewall rules

**Symptom (reported by an operator running Ollama qwen2.5-coder:14b)**:

Generated runbook claimed the host had a service called "system",
firewall rules from "ufw-before-login", "DNS lookups for both
IPv4 and IPv6 addresses of a domain", and a half-dozen other
details that **were not in the device snapshot at all**. The
generator was returning a JSON summary of *some* data and inventing
the rest from imagination, with no relationship to the actual host.

The same operator's 2.1.8 install on Anthropic Sonnet produced
correct runbooks. The hallucination only manifested on smaller
local models.

**Three root causes, in order of bite**:

1. **Ollama's OpenAI-compatible endpoint defaults to `num_ctx=2048`.**
   A typical RemotePower snapshot is 4–8K tokens. Ollama silently
   truncated the input at 2048 tokens, the model received only the
   head of the JSON, and filled in the rest from training-data
   priors about "what Linux server snapshots usually contain."
   Result: confident-sounding details about firewall rules and DNS
   resolvers that the snapshot never mentioned.

2. **The v2.1.7 runbook prompt was too elaborate.** Eight named
   sections with detailed format rules ("flag any port that looks
   unusual for the apparent role"). Large frontier models followed
   it; smaller coder-tuned models ignored the structure entirely
   and just summarised whatever JSON they could see — explaining
   why the actual output had headings like "Network", "Disk Usage",
   "CPU Utilization" instead of the requested Purpose / Stack /
   Services / etc.

3. **The snapshot itself was too big.** Up to 25 KB of nested JSON
   with full sysinfo, 40 journal lines, 15 commands at 500 chars
   each, 20 CVEs with 200-char summaries. Fine for Anthropic Sonnet's
   200K context, fatal at Ollama's default of 2048 tokens.

**Fix 1: pass `num_ctx=16384` on the Ollama / LocalAI body.**
`ai_provider.chat_openai_compatible` now adds an `options.num_ctx`
field when the provider is `ollama` or `localai`. Real OpenAI and
DeepSeek ignore unknown body keys, so this is safe across
providers. 16384 is comfortable for 80% of inputs we send and
works on any locally-runnable model from the last year. Operators
with tiny-context models (tinyllama, phi-2) may want to lower it;
that's a future toggle if it becomes an actual problem.

**Fix 2: rewrite the runbook prompt.** From ~2 KB / 8 sections to
~1 KB / 6 sections, with explicit `CRITICAL RULES` near the top:

```
- Use ONLY information from the snapshot. Do NOT invent services,
  ports, firewall rules, DNS configurations, or anything else not
  explicitly present in the data.
- If a section has no relevant data in the snapshot, write
  "No data captured." Do not fabricate.
- Do not summarise the snapshot structure. Write a runbook for an
  operator, not a description of the JSON.
```

Sections trimmed to: **Purpose / Current state / Watched services /
Containers / Recent activity / Health & risks / Operating notes**.
Each section has an explicit "if empty, write X" fallback so the
model isn't tempted to improvise.

**Fix 3: tighten the snapshot payload to ~8 KB.** New caps in
`_build_runbook_snapshot`:

| Field | Was | Now |
|---|---|---|
| Journal lines | 40 | 20 |
| Recent commands | 15 (output ≤500 chars) | 5 (output ≤200 chars) |
| CVE findings | 20 (summary ≤200 chars) | 10 (summary ≤100 chars) |
| Containers | 30 | 10 |
| Notes | 1000 chars | 500 chars |
| Sysinfo | full dump | only the 9 operator-relevant fields |
| Sysinfo `disks` | all | top 5 by usage |
| User-message cap | 48 KB JSON (indent=2) | 12 KB JSON (compact) |

Net: real-world snapshots that were 18–25 KB are now ~6–8 KB. Even
on a 4K-token context, the snapshot + system prompt + response
budget fits comfortably.

### Demo URL was demo.tvipper.com (should be demoremote.tvipper.com)

Operator runs the demo at `demoremote.tvipper.com`, not
`demo.tvipper.com`. Fixed across all `*.md` and `*.html` files in
the repo. Regression test in `test_v219.py` greps for bare
`demo.tvipper.com` and fails the build if any reappears.

## Tests

`tests/test_v219.py` (8 tests):

- `TestNumCtxWiring` — verifies `options.num_ctx` is set on Ollama
  + LocalAI bodies, NOT on OpenAI bodies. Calls
  `chat_openai_compatible` directly to bypass test pollution from
  earlier files that monkey-patch `ai_provider.chat`.
- `TestRunbookPromptAntiHallucination` — verifies prompt contains
  `CRITICAL RULES`, "do not invent / fabricate", `ONLY`,
  `No data captured`, and is under 3 KB total.
- `TestSnapshotSizeCap` — feeds a heavily-populated synthetic
  device (200 journal lines, 50 containers, 50 CVEs, 50 disks,
  10 KB junk field in sysinfo) and asserts the resulting snapshot
  is under 10 KB and the trimming caps are all respected.
- `TestDemoUrl` — file-system grep for any remaining bare
  `demo.tvipper.com` in `*.md` files.

Total suite: **754 tests, all passing.**

## Upgrading from 2.1.8

Drop-in. The first runbook generated after upgrade will be much
better. Existing stored runbooks in `runbooks.json` were written
under the bug — worth deleting and regenerating any you care
about. Each device's Runbook section on the detail modal has a
**Delete** button + **✨ Regenerate**.

## Honest caveats

1. **None of this guarantees a smaller model produces a *good*
   runbook.** It guarantees the input arrives intact and the prompt
   discourages fabrication. A 14B coder-tuned model still won't be
   as articulate as Claude Sonnet for this kind of structured
   writing. If runbook quality really matters, use a frontier
   provider; if free-and-local matters more, use this and accept
   the prose will be plainer.

2. **`num_ctx=16384` is a fixed compromise.** Larger models could
   take 32K or 128K; tinier models can't handle 16K and will OOM.
   We picked the middle. If your model OOMs on generate, the
   workaround today is to switch model in Settings → AI assistant.
   A configurable per-provider `num_ctx` is a future toggle if
   anyone hits this.

3. **Regression test for hallucination is weak by nature.** I can
   verify the prompt *says* "do not invent" and the input fits in
   context, but I can't unit-test that the model actually obeyed.
   Real verification is "regenerate one of the runbooks the
   operator reported as wrong and visually check the result." The
   tests catch the three causes; whether the cumulative fix
   eliminates the symptom on your specific Ollama setup is
   something only you can confirm with `✨ Regenerate`.
