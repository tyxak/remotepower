# v2.4.7 release notes

Released: 2026-05-18.

A "make fleet health visible" release — three features that
cohere: threshold alerting, a unified attention digest, and a
machine-readable status endpoint.

## Added

### Mailbox threshold alerting

A device's mailbox monitor can now carry an **alert threshold**
(Settings → Mailbox monitor). When a counted mailbox reaches that
many messages, RemotePower fires the new `mailbox_threshold`
webhook — routed through the same Discord / ntfy / Slack / JSON
pipeline as every other alert.

The check is **edge-triggered**: it fires once when a count
crosses from below the threshold to at-or-above it, not on every
heartbeat while it stays high. The alert re-arms when the count
drops back below — the same anti-fatigue pattern the metric and
service alerts use.

Built as a general comparison-and-trigger mechanism; the mailbox
count is the first value wired through it.

### Needs Attention digest

The Home dashboard's "Needs attention" panel is now a single
**ranked list** computed server-side (`/api/attention`). It merges
every fleet-wide signal into one prioritised view: offline
devices, critical and high CVEs, configuration drift,
pending-patch pileups, and mailbox threshold breaches — sorted
critical-first.

Previously this panel was computed in the browser and only knew
about offline devices, patches and drift — it missed CVE findings
and mailbox alerts entirely. It is now one source of truth, and
the `/api/status` endpoint below reads from the same place.
Unmonitored devices are excluded, the same gate the webhook
pipeline uses.

### Status endpoint for external dashboards

A new machine-readable fleet summary at **`/api/status`**, for
homelab and small-business dashboard tools — Uptime Kuma,
Homepage, a Grafana panel.

It is authenticated by a dedicated **status token**, not a login
session — so a monitoring tool can poll it — but it is not
public: without the token it returns 403. Generate, rotate or
disable the token under Settings → Advanced → Status endpoint.

The response is a rolled-up `health` word (`ok` / `warning` /
`critical`), device online/offline/total counts, and attention
counts by severity — enough for a dashboard to show a single
status dot or a small panel.

## Tests

`tests/test_v247.py` (14 tests): the mailbox threshold fires once
on crossing up and not again while high, re-arms after dropping
below, never fires without a threshold; the threshold is stored
and cleared via the endpoint; the attention digest merges and
ranks signals, is empty when all-clear, and excludes unmonitored
devices; the status endpoint requires a valid token, is disabled
without one, returns the summary with a token, and the token can
be generated and cleared.

Five pre-existing tests were updated for this release: the
webhook-event and fleet-event allowlists gained `mailbox_threshold`,
and two `test_v224` tests were rewritten — the attention digest
moved from a client-side computation to the `/api/attention`
endpoint, so the tests now verify the endpoint is consumed and
that the `monitored` gate is applied server-side.

**Total: 1035 tests, all passing.**

## Upgrading from 2.4.6

Drop-in, server-side. No data migration. The status endpoint is
off until you generate a token. Existing mailbox monitors keep
working; add a threshold to any of them to start alerting.

## Honest caveats

1. **The status endpoint is as private as its token.** Anyone
   with the token URL can read the fleet summary (counts and a
   health word — no device names, no detail). Treat the token
   like a password; rotate it if it leaks. It is disabled by
   default for exactly this reason.

2. **Threshold alerting depends on the agent reporting.** The
   mailbox count only updates every ~5 minutes (the agent's
   schedule), so a threshold crossing is detected on that cadence,
   not instantly.

3. **The digest reflects last-known state.** It aggregates stored
   data — CVE findings from the last scan, the last heartbeat's
   patch count, and so on. It is a fast read over existing data,
   not a live re-probe of the fleet.

4. **Browser-untested**, as usual — the threshold field, the
   rewired digest panel, and the status-token UI are covered by
   asset and unit tests, not a real browser load.
