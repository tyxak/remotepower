# v2.1.5 release notes

Released: 2026-05-14 (same-day follow-up to 2.1.4).

A polish release. Six fixes/improvements queued up from real-world
use of the 2.1.3/2.1.4 AI work plus a long-pending stderr-spam fix.

## Fixed

### "No Data Provided" from ✨ Investigate even when the device had data

**Symptom**: clicking ✨ Investigate on a device with a full
sysinfo + journal would return:

> No Data Provided: It seems that the provided snapshot is empty or
> incomplete, as there are no metrics, recent commands, or syslog
> entries to analyze.

**Root cause**: the JS was fetching `GET /api/devices/<id>` — a
route that doesn't exist. The handler split happens at sub-paths
(`/sysinfo`, `/output`, `/metrics`, etc.) and there's no top-level
device-detail route. The fetch silently returned null and the AI
was sent an empty payload.

**Fix**: assemble the snapshot from the actual endpoints, in
parallel:

- `GET /api/devices/<id>/sysinfo` — sysinfo dict + recent journal
- `GET /api/devices/<id>/output` — last 10 command outputs with rc
- `GET /api/devices` — to find top-level facts (`os`, `version`,
  `last_seen`, `ip`, `group`, `online`)

Bails visibly with `"No data available for this device yet — has
the agent checked in?"` if all four come back empty, rather than
sending a hollow request to the model. The previous behaviour was
worse — the model invented a placeholder response without anything
to base it on.

### Markdown in AI responses now renders properly

Models love their `**bold**` and `## headers` and `` `code` `` and
showing it as literal punctuation was jarring. Added a minimal
markdown→HTML renderer (`renderMarkdown()`) used in both the ✨
modal and the AI page chat. Supports:

- Code fences (`` ``` ``) and inline code
- Headers (`#`, `##`, `###`)
- Bold (`**` and `__`)
- Italic (`*` and `_`)
- Bullet lists (`-`, `*`) and numbered lists (`1.`)
- Blockquotes (`> `)
- Paragraph breaks

**Security**: HTML is escaped *before* any markdown transform
touches it, so user-supplied or model-supplied bytes can't inject
script tags. Tested against the standard repertoire — `<script>`,
`onerror=` handler injection, javascript: URLs — all neutralised.

Anything beyond the supported subset (links, tables, images) falls
through as escaped text. That's deliberate: keeps the renderer
small, keeps the attack surface narrow, and the common cases are
all covered.

### Routine heartbeat / lock_wait logs silenced by default

The `rp-silence-heartbeat-logs.sh` patch script that shipped with
2.1.2 is now redundant — its three behaviours are the default in
2.1.5:

| Log line | Now gated behind |
|---|---|
| `[remotepower] heartbeat dev=…` (per-request) | `RP_LOG_HEARTBEATS=1` |
| `[remotepower] heartbeat 202 …` (busy retry) | `RP_LOG_HEARTBEATS=1` |
| `[remotepower] lock_wait path=… mode=nb` | `RP_LOG_LOCK_WAITS=1` |
| `[remotepower] lock_wait path=… mode=block` | `RP_LOG_LOCK_WAITS=1` |

At 6 devices polling every 60s these add ~8,600 routine lines/day
to the nginx error log — drowning out actual errors. Set either
env var in the CGI environment (typically via fcgiwrap's
`spawn-fcgi -E`, or in `/etc/default/fcgiwrap`) to re-enable for
diagnostics.

**Not gated**: OFFLINE / ONLINE state-transition logs, since those
are rare and operationally important. Real-error stderr writes
(corrupt JSON, save failures) are also unaffected.

### About-page robustness (carry-over)

The 2.1.3 fix for "Latest release 2.0.0 ✓ up to date" is unchanged
— `local = SERVER_VERSION`, `latest = max(github, local)`. Mentioning
it because the bump from 2.1.4 → 2.1.5 means GitHub still serves an

## Changed

### AI Assistant moved to the Help section

Previously under Planning (the trip up the sidebar where Schedule /
Calendar / Tasks live). It's a help-and-documentation surface, not
a scheduling surface, so it now sits in Help between Documentation
and API Reference. URL hash `#ai` still works either way.

### Device-card dropdown is now grouped + collapsible

The pre-2.1.5 dropdown had 22 items in one vertical list — taller
than most device cards, which spilled off the bottom of the screen
on a busy view. The new layout:

- **Power** (always visible at top): Shut down, Reboot,
  Wake-on-LAN *(if MAC)*, Upgrade packages.
- **Inspect** (open by default, most-used group): System info,
  ✨ Investigate, Metrics, Update history.
- **Operate** (collapsed): Web terminal, Custom command, Run
  script…, docker compose, Agent update.
- **Configure** (collapsed): Edit tags, Set group, Notes, Poll
  interval, Metric thresholds, Allowlist, Icon, Enable/Disable
  monitoring.
- **Remove device** at the bottom in its own danger zone.

Native `<details>`/`<summary>` — no JS for the collapse logic.
Footprint drops from 22 visible items to 6 visible (4 Power + 4
Inspect, with two collapsed groups).

Both render sites (grid view and table view) now share a single
`deviceDropdownHtml()` helper, so the dropdown definition isn't
duplicated 1.5 KB twice anymore.

## Added — four new ✨ button surfaces

| Surface | Label | When it shows |
|---|---|---|
| Services → service detail | **✨ Diagnose** | When `active` is not `active` or `activating` (i.e. failed / inactive / degraded) |
| TLS → table row | **✨ Triage** | When status is `warning`, `critical`, or `error` |
| Patches → table row | **✨ Prioritise** | When the device has `upgradable > 0` |
| (Future) Container logs | — | Helper exists (`aiExplainContainerLogs`) but unused for now — the existing ✨ Explain on command output already handles container log output from the `container:logs` action |

The visibility conditions are deliberate: there's no value in asking
the model to triage a healthy cert or prioritise zero pending
updates. Buttons appear only where they add information.

Four new system prompt keys added to `ai_provider.SYSTEM_PROMPTS`:
`diagnose_service`, `explain_tls`, `prioritise_patches`,
`explain_container_logs`.

## Tests

`tests/test_v215.py` (3 tests):

- `TestNewSystemPrompts` — the four new prompt keys exist with
  non-trivial content and each prompt mentions its domain (service /
  TLS / package / container) so a future rename can't silently break
  a button.
- `TestEnvGatedLogging` — verifies the source contains
  `RP_LOG_HEARTBEATS` / `RP_LOG_LOCK_WAITS` gates near the right
  stderr.write sites, AND verifies OFFLINE/ONLINE logs are *not*
  gated. Source-grep style test because the actual logging happens
  inside hot-path functions that are hard to unit-test in isolation.

system-prompt registry test (so a future drift catches it).

## Upgrading from 2.1.4

Drop-in. No schema changes. The `rp-silence-heartbeat-logs.sh` patch
script from 2.1.2 is now a no-op — its changes are the default.
You can remove the script from your deployment automation.

If you previously set `RP_LOG_HEARTBEATS=1` to keep the noise
(for some reason) it still works as before.

## Upgrading from 2.1.3 or earlier

You still need the nginx `fastcgi_read_timeout` fix from 2.1.4 if
you're using a slow local Ollama model — that's a manual nginx
config edit, not something a tarball install can do. Snippet is in
[v2.1.4.md](v2.1.4.md) or [ai.md](ai.md#slow-local-models-and-nginx).
