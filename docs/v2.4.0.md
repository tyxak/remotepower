# v2.4.0 release notes

Released: 2026-05-17.

Proxmox snapshots, plus a CVE severity fix.

## Added

### Proxmox VM/LXC snapshots

The Virtualization page (QEMU) and the Containers page (LXC) now
have a **Snapshots** button on each guest. It opens a modal that
lists the guest's snapshots and supports:

- **Create** ‚Äî name + optional description. Disk-only (the VM's
  RAM state is not captured ‚Äî the common, fast case; no RAM toggle
  in this release).
- **List** ‚Äî name, when taken, description. The synthetic
  `current` pseudo-snapshot Proxmox always returns (the live
  state) is filtered out.
- **Rollback** ‚Äî *destructive*: discards all guest state since the
  snapshot. The UI requires typing the guest's name to confirm ‚Äî
  not just an OK click.
- **Delete** ‚Äî irreversible, but does not affect the running
  guest. Plain confirmation dialog.

Same server-to-API model as the rest of the Proxmox integration ‚Äî
the RemotePower server calls the Proxmox API directly. New client
methods in `proxmox_client.py`; new endpoints
`GET /api/proxmox/snapshots` and `POST /api/proxmox/snapshot`.

Snapshot names are validated locally (Proxmox's rule: a letter
followed by letters/digits/underscores) so a bad name is rejected
with a clear message instead of an opaque Proxmox 400. The
`current` name is reserved ‚Äî it can never be a rollback or delete
target.

The optional items from the feature request ‚Äî live CPU/RAM
adjustment and backup-trigger integration ‚Äî are **not** in this
release; they're larger and warrant separate work. VM
start/stop/shutdown already shipped in 2.3.0.

## Fixed

### CVE severity: Debian urgency was being shown as HIGH

`DEBIAN-CVE-2018-1000021` was displayed as HIGH while OSV rates it
CVSS 5.0 Medium. Root cause: when an OSV Debian entry carries no
CVSS vector, the severity chain falls back to the Debian Security
Tracker ‚Äî and the fallback mapped Debian's `urgency` field
straight to severity (`high` urgency ‚Üí `high`).

Debian `urgency` is **not** a CVSS severity. It is a
patching-*priority* signal ‚Äî Debian's triage of how soon they want
a fix out, which weighs exploit availability and exposure, not
just impact. Debian routinely marks something `high` urgency that
CVSS rates Medium.

Fixed: the Debian-urgency fallback is now **capped at `medium`** ‚Äî
it can never return `high` or `critical`. Claiming a HIGH severity
requires an actual CVSS score or a distro's explicit severity
rating, not an urgency hint. `high`/`medium` urgency ‚Üí `medium`;
`low`/`unimportant`/`negligible` ‚Üí `low`. The `**` postponed-marker
suffix Debian sometimes appends (`low**`) is now stripped. The
classification source for these is labelled `debian_urgency` (was
`debian_tracker`) to make the provenance honest.

## Tests

`tests/test_v240.py` (17 tests): Debian-urgency cap (every urgency
level, the `**` marker, the never-high guarantee); snapshot name
validation; `current` reserved; create/rollback/delete reject bad
names, unknown guest types, bad vmids, and `current`; the API
endpoints validate action/type/vmid and the unconfigured case;
the snapshot JS functions are present and rollback requires typed

modal is created dynamically (`document.createElement`), so its
IDs were added to that test's `KNOWN_DYNAMIC_IDS` set ‚Äî the same
mechanism the AI / runbook / drift modals already use.

Drop-in, server-side. No data migration. Snapshots appear once
Proxmox is configured (Settings ‚Üí Proxmox).

## Honest caveats

1. **Not tested against a live Proxmox node.** As with the 2.3.0
   Proxmox work: the 968 unit tests cover the logic ‚Äî name
   validation, the action gate, endpoint input validation ‚Äî but
   they do not verify the snapshot API request shapes against a
   real Proxmox 8.x. Test create ‚Üí list ‚Üí rollback ‚Üí delete on a
   throwaway VM before relying on it.

2. **Snapshot actions are fire-and-forget.** Proxmox snapshot
   operations return a task ID and run asynchronously. RemotePower
   sends the request, waits ~1.5s, then refreshes the list ‚Äî it
   does not poll the task to confirm completion or surface task
   errors. A slow snapshot may not appear in the list on the first
   refresh; reopen the modal. Task-status polling remains a
   sensible future addition.

3. **Rollback confirmation is name-typing, not a second factor.**
   Typing the guest name stops an accidental click, but anyone
   with an admin token can still roll back a VM. Admin token
   hygiene is the real control. The action is recorded in the
   fleet event log.

4. **Disk-only snapshots.** RAM state is never captured. For a
   running VM this means a rollback returns it to a
   crash-consistent disk state, not a live-memory state. This is
   the right default for most uses but worth knowing if you
   snapshot running VMs expecting live-memory restore.

5. **CVE severity recomputes on the next scan.** The Debian
   urgency fix applies as cache entries refresh ‚Äî findings already
   cached keep their old severity until re-scanned. Clear the CVE
   cache to apply it immediately.
