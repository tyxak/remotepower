# RemotePower v3.0.6 — production-readiness polish

> Released 2026-05-25, the same day as v3.0.5. Closes the four
> "nice-to-have but worth doing properly" items the v3.0.5 release
> notes left open: a dedicated health endpoint, CSP violation
> reporting, Subresource Integrity on bundled vendor libraries, and
> GitHub Actions CI. No schema changes, no migrations. Drop in the
> new tarball, copy the new nginx config (or just tweak the CSP
> header — see Upgrade), reload nginx, hard-reload the browser once.

This release is incremental. Operators who deployed v3.0.5 cleanly
don't *need* v3.0.6 — the dashboard works the same. What they get
in exchange for one more deploy is better signals: a cheap probe
for orchestrators, browser-reported CSP violations as audit-log
entries, tamper detection on the bundled JS/CSS, and a CI workflow
that runs the strict-CSP fidelity suite on every push.

## What changed

### `/api/health` — cheap liveness probe

A new unauthenticated endpoint that returns

```json
{"status": "ok", "version": "3.0.6"}
```

and nothing else. Replaces `/` in the Docker `HEALTHCHECK` (the
previous probe loaded the whole SPA on every poll) and gives external
orchestrators / load balancers / uptime monitors something cheap to
hit. Exposes only the version — no deployment detail, no fleet count,
no auth state. The endpoint is in `_PWCHG_ALLOWED_PATHS` so it works
even from a session that's pending forced password change.

The Docker `HEALTHCHECK` was already non-trivial post-v2.2.6 (it uses
Python's `urllib.request` rather than curl, which the slim image
doesn't ship). The change is one line — probe `/api/health` instead
of `/`.

### CSP violation reporting

The strict CSP shipped in v3.0.5 (`script-src 'self'; style-src
'self'` — no `'unsafe-inline'`) blocks any future inline script or
style. The browser's CSP machinery has been silently rejecting them
since the v3.0.5 deploy. v3.0.6 connects the rejection to the audit
log:

```
Content-Security-Policy: …; report-uri /api/csp-report;
```

When a browser blocks something the policy forbids, it posts a small
JSON report to `/api/csp-report`. The handler appends one audit-log
line per violation with the directive, blocked URI, source file, and
line number. So a regression that reintroduces inline code — anywhere
in `app.js`, a vendor library that ships an inline `<script>`, a
malicious payload trying to inject — surfaces as a logged event your
operations team can see, instead of a silent visual bug nobody finds
for weeks.

The endpoint is unauthenticated by design (the browser sends these
fire-and-forget, sometimes with `Origin: null`) and request size-
capped to 16 KB so it can't be used to flood the audit log. It's
the only `POST` allowed through the same-origin CSRF check; that
exception is documented inline.

`tests/test_v232.py::TestCSPMigrationFidelity` is still the primary
guardrail — `report-uri` is the runtime catch-net for anything the
test suite misses.

### Subresource Integrity on bundled vendor libraries

Every `/static/vendor/` script and stylesheet the dashboard loads
now carries an `integrity="sha384-…"` attribute pinning the
file's SHA-384 hash:

| Path | SHA-384 hash (prefix) |
|---|---|
| `vendor/xterm/xterm.min.js`               | `sha384-J4qzUjBl…` |
| `vendor/xterm/xterm.min.css`              | `sha384-tStR1zLf…` |
| `vendor/xterm-addon-fit/addon-fit.min.js` | `sha384-XGqKrV8J…` |
| `vendor/qrcode-generator/qrcode.min.js`   | `sha384-mZT2gIty…` |
| `vendor/swagger-ui/swagger-ui.min.css`    | `sha384-+R8fx41k…` |
| `vendor/swagger-ui/swagger-ui-bundle.min.js` | `sha384-5zikP7B6…` |

Static refs in `swagger.html` add the attribute directly on the
`<link>` / `<script>` tags. The dynamic loads in `app.js`
(`_loadXtermOnce`, `generateQRCode`) set `script.integrity` on the
element they create.

If the file on disk is ever overwritten — by a compromised deploy,
an accidental `rsync` from a different release, a sloppy edit, a
supply-chain attack against your build pipeline — the browser
refuses to execute it and logs the SRI mismatch to the console. Belt
and braces against a CSP that's already strict.

Updating a vendor version: replace the file in `static/vendor/`,
recompute the hash with
`openssl dgst -sha384 -binary <file> | openssl base64 -A`,
update the `integrity=` attribute (search for `sha384-` in
`swagger.html` and `app.js`).

### GitHub Actions CI

`.github/workflows/ci.yml` runs `python -m unittest discover -s tests`
on every push and pull request to `main`. The suite already includes
the `TestCSPMigrationFidelity` checks from v3.0.5, so a future commit
that reintroduces an inline event handler, an unresolved CSS
`${expr}`, a `javascript:` URI, a duplicate ID, etc. fails the
workflow before the PR can be merged. On main-branch pushes it also
runs `make dist` to make sure the tarball builds cleanly from a fresh
checkout.

Kept deliberately small: one Python version (3.12), the runtime
dependencies the test files actually import (bcrypt, cryptography,
dnspython), no separate lint job. The lint scope in this codebase
is currently two files and would add noise without value.

## Upgrade notes

1. `git pull` on the repo holding your checkout.
2. `sudo bash deploy-server.sh`. The script's per-file content-hash
   cache-bust (from v3.0.5) handles invalidation automatically.
3. **One-time nginx config update.** The CSP header changed — it now
   carries `report-uri /api/csp-report`. Either:
   - Copy `server/conf/remotepower.conf` to your nginx config dir, **or**
   - Edit your existing config and append `report-uri /api/csp-report;`
     to the `Content-Security-Policy` directive (just before the
     closing `"`).
   - Then `sudo nginx -t && sudo systemctl reload nginx`.
4. Hard-reload the browser once. The SW cache name bumped from
   `remotepower-shell-v3.0.5` to `remotepower-shell-v3.0.6`; the
   activate handler will evict the old cache on the next page load.
5. Verify the health endpoint:
   `curl https://your-host/api/health` should return
   `{"status":"ok","version":"3.0.6"}`.

## Regression coverage

Test suite has grown to **1,560+ tests, all passing**. New strict
pins live in `tests/test_v306.py`; the v3.0.5 pins in
`tests/test_v305.py` are loosened to `^3\.\d+\.\d+$` regexes — same
pattern test_v303.py / test_v304.py followed when their respective
versions shipped.
