# RemotePower v3.7.0 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.7.0 closes the remaining security/governance gaps and adds two infrastructure
features.

1. **2FA recovery codes** — one-time backup codes so a lost authenticator
   doesn't lock you out.
2. **Audit-log forwarding** — mirror every audit entry to a SIEM (HTTP) or
   syslog collector.
3. **Credential rotation reminders** — flag CMDB vault credentials older than a
   per-credential policy.
4. **Desired-state enforcement** — opt-in *correct-on-drift* for host config.
5. **Change approval (maker-checker)** — a second admin must approve arbitrary
   command runs.
6. **Proxmox QEMU VM create** — a create-VM wizard alongside the LXC one.
7. **Ansible playbook runner** — run playbooks against the fleet with the server
   as the control node.

Terraform: we deliberately did **not** build a bespoke provider — see
`docs/terraform-api.md` for driving the REST API from Terraform instead.

---

## 1. 2FA recovery codes

Enabling TOTP now also generates **10 one-time recovery codes**, shown once.
At login, a recovery code is accepted in place of the authenticator code (and
consumed on use). Settings → Security shows how many remain and offers
**Regenerate recovery codes** (password-gated). Codes are stored only as
password-hashes (reusing the existing bcrypt/PBKDF2 helper).

## 2. Audit-log forwarding (SIEM / syslog)

Settings → Security → **Audit log forwarding**: mirror each audit entry to an
external destination as it's written.
- **HTTP** — JSON `POST` to a URL, optional bearer token, SSRF-guarded.
- **Syslog** — RFC 5424 line over UDP or TCP (default port 514).

Best-effort and non-blocking — a forwarding outage never affects local logging.
A **Send test entry** button confirms the pipe. `POST /api/audit/forward-test`.

## 3. Credential rotation reminders

Each CMDB vault credential gains a **Rotate every N days** policy (0 = off).
When a credential's age since its last password change exceeds the policy it's
flagged on the dashboard (NA kind `cred_rotation`) and badged in the asset's
credential list. The rotation clock anchors on the password change, not metadata
edits.

## 4. Desired-state enforcement

The host-config modal adds **Correct on drift** beside the existing *Enforce on
host* toggle:
- **Enforce on host** (existing) — push desired every heartbeat (continuous).
- **Correct on drift** (new) — re-apply desired *only when* the host has drifted
  from it. Lighter than continuous enforcement; the corrective re-apply is
  audited (`host_config_enforce`).

## 5. Change approval (maker-checker)

Settings → Security → **Change approval**. When enabled, an arbitrary command
run (Run command) is parked as a pending confirmation instead of executing; a
**different** admin approves it on the Confirmations page. The requester cannot
approve their own change (separation of duties, toggleable). This generalises
the existing MCP confirmation queue with an `exec_command` action and a
self-approval guard.

## 6. Proxmox QEMU VM create

A **Create VM** button on the Virtualization page opens a wizard (name, VMID,
cores, memory, disk + storage, network bridge, optional boot ISO) and creates a
QEMU/KVM guest via the Proxmox API — mirroring the existing LXC create. Admin
only, audited. `POST /api/proxmox/qemu/create`,
`GET /api/proxmox/qemu/create-options`.

## 7. Ansible playbook runner

A new **Ansible** admin page: store playbooks and run them against a group /
tag / site / the whole fleet, with the RemotePower server acting as the Ansible
control node over SSH. Disabled when `ansible-playbook` isn't installed.
Defining playbooks is admin-only; running is gated on the `exec` permission and
audited. SSH credentials (key or password) are supplied per run and live only in
0600 temp files for the run's duration. `GET/POST /api/ansible/playbooks`,
`PUT/DELETE /api/ansible/playbooks/{id}`, `POST /api/ansible/playbooks/{id}/run`.

## Upgrade notes

- Agent bumped to 3.7.0 for lockstep; **no agent change** this release — all
  features are server/UI-side (the Ansible runner uses the server's own SSH,
  not the agent).
- The Ansible runner needs `ansible-core` (and `sshpass` for password auth) on
  the **server** to function; it's a clean no-op otherwise.
