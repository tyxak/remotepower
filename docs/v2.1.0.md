# v2.1.0 release notes

Released: 2026-05-13.

This is a maintenance release with two bug fixes and three features, all
focused on day-2 operations: making the heartbeat more resilient under
load, killing a long-standing JavaScript crash, and giving operators a
better story for running shell against the fleet.

If you're upgrading from 2.0.0, [docs/upgrading.md](upgrading.md) covers
the migration. No schema changes — `scripts.json` and `batch_jobs.json`
are created on first use; existing data files are read as-is.

---

## Bug fixes

### Flock offline fluctuation

**Symptom.** Devices flipping between online and offline every minute or
two on busy servers, even though the agents were running fine. The
`/var/log/nginx/error.log` showed nothing useful; the agents reported
heartbeats as "failed" because the response didn't come back before the
HTTP timeout, then the next one would succeed.

**Root cause.** `save()` in `api.py` held the per-file flock across the
entire write path — backup copy + `fsync()` + atomic rename. On a busy
ext4 / spinning disk `fsync` alone can take 50–200 ms. The heartbeat
handler chains six saves per request (devices, cmd_output, update_logs,
containers, config, commands) so a single slow writer stalled every
agent's heartbeat behind it. Once a request took longer than the agent's
HTTP timeout, the agent recorded the poll as failed and the server flipped
the device to offline — until the next heartbeat made it online again.

**Fix.** Move the expensive work *outside* the lock:

1. `save()` now writes the per-process unique `.tmp.<pid>.<nonce>` file
   **before** acquiring the lock. The unique filename was already what
   prevented corruption from concurrent writers, so the lock isn't doing
   anything during the write itself.
2. The lock is held only for the rolling-backup copy + atomic rename —
   both O(1) metadata ops. Critical section is microseconds rather than
   tens of milliseconds.
3. New `save(..., non_blocking=True)` mode that uses `LOCK_EX | LOCK_NB`
   and retries for ~100 ms total before raising `LockBusy`. The heartbeat
   handler catches `LockBusy` and returns HTTP 202 (Accepted), which the
   agent treats as "delivered, retry next cycle" — so a contended save
   no longer stalls the request long enough for the agent's timeout to
   fire.
4. Lock waits longer than 50 ms log to nginx's error log:
   `[remotepower] lock_wait path=devices.json waited_ms=87 mode=nb pid=…`.
   Operators can grep for these to see whether they're actually hitting
   flock contention or whether the next "fluctuation" report is something
   else entirely.

**Files touched.** `server/cgi-bin/api.py` (rewrote `save()` + added
`LockBusy` + `_acquire_lock` + 7 `_save_nb` substitutions in
`handle_heartbeat`); `client/remotepower-agent` (handles 202's
`{busy: true}` response without flipping state).

### Auto-refresh closes browser window / tab crash

**Symptom.** The dashboard would close itself or crash the tab during
the periodic auto-refresh, particularly on fleets with many devices.
Reproducible if you had a device named `O'Brien`, `Ada's-laptop`, or
similar.

**Root cause.** Two independent bugs combining:

1. `escHtml()` in `app.js` only escaped `&<>"`. It did **not** escape `'`,
   and the dropdown rebuild interpolated device names into inline
   `onclick="fn('${escHtml(d.name)}')"` strings. A name containing an
   apostrophe broke out of the JS string literal mid-attribute. When the
   periodic `loadDevices()` re-rendered the device grid, the parser hit
   `fn('O'Brien')`, threw a SyntaxError, and depending on what followed
   the quote could call other inline handlers — including, in the worst
   case, window-closing APIs. Effectively a self-inflicted XSS: the device
   name didn't even need to be malicious, just typographically realistic.

2. `setInterval` for the refresh cycle kept firing regardless of whether
   the user had a modal open or the tab was hidden. Combined with #1,
   the broken JS ran every 60 s under the operator's nose. Even on
   clean names, re-rendering the device grid while a modal was open
   destroyed event listeners the modal had captured by reference.

**Fix.**

1. Added a separate `escAttr()` that hex-escapes (`\x27` etc.) for
   JS-in-HTML-attribute contexts. Survives both HTML and JS parsing —
   no entity-decoding round-trip can turn it back into a quote. A script
   converted 73 inline-handler escape sites in `app.js` from `escHtml`
   to `escAttr`. `escHtml` keeps its old, safer-for-text semantics.
2. `startRefreshCycle` now pauses when `document.hidden` or any
   `.modal-overlay.active` exists. Countdown label shows "Refresh paused"
   so the operator knows it isn't broken.
3. `toggleDropdown` rewritten to clean up its click handler properly,
   look up the dropdown by ID on every click rather than capturing the
   DOM node by reference (survives re-renders cleanly).

**Files touched.** `server/html/static/js/app.js` only.

## New features

### Script library

Multi-line bash scripts, separate from the existing one-liner Command
Library. See **[docs/scripts.md](scripts.md)** for the full feature doc
and API reference.

The user-facing summary: a **Scripts** page in the sidebar, CRUD over
saved scripts, `bash -n` syntax check + an 11-pattern dangerous-command
detector that runs on save and on demand. Scripts can be queued from
the device dropdown (single device) or via the batch action bar
(multi-select). Dangerous patterns flag with a `⚠ DANGER` badge in the
listing and a confirmation gate at dispatch.

### Multi-select script execution

The batch action bar gets a new **Run script** button alongside the
existing shutdown/reboot/upgrade/update. Pick a saved script, fan it out
to every selected device, and a status modal polls
`GET /api/exec/batch/<id>` every 10 s to show per-device output as it
returns from each agent's next heartbeat.

Job records have a 1-hour TTL. After that the status modal returns 404,
but the agent's normal `cmd_output` history retains the output for
longer (subject to `MAX_CMD_OUTPUT`).

**New endpoints.** `POST /api/exec/batch` and `GET /api/exec/batch/<id>`.

### Docker compose dropdown

If the agent finds `docker-compose.yml` / `compose.yml` under any of
`/opt`, `/home`, `/docker`, `/srv` (max depth 4, hard cap 50 projects),
it reports them in the heartbeat. The dashboard renders a **docker
compose** entry in the device dropdown showing the project count.
Clicking it opens a modal with **Up / Down / Restart / Pull / Logs**
buttons.

See **[docs/compose.md](compose.md)** for what gets scanned, the
security boundary on dir validation, and the full agent and server
implementation notes.

## Upgrading

This release is forward-compatible. Old agents continue to work against
a new server; new agents continue to work against an old server (they
just won't get the 202-on-busy behaviour or report compose projects).

`make dist` builds a release tarball:

```
$ make dist
==> Built dist/remotepower-2.1.0.tar.gz
-rw-r--r-- 1 root root 4.5M  dist/remotepower-2.1.0.tar.gz
-rw-r--r-- 1 root root   91  dist/remotepower-2.1.0.tar.gz.sha256
```

Standard upgrade path is in [docs/upgrading.md](upgrading.md).

The new tests cover the `LockBusy` retry behaviour under real flock
contention, every dangerous-command regex pattern with positive and
negative cases, the batch endpoint's dispatch and TTL pruning, and the
compose security boundary (rejecting `dir` values not in the device's
reported list).
