# RemotePower v3.6.0 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.6.0 is a seven-feature batch that closes the gap between *seeing* a host's
state and *acting* on it — remote files, account/key/firewall management, backup
orchestration, AV posture, scheduled auto-patching — plus a Proxmox backup
recency check.

1. **Remote file manager (SFTP)** — browse and transfer files in the browser.
2. **Backup orchestration** — define, run, and schedule backup commands.
3. **Host user & SSH-key management** — add/lock/unlock/delete users, add/revoke keys.
4. **Endpoint AV/malware posture** — ClamAV / rkhunter status + on-demand scans.
5. **Host firewall rule management** — ufw / firewalld allow/deny/delete.
6. **Auto-patch policy** — scheduled fleet updates by group/tag/site.
7. **Proxmox backup check** — per-guest vzdump recency, flagged in attention.

---

## 1. Remote file manager (SFTP)

A new **Files** action on the device drawer opens an SFTP session tunnelled over
the same web-terminal daemon + ticket + SSH path as the terminal and VNC viewer
(no new transport, no inbound ports). Browse directories, download and upload
files (up to 32 MB per transfer), create folders, and delete. Runs as the SSH
user, so the OS enforces permissions. The daemon gained a `mode: 'sftp'` that
speaks a small JSON request/response protocol (`list`/`read`/`write`/`delete`/
`mkdir`/`rename`) over the WebSocket.

## 2. Backup orchestration

Closes the loop on the existing `backup_stale` *monitoring*: under **Planning →
Backups**, define a backup command per device (restic / borg / rsync / …), run
it on demand, or attach a cron schedule. Job definitions are admin-only (the
command is arbitrary shell, like a saved script); running is gated on the `exec`
permission and queued through the normal command path (quarantine still blocks
it). A per-request sweep fires due cron jobs.

`GET/POST /api/backup-jobs`, `PUT/DELETE /api/backup-jobs/{id}`,
`POST /api/backup-jobs/{id}/run`.

## 3. Host user & SSH-key management

Beyond monitoring `authorized_keys` drift, the **Users & keys** device action
queues the standard account operations: add user (`useradd -m`), lock/unlock
(`usermod -L/-U`), delete (`userdel`, never `-r`), and add/revoke an SSH public
key in the user's `authorized_keys`. Every operation is `exec`-gated, audited,
and quarantine-blocked. Inputs are strictly validated — usernames against
`[a-z_][a-z0-9_-]{0,31}`, SSH keys against a known-type + base64 pattern — so
nothing the operator types can break out of the constructed command.

`POST /api/devices/{id}/user-action`.

## 4. Endpoint AV/malware posture

The agent reports ClamAV / rkhunter presence, ClamAV signature-DB age, and a
parse of the latest scan/log (infected count, rkhunter warnings) on its slow
cadence. The **AV scan** device action shows that posture and queues an
on-demand ClamAV or rkhunter scan (output to the device command history).
Infections → a **critical** attention item; a stale signature DB (>7d) or
rkhunter warnings → **warning**. Silenceable as the NA kind `av_posture`.

`GET /api/devices/{id}/av`, `POST /api/devices/{id}/av-scan`.

## 5. Host firewall rule management

The agent already *reported* ufw/firewalld/nftables status; the **Firewall**
device action now *manages* simple rules — allow / deny / delete a port for ufw
or firewalld — mirroring the OPNsense/RouterOS firewall management already in the
product. `exec`-gated and audited.

`POST /api/devices/{id}/firewall-action`.

## 6. Auto-patch policy

**Planning → Auto-patch**: apply package updates automatically on a cron
schedule across a group, tag, site, or the whole fleet, optionally rebooting if
required. Queued upgrades flow through the normal command dispatch, so
maintenance-window execution gating and device quarantine apply for free. Run a
policy on demand or let the sweep fire it. Policy mutation is admin-only.

`GET/POST /api/autopatch`, `PUT/DELETE /api/autopatch/{id}`,
`POST /api/autopatch/{id}/run`.

## 7. Proxmox backup check

The Proxmox integration now queries each storage's `content=backup` listing,
groups vzdump archives by guest, and caches the newest backup timestamp per
guest (refreshed opportunistically when the Virtualization page loads). Guests
with no backup, or whose newest backup is older than
`proxmox_backup_warn_days` (default 7), are flagged as attention items
(NA kind `proxmox_backup`) — the backup analogue of the existing stale-snapshot
check.

---

## Upgrade notes

- The agent is bumped to 3.6.0 and gains one new optional collector (`av`); all
  other features are server/daemon/UI-side. No migration.
- The destructive host operations (user/firewall/backup/AV-scan) all run through
  the existing audited, `exec`-permission-gated, quarantine-aware command queue
  — they are not a new privilege path.
