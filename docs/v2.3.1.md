# v2.3.1 release notes

Released: 2026-05-17.

A small security release hardening how the Proxmox API token (and
the other stored secrets) are handled.

## Changed

### Proxmox token secret can be supplied via environment variable

The Proxmox API token secret can now be set in the
`RP_PROXMOX_TOKEN_SECRET` environment variable — in the systemd
unit (`Environment=` / an `EnvironmentFile=` with mode 600) or the
container environment. When that variable is set it takes
precedence over the value in `config.json`.

Why this is better: the secret is no longer in the data directory.
It is not in `config.json`, it is not swept into the backup export,
and it isn't sitting in a file that a misconfigured backup job or
file sync might copy somewhere world-readable. The `config.json`
value remains a working fallback for setups that don't use the env
var, so this is opt-in and nothing breaks on upgrade.

The Settings → Proxmox page detects when the secret is coming from
the environment: it shows a confirmation hint and disables the
config field (which would be ignored anyway).

### Backup export redacts secrets from config.json

Before 2.3.1 the backup ZIP (`Settings → Advanced → Export backup`)
included `config.json` verbatim — so a backup file carried live
credentials: the Proxmox API token secret, the SMTP password, and
the LDAP bind password. Anyone who got hold of a backup got those
secrets.

The export now redacts all three — each is replaced with
`(redacted)` in the `config.json` inside the ZIP. The keys are kept
(not dropped) so a restored backup is structurally intact and the
operator can see which secrets need re-entering. This matches how
`apikeys.json` key values were already redacted.

Note this means a restored backup needs its secrets re-entered —
that is the correct trade-off. If you use `RP_PROXMOX_TOKEN_SECRET`
the Proxmox secret isn't in `config.json` to begin with, so it
survives a backup/restore cycle untouched in the environment.

## Upgrading from 2.3.0

Drop-in, server-side only. To move the Proxmox secret out of
`config.json`: set `RP_PROXMOX_TOKEN_SECRET` in the server's
environment, then clear the token-secret field in Settings →
Proxmox and save. Verify with the Test-connection button.

## Honest caveats

1. **This raises the bar; it does not eliminate the risk.** For
   the server to poll Proxmox unattended it must be able to read
   the secret at runtime. The environment-variable approach keeps
   the secret out of the data directory and the backup — it
   defends against the realistic leak paths (a readable
   `config.json`, a secret in a backup ZIP, a file sync). It does
   **not** defend against an attacker who can already read the
   service process's environment (e.g. root, or the service user).
   That would need an OS-level secret store (systemd credentials,
   a vault) — a larger piece of work, deliberately not in this
   release.

2. **The env var is visible to the process and its children.**
   `RP_PROXMOX_TOKEN_SECRET` is readable via `/proc/<pid>/environ`
   by the process owner and root. This is normal for service
   credentials but worth knowing — it is not a secret store, it is
   a better-than-config-file location.

3. **Browser-untested**, as usual — the settings-page env hint and
   the disabled-field behaviour were verified by unit test (the
   asset/JS checks) but not loaded in a real browser.

4. **Still no encryption at rest** for the `config.json` fallback
   path, and SMTP / LDAP passwords still live in `config.json`
   when set (they're now redacted from *backups*, but the live
   file still has them at mode 600). An env-var option for those
   two, mirroring the Proxmox one, would be a reasonable follow-up
   if you want it.
