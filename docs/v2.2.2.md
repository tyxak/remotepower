# v2.2.2 release notes

Released: 2026-05-15.

A small hotfix for three things found in the first browser run of
2.2.1. No new features — three real-world bugs that needed real
hands on the dashboard to surface.

## Fixed

### Hover-action focus ring clipped on the right edge

In 2.2.1 the hover-revealed `Detail · Logs · Run` strip lived inside
the last `<td>` (the actions cell) which is only 50 px wide. When
the `Run` button took keyboard focus, the system focus outline
extended beyond the cell's right edge and got visibly clipped, with
the outline trailing off the right side of the row.

**Fix**:

- The action strip now lives in the **first cell** (next to the
  row's checkbox). Absolute positioning relative to the row keeps
  the visual placement unchanged.
- `right: 12px` → `right: 24px` — more breathing room from the row
  edge.
- `z-index: 2` so it always renders above adjacent cells.
- `:focus / :focus-visible` outline suppressed on these buttons,
  replaced with a softer accent border. Keyboard navigation still
  works; the harsh system outline doesn't.
- `td { overflow: visible }` on rows with hover actions, defensive
  against future grandparent-clipping.

### Home dashboard: webhook activity panel hit a 404

`loadHome()` called `/api/webhook-log?limit=20` — wrong path, the
hyphen should have been a slash. The correct endpoint is
`/api/webhook/log`. 404 in the network panel; activity panel
showed empty.

**Fix**:

- Endpoint changed to `/webhook/log`.
- `_renderHomeActivity()` now handles the actual response shape
  (a flat list, newest-first reversed by the server) rather than
  the `{events: [...]}` shape I assumed.
- The endpoint requires admin auth; viewer-role operators see
  `null` here and get the friendly empty state. Same handling for
  any other transient failure.

### Pre-existing: `handle_webhook_log` crashed 500 on a bare-list
webhook_log.json

While debugging the above I noticed `/api/webhook/log` was also
returning 500 in the screenshot logs. Root cause: the handler did
`wl.get('entries', [])` against `wl = load(WEBHOOK_LOG_FILE)`, but
some deployments (older releases, hand-edited files, or tooling
that wrote the bare list) have the file as a `[…]` instead of a
`{entries: [...]}`. `list.get('entries')` → `AttributeError` → 500.

**Fix**: handler now accepts both shapes — `{entries: [...]}` or
bare list — and normalises in the response. Disk file is left as-is
(operators may have other tooling assuming the bare-list shape;
auto-rewriting on read would be a bigger change than this hotfix
warrants).
