# v2.4.10 release notes

Released: 2026-05-18.

Two bug fixes — a broken forced package scan, and a fake 7-day
status stripe.

## Fixed

### Forced package scan was silently skipped on stable hosts

The "Send package list now" button (CVE page) sets a flag; the
agent picks it up and runs `send_package_list()`. But that
function has an unchanged-list **hash gate**: if the installed
package list is identical to the last submission, it skips the
upload — a sensible bandwidth saving for the routine 6-hourly
send.

That gate also applied to the *forced* send. On a stable host the
package list genuinely does not change for days, so the hash
matched and every forced scan was silently dropped. The agent log
showed "Forced package scan (operator requested)" — it really did
run — but `packages.json` on the server never updated. (Confirmed
on `tviweb01`: the agent ran the forced scan, `packages.json`
stayed stamped from the previous day.)

Fixed: `send_package_list()` takes a `force` argument; the forced
heartbeat path passes `force=True`, which bypasses the hash gate.
A forced scan now always uploads, regardless of whether the list
changed.

### The 7-day status stripe was not real

The dashboard's "Fleet roster — 7-day status" stripe showed every
day except today as `unknown`. The stripe was hardcoded — six
`unknown` cells plus a single live cell for today. A device up
for a week still showed five blank days.

The underlying cause ran deeper: `_record_uptime()` was only ever
called with `online=True` (from the heartbeat). The offline-
detection sweep fired the `device_offline` webhook but never
recorded the transition. So `uptime.json` only ever held a single
online event per device — there was no up/down history for any
stripe to draw on.

Two fixes:

- The offline-detection sweep now records both transitions —
  offline *and* the later recovery — in `uptime.json`.
- A new `/api/fleet/uptime7d` endpoint derives a real 7-day daily
  up/down status from those events, and the dashboard stripe
  reads from it.

**Important honesty:** this fixes the stripe *going forward*. It
cannot reconstruct history that was never recorded — days before
the upgrade still show `unknown`, correctly, because RemotePower
genuinely has no record for them. The stripe becomes fully real
once a week of post-upgrade data accumulates. `unknown` now means
"genuinely no data", not "feature not implemented".

## Tests

`tests/test_v2410.py` (8 tests): `send_package_list` takes `force`
and the hash gate is guarded by `not force`; the forced call
passes `force`; both uptime transitions are now recorded; the
day-status helper classifies up / down / unknown correctly; the
`/api/fleet/uptime7d` endpoint returns 7 cells, all `up` for a
long-online device, and `unknown` for days predating a device's
history.

**Total: 1047 tests, all passing.**

## Upgrading from 2.4.9

Drop-in. Deploy the updated agent so forced scans bypass the hash
gate. The 7-day stripe starts filling with real data immediately;
expect it to be fully populated after seven days of uptime
recording.

## Honest caveats

1. **The stripe stays mostly `unknown` for a week.** Covered
   above — the history did not exist before this release and
   cannot be invented. This is the honest state, not a bug.

2. **Agentless devices may show `unknown` indefinitely.** Devices
   that do not run an agent (e.g. `opnsense01`) do not heartbeat,
   so the offline-detection sweep does not evaluate them and they
   accrue no uptime events. Their stripe reflects current status
   only. Giving agentless devices a real uptime history would
   need active probing (the ping/TCP monitor already does this
   for explicit targets) — out of scope here.

3. **The forced-scan fix needs a 2.4.10 agent.** An older agent
   still applies the hash gate to forced sends. Update the agent
   on any host where a forced scan must work.

4. **Browser-untested**, as usual — the stripe rewrite is covered
   by unit and asset tests, not a real browser load.
