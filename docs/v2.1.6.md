# v2.1.6 release notes

Released: 2026-05-14 (same-day hotfix for 2.1.5).

Two real bugs in 2.1.5's Patches page, plus a regression test to keep
this exact class of bug from recurring.

## Fixed

### Patches page → Detail button threw "can't access property textContent of null"

Two compounding causes:

1. **2.1.5 layout regression.** Adding the ✨ Prioritise button next
   to Detail used `<td style="display:flex;gap:4px">` to align them.
   But `display:flex` on a `<td>` knocks the cell out of its
   `display:table-cell` behaviour, so the cell escapes the table
   grid — visible in screenshots as the Detail buttons hanging off
   the right edge of the table card. Fixed: wrap the two buttons
   in a `<div style="display:flex">` *inside* the cell, leaving
   `<td>` to do its own thing.
2. **Pre-existing bug uncovered by the new button.** The Detail
   handler `openDevicePatchReport()` references `#device-patch-title`,
   `#device-patch-body`, and `#device-patch-modal` — but **those
   elements were missing from `index.html` entirely**. The function
   had been broken for several releases, but the v2.1.5 ✨ Prioritise
   button drew attention to it. Restored the missing modal alongside
   the other detail modals.

The pre-existing bug had been silent because nobody clicked Detail
on the Patches page — the inline column already showed everything
useful. The 2.1.5 ✨ Prioritise button sits right next to Detail,
made it visible, and clicking either one triggered the underlying
HTML-element-missing crash.

## Added — regression test

`tests/test_v215.py::TestHtmlIdReferences` — scans `app.js` for
every `getElementById(...)` and `(open|close)Modal(...)` call, then
verifies the referenced ID exists in `index.html`. Restricted to
modal/title/body IDs to avoid noise from per-row dynamic IDs.

A `KNOWN_DYNAMIC_IDS` allow-list covers things that are intentionally
created in JS (the AI modal that gets generated by `_ensureAIModal()`
on first ✨ click, the toast container). Any new "JS function
references an HTML element that doesn't exist" bug will now fail
this test at build time instead of throwing in production.

Verified the test actually catches the bug by temporarily removing
the restored modal and re-running — got the expected failure
listing all three missing IDs.

## Tests

- Total suite: **715 tests, all passing.**
