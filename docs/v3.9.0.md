# RemotePower v3.9.0 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.9.0 is a second bind-it-together and hardening sweep on top of v3.8.0. It
wires more dropped agent data into the UI, fixes a handful of correctness bugs
in the alerting and patch-verification paths, closes an SSRF gap in the uptime
monitor, and rounds off the front-end with sortable tables, Lucide icons, and
accessibility labels. No new headline features — a polish and correctness
release.

## Security

- **HTTP uptime-monitor SSRF closed.** The `http`/`https` monitor target was
  validated only by a literal string-prefix blocklist (`127.`, `10.`, `192.168.`
  …) and then fetched with a bare `urllib.request.urlopen`. That left three
  holes: an IPv6 loopback (`http://[::1]/`), an alternately-encoded IPv4
  (`http://2130706433/` = 127.0.0.1, plus octal/hex), and DNS rebinding — a
  hostname that resolves to a public address for the pre-flight check but to a
  metadata/loopback address when the request actually connects. The check now
  routes through the same connect-time SSRF guard used by the webhook, audit-
  forward, and OIDC back-channels: the *connected* peer IP is re-classified
  (anti-rebinding), 3xx redirects are refused, and the shared IP classifier
  replaces the string list. RFC1918 LAN targets remain allowed by design (the
  whole point of a fleet monitor); cloud-metadata / link-local is always blocked,
  and loopback only when the operator opts in via `allow_internal_monitors`.
- **Inbound-webhook alert links are scheme-validated.** A link attached to an
  inbound alert is now run through the same `http(s)`-only validator the operator
  quick-links and CVE reference-links use, rejecting `javascript:`/`file:` and
  quote characters at ingest — defence-in-depth so a future renderer that makes
  these clickable can't be tricked.

See `docs/security-review-3.9.0.md` for the full review.

## Fixes

- **Post-upgrade verification no longer false-alarms.** The patch report's
  per-device verification badge flipped to *stalled* — shown as "didn't take" —
  whenever the host's pending-update count failed to strictly decrease within an
  hour of an upgrade being queued. That wrongly fired in two common cases:
  - a **fleet-wide upgrade** that also touches already-patched hosts (nothing was
    pending, so nothing could drop), and
  - an **offline host** whose upgrade command is still queued and undelivered.

  Both now read as "nothing to verify" (no badge) or "still pending" rather than
  a failure. A genuinely stalled upgrade on an online host that *did* have
  pending updates still surfaces as before.

  The badge could also get **stuck on "verifying…" indefinitely** when the host
  had no baseline patch count at queue time (the agent hadn't reported one yet),
  because the before/after comparison had nothing to compare against. That now
  resolves to "nothing to verify" too, and the patch report shows a **Re-check**
  button that forces a fresh package scan on demand rather than waiting for the
  periodic one. The verification simply compares the pending-update count
  before the upgrade against the count after the host's next package scan — its
  tooltips now spell that out.
- **Metric thresholds could skip disk alerting.** When a device's CPU load eased
  back down through the recovery hysteresis band, the inline CPU branch executed
  a `return` that exited the entire threshold function — so that heartbeat
  skipped the per-mount disk threshold checks and discarded the metric state it
  had just computed for memory/swap. It now declines to transition the CPU level
  without bailing out of the function.
- **TLS-expiry alerts had a fixed severity and a broken title.** The severity
  mapper and the alert title read a `days` field the `tls_expiry` event never
  sends — it carries `days_left` — so every certificate-expiry alert landed in
  the inbox as `high` and was titled "TLS expires in ?d". Severity now scales
  correctly with the real days remaining (critical < 3 days, etc.).

## Bind it together

- **CPU-load history on the Trends page.** Load average was collected on every
  heartbeat but had no persisted time series anywhere — it could only ever be
  read as a live value. The daily sampler now keeps the load average and core
  count, and the Trends device chart plots a **CPU load (saturation %)** series
  (load ÷ cores, where 100% = load equal to the core count) next to memory,
  swap, and busiest-mount disk.
- **Swap on the metrics sparkline.** The per-device metrics modal showed
  CPU / memory / disk sparklines but not swap, even though swap percentage was
  being collected. It now has its own track.
- **rkhunter last-run time.** The AV-posture panel showed whether rkhunter is
  installed and its warning count, but not *when it last ran* — the one signal
  that tells you the posture is current. The agent already reported it; it now
  appears on the pill.
- **systemd alias resolution.** When you watch `mysql.service` but the host
  actually runs `mariadb.service`, the agent resolves the canonical unit name.
  That resolution was being dropped by the heartbeat sanitiser; the Services
  table now shows the canonical unit alongside the watched name.
- **Livepatch state.** The kernel pill showed the livepatch provider and whether
  a patch is applied, but not the provider's *state* (`checked`, `check-failed`,
  …) — the field that explains why a host shows "not applied". It's now shown
  when no patch is applied.

## Polish

- **Three tables gained their missing sort wiring:** the Log Alert per-device
  rules, the Log Alert fleet-wide rules, and the Maintenance suppression log.
  Each advertised sortable columns but clicking the headers did nothing.
- **Lucide SVG icons** replace the typographic glyphs (`▶`, `⬇`, `✕`, `✓`, `ℹ`)
  on the prominent action buttons — Run this command, Fetch current, Download
  server debug log, Clear — and on the toast notifications, matching the
  project's inline-SVG icon convention.
- **Accessibility:** the icon-only close buttons on the device drawer, the
  RouterOS / OPNsense console modals, and the mobile menu now carry an
  `aria-label` so a screen reader announces them.
- **Command Queue completeness.** ACME renew/revoke/issue actions are now
  recorded in the command history, so they show in the queue's *recently
  dispatched* log like every other queued command — previously they waited for
  the agent invisibly. The Command Queue also gained a **Clear all pending**
  button (clears every device's queue at once) and a **Clear log** button for
  the dispatched-command history, beside the existing per-command cancel and
  per-device clear.
- **One-click container image update.** The Image Updates page now has an
  **Update** button on stale, compose-managed rows that runs `docker compose
  pull` then `up -d` on the affected host(s) — fetching the new image and
  recreating the container in one go (a new compose `update` action). Two agent
  improvements make it work and make the page clearer:
  - it captures each container's **compose working directory** (from the
    `com.docker.compose.project.working_dir` label), so the server knows where
    to run the update; and
  - it **recovers the real image name when `docker ps` reports a bare untagged
    image ID** — which happens right after a `compose pull` that hasn't been
    followed by `up -d`, where the still-running container references the old,
    now-untagged image. Previously such a container showed as `sha256:…` and
    "Local"; now it resolves to e.g. `ghcr.io/seerr-team/seerr:latest`, and the
    rows also show the **container name** so they're identifiable regardless.

  The check stays notify-only — RemotePower never pulls or restarts on its own;
  the Update button is an explicit, audited, exec-gated operator action.

## Upgrading

Drop-in for v3.8.0 — no schema or config migrations. Agents self-update on
their next check-in (or opt into `/etc/remotepower/require-signed-updates` for
fail-closed signature enforcement). The new CPU-load and swap history starts
accumulating from first heartbeat after the upgrade; older devices simply show
those series once enough samples exist.
