# v2.4.8 release notes

Released: 2026-05-18.

A small dashboard fix.

## Fixed

### Recent activity feed no longer floods on one noisy host

The Home dashboard's "Recent activity" feed showed the most recent
8 fleet events with no de-duplication. A host repeatedly emitting
the same alert — the trigger here was a `postfix.service`
`log_alert` firing roughly hourly on one mail host — filled all 8
rows with the same line, burying every other host's activity.

The feed now collapses repeated entries that share the same
event, host, and subject (the watched unit, file, metric, CVE id,
etc.) down to their single most-recent occurrence. Two different
hosts, or two different subjects on one host, still appear
separately — only an exact repeat is folded.

This is a display change only. The server's fleet event log still
records every individual event; the audit trail is unchanged.
De-duplication happens before the feed is trimmed to 8, so the
feed now shows 8 *distinct* recent things rather than 8 rows that
might all be the same.

## Tests

`tests/test_v248.py` (5 tests): de-dup runs before the slice; the
key combines event + host + subject; a Set backs the lookup; the
event allowlist filter still runs; and a Python re-implementation
of the key logic proves the reported case (8x identical) collapses
to one while a genuinely different entry survives.

One pre-existing `test_v223` test had its source-search window
widened — the de-dup block lengthened the function past the old
2000-character window.

**Total: 1039 tests, all passing.**

## Upgrading from 2.4.7

Drop-in. Frontend only — no API, agent, or data changes.

## Honest caveats

1. **The underlying noise is not addressed.** This makes the feed
   readable; it does not stop `pmg01`'s postfix unit from emitting
   a `log_alert` every hour. If those alerts are not actionable,
   tune the log-watch pattern for that unit, or mark it handled —
   the de-dup just stops one noisy source from hiding everything
   else.

2. **De-dup is scoped to the dashboard feed.** The fleet event log
   page, webhook deliveries, and the event count are all
   unchanged — every event is still recorded and still delivered.

3. **Browser-untested**, as usual — covered by asset and logic
   tests, not a real browser load.
