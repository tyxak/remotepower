# v2.3.2 release notes

Released: 2026-05-17.

A security release — no new features. This release is the result of
a focused security review of the authentication, command-execution,
secrets, transport, and frontend surfaces. The full review, including
findings that were *not* changed and why, is in
**docs/security-review-2.3.2.md**.

## Fixed

### Unsalted SHA-256 password fallback → salted PBKDF2

When the `bcrypt` library wasn't installed, password hashing fell
back to a bare, **unsalted** `hashlib.sha256`. Unsalted SHA-256 is
fast and rainbow-table-able — a stolen `users.json` would be
trivially crackable. (`bcrypt` is present in the Docker image and
the documented bare-metal install, so this was a fallback path —
but a fallback that silently downgrades to "crackable if leaked"
isn't acceptable.)

The fallback is now **salted PBKDF2-HMAC-SHA256** — random 16-byte
per-hash salt, 600,000 iterations (OWASP floor), pure stdlib, no new
dependency. Hashes are self-describing: `pbkdf2$<iters>$<salt>$<hash>`.

Backward compatible: existing unsalted SHA-256 hashes still verify,
and they're upgraded automatically (to bcrypt if available, else
PBKDF2) on the user's next successful login.

### Default-password warning on bare-metal installs

A fresh bare-metal install seeds an `admin` account with the
well-known password `remotepower`. (Docker already generates a
random password — fixed in 2.2.6.)

Two changes: the seeded admin is now stored with a proper salted
hash instead of a bare SHA-256, and it carries a
`must_change_password` flag. While that flag is set the UI shows a
persistent red banner — "This account is using the default
password, change it now" — linking to the password-change form. The
banner clears automatically once the password is changed.

This makes running on the default impossible to miss. It does not
*force* a change — a hard forced-change flow has lockout edge cases
and is left as a future hardening; the loud banner is the safe,
non-breaking nudge.

## Tests

`tests/test_v232.py` (11 tests): PBKDF2 round-trip and salting,
iteration-count floor, legacy SHA-256 backward compatibility,
`hash_password` never emits a bare digest, corrupt-hash handling,
default-user re-seed format and flag, the UI banner, and a
regression guard that both nginx configs still carry their security
headers.

**Total: 940 tests, all passing.**

## Reviewed and found OK

The review confirmed a number of existing controls are sound and
left them unchanged: login rate-limiting + lockout, constant-time
credential comparison, the username-timing-oracle defence, TOTP,
agent-to-server TLS verification, the SSRF guard on monitor targets,
strict resource-ID validation, `0600` data files, and the HTTP
security headers (incl. CSP) on both nginx configs. Details — and
the accepted limitations (the CSP's `'unsafe-inline'`, plaintext
secrets in `config.json`, the CSRF posture) — are in
docs/security-review-2.3.2.md.

## Upgrading from 2.3.1

Drop-in, server-side only. No data migration. Existing users are
unaffected — their password hashes keep working and upgrade silently
on next login. A fresh bare-metal install will show the
default-password banner until the admin password is changed.

## Honest caveats

1. **This is a focused review, not an exhaustive audit.** It
   concentrated on the high-risk surfaces — auth, command execution,
   secrets, transport, frontend escaping. It is not a line-by-line
   audit of all ~11k lines of `api.py`. A finding count of "two" is
   not proof the code is clean; it's the result of where the review
   looked.

2. **The default-password fix is a warning, not a wall.** A bare-
   metal operator who ignores a large red banner can still run on
   `remotepower`. If you deploy bare-metal: change the admin
   password immediately, as you always should.

3. **Browser-untested**, as with every release this series. The
   PBKDF2 work and the flag logic are unit-tested (940 tests); the
   banner itself was verified by an asset test, not by loading it in
   a browser.

4. **Plaintext secrets remain.** SMTP and LDAP passwords still live
   in `config.json` (mode 0600). 2.3.1 redacts them from backups;
   2.3.2 does not change the live-file storage. Extending the
   `RP_PROXMOX_TOKEN_SECRET` env-var pattern to them is the
   recommended next step — see the review document.
