# v2.4.1 release notes

Released: 2026-05-17.

A bugfix release — the CVE severity fixes from 2.3.4 / 2.4.0 were
correct but couldn't reach findings already in the cache. This
release makes them self-healing.

## Fixed

### Stale CVE cache served wrong (often "critical") severities

After 2.3.4 and 2.4.0 fixed the CVE severity classifier, some
findings still showed the wrong severity — e.g. several
`DEBIAN-CVE-*` packages reported as **critical** when OSV and
other sources rate them Medium.

The tell was in the data: every wrong finding had
`"severity": "critical"` together with `"severity_source":
"unknown"`. That pairing is impossible from the current code —
`_severity_from_vuln` produces both fields together and never
returns a real severity with an "unknown" source. So the severity
was not coming from the current classifier at all.

Root cause: the **CVE details cache** (`cve_details_cache.json`).
Cache entries written by a pre-2.3.4 RemotePower carry a
`severity` computed by the OLD, BUGGY classifier (the
`AC:H`-matched-as-`C:H` substring bug; the un-capped Debian
urgency) and have NO `severity_source` field — that field didn't
exist before 2.3.4. The scan's refresh gate only re-fetched an
entry if it was missing or past its TTL. A pre-2.3.4 entry that
was still within TTL looked "fresh", so it was reused as-is — the
stale wrong severity re-served on every scan, and `severity_source`
read back as the `"unknown"` default because the key was absent.
A manual "rescan" didn't help: the TTL kept the bad entry alive.

(Confirmed on a production cache:
`grep -c severity_source cve_details_cache.json` returned `0` —
the entire cache predated 2.3.4.)

Fixed: a cache entry without a `severity_source` field is now
recognised as pre-2.3.4 and treated as **stale regardless of
TTL** — it is re-fetched from OSV and re-classified with the
current code on the next scan. This is self-healing: no manual
cache wipe, no config change. After upgrading, the first scan of
each device rewrites its stale entries; subsequent scans use the
normal TTL.

The fix is narrow: it only forces a re-fetch for entries missing
`severity_source`. Modern entries (2.3.4+, which have the field)
still use the normal TTL — the cache is not defeated.

## Tests

`tests/test_v241.py` (3 tests), with OSV network calls stubbed:

- A pre-2.3.4 entry (stale `critical`, no `severity_source`,
  `cached_at` fresh/within TTL) IS re-fetched and re-classified —
  the finding comes back with the correct severity and a real
  `severity_source`.
- A modern entry (has `severity_source`, fresh) is NOT re-fetched
  — the fix doesn't defeat the cache.
- A modern entry past its TTL is still refreshed — normal TTL
  behaviour intact.

**Total: 971 tests, all passing.**

## Upgrading from 2.4.0

Drop-in, server-side. No data migration and no need to touch
`cve_details_cache.json` — the stale entries are detected and
refreshed automatically. Expect the first post-upgrade scan of
each device to take a little longer than usual (it re-fetches the
pre-2.3.4 entries from OSV); scans after that are normal speed.

If you want the corrected severities immediately rather than
waiting for each device's next scheduled scan, trigger a scan —
or, equivalently, deleting `cve_details_cache.json` still works
and forces a clean rebuild.

## Honest caveats

1. **The corrected severities appear on the next scan, not at
   upgrade time.** The fix changes what the *scan* does; it
   doesn't rewrite the cache on deploy. Until a device is
   re-scanned its CVE page still shows the old values.

2. **First scan per device is slower.** Every pre-2.3.4 cache
   entry for that device is re-fetched from OSV in one pass. On a
   large package set with many findings this is noticeably longer
   than a normal cached scan. It happens once per device.

3. **This depends on OSV being reachable at scan time.** The
   re-fetch goes to OSV. If OSV is unreachable the stale entry is
   kept (fail-safe — better a stale finding than none) and will be
   retried on the following scan.

4. **Browser-untested**, as usual — though this release is
   server-side scanner logic only, covered by the 971 unit tests.
