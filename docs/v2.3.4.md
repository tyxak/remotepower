# v2.3.4 release notes

Released: 2026-05-17.

A fleet-issues bugfix release — four fixes from a reported issue
list, plus an honest note on two items that are not changed.

## Fixed

### CVE severity misclassification (LOW/MEDIUM shown as HIGH)

The CVE scanner was upgrading low-severity CVEs to HIGH. Root
cause: the CVSS scorer did naive substring matching on the vector
string — it tested `'c:h' in vector`, which matched **`AC:H`**
(Attack Complexity: High) as if it were `C:H` (Confidentiality:
High). The result: *any* CVE with high attack complexity scored
7.5 / HIGH regardless of its real impact. A genuine LOW CVE
(CVSS 2.9) with `AC:H` — and high attack complexity is often
exactly *why* a CVE is low severity — came out HIGH.

Fixed: the CVSS vector is now properly tokenised (split on `/`
and `:`, no substring matching) and the real **CVSS v3.1 base
score formula** is applied. A CVSS score below 4.0 can no longer
become HIGH. Each finding now also carries a `severity_source`
field recording where the classification came from
(`cvss_v3`, `ubuntu_priority`, `database_specific`,
`debian_tracker`, …) per the spec's "log classification source"
requirement.

### Unmonitored devices appeared in Recent Activity

Devices explicitly set `monitored: false` were still showing up in
the Home activity feed. The fleet-events endpoint now filters them
out — at read time, so it reflects the *current* monitored state
(re-monitoring a device brings its history back) and never
silently discards events.

### Drift false positives — files can now be ignored

A watched file that's legitimately absent on a particular host
(the reported example: `/etc/pam.d/common-auth`) was flagged as
drift with no way to silence it. Each watched file now has an
**ignore** toggle in the drift detail view. An ignored file is
non-critical: it drops out of the device's drift / missing counts
and no longer drives a red status, but stays visible (shown muted,
with its underlying state and an optional ignore reason). This is
the explicit-decision counterpart to the automatic, time-based
`dormant` state added in 2.2.6.

### Services and Logs moved to the Main nav group

Services and Logs were under the collapsible Security group. They
are routine operational views, not security-specific tooling, so
they've moved up to Main (right after Monitor).

## Investigated — no change

### Dashboard time ranges

The report flagged "only yesterday visible, 2D/3D possibly broken,
may be a regression." Investigated: the Home dashboard activity
feed has no multi-day range selector — it fetches the most recent
events, full stop, and that hasn't changed. What is most likely
being seen as "only yesterday" is the **7-day status stripe**
showing `unknown` for most cells — and that is the long-standing
limitation that there is no server-side uptime history (it has
been on the deferred list for many releases), **not** a regression
introduced by an upgrade. The real fix is building server-side
uptime/metrics history, which is a separate feature, not a bugfix.
Nothing was changed here rather than fake a patch.

### Mobile rendering

Per the issue list, the mobile render problem resolved by
switching browsers — pointing to a browser-specific quirk or a
stale cache rather than a code defect — and was marked
deprioritised. Not changed in this release. It remains worth
capturing the browser console error if it recurs.

## Tests

`tests/test_v234.py` (11 tests): CVSS parser (the AC:H false-HIGH
bug, genuine high/critical, scope-changed, numeric passthrough,
the "<4.0 never HIGH" rule, severity-source logging, CVSS
roundup); unmonitored-device exclusion from the fleet feed; drift
ignore toggle and ignored-file count exclusion; the nav move.

Three pre-existing `test_cve_scanner` severity tests were updated:
`_severity_from_vuln` now returns `(severity, source)` rather than
a bare string, and a bare numeric CVSS score is now accepted (it
is a valid upstream score).

**Total: 951 tests, all passing.**

## Upgrading from 2.3.3

Drop-in, server-side. No data migration. Existing drift state and
CVE caches are compatible — the CVE severity recomputes on the
next scan; the drift `ignored` flag simply doesn't exist on
existing entries until set.

## Honest caveats

1. **Browser-untested.** 951 unit tests pass; the drift ignore
   button, the nav move, and the muted ignored-row styling were
   not loaded in a real browser.

2. **CVE severity recomputes on next scan, not retroactively.**
   Findings already in the CVE cache keep their old (possibly
   wrong) severity until the cache entry is refreshed by the next
   scan cycle. If you want the corrected severities immediately,
   clear the CVE cache.

3. **The CVSS v2 path is still approximate.** The proper formula
   is implemented for CVSS v3.x (the common case). CVSS v2 vectors
   get a conservative impact-metric-based estimate, not the full
   v2 formula. v2 advisories are increasingly rare.

4. **#3 (time ranges) was investigated, not fixed** — see above.
   If you are in fact seeing a genuine multi-day regression
   somewhere specific (not the status stripe), point me at the
   exact page and I'll look again.
