# v2.2.0 release notes

Released: 2026-05-15.

The first minor-version bump since 2.1.0. Two big new features, both
landing as their own surface and both tied to current themes in the
fleet-management space:

1. **Configuration drift detection** — per-device file integrity
   monitoring. The compliance theme of 2026 in one feature.
2. **MCP server** — natural-language fleet queries through Claude
   Desktop / Cursor / VS Code Copilot via the Model Context Protocol.

Plus the usual housekeeping and a small batch of fixes.

## Added

### Configuration drift detection

New **Drift** page under the Security sidebar group, plus per-device
detail. Watches a list of operationally-significant config files
(default: `/etc/ssh/sshd_config`, `/etc/sudoers`, `/etc/fstab`,
`/etc/crontab`, `/etc/hosts`, `/etc/resolv.conf`,
`/etc/nsswitch.conf`, `/etc/pam.d/sshd`).

**How it works**:

1. The agent computes SHA-256 hashes of the watched files every few
   heartbeats and ships them in the heartbeat payload.
2. The server stores baselines (first-seen hash per file per
   device) and detects when a current hash diverges.
3. On divergence, fires a `drift_detected` webhook event.

**Hash-only by design.** File contents never cross the wire on
routine polling — `/etc/shadow` and `/etc/sudoers` can be watched
without privacy concerns. To see what actually changed, the
operator can queue a `cat` command through the existing exec
mechanism (subject to the usual audit + permission checks).

**UI**:

- **Drift page**: fleet-wide table — devices sorted with drifted
  ones at top, columns for total / drifted / missing / last check.
- **Per-device modal**: row per watched file with status (baseline
  / drifted / missing), drift count, history (last 20 changes).
- **Accept-as-baseline button**: when you've reviewed a change and
  decided it's legitimate, click to make the current hash the new
  baseline. Audit-logged with operator and timestamp.

**Endpoints**:

```
GET    /api/drift                          — fleet overview
GET    /api/devices/<id>/drift             — per-device state
POST   /api/devices/<id>/drift/baseline    — accept current as baseline
                                              body: {paths: [...]} or {all: true}
DELETE /api/devices/<id>/drift             — wipe state
```

**Webhook event**: `drift_detected` (new; defaults to enabled).
Per-file debounced — fires once when crossing the baseline, *not*
on every poll that reports the same new hash.

**Storage**: `data/drift_state.json`, hash-only payload, per-file
history capped at 20 changes.

**Agent requirement**: needs **v2.2.0+** agent to report drift.
Older agents simply don't send the `drift` field; their devices
show "no data" on the Drift page until upgraded.

**Compliance angle**: configuration drift detection is an expected
control for SOC 2 (CC6.1, CC6.6), ISO 27001 (A.12.4.3, A.14.2.4),
HIPAA (164.312(c)), PCI DSS (11.5), FedRAMP. The audit-log entries
for baseline acceptances (`drift_baseline` events with actor +
timestamp) are designed to be readable as evidence.

Full reference: **[docs/drift.md](drift.md)**.

### MCP server — natural-language fleet queries

RemotePower now ships an [MCP](https://modelcontextprotocol.io)
server (`mcp/remotepower-mcp.py`, ~470 lines, pure stdlib Python).
Connect an MCP-compatible AI host (Claude Desktop, Cursor, VS Code
Copilot, etc.) and query the fleet in plain English:

> *Which of my devices have pending security updates?*
>
> *Show me the journal for pmg01 from the last hour.*
>
> *Is the SSH config on tviweb01 still at its baseline?*

**Architecture**: the MCP server runs on the **operator's laptop**,
not on the RemotePower server. The AI host spawns it as a subprocess
and exchanges JSON-RPC 2.0 frames over stdin/stdout. The MCP server
makes HTTPS calls to RemotePower's REST API on behalf of the AI.
Credentials live in the host's config file and never reach the AI
provider — only tool *results* do.

**Read-only by design.** 12 tools, all read-only:

| Tool | What it returns |
|---|---|
| `list_devices` | Every device with status, OS, group, tags, IP |
| `get_device` | Full detail for one device |
| `get_journal` | Recent systemd journal entries |
| `get_services` | Watched systemd unit states |
| `get_containers` | Docker / Podman container state |
| `get_cves` | CVE findings (per-device or fleet-wide) |
| `get_drift` | Configuration drift (per-device or fleet-wide) |
| `get_recent_commands` | Recent commands with output |
| `get_runbook` | AI-generated device runbook (v2.1.7+) |
| `get_patches` | Pending OS package updates |
| `get_tls` | TLS / DNS expiry watchlist |
| `search_devices` | Free-text search across name / OS / tags / notes |

**No write tools, deliberately.** No `run_command`, no `run_script`,
no `reboot_device`. The test suite explicitly asserts no
write-shaped names slipped in. Write tools need a server-side
allow-list + per-MCP-token role + confirmation flag for prod
devices — not the kind of thing to bolt on as an MVP feature.

**Device-name resolution**: exact → prefix → substring → ambiguity
error. The tools accept friendly names like `web01` and resolve
them to the right device.

**Protocol version**: `2024-11-05` (the widely-supported version
hosts standardised on after Anthropic published the spec).

Setup, security model, and host-specific config snippets:
**[docs/mcp.md](mcp.md)**.

## Upgrading from 2.1.9

Drop-in for the server. The new `data/drift_state.json` file is
created on the first agent drift report — no migration needed.

For the agent: **v2.2.0+ is required to report drift**. The standard
agent push-update flow (Settings → Agent updates) handles this like
any other agent release. Older agents continue to work, just
without drift data flowing to the new page.

For the MCP server: copy `mcp/remotepower-mcp.py` to your laptop
and configure your AI host per [docs/mcp.md](mcp.md). Setup takes
about two minutes once you've generated an API token in Settings.

## Honest caveats

1. **Browser-untested.** All 778 unit tests pass and the JS parses,
   but I haven't clicked through the Drift page in a real browser.
   The "accept baseline" + history viewer modal is where I'd look
   first if anything misbehaves.

2. **MCP server has never been spawned by Claude Desktop in real
   use.** The protocol-level tests verify the handshake, tool
   listing, and error handling. They don't verify a real host
   will like our specific framing. Possible surprises: a host
   expecting a different protocol version, capability negotiation
   quirks, stdout buffering during long responses. First user to
   try this will hit at least one of those.

3. **Drift detection is hash-only in v1.** "Show me what actually
   changed" — a diff view that queues a `cat` command via the
   existing exec path — is a future feature. Today, you see
   "drifted" without seeing the actual diff.

4. **Default watched-files list is conservative.** It's the
   small set that's almost always operationally significant. If
   your environment cares about `/etc/passwd`, `/etc/group`,
   `/etc/shadow`, or anything else, add them via
   `cfg['drift']['default_watched_files']` or the per-device
   `watched_files` override.

5. **MCP is read-only by deliberate design.** Write tools will
   land when the server-side allow-list, MCP-specific role, and
   audit hooks are in place — not earlier. If you want
   *"reboot pmg01"* via Claude Desktop today, the answer is no.
   Run it from the dashboard, where the existing audit and
   permission checks apply.

6. **The MCP server runs on the operator's laptop.** It doesn't
   ship as part of the RemotePower server install — it's a
   separate file you scp to your dev machine. This is the right
   architecture (credentials stay local, no exposed network
   service) but means there's no "one-click install" path. If
   you'd rather not deal with that, every other RemotePower
   feature is still reachable from the dashboard.
