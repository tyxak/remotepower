# v5.8.0 — "WatchMatters" (unreleased, test)

A monitoring-reliability + integrations release: TLS/DANE expiry probing now
runs on the server's own schedule (it was silently cron-only), and a new
GitHub Issues connector alerts on newly opened issues in repositories you
watch. No breaking changes.

## Fixed

- **TLS/DANE expiry checks now actually run on schedule.** Periodic probing of
  the TLS watchlist existed only as an optional cron script the installer
  merely *suggested* — and that script read the watchlist as a raw file, so it
  saw nothing under the SQLite/Postgres storage backends even where it was
  installed. The server now owns the cadence like every other monitor:
  `run_tls_scan_if_due()` re-probes each target once its last result is older
  than ~6 hours (oldest first, bounded to 10 targets / 15 s per sweep pass so a
  big watchlist never stalls a request), fires `tls_expiry` webhooks
  edge-triggered on warn/critical crossings honouring the per-target
  `warn_days`/`crit_days`, and prunes results for deleted targets. The
  standalone `remotepower-tls-check` runner is now optional (legacy) and
  backend-aware. **No action needed** — existing watchlists just start
  refreshing.

## Added

- **GitHub issue monitor** (Settings → Integrations → *GitHub Issues*). Watch
  one or more repositories — `owner/repo, owner/repo` (up to 10 per instance) —
  for **newly opened issues**:
  - URL is the API root: `https://api.github.com`, or your GitHub Enterprise
    `/api/v3` root.
  - The access token is optional — needed for private repos, and it lifts the
    anonymous rate limit. Stored server-side and redacted like every
    integration secret.
  - A new issue raises a **`github_new_issue`** alert (low severity) in the
    Alerts inbox and the dashboard activity feed. Pull requests are ignored,
    and the first poll only baselines, so attaching a repo never floods the
    inbox with its existing backlog. Bursts are capped per poll with an
    explicit "…and N more" summary alert.
  - The new **GitHub new issues** channel kind is deliberately quiet by
    default: Alerts inbox + activity feed only. Enable webhook / push /
    needs-attention routing in **Settings → Notifications** if you want to
    page on it.
  - Tiles on the dashboard widget and Integrations page show repo +
    open-issue counts like every other connector; transient GitHub outages
    surface as a warning status without resetting the new-issue baseline.

## Upgrade notes

Nothing to do. The TLS schedule activates on its own; the GitHub connector
only polls once you add an instance. An existing `remotepower-tls-check` cron
entry keeps working (it now reads through the storage backend) but is
redundant and can be removed.
