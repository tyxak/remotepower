# RemotePower v3.4.0 — Release Notes

**Status:** unreleased (dev)

## Overview

Adds **RAG over your infrastructure**: the AI assistant now answers from
*your* fleet — its device state, watched services, CVEs, containers, CMDB
metadata and docs, runbooks, recent commands and alerts, plus the
RemotePower product docs — instead of generic Linux knowledge. Every chat
request retrieves the most relevant indexed snippets and injects them into
the prompt as a cited `<retrieved_context>` block.

Pure stdlib, no new pip dependencies. Works with **every** provider —
including Anthropic, which has no embeddings endpoint — because lexical
(BM25) retrieval is the always-on base. Embeddings are an optional
semantic-rerank layer that turns on when you run an embedding-capable
provider (OpenAI / Ollama / LocalAI).

## How it works

1. **Index.** A background corpus builder normalises each source into
   chunked documents with metadata (source, device, timestamp). Markdown
   (product docs, runbooks, CMDB asset docs) is split on headings; live
   state becomes one chunk per `(device, facet)` — summary, hardware,
   services, CVEs, containers, SNMP. The index — lexical postings, the
   chunks, and the embedding cache — persists to
   `/var/lib/remotepower/rag_index.json`.
2. **Retrieve.** The operator's question is scored with BM25 over an
   inverted index. When embeddings are enabled, the question is embedded
   once and cosine-ranked against the cached chunk vectors, then the two
   rankings are fused with Reciprocal Rank Fusion. Top chunks (default 6,
   bounded by a char budget) are injected into the system prompt.
3. **Refresh.** The index rebuilds lazily on chat whenever an enabled
   source file is newer than the last build (cheap — the corpus is small),
   and on demand via **Rebuild index**. Embeddings are incremental: only
   chunks whose text changed (by content hash) are re-embedded.

## Privacy & safety

- **The credentials vault is never indexed.** `cmdb_vault` ciphertext and
  any `credentials` / `secrets` / `password` subtree are excluded by key —
  metadata and asset docs only.
- **Redaction at index time.** History chunks pass through the same
  privacy-toggle redaction as chat, so hostnames / IPs / secret-shaped
  tokens are scrubbed before they ever reach the index — or, with
  embeddings, a cloud provider.
- **Embeddings egress is opt-in.** Off by default for cloud providers
  (embedding content is data egress); the toggle is pre-checked only for
  local providers (Ollama / LocalAI), where nothing leaves the building.

## Where you see it

- **Settings → AI → Knowledge index (RAG).** Enable/disable, choose which
  sources to index, turn embeddings on, set max chunks and history
  retention, and **Rebuild index** with a live status line (chunk count,
  build age, embedded count, staleness).
- **Test retrieval** box: type a question, see exactly which chunks it
  pulls in (id · kind · device · excerpt, sortable) with no model call and
  no tokens spent.
- **Chat answers** cite their sources by bracketed id, e.g.
  `[live/web01#cves]`, so any claim traces back to the indexed source.

## Endpoints

- `GET  /api/ai/rag/status` — index freshness, counts, embeddings state.
- `POST /api/ai/rag/reindex` — rebuild the corpus + index (admin).
- `POST /api/ai/rag/search` — standalone retrieval, no LLM (any authed user).

## Deployment note

`deploy-server.sh` / `install-server.sh` now install the product
Markdown docs into `/var/lib/remotepower/docs/` (read-only, not web-served)
so the indexer can answer "how do I do X in RemotePower" questions. If the
directory is absent the docs source is simply skipped — never an error.

## Compatibility

No schema changes, no breaking changes. RAG is on by default but degrades
gracefully: with no embedding-capable provider it runs lexical-only, and
any retrieval error is swallowed so the chat path is never broken.

See [rag.md](rag.md) for the full reference.

# Fleet health & insights batch

A second wave of v3.4.0 adds hardware health, forecasting, on-demand
diagnostics, AI insights, device quarantine, Helm visibility, and compliance
reporting. All of it reuses the existing agent heartbeat, command channel,
event system, and AI provider layer — no new dependencies.

## Hardware & health (agent probes)

Three best-effort probes ride the slow (container-cadence) heartbeat, each
silently omitted when its tool is missing or unreadable:

- **SMART** (`smartctl -H -A -i` per disk) — overall health plus reallocated /
  pending / offline-uncorrectable sectors, CRC errors, temperature, power-on
  hours. A `FAILED` result or any pre-fail sector count fires the
  edge-triggered **smart_failure** event.
- **Kernel / livepatch** — running `uname -r` vs newest installed kernel
  package (dpkg/rpm); fires **kernel_outdated** when a reboot would pick up a
  newer kernel. Reads `canonical-livepatch` / `kpatch` status when present.
- **Passive inventory** — DIMMs, system serial (`dmidecode`), temperatures
  (`lm-sensors`), software-RAID arrays (`/proc/mdstat`).

Stored in `hardware.json`; shown in the drawer's **Health & Hardware** card.
`GET /api/devices/<id>/hardware`.

## Resource forecasting & "what changed"

`_maybe_sample_metrics` writes one compact sample per device per UTC day
(per-mount used/total GB, memory, swap, and a state fingerprint) to
`metrics_history.json` (≈6 months retained). `forecast.py` is pure stdlib:

- **Forecast** (`GET /api/devices/<id>/forecast`) — least-squares fit of
  used-GB over time per mount, extrapolated to capacity → days-to-full + a
  fill date. Flat/shrinking mounts report no fill.
- **What changed** (`GET /api/devices/<id>/changes?days=1|7`) — diffs the
  oldest-in-window snapshot against the latest: package deltas, opened/closed
  ports, failed/recovered units, reboot edges, per-mount growth.

## On-demand diagnostics

- **Speed test** — `POST /api/devices/<id>/speedtest` queues a `speedtest`
  command; the agent runs `librespeed-cli --json` and reports Mbps up/down +
  ping/jitter back through the command-result channel (`speedtest.json`).
- **Network discovery** — `POST /api/devices/<id>/netscan` (optional
  `{"subnet": "192.168.1.0/24"}` for an `nmap -sn` sweep, else passive
  ARP/neighbour table). Results land in `discovery.json`; `GET /api/discovery`
  aggregates unmanaged hosts across all scanning agents, cross-referenced
  against known device IPs.

## Device quarantine

`PATCH /api/devices/<id>/quarantine {quarantined: true|false}` (admin). A
quarantined device has **all** queued commands dropped at the heartbeat
dispatch chokepoint — defence-in-depth that catches every action path, not
just the common one — and `_queue_command` rejects new actions at queue time.
Poll-interval changes are the one exception (purely local to the agent).
Every change is audited.

## Helm releases

Where `helm` and a kubeconfig are present, the agent reports `helm list -A -o
json` (visibility only — never deploy/upgrade/rollback) to `helm.json`.
`GET /api/devices/<id>/helm`.

## On-demand AI insights

`ai_insights.py` holds pure prompt-builders + response parsers (so they
unit-test without a provider); the handlers call the shared `ai_provider.chat`
with the AI-enabled + rate-limit gates.

- **Fleet anomaly scan** — `POST /api/ai/anomaly`. Assembles a compact
  per-device metric line for the fleet, asks the model for ranked anomalies.
- **Cron builder** — `POST /api/ai/cron`. Plain English → cron expression,
  then validated and previewed locally by a stdlib 5-field cron evaluator
  (handles `*/n`, ranges, lists, and the DOM/DOW OR quirk) returning the next
  5 run times.
- **Runbook suggestions** — `POST /api/devices/<id>/runbook`. RAG-aware: pulls
  matching runbook/doc context when the index is enabled.
- **CMDB doc drafts** — `POST /api/devices/<id>/doc-draft`. Generates a
  Markdown asset page from observed state for a human to review.

## Compliance reports

`GET /api/compliance?frameworks=pci,hipaa,soc2`. `compliance.py` maps each
control to data RemotePower already collects and resolves it to **pass / fail
/ N-A**, with evidence and remediation. The score is pass ÷ (pass + fail),
ignoring N-A — an honest "of what we can measure, how much passes." This is an
audit-prep aid, not a formal attestation.

## New events

`smart_failure` and `kernel_outdated` join the channel matrix under a new
**Hardware health** kind, routable to any channel like every other event.

## Compatibility

Additive only. New per-device data files (`hardware.json`, `speedtest.json`,
`discovery.json`, `metrics_history.json`, `helm.json`) are created lazily; old
agents simply don't populate the new payload keys and the new cards show
"no data yet". All probes and AI calls degrade to a clean empty/disabled
state when their tooling or provider is absent.
