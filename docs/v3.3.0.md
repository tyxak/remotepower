# RemotePower v3.3.0 — Release Notes

**Released:** 2026-05-27

## Overview

Audit follow-up release. Three Explore agents (bugs / performance /
security) swept the codebase; this release ships the agreed plan
(Phases A–D) end-to-end plus a hash-driven agent self-update.

No breaking changes. Defaults that previously prioritised
compatibility now prioritise safety where it doesn't cost operator UX
(loopback webhooks still work; same-username login still rate-limited).

## Highlights

### Hash-driven agent self-update

Agents now decide whether to update based on **binary sha256
comparison**, not version-string comparison. The previous logic
silently skipped same-version rebuilds (think: hotfix backported but
version intentionally left at the same string) and refused
downgrades. The new flow updates whenever the server's canonical
binary differs from the agent's, in either direction.

The server caches the canonical agent sha256 to a sidecar
`.sha256` file so CGI requests don't re-hash on every poll. Hash
validation of the downloaded bytes remains constant-time
(`hmac.compare_digest`).

### Home dashboard 7× → 1× round-trip

`loadHome()` previously fired seven parallel `/api/*` requests on
every 60s refresh. Under CGI each one is a fresh Python process. The
new `/api/home` endpoint returns everything the Home page renders in
one shot — slim devices, drift summary, CVE counts, fleet events,
mailwatch, links, attention digest, and the handful of config flags
the page reads. A 50-device fleet with 5 dashboard operators now
spawns ~5 CGI processes per minute for Home, not ~35.

### Per-IP rate limits on enrollment and login

`POST /api/enroll/register` is throttled at 10 attempts per minute
per source IP. The PIN namespace is 10⁶ with a 10-minute window;
without throttling, brute-forcing the whole space from one IP took
minutes.

`POST /api/login` is throttled at 20/min/IP, layered on the existing
per-username lockout. The username gate stops single-account
brute-force; the IP gate stops credential-stuffing across thousands
of usernames from the same source.

### SSRF default flipped

`webhook_block_local` now defaults to **True**. Webhook deliveries to
link-local targets (AWS/GCP/Azure metadata at 169.254.169.254) and
unspecified addresses (0.0.0.0) are blocked by default. Loopback
(127.0.0.1, ::1) remains allowed via a separate
`webhook_allow_loopback` flag (default True) so homelab Gotify and
ntfy sidecars keep working out of the box.

## What's new in detail

### Phase A — correctness

| Bug | Fix |
|---|---|
| `process_service_report` raced on `services.json` | Wrapped in `_LockedUpdate`; webhooks fire after release |
| `_record_service_transition` raced on `service_history.json` | Same fix |
| `/api/status` `'high load'` flag always fired when `os.cpu_count()` returned None | Reshaped ternary with explicit cpu-count guard |
| `handle_device_delete` cleanup used bare load/save on 12 files | All cleanup steps under `_LockedUpdate` |
| `_compute_attention` loaded `CONFIG_FILE` four times | Loaded once at top, reused |

### Phase B — performance

| Endpoint | Change |
|---|---|
| `GET /api/devices?slim=1` | Omits `sysinfo`, `listening_ports`, `brute_force_active`, SNMP metrics. ~1.5 MB → ~50 KB per response on a 50-device fleet |
| `GET /api/home` | One round-trip Home dashboard endpoint |
| `GET /api/devices/sysinfo?ids=a,b,c` | Batch sysinfo for Monitoring page (was 1 + N) |
| `GET /api/attention` | File-backed 10s cache; busts on underlying data change |
| `_detect_brute_force` | Short-circuits before file load when no lines match |

### Phase C — security

| Surface | Change |
|---|---|
| `/api/enroll/register` | 10/min/IP throttle |
| `/api/login` | 20/min/IP throttle, layered on existing per-username lockout |
| Webhook SSRF | `webhook_block_local=true` by default; new `webhook_allow_loopback` flag |
| Alert mutation | New `viewers_can_ack_alerts` flag (default True for back-compat); opt in to admin-only |
| Re-enrollment | Device token rotates on success; both `reenroll` and `reenroll_denied` audit-logged |

### Phase D — reliability + agent update

| Change | Impact |
|---|---|
| Hash-based agent self-update | Same-version rebuilds and operator re-pushes now reliably update |
| Server-side agent sha256 cache | One hash per binary push, not per agent poll |
| `_record_fleet_event` archive write outside flock | Heartbeat-driven event fires no longer serialise behind gzip append |
| `GET /api/devices?limit=N&offset=N` | Optional pagination for very large fleets |

## Configuration

New config keys (all optional):

```jsonc
{
  // SSRF defaults
  "webhook_block_local":    true,   // v3.3.0 default — flipped from false
  "webhook_allow_loopback": true,   // v3.3.0 new — keeps 127.0.0.1 sidecars working

  // Alert mutation gating
  "viewers_can_ack_alerts": true    // v3.3.0 new — set false for strict least-privilege
}
```

## Upgrading

1. Agents in the wild will pick up the new binary automatically on
   their next poll because the hash differs from the freshly-built
   v3.3.0 binary. No manual action required.
2. Existing webhook destinations on 127.0.0.1 / ::1 keep working
   (loopback allowed by default). Destinations resolving to
   169.254.169.254 will start being blocked — flip
   `webhook_block_local: false` if your deployment genuinely needs
   to reach link-local addresses (rare).
3. The Home dashboard now reads from `/api/home`; clients still
   hitting the legacy individual endpoints work unchanged.
