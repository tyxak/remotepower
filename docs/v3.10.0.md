# RemotePower v3.10.0 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.10.0 is a third bind-it-together and security sweep on top of v3.9.0. It
moves agent data that was collected but stuck at zero into the alerting path
and the UI, closes two real outbound-fetch / secret-disclosure gaps, and fixes
a couple of alert-label bugs. No new headline features — a correctness,
binding, and hardening release.

Hard-reload once after upgrading (service-worker cache
`remotepower-shell-v3.10.0`).

## Security

- **Container image-registry SSRF closed.** The image-update scanner
  (`server/cgi-bin/image_registry.py`) was the one outbound path that did not
  use the connect-time SSRF guard the rest of the server adopted in v3.8.0–
  v3.9.0. Three distinct holes:
  1. **Redirect following** — a malicious registry could answer the manifest
     request with `302 → http://169.254.169.254/…` and the bare `urlopen`
     would follow it.
  2. **DNS-rebinding TOCTOU** — the caller pre-flighted the manifest hostname,
     but `urlopen` re-resolved independently at connect time, so DNS could
     return a public IP for the check and an internal IP for the fetch.
  3. **Unchecked token-realm fetch** — on a `401`, the client parses the
     `Www-Authenticate: Bearer realm="…"` header (fully registry-controlled)
     and fetches that realm URL to mint a token, with **no** SSRF check at all.
     If registry credentials were configured, they were sent as Basic auth to
     that attacker-chosen realm — a credential-exfiltration path.

  The fix routes **every** fetch the client makes — the manifest *and* the
  token realm — through the shared SSRF-safe opener (the connected peer IP is
  re-validated, 3xx redirects are refused), pre-flights the realm URL against
  the same IP classifier as the manifest, and forces the realm to HTTPS.
  Image references come from container metadata reported by managed hosts, so a
  compromised/malicious host was the realistic adversary here.

- **`GET /api/config` secret-scrub backstop.** The endpoint is readable by
  `viewer` and read-only `mcp` keys (not just admins) and redacted secrets with
  a *denylist* — popping specific known keys (SMTP/LDAP/Proxmox/OIDC passwords,
  webhook tokens, the status token). It never popped `cfg['ai']` (which holds
  the AI-provider `api_key` in cleartext) or `cfg['registry_credentials']`, so
  both leaked to any authenticated viewer/MCP key. Beyond fixing those two, the
  endpoint now runs a **recursive secret scrub** as a structural backstop: any
  field whose name ends in a secret word (`password`, `secret`, `api_key`,
  `token`, `credentials`, …) is removed at any nesting depth before the
  response is built, while every `*_set` / `*_from_env` indicator and non-secret
  `*_id` field is preserved. Only `ai_configured` and `registry_credentials_set`
  booleans are surfaced. A newly-added config secret can no longer leak before
  someone remembers to redact it by hand.

- **TCP uptime monitor + Healthchecks.io ping hardened.** The `http` monitor
  branch already ran through the IP classifier and the connect-time peer
  recheck; the **`tcp`** branch did neither — it only stripped odd characters
  from the host and validated the port, then opened a socket. That made a TCP
  monitor a blind internal port scanner with an open/closed oracle, ignoring the
  `allow_internal_monitors` opt-in and reachable for cloud-metadata probing. The
  tcp branch now resolves and classifies the target exactly like the http
  branch and re-checks the connected peer (closing the rebinding window). The
  **Healthchecks.io** watchdog ping moved off a bare `urllib.urlopen` onto the
  SSRF-safe, no-redirect opener for consistency.

See `docs/security-review-3.10.0.md` for the full review.

## Bind it together

- **Container restart tracking now works fleet-wide.** The agent's docker/podman
  listing reported `restart_count`, `started_at` and `uptime_seconds` hardcoded
  to `0` (only the Kubernetes path filled them), which left the
  `container_restarting` alert permanently dead and the device drawer's
  container-age column blank on every non-Kubernetes host. The agent now fills
  all three from a single batched `docker inspect` per heartbeat (one call for
  the whole set, like the existing stats/digest samples), so the alert fires
  everywhere and age renders.

- **ClamAV last-scan time + per-interface MAC.** The agent now parses the
  clamscan SCAN SUMMARY block for the last-scan timestamp (it previously only
  reported signature-DB age), and the device drawer shows it on the ClamAV pill.
  Each network interface's MAC address — already collected and stored — now
  shows next to its IP in the drawer's network badges.

## Fixes

- **Config-drift alert titles.** `_alert_title` read a `files` field that neither
  drift event sends (`drift_detected` carries `path`; `config_drift` carries a
  `sections` list), so every alert in the inbox, audit log, and AI-investigate
  prompt read "Config drift on host: ? file(s)". Titles now name the file that
  changed (file-integrity drift) or the number of sections that drifted
  (host-config drift).

- **Devices table-view Hostname sort.** The table's Hostname column header was
  clickable and painted a sort arrow, but the column getter had no `hostname`
  key, so clicking it reordered nothing; a stale `enrolled` key with no header
  was the mirror image. Both are aligned now, so Hostname sorts.

## Upgrade notes

- No configuration changes are required.
- The agent change is backward-compatible; older servers ignore the now-populated
  fields, and the new `docker inspect` is best-effort (a failure degrades to the
  previous zeros rather than dropping the listing).
- Hard-reload the web UI once so the new service-worker cache
  (`remotepower-shell-v3.10.0`) takes effect.
