# RemotePower v3.0.5 — strict CSP + UX polish

> Released 2026-05-25, one day after v3.0.4. The Content Security Policy
> `'unsafe-inline'` finding is now closed, several user-visible bugs the
> migration exposed are fixed, and the deploy pipeline picks up
> per-file changes correctly across browser caches. Drop in the new
> tarball, copy the new `server/conf/remotepower.conf` (or just update
> the CSP directive in your existing one), reload nginx, and hard-reload
> the browser once for the service worker to pick up the new cache name.
> No schema changes, no migrations.

This is a security + UX release. The headline change is enforcement: the
browser now blocks injected inline scripts and styles, not just our
`escHtml` / `escAttr` discipline. The fallout — auto-converted CSS rules
breaking some JS reveal paths, a stale "Install app" button on Chrome —
is patched here.

## What changed

### Strict CSP — `'unsafe-inline'` removed

The L1 finding from the v2.3.2 security review is closed. Both nginx
configs now ship:

```
script-src 'self'; style-src 'self'; img-src 'self' data:;
worker-src 'self'; frame-ancestors 'none'; base-uri 'self';
form-action 'self';
```

Before, an injected inline `<script>` or `onclick=` payload would have
executed — a single missed escape was the only barrier. After, the
browser refuses them outright. The migration touched every inline event
handler, every inline `style=` attribute, every `<style>` block, and the
inline JS that bootstrapped the service worker. All are now in external
files using `data-action` event delegation.

The same audit caught a stray `javascript:` URI in `index.html` and a
runtime `setAttribute('onclick', …)` in the per-user host-config flow —
both would have silently failed under the new policy. Both fixed.

### Vendor libraries self-hosted

xterm.js 5.5.0 + addon-fit (web terminal), qrcode-generator 1.4.4 (TOTP
QR), Swagger UI 5.17.14 (`/swagger.html`), and the Inter / JetBrains
Mono web fonts now live under `/static/vendor/` rather than being
fetched from cdn.jsdelivr.net, cdnjs.cloudflare.com, and bunny.net.

Two reasons. First, the new CSP blocks every external origin — same
strictness that closes the inline-script hole also closes the
remote-script hole. Second, those CDN fetches were *already* blocked
by the previous CSP (`script-src 'self' 'unsafe-inline'` doesn't allow
cross-origin), so the web terminal and TOTP setup had been silently
broken for any operator who tried to use them. Worth verifying after
upgrade.

The fonts directory ships 92 woff/woff2 subset files (~1.4 MB on disk)
but browsers only fetch the latin subset they actually render — typical
page load is ~30 KB of fonts.

### AI Assistant page no longer renders blank

When the CSP migration ran the auto-class generator over inline
`style="display:none"`, the resulting CSS rules had the same display
value as the class. JS code that reveals these elements via
`element.style.display = ''` was now a no-op: clearing the inline
property let the class rule keep the element hidden. Same pattern as
the v3.0.3 `#pwa-install-btn` fix, now applied to the eight more sites
the migration exposed (AI page status + chat panel, Services
maintenance badge, Mitigation modal AI fix panel). All reveals now set
explicit display values.

### Install button no longer stuck visible on Chrome

Two layered causes. A leftover `#pwa-install-btn { display: flex; }`
rule from the v2.8.1 anti-FOUC mechanism (the inline `<style>` it was
paired with had been removed in this release's CSP migration, but the
override rule stayed and outranked the class-based initial-hide via
ID specificity) — removed. And Chrome keeps firing
`beforeinstallprompt` in regular browser tabs on profiles where the
PWA is already installed, with no web-page-readable signal for "I'm
installed" — a `localStorage` flag now persists the installed state
the first time the PWA opens in standalone mode (which the OS always
does at least once after install), and subsequent regular-tab loads on
that profile suppress the install button.

### Mobile sidebar — ✕ and Collapse buttons removed

The drawer dismisses via tap-outside (the scrim handler in `app.js`).
The ✕ button was redundant with that path, the Collapse button never
visually worked at mobile width (its CSS was gated on `min-width:
769px`), and operators reported the dead Collapse button as
confusing. Both gone at ≤720px.

When running as an installed PWA above the mobile breakpoint
(`@media (display-mode: standalone) and (min-width: 721px)`) the
docked-sidebar collapse-rail rules now apply at every width — the
previous gap at 721–768px left the Collapse button visible but inert.

### Login + Change Password + Add User flows now use real `<form>` elements

So browser password managers can reliably extract / autofill
credentials. `[DOM] Password field is not contained in a form`
warnings disappear for those three flows. The remaining 13 warnings
on admin-config secrets (SMTP / LDAP / Proxmox tokens, CMDB vault
keys) are intentional — password managers shouldn't be autofilling
service-account credentials.

### Per-file content-hash cache-busting

`deploy-server.sh` now rewrites the `?v=...` query strings on
`<script>` and `<link>` references in the deployed `index.html` to a
12-character SHA-256 prefix of each file's content. Previously the
version marker only changed when `SERVER_VERSION` did, so between-
release JS or CSS fixes silently shipped to browsers running the
cached old version. With this in place, every deploy invalidates
exactly the files that changed, and an operator never needs to
manually clear browser cache after a fix. The source-tree
`index.html` keeps `?v=<SERVER_VERSION>` so the regression tests pass;
the rewrite happens at install time.

### Service Worker registration hardened

The `InvalidStateError` seen on Chrome and Brave after BFCache
restore or partitioned-storage hiccups now has a real recovery path:
4 retries on the error with linear backoff, then a last-resort
unregister-all-and-retry from a clean slate (capped at one attempt
per document so it can't loop). Also fixed the duplicate-fire on
initial load (`pageshow` event and the synchronous bootstrap both
fire — now gated by a `_kicked` flag) and the duplicate-warn that
followed.

### `docs/Manual.html` is actually deployed now

The dashboard's `<a href="/Manual.html">` link silently 404'd in
production. The file lived in `docs/` in the repo, but neither the
Dockerfile nor `deploy-server.sh` nor `install-server.sh` copied it
to the web root. All three deployment paths now do.

## Upgrade notes

1. `git pull` on the repo holding your checkout.
2. `sudo bash deploy-server.sh`. The deploy script handles the
   `?v=...` cache-bust rewrite as part of its index.html publish step.
3. **One-time nginx config update.** The CSP header changed.
   Either:
   - Copy `server/conf/remotepower.conf` to
     `/etc/nginx/sites-available/remotepower` (overwrites your TLS /
     rate-limiting customisations — don't do this if you've diverged),
     **or**
   - Edit `/etc/nginx/sites-available/remotepower` by hand and replace
     the `add_header Content-Security-Policy` line with:
     ```
     add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; connect-src 'self'; worker-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'self';" always;
     ```
   - Then `sudo nginx -t && sudo systemctl reload nginx`.
4. Hard-reload the browser once (Ctrl+Shift+R) — the SW cache name
   bumped to `remotepower-shell-v3.0.5`, the activate handler will
   evict the old cache automatically.

## Regression coverage

`tests/test_v232.py` now scans every shipped HTML file (`index.html`,
`swagger.html`, `Manual.html`) for inline scripts, inline styles,
inline event handlers, `style=` attributes, `javascript:` URIs, and
any auto-loaded external resource. A vendor-libs-present check ensures
the bundled copies stay in tree. The test suite is at ~1,545 tests, all
passing on the released tree.
