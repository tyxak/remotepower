# v6.1.2 — "AfterglowMatters"

A correctness-and-fit release. Three reported bugs — two of which meant a shipped
feature simply did not work — a sweep of the frontend modules, the first wave of
the performance programme, and the ability to switch off the enterprise modules a
small homelab never opens.

No breaking API changes. No configuration change is required: every new switch
defaults to today's behaviour.

## Reported bugs

### Muting an alert now lifts the health score

A per-(host, event) mute is documented as silencing the Alerts inbox **and**
Needs-Attention. In practice only the alert-firing path consulted the mute list —
the Needs-Attention digest never did. Since fleet health is derived purely from
Needs-Attention signals, a muted alert kept its item and went on depressing the
host's (and therefore the fleet's) score indefinitely. That is now fixed at the
digest, so the inbox, Needs-Attention and health all agree.

Mutes are matched by **event and severity**, not by the item's UI grouping, so
silencing `av_warning` does not also hide a critical `av_infected` on the same
host. History and the Tuning page are unaffected — they read the event log, which
still records everything.

### Container image CVEs (trivy) can actually be scanned

The agent has always honoured a `force_image_scan` flag in the heartbeat
response. Nothing on the server ever set one. That left the agent's own cadence
gate — `poll_count % 1440` — as the only path to a scan, and `poll_count` is
process-local: it resets to zero on every agent restart (self-update, reboot,
unit restart). Any host whose agent restarted more often than once a day never
reached the modulo and never scanned, so the page stayed empty no matter how much
trivy was installed.

Three changes make the feature reachable:

- the agent's cadence is now a **persisted wall-clock interval** that survives
  restarts, so "every ~24h" means what it says;
- a **Scan now** button (`POST /api/image-cves/scan`, admin) sets the one-shot
  flag on every host that runs containers — trivy then runs on the next heartbeat
  and reports with the one after;
- the empty state distinguishes *off*, *nothing reported yet* and *reported and
  clean*, which previously all rendered as the same dead end.

### `POST /api/cve/scan` no longer 500s on PostgreSQL

The handler writes its status marker with a **non-blocking** lock so a wedged
writer can never hang the request. Postgres, however, was the only backend whose
non-blocking acquire had no retry budget at all: the JSON backend retries for
100 ms (20 × 5 ms) and SQLite sets `busy_timeout=100`, but `pg_try_advisory_xact_lock`
gives up on the first contended attempt. The detached CVE scan runner writes that
marker continuously, so a perfectly ordinary collision raised `LockBusy` — and
nothing caught it.

- The three backends now agree on the retry budget.
- `handle_cve_scan` handles contention explicitly (a retryable 409).
- An uncaught `LockBusy` anywhere now renders as **503 + `Retry-After`** instead
  of an opaque 500. It is a transient condition, not a server fault.

## Frontend fixes

A full review of the ~38 JS modules surfaced these:

| Fix | Why it mattered |
|---|---|
| Billing rate-card / recurring-fee deletes | Rows were addressed by their render-time index and the table was never re-rendered after a delete, so removing any non-last row made every surviving button point one row too far — the next delete removed the **wrong** row, and Save persisted it. Rows now carry a stable id. |
| Backups "Restore drill" column | Sortable in appearance only — the header had no matching sort key, so it showed an active arrow and never reordered. |
| OPNsense config save | Repainted the drawer card instead of the console you were looking at (and nothing at all if the drawer was closed). Now uses the same active-surface helper every other OPNsense action does. |
| Dead runbook renderer | Unreachable code targeting an element no code has created for several releases, with comments claiming otherwise. Removed; the live runbook path is untouched. |
| TLSA/DANE fields, containers loading row, dead ternary | Escaping and markup correctness. |

## Performance

The first wave of the performance programme.

| Change | Effect |
|---|---|
| Needs-Attention cache | It invalidated whenever `devices.json` changed — which every heartbeat does — so on any fleet whose hosts beat faster than its 10 s TTL the cache **never hit**, and the O(fleet) digest recomputed on every poll and for every health/risk consumer. It now honours its TTL and busts on the inputs an operator actually changes. |
| Six heartbeat sweeps | Scheduled backup, disk watchdog, OTLP push, scheduled scans, netscan schedules and audit-HMAC rotation were hard-wired into the heartbeat handler, bypassing the scheduler gate — they ran on every beat from every agent even when a dedicated out-of-band scheduler was already doing that work. Now ordinary gated sweeps. |
| OTLP hook | No longer deep-copies the entire config on every heartbeat to read one boolean that is off by default. |
| Monitor sweep | Reads and writes only the monitors that ran, instead of reconstructing and rewriting every configured monitor's history. |
| PostgreSQL connections | No longer re-run the full schema DDL (~9 catalog round-trips) on every new connection. |

## Optional modules

RemotePower has grown a lot of surface that a small homelab simply does not want.
**Settings → Advanced → Optional modules** now lets you switch off:

| Module | Default | Off means |
|---|---|---|
| Alerts | on | No alert-inbox rows are recorded. **Webhook and email notifications still fire**, and the event history is untouched — this is the setup for operators who page themselves through ntfy or Discord and don't want a queue to curate. |
| Tickets | on | The built-in helpdesk (incidents, requests, changes, SLAs, email ingestion) disappears. |
| Billing | off | Worksheet, invoices, rates & fees. Logging hours and the Timesheet stay available regardless. |
| Knowledge base | off | Operator-authored SOPs and how-tos. |
| Compliance | on | CIS/PCI/HIPAA/SOC 2 scoring and OpenSCAP scans. |
| Pentest | on | The authorized vulnerability-scan orchestrator. |

A disabled module is **actually** off: its whole API prefix 404s at the request
dispatcher, so the UI is not the enforcement boundary. Nothing is deleted —
switch it back on and your data is exactly where you left it. The defaults above
reproduce current behaviour, so an install that never touches this page is
unchanged.

(v6.0.0 had declared tickets/billing/KB "standard, always-on" and removed their
toggles. This restores them — as opt-*outs*, not a return to the pre-v6 opt-in.)
