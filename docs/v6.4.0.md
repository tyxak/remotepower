# v6.4.0 — "Sh1eldMatters"

The security-and-protection release. Integrity-guard and baseline checks that
behave exactly as an operator expects — disable one and it stops evaluating on
the host, not just in the UI; accept a legitimate change with one click. A
full, public security review and pentest pass backs it (nothing Critical, High
or Medium ships). And agentic diagnosis, used defensively: when a host is
misbehaving and nobody knows where to look, RemotePower does what a good
on-call engineer does — sweep the recent logs, follow the evidence, and write
down a verdict, with hard budgets, secret redaction and a visible evidence
trail at every step.

Two features, designed to compose:

## Hail-mary log sweep — "Diagnose from logs"

A new device-drawer action for the *"something is wrong but I don't know
what"* moment.

**Collect.** The server sets a one-shot `force_log_sweep` flag (the same
delivery contract as `force_package_scan`); on its next heartbeat the agent
sweeps `/var/log/*` and `/var/log/*/*`:

- only files **modified in the last 24 h** — the sweep is about what's
  breaking *now*;
- compressed (`.gz`/`.xz`/…), rotated (`.1`), and binary files
  (wtmp/btmp/journal/ASL) are skipped;
- at most **12 KB tail per file, 40 files, 256 KB total**;
- files are ranked by **error density + recency**, so the budget goes to the
  log that is actually screaming;
- the same path deny-list as log watching applies (symlink-resolved;
  shadow/sudoers/SSH keys can never qualify).

macOS sweeps `/var/log` the same way. Windows has no `/var/log`, so its sweep
is the last 24 h of Error/Warning events from the System and Application
event logs, shaped as pseudo-files (`eventlog:System`).

**Redact, then store.** At ingest the server re-caps every size and scrubs
credential material line-by-line — `password=…`, `Authorization: Bearer …`,
`api_key: …`, `https://user:pass@host` all become `[REDACTED]` before the
sweep is stored. One sweep is kept per device (`log_sweep.json`).

**Diagnose.** The **Diagnose** button hands a ≤48 KB excerpt (highest-scored
files first) to your configured AI provider under the new `log_sweep_rca`
prompt and renders the result: a verdict, per-file findings quoting the key
evidence lines, and conservative next steps. Like the mitigate flow, this is
an **explicit-action consent model** — log content goes to the AI provider
only when you click, and the modal says exactly that.

Endpoints: `POST /api/devices/<id>/log-sweep/run` (admin),
`GET /api/devices/<id>/log-sweep`,
`POST /api/devices/<id>/log-sweep/diagnose` (write role). All device-scope
and tenant gated.

## Agentic alert triage

`POST /api/alerts/<id>/ai-triage` — the **Triage** button in the alert inbox —
runs a bounded, MXDR-style investigate loop for one alert. Instead of
answering from a static context blob, the model *investigates*: it is offered
a fixed menu of read-only, device-scoped evidence tools —

- device summary (load, memory, disk, failed units, last-seen),
- journal tail,
- watched service states,
- the host's **other open alerts** (correlation),
- recent command output,
- a regex search over the 6 h log buffer,
- the latest hail-mary log sweep, if one exists —

and must reply with strict JSON each round: **one tool call or a final
verdict**. At most 4 tool calls; every tool result is clipped and
secret-redacted; the model **never executes anything** — acting on a verdict
stays on the existing approval-gated paths (mitigate playbooks, the governed
AI executor).

The verdict — root cause, confidence, evidence lines, recommended action —
plus the list of tools the model actually used is stored on the alert and
shown in the inbox: an **"AI verdict" badge** on the row reopens it any time.
Cross-tenant and out-of-scope alert ids 404.

Both new prompt keys (`log_sweep_rca`, `alert_triage`) are tunable under
Settings → AI Assistant like every other prompt.

## Automatic alert triage (opt-in)

Settings → AI Assistant → **Automatic alert triage** turns the Triage button
into a background habit: a cadence sweep picks one new, open, untriaged alert
per tick (severity floor configurable, default *High and up*), runs the same
bounded investigate loop, and stores the verdict as `by: auto` — alerts land
in the inbox already carrying an **AI verdict** badge. Hard limits: one alert
per tick, a configurable **daily run cap** (default 20; each run is a few
provider calls), and the loop's own 4-tool budget. **Off by default.** With
the out-of-band scheduler active the runs happen fully off the request path.

## The feedback loop

Every verdict shows **Was this verdict helpful?** (👍/👎). Ratings are stored
on the alert and aggregated in `GET /api/ai/stats` (triaged / auto / up /
down) — a quick read on whether auto-triage is earning its keep, and the
strongest tuning signal when it isn't.

## From verdict to fix — the governed bridge

When the **AI executor** module is enabled, a verdict with a recommended
action offers **Propose fix (governed executor)**: the verdict becomes the
proposal context, the executor may only pick from remediation actions a
human already wrote and saved (never author a command), and the pick parks
in **Confirmations** for a human to approve. Tickets opened from a triaged
alert carry the full verdict in their seeded note.

## Evidence menu (9 tools)

Wave 2 adds `cves` (open CVE findings, top by severity) and `metrics_trend`
(recent daily disk/metric samples) to the triage menu. And the `log_sweep`
tool is now self-provisioning: a missing or >30-minute-old sweep makes the
server set the one-shot `force_log_sweep` flag itself, so the evidence is
ready for the next run.

## SLA / SLO objects for remote probes

Named availability targets you attach to probes. Define an SLA/SLO object on
the Monitoring page — a name, a target percentage (say 99.9), a rolling window
in days — then tick it on each remote probe whose checks should count toward
it (any probe type, including multi-step HTTP flows; a tag/group probe's
fanned-out per-device checks all count). The new **SLA / SLO objects** panel
shows each object's check-weighted availability across its attached probes,
the error budget remaining and the burn rate over the object's own window —
filterable by name/probe and by compliance (breached / meeting / no data),
sortable on every column. An object with no measured checks reads **no data**,
never a fake breach; deleting one detaches its probes and keeps their history.

`GET /api/slo` gains an `objects` list, and the same numbers export as
Prometheus gauges (`remotepower_slo_object_target_percent`,
`…_availability_percent`, `…_budget_remaining_percent`) for Grafana SLO
dashboards. See [monitors.md](monitors.md).

Two adjacent probe bugs fixed while wiring the editor: a **paused** monitor
produced no row at all in `GET /api/monitor` (pausing made it vanish from the
Remote Checks table, stranding the v6.1.2 Resume button), and the row
Edit/Clone/Delete actions resolved the config entry by table position — which
desyncs whenever a paused, satellite or fan-out row is present. Both resolve
correctly now, and editing a paused monitor no longer silently resumes it.

## WordPress connector — last logins with IP + geo

The 44th integration connector watches a WordPress site: REST reachability and
site identity out of the box; add a username + **Application password** (Users
→ Profile → Application Passwords) and it verifies the credentials and — via
the free **Simple History** plugin's REST API — shows the **last 5 successful
logins** (timestamp, account, source IP) in a *recent logins* panel on the
Integrations page. Configure a GeoIP MMDB (Settings → Security) and each
login IP is annotated with its country. Without the plugin the connector
still monitors the site; rejected credentials read *warning*, not a fake
outage. Login rows are strictly sanitized server-side (IP-shape validation)
and rendered inert.

## Client-side JS errors — collected since v5.4.1, finally visible

The `window.onerror` beacon has quietly collected browser-side errors into a
capped ring since v5.4.1 — with no UI ever showing them. The **Server-status
page** now renders the ring (admin-only; newest first) with a **Clear**
action, so a frontend failure that dies in some operator's console becomes a
row an admin actually sees. After a fix ships, the list staying empty is the
verification.

## Auto-patch "Reboot if required" is finally conditional

The policy checkbox has said *if required* since it shipped — but every path
rebooted after every clean run. Now the reboot fires only on a real need
signal: Linux checks the Debian/Ubuntu `reboot-required` marker,
`needs-restarting -r` (dnf/yum), or a newer installed kernel than the running
one; Windows consults its pending-reboot registry state via a new
`reboot-if-required` agent verb; macOS honestly never reboots for a brew
upgrade. A patch window that changed nothing keeps the host up, with "reboot
SKIPPED" in the log. Applies to policies, rollout rings and scheduled jobs
alike; the explicit Reboot action stays unconditional.

## Every alert auto-resolves — the gap list is empty

The last five stateful alerts got real recovery observers: `cve_cleared`,
`patch_ok`, `tls_renewed` (matched host+port), `ecc_stable` (24 quiet hours
after the last counter rise) and `secret_cleared`. Every alert that records
an ongoing condition now closes itself when the condition clears; the
point-in-time ones say **confirm** on the row so you know nothing will
self-heal.

## Alerts tell you more — and where to act

Every whitelisted fact an alert carries now renders as a compact key:value
detail line in the inbox. Protect/baseline check failures additionally say
exactly **where to accept a legitimate change** — Monitoring → Checks → the
host's row → *Accept change* — in both the notification text and the inbox
row (click-through included). And the AV/rootkit mitigation AI is pinned to
the tool that raised the alert: no more `rkhunter --update` suggested for a
ClamAV signature problem.

## Every button gives feedback

Any button whose handler does async work shows a working state — dim, a
small spinner, `aria-busy` — while it runs, and a brief accent pulse when it
settles. Both dispatcher paths are covered, so no control is silently inert
while something happens under the hood.

## Five cross-reference gates + an error sweep

The test suite gained five structural gates that turn "fails silently"
classes into build failures: UI wiring parity (every `data-action`, element
id, panel constant and client `api()` call must resolve against its
counterpart), a JS undefined-global gate (eslint `no-undef` over the
load-order bundle — the client-side `ruff F821`), AST error-path guards
(success-`respond()` swallowed by `except Exception`, `Path.exists()` on
storage keys, unreviewed bare-auth mutations), CSS class parity, and
response-shape contracts for the hottest endpoints. Their first full run —
the v6.4.0 error sweep — found and fixed real shipped bugs, including a
netmap device-click that always fell through to the CMDB view, a
posture-digest test send that returned 500 on success, and two host-config
textareas that had never received their styles.

## Server-status: syslog-receiver watcher

The Distributed-subsystems card on the Server-status page now shows the
v6.3.0 agentless syslog receiver: enrolled sources, last intake, and a
best-effort local unit probe. The row is **informational only** — never a
warning or a health input, because the receiver is optional and may run on a
different host. If you *want* alerting on it, apply the new opt-in
baseline-checks catalog row **`rp_syslogd_running`** (tag your server host
`rp-server`) for a critical Checks row when the unit stops.

## Guarded, verified auto-remediation

Wave 3 closes the fix loop. Automation rules could already run a saved script
on an event; now that channel is safe to trust with real fleets:

- **Per-host cooldown** and a **max-hosts-per-hour blast cap** bound every
  rule's blast radius (a flapping alert can't become a restart loop; an event
  storm can't run a script fleet-wide).
- **Verification**: set *Verify within N seconds* on a rule and the verify
  sweep checks that the triggering alert actually cleared after the fix ran.
  Still open → a **`remediation_failed`** alert fires (rule, host and script
  named), and **N consecutive failures auto-disable the rule**.
- Every firing — including guard suppressions — lands in the **Recent
  auto-remediations** ledger on the Automations page.
- A `run_script` action never fires on `remediation_failed` itself; escalate
  with notify/ticket actions instead.

See [automations.md](automations.md) for the full knob table.

## Signed command channel (opt-in)

The trust-model upgrade: the server detach-signs **every dispatched command**
with the release signing key, binding it to the target device and an issue
timestamp. Pin `release.pub` on a host and touch
`/etc/remotepower/require-signed-commands` (an operator-owned file the server
can never clear, like `audit-mode`) and that agent refuses anything unsigned,
tampered, aimed at another device, or older than 15 minutes — reporting the
refusal as command output so it's visible, never silently dropped. All three
agents enforce identically.

Database or command-queue tampering, and replay of captured commands, can no
longer execute anything on a pinned fleet — an attacker needs the signing
key, not just DB write access. See
[agent-commands.md](agent-commands.md) for setup and the honest boundary
(a full app-server compromise still signs in the convenient mode).

## Compliance you can back up

The Compliance page gains the discipline that a posture surface actually needs:
a control never reads **Pass** off an empty offenders list when the capable
source never ran. Patch/CVE/EOL/reboot/privileged-access controls gate on a
coverage fact and return **Not assessed** — disclosed, never a silent green —
when no host reported the signal. The framework score ignores Not-assessed, so
"unmeasured" can't inflate it. Two frameworks join PCI/HIPAA/SOC 2: **ACSC
Essential Eight** (process controls it can't observe are honestly Not assessed)
and **SMB1001:2026**. See [compliance.md](compliance.md).

## ATT&CK on triage verdicts

Agentic triage verdicts can now carry **proof-labelled MITRE ATT&CK
techniques** — observed / inferred / theoretical, validated server-side so the
model can't invent ids or overstate proof — shown as linked chips only when the
evidence genuinely points at adversary behaviour.

## Metrics incident-zoom band

A **5-minute metric roll-up tier** (kept ~8 days) fills the resolution gap
between the raw 1-minute window (only 24 h) and the hourly tier (30 days) —
the band you reach for when investigating an incident a few days old, where
hourly averages smooth away the spike and the raw window has rolled off. New
"7 days (5-min avg)" option on the Trends range selector and
`?tier=fivemin` on the roll-up endpoint. The agentic-triage `metrics_trend`
evidence tool now reads this roll-up series (CPU/mem/swap/disk min/avg/max),
not the daily disk-mount store — so triage sees the shape of the event.

## Network flow visibility (agentless)

`remotepower-flowd` is the flow sibling of the syslog receiver — the fleet's
**traffic** view. Routers, firewalls and L3 switches can't run an agent but
almost all export NetFlow/IPFIX; this UDP sidecar (udp/2055) maps each
exporter's source IP to its enrolled device, aggregates the flows (top talkers,
top conversations, protocol breakdown), and forwards a compact rollup to
`POST /api/flow/in/<token>` (a new `kind=flow` inbound token). It parses
**NetFlow v5, v9, IPFIX and sFlow v5** (template-caching for v9/IPFIX;
sampled-header dissection for sFlow). The
device drawer's **Network flows** view shows the latest window; the server
re-caps every rollup on ingest. See [flow.md](flow.md).

## Service-dependency link verification

The flow receiver's payoff. RemotePower already suggests `depends_on` edges from
observed traffic; now it **verifies** each declared edge. When a dependency link
that was carrying observed traffic (agent peer-connections or the NetFlow/IPFIX
receiver) goes silent while **both** hosts stay online, a **`dependency_missing`**
alert fires — the silent firewall/route/service break that the device-offline
alert can never catch — and clears automatically (`dependency_restored`) when
traffic returns.

Safe by construction: **off by default** (Settings → Alerts), edge-triggered only
on links seen live first (so a fleet with no flow/peer visibility never
false-alarms — those edges read *Unverified*), and both-online guarded (a link to
a host that is simply down reads *Silent*, not *Broken*, and doesn't duplicate the
offline alert). New read-only `GET /api/dependency-health` and a **Dependency link
health** card on the Network page. See [flow.md](flow.md).

## Cross-fleet incident outcome memory

The compounding half of agentic triage: RemotePower watches the whole fleet, so
every resolved+triaged incident becomes memory the next triage can learn from. A
cadence sweep harvests resolved alerts carrying an AI verdict into a durable,
tenant-tagged outcome store (it outlives the alert), and the investigate loop
gets a new **`prior_incidents`** evidence tool that surfaces the most similar
priors *in the same tenant* — the verdict and what actually cleared them, ranked
by operator-confirmed-helpful then recency. Read via `GET /api/ai/incident-memory`
(tenant-scoped) with a count in `GET /api/ai/stats`. Retrieval never crosses
tenants; harvesting is pure bookkeeping (no extra AI calls) and runs off the
request path.

## Detection self-test — "silence isn't clearance"

A green dashboard proves the monitoring found nothing, not that it *would*. The
Server-status page gains a **Detection self-test** card (`GET /api/detection-selftest`,
admin) that verifies the detection→routing→delivery chain is intact for every
alertable event type and surfaces the silent gaps that live only in your live
config: an alert kind muted on every actionable channel (distinguished from the
intentionally-quiet defaults), `notifications_test_mode` left on, a webhook-routed
kind with no destination, or a recover event that can never close its alert. It's
read-only and fires nothing (the Settings "send a test" button covers active
delivery). New bound module `server/cgi-bin/detection_selftest_handlers.py`.

## Reset baseline actually works — and everything that recovers resolves

Integrity-guard and baseline checks now behave the way an operator expects. The
root cause of "Reset baseline does nothing": guard directives were written to
the whole-devices collection instead of the device row, so no guard action had
ever reached an agent. Fixed end to end — and acceptance is now
server-authoritative, so accepting a legitimate change suppresses the check
**instantly** (surviving a refresh, not waiting for the agent round-trip) and
re-fires only on a genuinely new change. Accepting, disabling or deleting a
check now **resolves its open alert**, and the agent re-baselines on-host.
Disable now disables *everywhere* — the heartbeat push and the alert ingest are
both gated, so a disabled check stops evaluating (and stops quarantining) on the
host, not just in the UI. `file_hash`/`file_present`/`file_absent`/`job_fresh`
accept several `|`-separated paths (and globs) and match whichever exists — a
wp-config under any site root, ClamAV's `daily.cvd` vs `daily.cld`, the apt
stamp vs the package-list mtime — with a one-time migration repairing
already-applied checks. Failure text is now actionable, the baseline catalog is
text-filterable and un-applyable per scope from its card, and a Reset-baseline
button sits right on the Checks row. Full guide: [integrity-guard.md](integrity-guard.md).

## Every status that returns to good closes its alert

A lifecycle audit of all **100 alertable events**: 63 auto-heal, 29 are
point-in-time (an operator confirms them away), and the remaining 8 are listed
as bounded, visible gaps (`_AUTOHEAL_GAPS`) rather than silently lingering.
Deleting a device, custom check or script now resolves the associated open
alerts (they previously lingered forever); a check that was already failing at
upgrade time now alerts exactly once instead of staying permanently silent; and
undo/redo now genuinely covers the mute/ignore/toggle/ack/resolve/delete
families with a visible **Undo** toast. Guardrail: `tests/test_alert_autoheal.py`.

## "Clear this log line" now actually clears

Clearing a matched log line stores the line's *signature* (timestamps, pids,
ids, addresses folded out) and drops future matches. Three field-reported bugs
stopped it working on real logs — all fixed: the dashboard button escaped `"`
into the literal text `\x22` so a dockerd `msg="…"` line's signature never
matched (the server now un-escapes before signing); the clearable sample was
truncated to 200 chars while the rule matches the full line; and host/domain
noise didn't fold, so a postfix `dnsblog` line was a different signature per
blocklist. Signatures now also fold 3+-label hostnames/FQDNs to `<host>`
(sparing 2-label unit names like `docker.service` and filenames like `app.js`),
so clearing one blocklist clears the whole class. Cleared lines are surfaced
under **Settings → Ignored items** as well as Logs → Cleared lines. Guide:
[log-watch.md](log-watch.md).

## Ignored items stop piling up

A permanent × ignore used to live forever — even after its condition cleared,
its device was removed, or its point-in-time event aged out — so on a large
fleet the ignore list grew without bound and could hide a real recurrence. Five
counter-measures: an hourly sweep **garbage-collects** any Needs-Attention
ignore whose condition has been gone past a grace window; deleting or
decommissioning a device **drops its ignores**; a recheck sweep **auto-resolves**
open `patch_alert` / `cve_found` alerts when current state is clear (so fewer
things need ignoring at all); **class-level suppression rules**
(`GET/POST /api/na-suppress`) silence a whole kind across the fleet / a group / a
tag / a device as one entry instead of hundreds; and each ignore shows **when it
was last active** with a bulk **Restore all** per category. See
[alerts.md](alerts.md).
