# v2.4.2 release notes

Released: 2026-05-17.

A small features release: a per-user default SSH username, a quick
SSH link on the Devices page, and expanded in-app documentation.

## Added

### Default SSH username (per-user setting)

Settings → Security → SSH preferences. Set your SSH login name
once; it's stored per-user (in your ui_prefs, not shared) and
reused by the quick SSH link below. The value is validated as an
SSH-safe username — letters, digits, dot, dash, underscore, max 32.

### Quick SSH link on the Devices page

Each device row now shows a small SSH icon next to its hostname.
Clicking it:

- builds an `ssh://<user>@<host>` link and hands it to the OS, and
- copies the plain `ssh <user>@<host>` command to your clipboard.

The host is the device's IP when known, falling back to its
hostname. The username is your default from SSH preferences (if
you haven't set one, the icon tells you to).

**Honest note on the ssh:// link:** a web app cannot open a
terminal itself — it can only hand an `ssh://` URL to the
operating system. Whether that opens anything depends on your
machine having an `ssh://` handler registered (the OS, PuTTY, a
terminal emulator). That's why the action also copies the
`ssh user@host` command — the copy works everywhere regardless of
handler setup. Think of the link as a convenience for set-up
machines and the copy as the reliable path.

### Documentation — Proxmox, snapshots, quick SSH

Four new cards on the Documentation page:

- **Proxmox virtualization** — connecting a node, the API token,
  the Virtualization page.
- **Proxmox LXC containers** — where LXC guests appear.
- **Snapshots & rollback** — the create / rollback / delete
  workflow, including why rollback asks you to type the guest
  name and what "disk-only" means.
- **Quick SSH from the Devices page** — the setting and the link.

valid `default_ssh_username`, rejects bad ones without discarding

**Total: 982 tests, all passing.**

## Upgrading from 2.4.1

Drop-in, server-side. No data migration — the SSH username is a
new optional key in each user's existing ui_prefs document.

## Honest caveats

1. **The ssh:// link depends on the client machine.** Covered
   above — it is not a RemotePower limitation that can be fixed
   server-side; a browser genuinely cannot launch a terminal. The
   clipboard copy is the universal fallback.

2. **Browser-untested.** 982 unit tests pass and the assets are
   verified by test, but the SSH icon, the settings field, and the
   doc cards were not loaded in a real browser.

3. **No SSH key handling.** This is username + host only. It does
   not manage SSH keys, known_hosts, jump hosts, or ports — it
   builds a basic `ssh user@host`. Anything beyond that is your
   SSH client's own configuration.
