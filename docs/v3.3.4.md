# RemotePower v3.3.4 — Release Notes

**Status:** unreleased (dev)

## Overview

Adds **container image-update detection**: RemotePower now tells you when
a running container is behind the registry's current image for its tag.
Notify-only — it flags staleness, you decide when to pull. No schema
changes, no breaking changes.

## How it works

1. **Agent** reports each container's pulled image digest
   (`docker images --digests` / podman equivalent) as `repo_digest`,
   alongside the existing container fields.
2. **Server** resolves the registry's *current* manifest digest for each
   unique `repo:tag` (registry v2 API + token auth), **deduped across the
   fleet** so the same `lscr.io/linuxserver/*` image on five hosts costs
   one registry call. A container is stale when its pulled digest differs
   from the registry's current one.
3. The check runs as a bounded background sweep (default every 12h, capped
   per run with a wall-time budget and a short per-request timeout) plus an
   on-demand **Scan now** button. Multi-arch tags compare on the
   manifest-list (index) digest, which is what Docker stores locally — so
   the comparison is apples-to-apples.

Supported registries: Docker Hub, GHCR, lscr.io, Quay, and any generic
v2 registry. Docker Hub throttles anonymous manifest lookups (~100/6h per
IP); central dedup + the long sweep interval keep well under that, and a
registry credential (a free Docker Hub login) raises the ceiling.

## Where you see it

- **Containers page → Image updates** table: Image · Tag · Hosts · Status
  (Up to date / Update available / Unknown) · Registry · Last checked,
  sortable, with a summary line and a **Scan now** button.
- **Alert inbox + webhooks**: a stale image fires a low-severity
  `image_update_available` event, debounced on the digest last alerted
  (no re-spam across sweeps) and re-fired when upstream publishes a newer
  digest. Once every host has caught up, `image_updated` auto-resolves the
  open alert. Routable independently via the channel matrix
  ("Container image updates").

## Notes

- **Notify-only.** RemotePower never pulls or restarts a container — that
  stays your call (or your compose/watchtower workflow's).
- **Default on.** Set `image_updates_enabled=false` to stop the server
  reaching out to registries.
- Locally-built images (no registry digest) and digest-pinned refs are
  skipped rather than guessed — they show as "Not checked".
- SSRF-guarded: a crafted image reference can't point the server at a
  link-local / metadata address (reuses the webhook SSRF check).
- As with every cache-first PWA change, the bump to v3.3.4 (`?v=` +
  `CACHE_NAME`) is what delivers the new UI to installed clients.

## Compose stacks (upload + deploy)

The Containers page also gains a **Compose stacks** section: upload or
paste a `docker-compose.yml` as a named stack targeted at a device, then
**up / down / redeploy** it. This is the first feature that lets
RemotePower *create / build / deploy* containers rather than just observe
and restart them.

How it works:

1. Create a stack (admin) — name, target device, and the compose YAML.
   Stored server-side in `compose_stacks.json`.
2. Deploy queues a short `compose_deploy:<action>:<stack_id>` command. The
   agent fetches the YAML itself via `POST /api/compose/fetch` (device
   token), so the file — and any secrets in it — never lands in the
   command queue/log.
3. The agent writes it under `/var/lib/remotepower/stacks/<name>/` and runs
   `docker compose -p <name> <action>` via argv (no shell). `redeploy` is
   `pull` then `up -d`.
4. Status (up / down / error) flows back through the normal command-output
   channel and shows on the stack row.

### Security

Deploying runs an arbitrary compose file **as root** on the host (a
compose file can mount the Docker socket, bind host paths, or run
privileged). So it is:

- **Admin-only** and **audit-logged** (create, deploy, delete).
- **Gated behind a per-device opt-in.** Each device has a
  `compose_enabled` flag, **off by default**; a stack can't be deployed to
  a host until an admin enables it there (mirrors the `require_confirmation`
  pattern). Enable it from the stack row or the device.

### Scope

Lifecycle-core: upload + up / down / redeploy + status, plus a read-only
View (YAML + last run output). RemotePower runs the file as-is — it does
**not** sandbox it, build images for you, lint the compose, or auto-redeploy
on change. Deleting a stack only forgets the definition; run **Down** first
if you want to tear down the containers.
