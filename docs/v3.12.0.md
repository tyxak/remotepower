# RemotePower v3.12.0 — Pluggable storage backend (SQLite)

RemotePower has always persisted its state as flat-JSON files in the data
directory (`/var/lib/remotepower` by default), read and written through a small
set of helpers. That design is simple and robust, but **every write rewrites a
whole file**. On a busy fleet, `devices.json` is rewritten on *every* heartbeat —
an O(N) write per heartbeat for N devices, serialized behind a single file lock.

v3.12.0 adds an **optional embedded SQLite backend** you can switch to under
**Settings → Advanced → Storage backend**. It uses the Python standard library
`sqlite3` in WAL mode — **no new dependencies**, no database server to run (a
good fit for the CGI/fcgiwrap runtime, which is one process per request).

Flat JSON remains the default. Nothing changes for existing installs until an
operator opts in, and the switch is **fully reversible**.

---

## What changes under the hood

A new module, `server/cgi-bin/storage.py`, sits *behind* the existing
`load()` / `save()` / `_save_held()` / `_locked_update()` helpers in `api.py`.
The ~1000 call sites across the codebase are untouched — they keep their
whole-document semantics — but when the active backend is SQLite, those helpers
route to the decomposed store instead of the filesystem.

**File classification:**

| Class | Files | SQLite representation |
|-------|-------|-----------------------|
| Cold (most files) | config, users, schedule, … | one JSON blob row in `kv` |
| Dict-of-entity | `devices`, `cmd_output`, `metrics`, `monitor_history`, `metrics_history` | one row per key (devices has its own table with an indexed `last_seen`) |
| Wrapped-list | `history` (`entries`), `alerts` (`alerts`), `fleet_events` (`events`) | one row per list element; sibling metadata kept in a `…#meta` blob |

`load()` reconstructs the exact same Python structure the JSON backend would
have returned, so callers can't tell the difference.

**Why it's faster on big fleets:**

- A `save(DEVICES_FILE, devices)` only writes the device rows whose serialized
  form actually changed — a heartbeat persists *one* row instead of rewriting
  the whole file.
- Appends to the history / fleet-event logs are O(1) (`list_append`) instead of
  rewriting the whole log.
- WAL mode allows concurrent readers alongside a single writer; the per-file
  `flock` read-modify-write pattern maps to a `BEGIN IMMEDIATE` transaction,
  which takes the write lock at statement start and structurally removes the
  lost-update race the flock design existed to fix.
- The `devices.json` last_seen monotonic guard is preserved (a stale write can
  never move a device's clock backwards).

## Switching backends

### In the UI

**Settings → Advanced → Storage backend** shows the active backend and a file
count. Pick a target, then:

- **Preview** — a dry run that reports how many files *would* migrate, changing
  nothing.
- **Migrate & switch** — runs the migration in-process:
  1. takes a **rollback snapshot** (a tarball under `<data-dir>/backups/`),
  2. migrates every file into the target backend,
  3. **verifies** a full round-trip (counts + content of every file),
  4. flips the active backend **only if verification passes**.

If verification fails, your data is left untouched and the backend is *not*
switched; the snapshot path is reported so you can restore if needed.

### From the CLI

```
tools/migrate_storage.py --to sqlite            # JSON  -> SQLite
tools/migrate_storage.py --to json              # SQLite -> JSON (revert)
tools/migrate_storage.py --to sqlite --dry-run        # preview only
tools/migrate_storage.py --to sqlite --verify-only    # compare both views
tools/migrate_storage.py --to sqlite --data-dir /path # explicit data dir
```

Exit code is non-zero on verification mismatch — the active backend is only
switched after verification passes.

### How the active backend is selected

Selection cannot live in `config.json` (config itself is stored in whichever
backend is active). Precedence:

1. `RP_STORAGE_BACKEND=json|sqlite` environment variable (lets operators pin a
   mode; when set, the UI refuses to switch),
2. the marker file `<data-dir>/storage_backend.json` (always flat JSON), flipped
   by a successful migration,
3. default: `json`.

## Operational notes

- **Network filesystems.** SQLite's WAL relies on a shared-memory index that is
  broken over NFS/CIFS, which can cause corruption or `database is locked`
  storms. The backend detects a network filesystem and automatically falls back
  to a rollback journal (slower, but safe). **A local disk is strongly
  recommended** for the data directory when using SQLite.
- **Backups.** Backup/export, the scheduled tarball backup, and the self-status
  disk report all go through the backend seam now. Under SQLite the tarball
  captures a *consistent* online-backup snapshot of the database rather than the
  live WAL files.
- **The database file** lives at `<data-dir>/remotepower.db` (plus `-wal` /
  `-shm` sidecars in WAL mode).

---

## Compatibility

- Existing installs keep using flat JSON until you opt in.
- The migration is reversible at any time (`--to json` or the UI).
- Every existing feature behaves identically on both backends — only the on-disk
  representation differs.

## Quieter posture alerts (listening-port & firewall audit)

The v3.11.0 listening-port / firewall-drift audit fires `new_port_detected`,
`port_exposed_world` and `firewall_changed`. On Docker hosts `docker-proxy`
publishes every container port to `0.0.0.0`, which floods the inbox — so in
v3.12.0 the whole audit is **off by default** and opt-in under **Settings →
Security → Listening-port & firewall audit** (config `port_audit_enabled`).

If you want the audit on but a specific service quiet, use **surgical mutes**:

- A per-process **Mute** button on the **Exposure** page, or a managed list
  (with proto/port rules) in Settings → Security.
- `POST /api/exposure/mute` `{action, process?, proto?, port?}`; config
  `exposure_mutes`.
- Adding a mute **resolves matching open alerts** in the same action; turning
  the audit off resolves the whole `new_port_detected` / `port_exposed_world` /
  `firewall_changed` backlog at once.
- The baselines keep updating while suppressed, so enabling later never fires a
  catch-up burst, and the Exposure page always lists every socket (a banner
  links back to the toggle when alerting is off).

## Database maintenance & integrity

The SQLite backend self-maintains via the per-request sweep
`_sqlite_maintenance_if_due` (no-op under JSON):

- **Hourly:** `PRAGMA wal_checkpoint(TRUNCATE)` so the WAL can't grow unbounded.
- **Weekly:** `VACUUM` (reclaim free pages) + `PRAGMA integrity_check`.

DB size and the last integrity verdict show on **Server Status**. A failed
integrity check is logged loudly and raises a **critical `db_integrity_failed`
alert** (webhook + inbox) so a corrupt database pages you instead of sitting
silent — restore from the most recent backup/snapshot.
