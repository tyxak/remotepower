# v3.0.3 release notes

Released: 2026-05-24.

A small, focused security + UX patch on top of v3.0.2. **Recommended
for all operators**, especially anyone whose "Install RemotePower"
button stopped working in Chrome or Brave.

## Fixed

### "Install as app" button never appeared in Chrome or Brave

Three layered causes, each enough on its own to keep the button
hidden — fixing only one wouldn't have been enough.

1. **A stylesheet rule with ID-selector specificity was overriding
   the JS reveal.** When the browser fired `beforeinstallprompt`,
   the JS cleared the inline `style="display:none"` on the button,
   but the `<style>` block in `<head>` carried
   `#pwa-install-btn { display: none; }` — ID specificity 0,1,0,0 —
   which immediately took over and kept the button hidden.

2. **A timing race against `DOMContentLoaded`.** Chrome can fire
   `beforeinstallprompt` *before* `DOMContentLoaded` on warm reloads
   (when the manifest and service worker are already cached). The
   old code captured the button reference in a `DOMContentLoaded`
   handler and silently bailed if it was still `null` when the
   event arrived. `beforeinstallprompt` only fires once per session,
   so the button never came back.

3. **Combined `"any maskable"` icon purpose.** Both PWA icons
   declared `"purpose": "any maskable"`. Some Chrome and Brave
   builds treat that as maskable-only, which doesn't satisfy the
   installability gate that requires at least one pure-`any` icon.
   The manifest now declares the two purposes as separate entries.

The service worker cache name was bumped to
`remotepower-shell-v3.0.3` so existing installs evict the stale
shell on first reload. If the button is still missing after
upgrading, do **one hard reload** (Ctrl+Shift+R / Cmd+Shift+R) so
the new service worker takes over.

### Mobile UI polish

- **Mobile drawer scrim rendered as a half-sized floating rectangle.**
  Opening the mobile nav drawer dropped a partial translucent
  black box near the top of the screen instead of dimming the
  whole page behind the drawer, and tap-outside-to-close did
  nothing. Two `body::after` rules collided — the ambient
  blue-glow effect at the top of every page is a fixed-position
  800 × 400 pseudo-element with `translateX(-50%)` and
  `pointer-events: none`. The mobile-nav scrim rule set only
  `inset: 0`, `background` and `z-index`, expecting that to
  override the glow. It didn't: `inset: 0` only resets
  `top/right/bottom/left`, leaving `width`, `height`,
  `transform` and `pointer-events` inherited from the glow. The
  scrim now explicitly resets each of those properties so it
  fills the viewport and catches taps. A regression test
  (`TestMobileNavScrimNoCollision`) keeps the resets in place.
  Mobile-only — desktop never hit the scrim path.
- **Hamburger button on mobile no longer shows a square border.**
  The `.mobile-burger` style carried a 1 px border that rendered
  as a discrete framed button next to the logo, easy to mistake
  for a separate clickable element. Removed — the hamburger glyph
  reads fine on its own.
- **Quick-SSH icon next to hostnames is now readable on dark mode.**
  The icon was inheriting the browser's default link colour (blue),
  which clashed with the dark sidebar. It now uses the theme's
  primary text colour: near-white in dark mode, near-black in
  light mode.

## Added

### `RP_SMTP_PASSWORD` and `RP_LDAP_BIND_PASSWORD` environment variables

The two remaining plaintext secrets in `config.json` can now be
supplied via the environment — the same pattern
`RP_PROXMOX_TOKEN_SECRET` introduced in v2.3.1. When the env var
is set:

- It **takes precedence** over the value in `config.json`.
- The secret **stays out of `/var/lib/remotepower/`** entirely.
- The secret is **excluded from the backup export**, so the
  redacted tarball is safe to share with support.

Set them in your systemd unit or container environment:

```ini
# /etc/systemd/system/remotepower.service (or your drop-in override)
[Service]
Environment=RP_SMTP_PASSWORD=…
Environment=RP_LDAP_BIND_PASSWORD=…
```

```yaml
# docker-compose.yml
services:
  remotepower:
    environment:
      RP_SMTP_PASSWORD: "${RP_SMTP_PASSWORD}"
      RP_LDAP_BIND_PASSWORD: "${RP_LDAP_BIND_PASSWORD}"
```

When the env var is set, the Settings page shows a green hint
above the (now disabled) input:

> ✓ Password is currently being read from `RP_SMTP_PASSWORD`. The
> field above is ignored.

If you continue to keep the password in `config.json`, nothing
changes — the env var is opt-in.

### Forced password change for default-credential accounts

A fresh install seeds an `admin` account with password
`remotepower` and the `must_change_password` flag set. Since v2.3.2
a red banner has nagged the operator to change it — but the app
remained fully usable on the default password. The banner could be
ignored.

v3.0.3 makes this a hard backstop. **Every API call returns 403
until the password is changed**, with only two endpoints reachable
through the gate:

- `POST /api/users/passwd` — the password change itself.
- `GET  /api/public-info`  — bare server-info used by the login screen.

The dashboard catches the 403, surfaces a clear toast
(*"Change your password first — every other action is blocked"*),
and routes the user straight to **Settings → Account**. The instant
the password is changed, the flag clears and the rest of the UI
unlocks — no restart needed.

The gate fires **only** for accounts that still carry the
`must_change_password` flag. Once cleared, never enforced again.
API keys are unaffected (a default-password admin couldn't have
created one anyway, since the create-key endpoint is in the
blocked set).

## Upgrading from v3.0.2

Drop-in. Pull and redeploy:

```bash
cd /path/to/remotepower
git pull origin main
sudo bash deploy-server.sh
```

- Agents auto-upgrade on next heartbeat. No data migration. The
  `config.json` schema is unchanged.
- To move SMTP / LDAP secrets to the environment, edit your unit
  or compose file, restart the server, then **clear the password
  fields** in Settings → Notifications and Settings → Security
  and save once — that removes the plaintext from `config.json`.
  The backup export will then no longer contain those secrets at
  all.
- If your dashboard's Install button has been missing, force one
  hard reload after upgrading so the new service worker takes
  over.

## Honest caveats

1. **The PWA fix is browser-untested in CI** — there is no
   headless-browser harness in the test suite. The fix is validated
   by static checks against the shipped HTML / manifest / SW
   (the rule that defeated the reveal is gone, icons are split, JS
   uses an explicit display value, SW cache name advanced). The
   actual install prompt firing depends on Chrome's heuristic gate
   (HTTPS, user engagement, valid manifest, fetch-capable SW), all
   of which the build now satisfies.

2. **The forced password change is a UI-level gate, not an
   authentication boundary.** A `must_change_password` admin still
   has a valid session token; the interceptor relies on the
   `/api/users/passwd` and `/api/public-info` allowlist being
   correct. The set is deliberately small for that reason, and
   tested directly.

3. **Env-var secrets are opt-in.** No upgrade automatically removes
   secrets from `config.json` — that's a deliberate operator
   action. Until you clear the field, both sources exist; the env
   var wins at runtime.
