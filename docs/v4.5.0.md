# RemotePower v4.5.0 — "TrustMatters"

TLS onboarding made easy. RemotePower has always *required* HTTPS to enrol a real
agent, but getting a cert onto an internal / airgapped instance with no public DNS
was a manual chore. This release adds a one-command **self-signed CA**, a
fingerprint-verified way to roll its trust out to agents, and an opt-in TLS mode
for the Docker image — while still recommending a real cert where you can use one.
**No breaking changes**: existing HTTP-only installs and real-cert (Let's Encrypt)
setups are untouched.

After upgrading, hard-reload the dashboard once so the new front-end loads
(service-worker cache `remotepower-shell-v4.5.0`).

The full decision tree and the self-signed → real migration live in
[tls-selfsigned.md](tls-selfsigned.md).

## A self-signed CA, not a bare self-signed cert

`tools/gen-ca.sh` (or `make tls-selfsigned HOST=rp.internal`) generates a small
private **CA** and a server **leaf** signed by it — ECDSA P-256, a correct SAN
(DNS + IP), `serverAuth` EKU, sane validity (CA ~10y, leaf 397d). The leaf goes
to nginx; the CA's public cert (`ca.crt`) is what you distribute to agents.

Why a CA and not a one-off self-signed cert: **agents trust the CA, not the
leaf.** So you can re-issue or rotate the server cert at any time —
`make tls-renew` — and **not a single client needs to change**. The CA's private
key stays `0600`, root-only, and the script offers to move it off-box after
issuing. The CA's SHA-256 fingerprint is printed for out-of-band pinning.

## Rolling trust out to agents — verified, not blind

The agent already trusted a custom CA via `RP_CA_BUNDLE` (full verification:
`CERT_REQUIRED` + hostname check + TLS 1.2 floor — none of that is weakened).
This release adds the distribution layer:

- nginx serves the CA at **`/ca.crt`** (over plain HTTP — that's fine, see below).
- `install-client.sh --server https://rp.internal --ca-fingerprint <sha256>`
  fetches `/ca.crt` and **refuses to trust it unless its fingerprint matches**
  the one you printed from `gen-ca.sh`. A man-in-the-middle on that first,
  pre-TLS fetch can't substitute its own CA. The macOS (`install-macos.sh
  --ca-fingerprint`) and Windows (`install-windows.ps1 -CaFingerprint`)
  installers do the same.
- The CA is wired into the running agent via `RP_CA_BUNDLE` — a systemd
  `EnvironmentFile` on Linux, the launchd plist's `EnvironmentVariables` on
  macOS, a machine env var on Windows. As a backstop, all three agents now also
  fall back to the conventional CA path (`/etc/remotepower/ca.crt`, or
  `%ProgramData%\RemotePower\ca.crt`) when the env var isn't set.

## One nginx routing snippet (no HTTP/HTTPS drift)

The location blocks moved into a shared `remotepower-locations.conf` that **both**
the HTTP and the HTTPS server blocks include (the Docker image too). Enabling
HTTPS is now a clean uncomment instead of a copy-paste of 100 lines that slowly
drift apart. `gen-ca.sh --nginx` writes the matching `ssl` snippet.

## Docker: opt-in self-signed TLS

Set `RP_TLS_SELFSIGNED=1` (and `RP_TLS_HOST=<name>`), map host `443:8443`, and the
container generates a CA+leaf into the persistent data volume on first boot and
serves HTTPS itself — printing the CA fingerprint to `docker logs` for agent
enrolment. HTTP-only (terminate TLS at your own reverse proxy) stays the default.

## Switching self-signed → real is a server-only change

Because the agent trusts the **system root store *and* your CA at the same time**
(`RP_CA_BUNDLE` *adds* to the defaults), migrating to a real cert is painless:

1. Get a real cert: `sudo certbot --nginx -d your.domain` (or acme.sh DNS-01).
2. Point nginx at it and reload.
3. Done. Browsers trust it immediately; agents already trust public roots, so
   they keep verifying with **zero changes** — `RP_CA_BUNDLE` just sits there
   harmlessly. Drop it from agents later, at your leisure.

No overlap window, no agent redeploy.

## Still: prefer a real cert

A self-signed CA means browsers warn until you import `ca.crt` into your OS/browser
trust store, and you own renewal. Use it for airgapped / internal-only / no-public-
DNS deployments. If the server has a public hostname, Let's Encrypt (HTTP-01 via
certbot, or DNS-01 via acme.sh) is less work and trusted everywhere.
