# v5.1.1 — "ClusterMatters"

Status: released 2026-06-26. No breaking changes.

A small follow-up to v5.1.0 "UnityMatters" centred on the Proxmox integration,
plus the first round of community-reported AI/UX fixes and the test/polish gaps
from the feature that shipped it. No breaking changes.

## Proxmox: see the whole cluster, not just one node

The Proxmox integration only ever queried the single configured node, so on a
cluster you saw one node's guests and nothing else. It now lists guests
cluster-wide via `/cluster/resources`, with each guest tagged by its owning
node (shown as a badge in the UI). Start / shutdown / reboot / snapshot /
migrate all resolve the owning node per guest, so an action targets the right
host even when the guest lives on a node other than the configured one.

- **Backwards compatible.** A token with only node-scoped read permission falls
  back to the original single-node listing, so existing setups are unchanged.
  Cluster-wide visibility needs the API token to have cluster-scope read
  (`Sys.Audit` on `/`).
- **Backups across the cluster.** vzdump archives are now enumerated on every
  node a guest lives on (a cluster writes backups to per-node local storage), so
  cross-node guests are no longer falsely flagged as having no backup.
- **Hardened against a hostile cluster.** Node names from the cluster response
  are validated as hostnames before they can ever reach a `/nodes/<node>/…`
  request path, and the node is URL-quoted at the sink, so a misbehaving or
  hostile Proxmox cannot steer an authenticated call to an unintended endpoint.

Contributed by **@tbouquet** ([#9](https://github.com/tyxak/remotepower/pull/9)).

## Hardening the cluster feature

This release also closes the follow-up gaps from #9:

- **Regression tests** for the new code path: the node-name validator drops
  path-separator / traversal / newline injection, the cluster listing falls back
  to single-node on a permission error or empty response, owning-node resolution
  is correct (and falls back to the configured node), and the cluster member
  listing degrades to an empty list rather than raising.
- The Proxmox **node badge** now has its own style, distinct from the tags badge.
- The cluster **node summary** returned by the API is now surfaced in the UI
  instead of being unused.

## Community fixes (AI assistant + UX)

The first wave of external issues/PRs against the public repo:

- **Restore the page from the URL hash on refresh** — reloading the app on a deep
  page (e.g. `#containers`) dropped back to home. The page named in the hash is
  now restored on boot; the existing `#device/<id>` and `#devices?view=`
  deep-link handlers are left untouched. Contributed by **@tbouquet**
  ([#12](https://github.com/tyxak/remotepower/pull/12)).
- **LocalAI API keys are now accepted** — the AI-provider API-key field was
  disabled for LocalAI (and Ollama). LocalAI added API keys as a way to track
  per-app usage, so the field is now optional-but-editable for local providers
  and the key is forwarded as a `Bearer` token (the backend already did this; the
  field was simply blocked in the UI). Reported by **@loryanstrant**
  ([#10](https://github.com/tyxak/remotepower/issues/10)).
- **Run embeddings on a different service than chat** — *Settings → AI assistant →
  Knowledge index (RAG)* gains optional **Embedding provider**, **Embedding base
  URL** and **Embedding API key** fields. Embeddings default to reusing the chat
  provider; set these to point semantic search at a separate endpoint — e.g. a
  dedicated, less-contested GPU box. The embedding endpoint gets the same SSRF
  pre-flight (set-time) and connect-time peer-IP guard as the chat endpoint; its
  key is withheld and masked on read like the main key; and changing the
  provider/URL/model correctly invalidates the cached vectors (the embedding
  fingerprint covers the override). Semantic search can now also be enabled when
  the *chat* provider has no embeddings endpoint — e.g. Anthropic for chat with
  LocalAI for embeddings. Requested by **@loryanstrant**
  ([#11](https://github.com/tyxak/remotepower/issues/11)).

## Upgrade notes

No breaking changes. Hard-reload once after upgrading so the browser picks up
the new cache (`remotepower-shell-v5.1.1`).
