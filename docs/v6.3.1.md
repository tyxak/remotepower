# v6.3.1 — "Tr1ageMatters"

The agentic-diagnosis release. When a host is misbehaving and nobody knows
where to look, RemotePower can now do what a good on-call engineer does:
sweep the recent logs, follow the evidence, and write down a verdict — with
hard budgets, secret redaction and a visible evidence trail at every step.

Two features, designed to compose:

## Hail-mary log sweep — "Diagnose from logs"

A new device-drawer action for the *"something is wrong but I don't know
what"* moment.

**Collect.** The server sets a one-shot `force_log_sweep` flag (the same
delivery contract as `force_package_scan`); on its next heartbeat the agent
sweeps `/var/log/*` and `/var/log/*/*`:

- only files **modified in the last 24 h** — the sweep is about what's
  breaking *now*;
- compressed (`.gz`/`.xz`/…), rotated (`.1`), and binary files
  (wtmp/btmp/journal/ASL) are skipped;
- at most **12 KB tail per file, 40 files, 256 KB total**;
- files are ranked by **error density + recency**, so the budget goes to the
  log that is actually screaming;
- the same path deny-list as log watching applies (symlink-resolved;
  shadow/sudoers/SSH keys can never qualify).

macOS sweeps `/var/log` the same way. Windows has no `/var/log`, so its sweep
is the last 24 h of Error/Warning events from the System and Application
event logs, shaped as pseudo-files (`eventlog:System`).

**Redact, then store.** At ingest the server re-caps every size and scrubs
credential material line-by-line — `password=…`, `Authorization: Bearer …`,
`api_key: …`, `https://user:pass@host` all become `[REDACTED]` before the
sweep is stored. One sweep is kept per device (`log_sweep.json`).

**Diagnose.** The **Diagnose** button hands a ≤48 KB excerpt (highest-scored
files first) to your configured AI provider under the new `log_sweep_rca`
prompt and renders the result: a verdict, per-file findings quoting the key
evidence lines, and conservative next steps. Like the mitigate flow, this is
an **explicit-action consent model** — log content goes to the AI provider
only when you click, and the modal says exactly that.

Endpoints: `POST /api/devices/<id>/log-sweep/run` (admin),
`GET /api/devices/<id>/log-sweep`,
`POST /api/devices/<id>/log-sweep/diagnose` (write role). All device-scope
and tenant gated.

## Agentic alert triage

`POST /api/alerts/<id>/ai-triage` — the **Triage** button in the alert inbox —
runs a bounded, MXDR-style investigate loop for one alert. Instead of
answering from a static context blob, the model *investigates*: it is offered
a fixed menu of read-only, device-scoped evidence tools —

- device summary (load, memory, disk, failed units, last-seen),
- journal tail,
- watched service states,
- the host's **other open alerts** (correlation),
- recent command output,
- a regex search over the 6 h log buffer,
- the latest hail-mary log sweep, if one exists —

and must reply with strict JSON each round: **one tool call or a final
verdict**. At most 4 tool calls; every tool result is clipped and
secret-redacted; the model **never executes anything** — acting on a verdict
stays on the existing approval-gated paths (mitigate playbooks, the governed
AI executor).

The verdict — root cause, confidence, evidence lines, recommended action —
plus the list of tools the model actually used is stored on the alert and
shown in the inbox: an **"AI verdict" badge** on the row reopens it any time.
Cross-tenant and out-of-scope alert ids 404.

Both new prompt keys (`log_sweep_rca`, `alert_triage`) are tunable under
Settings → AI Assistant like every other prompt.

## Automatic alert triage (opt-in)

Settings → AI Assistant → **Automatic alert triage** turns the Triage button
into a background habit: a cadence sweep picks one new, open, untriaged alert
per tick (severity floor configurable, default *High and up*), runs the same
bounded investigate loop, and stores the verdict as `by: auto` — alerts land
in the inbox already carrying an **AI verdict** badge. Hard limits: one alert
per tick, a configurable **daily run cap** (default 20; each run is a few
provider calls), and the loop's own 4-tool budget. **Off by default.** With
the out-of-band scheduler active the runs happen fully off the request path.

## The feedback loop

Every verdict shows **Was this verdict helpful?** (👍/👎). Ratings are stored
on the alert and aggregated in `GET /api/ai/stats` (triaged / auto / up /
down) — a quick read on whether auto-triage is earning its keep, and the
strongest tuning signal when it isn't.

## From verdict to fix — the governed bridge

When the **AI executor** module is enabled, a verdict with a recommended
action offers **Propose fix (governed executor)**: the verdict becomes the
proposal context, the executor may only pick from remediation actions a
human already wrote and saved (never author a command), and the pick parks
in **Confirmations** for a human to approve. Tickets opened from a triaged
alert carry the full verdict in their seeded note.

## Evidence menu (9 tools)

Wave 2 adds `cves` (open CVE findings, top by severity) and `metrics_trend`
(recent daily disk/metric samples) to the triage menu. And the `log_sweep`
tool is now self-provisioning: a missing or >30-minute-old sweep makes the
server set the one-shot `force_log_sweep` flag itself, so the evidence is
ready for the next run.

## Server-status: syslog-receiver watcher

The Distributed-subsystems card on the Server-status page now shows the
v6.3.0 agentless syslog receiver: enrolled sources, last intake, and a
best-effort local unit probe. The row is **informational only** — never a
warning or a health input, because the receiver is optional and may run on a
different host. If you *want* alerting on it, apply the new opt-in
baseline-checks catalog row **`rp_syslogd_running`** (tag your server host
`rp-server`) for a critical Checks row when the unit stops.

## Guarded, verified auto-remediation

Wave 3 closes the fix loop. Automation rules could already run a saved script
on an event; now that channel is safe to trust with real fleets:

- **Per-host cooldown** and a **max-hosts-per-hour blast cap** bound every
  rule's blast radius (a flapping alert can't become a restart loop; an event
  storm can't run a script fleet-wide).
- **Verification**: set *Verify within N seconds* on a rule and the verify
  sweep checks that the triggering alert actually cleared after the fix ran.
  Still open → a **`remediation_failed`** alert fires (rule, host and script
  named), and **N consecutive failures auto-disable the rule**.
- Every firing — including guard suppressions — lands in the **Recent
  auto-remediations** ledger on the Automations page.
- A `run_script` action never fires on `remediation_failed` itself; escalate
  with notify/ticket actions instead.

See [automations.md](automations.md) for the full knob table.

## Signed command channel (opt-in)

The trust-model upgrade: the server detach-signs **every dispatched command**
with the release signing key, binding it to the target device and an issue
timestamp. Pin `release.pub` on a host and touch
`/etc/remotepower/require-signed-commands` (an operator-owned file the server
can never clear, like `audit-mode`) and that agent refuses anything unsigned,
tampered, aimed at another device, or older than 15 minutes — reporting the
refusal as command output so it's visible, never silently dropped. All three
agents enforce identically.

Database or command-queue tampering, and replay of captured commands, can no
longer execute anything on a pinned fleet — an attacker needs the signing
key, not just DB write access. See
[agent-commands.md](agent-commands.md) for setup and the honest boundary
(a full app-server compromise still signs in the convenient mode).

## Compliance you can back up

The Compliance page gains the discipline that a posture surface actually needs:
a control never reads **Pass** off an empty offenders list when the capable
source never ran. Patch/CVE/EOL/reboot/privileged-access controls gate on a
coverage fact and return **Not assessed** — disclosed, never a silent green —
when no host reported the signal. The framework score ignores Not-assessed, so
"unmeasured" can't inflate it. Two frameworks join PCI/HIPAA/SOC 2: **ACSC
Essential Eight** (process controls it can't observe are honestly Not assessed)
and **SMB1001:2026**. See [compliance.md](compliance.md).

## ATT&CK on triage verdicts

Agentic triage verdicts can now carry **proof-labelled MITRE ATT&CK
techniques** — observed / inferred / theoretical, validated server-side so the
model can't invent ids or overstate proof — shown as linked chips only when the
evidence genuinely points at adversary behaviour.
