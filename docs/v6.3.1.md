# v6.3.1 — "Tr1ageMatters"

The agentic-diagnosis release. When a host is misbehaving and nobody knows
where to look, RemotePower can now do what a good on-call engineer does:
sweep the recent logs, follow the evidence, and write down a verdict — with
hard budgets, secret redaction and a visible evidence trail at every step.

Two features, designed to compose:

## Hail-mary log sweep — "Diagnose from logs"

A new device-drawer action for the *"something is wrong but I don't know
what"* moment.

**Collect.** The server sets a one-shot `force_log_sweep` flag (the same
delivery contract as `force_package_scan`); on its next heartbeat the agent
sweeps `/var/log/*` and `/var/log/*/*`:

- only files **modified in the last 24 h** — the sweep is about what's
  breaking *now*;
- compressed (`.gz`/`.xz`/…), rotated (`.1`), and binary files
  (wtmp/btmp/journal/ASL) are skipped;
- at most **12 KB tail per file, 40 files, 256 KB total**;
- files are ranked by **error density + recency**, so the budget goes to the
  log that is actually screaming;
- the same path deny-list as log watching applies (symlink-resolved;
  shadow/sudoers/SSH keys can never qualify).

macOS sweeps `/var/log` the same way. Windows has no `/var/log`, so its sweep
is the last 24 h of Error/Warning events from the System and Application
event logs, shaped as pseudo-files (`eventlog:System`).

**Redact, then store.** At ingest the server re-caps every size and scrubs
credential material line-by-line — `password=…`, `Authorization: Bearer …`,
`api_key: …`, `https://user:pass@host` all become `[REDACTED]` before the
sweep is stored. One sweep is kept per device (`log_sweep.json`).

**Diagnose.** The **Diagnose** button hands a ≤48 KB excerpt (highest-scored
files first) to your configured AI provider under the new `log_sweep_rca`
prompt and renders the result: a verdict, per-file findings quoting the key
evidence lines, and conservative next steps. Like the mitigate flow, this is
an **explicit-action consent model** — log content goes to the AI provider
only when you click, and the modal says exactly that.

Endpoints: `POST /api/devices/<id>/log-sweep/run` (admin),
`GET /api/devices/<id>/log-sweep`,
`POST /api/devices/<id>/log-sweep/diagnose` (write role). All device-scope
and tenant gated.

## Agentic alert triage

`POST /api/alerts/<id>/ai-triage` — the **Triage** button in the alert inbox —
runs a bounded, MXDR-style investigate loop for one alert. Instead of
answering from a static context blob, the model *investigates*: it is offered
a fixed menu of read-only, device-scoped evidence tools —

- device summary (load, memory, disk, failed units, last-seen),
- journal tail,
- watched service states,
- the host's **other open alerts** (correlation),
- recent command output,
- a regex search over the 6 h log buffer,
- the latest hail-mary log sweep, if one exists —

and must reply with strict JSON each round: **one tool call or a final
verdict**. At most 4 tool calls; every tool result is clipped and
secret-redacted; the model **never executes anything** — acting on a verdict
stays on the existing approval-gated paths (mitigate playbooks, the governed
AI executor).

The verdict — root cause, confidence, evidence lines, recommended action —
plus the list of tools the model actually used is stored on the alert and
shown in the inbox: an **"AI verdict" badge** on the row reopens it any time.
Cross-tenant and out-of-scope alert ids 404.

Both new prompt keys (`log_sweep_rca`, `alert_triage`) are tunable under
Settings → AI Assistant like every other prompt.
