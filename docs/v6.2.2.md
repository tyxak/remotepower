# v6.2.2 — "Pu1seMatters"

A performance and polish release built around the agent heartbeat, plus a new
always-on health check that catches the v6.2.1 class of problem **before**
patch day instead of at reboot time.

## Kernel-module visibility check (always on)

v6.2.1 made the upgrade command refuse to run when the agent's execution
context can't see `/lib/modules` (the condition that produces module-less,
unbootable initrds). v6.2.2 moves that from a runtime refusal to a **standing
health signal**:

- The Linux agent reports whether `/lib/modules/<running kernel>` is visible
  from its own service context on every sysinfo heartbeat. It's a single
  directory check — effectively free.
- A host where the modules are hidden gets a **critical "Kernel modules
  visible" row** on the Checks page and a `modules_hidden` alert. Unlike most
  tripwires this fires on **first contact** too: a freshly enrolled host with
  the problem is exactly the host you need to hear about.
- The alert auto-resolves (`modules_visible_restored`) once the sandbox is
  fixed.
- The check **cannot be disabled per host** — deliberately. A signal that
  says "the next package upgrade may leave this machine unbootable" must not
  be muteable into invisibility.
- Hosts with no initramfs generator at all (WSL, minimal containers) don't
  report the field and are never flagged.

## Performance

### Delta sysinfo

Agents no longer re-send heavy, slow-moving inventory on every heartbeat.
Package lists, listening ports, network interfaces, SSH host-key
fingerprints, USB inventory and auto-update posture are hashed agent-side and
**omitted when unchanged**; the server merges its stored copy back in at
ingest, so every consumer (checks, alerts, the UI, the API) still sees a
complete picture.

The protocol is negotiated and self-healing in both directions:

- The agent only starts omitting after the server advertises support in a
  heartbeat response — a new agent against an older server keeps sending full
  payloads.
- If the server has nothing stored to merge (fresh install, restored backup),
  it asks for exactly those fields back, full, on the next beat.
- The agent only trusts a send the server confirmed — a dropped or busy beat
  can never leave the server holding stale data that the agent then stops
  sending.

On a steady fleet this removes the large majority of heartbeat bytes.

### Agent HTTP keep-alive

The agent now keeps its HTTPS connection to the server open across heartbeats
instead of performing a full TCP + TLS handshake every beat. Certificate
verification, self-signed-CA bundles and mTLS client certificates work
exactly as before, and redirects remain refused. Environments using an HTTPS
proxy automatically keep the previous per-request behavior, and
`RP_NO_KEEPALIVE=1` in the agent's environment opts a host out entirely.
`remotepower-wsgi.service` and the Docker entrypoint pass gunicorn
`--keep-alive 75` so the server side holds connections open past the default
60 s poll.

### Faster large-fleet tables

The Checks, Alerts (flat view), Processes, Exposure and CMDB asset tables now
render their first 300 rows immediately and stream the rest in as you scroll.
Filtering and sorting stay instant regardless of fleet size.

### Leaner first load

Ten page modules (Checks, Power/Thermal/Disk health, Billing/Timesheet,
Provisioning, Rollouts, Tuning, Knowledge base, GPUs, DMARC, WG Access) load
on first navigation to their page instead of at boot, shrinking the initial
JS payload for every session.

### Storage

Added the index the metric-retention sweep needed on the PostgreSQL and
SQLite backends — pruning old samples no longer scans the whole
`metric_samples` table. Applied automatically and idempotently on upgrade.

## Installer & agent lifecycle

Re-running `install-client.sh` on an already-enrolled host is now an
**in-place upgrade**:

- The agent binary and systemd unit are refreshed; enrollment and credentials
  are kept (no token churn, no duplicate device).
- A unit file you customized is backed up to
  `remotepower-agent.service.rp-old` before being replaced.
- The agent is **restarted** into the new unit — not `enable --now`, which is
  a no-op on a running service and would leave the old unit definition live.
- If the installer is being run *through* the agent's own exec channel, the
  final restart is detached into a transient systemd unit so the script can't
  terminate itself mid-run.
- `--re-enroll` restores the previous always-enroll behavior.

## Alert parameters — one home for every threshold

A new **Settings → Alert parameters** page consolidates the numeric thresholds
that decide when an alert fires and how a host is graded and coloured. Roughly
**70** values that used to be scattered — some in other Settings forms, many only
reachable by hand-editing the config file, a few hard-coded — are now in one
filterable place: firing thresholds (NIC errors, SNMP, temperature, clock skew,
inode/fd/conntrack, TLS/cert expiry, disk-fill forecast, mail-queue, and more),
the **health-grade / risk-level / reliability** score cutoffs (the dashboard
paints its colours at exactly these boundaries), and the **per-factor score
weights** (set one to 0 to ignore that signal). A filter box narrows the list as
you type; leaving a field blank restores its default. It is distinct from
Monitoring → Tuning, which mutes noisy alerts. Full guide:
[alert-parameters.md](alert-parameters.md).

Alerts also got a **description overhaul**: every alert, Needs-Attention item and
activity-feed row now names the specific resource and values ("Service failed on
host: nginx.service", "UPS battery CRITICAL: 12%") instead of a generic label,
and a custom-metric threshold breach now reaches the inbox (it previously only
sent a webhook).

## UX

- **Keyboard-driven alert inbox** — `j`/`k` move the selection, `a`
  acknowledges, `r` resolves, `m` mutes, `x` toggles the row checkbox, `?`
  shows the map, `Esc` clears. Only active on the Alerts page, never while a
  modal is open or focus is in a form field.
- **Device hover cards** — hover a hostname on the Alerts, Checks or Exposure
  pages for a quick-peek card: online state, OS, group, IP.
- **Tab-level device deep links** — `#device/<id>/<tab>` opens the device
  drawer on a specific tab (`actions` or `audit`), and a copy-link button in
  the drawer header puts the current view's link on the clipboard for
  tickets, runbooks and chat.
- **Skeleton loading everywhere** — the last tables that still flashed a bare
  "Loading…" (containers, host-config results, network map) now use the same
  skeleton placeholders as the rest of the app.
