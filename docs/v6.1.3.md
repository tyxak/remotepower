# v6.1.3 — "SentinelMatters"

Closing named gaps against comparable RMM products. Almost everything here is
built on signals RemotePower was **already collecting and never acting on** — the
work was the baseline, the diff and the honesty, not the telemetry.

---

## Privileged-group tripwire

Someone landing in `sudo`/`wheel` (Linux) or `Administrators` (Windows) now
raises an edge-triggered **`priv_group_added`** alert (high).

Both agents have always reported this. The Linux agent's `get_local_accounts()`
parses `/etc/group` for `sudo`/`wheel`/`admin` members; the Windows agent runs
`Get-LocalGroupMember -Group Administrators`. Both set a per-account `sudo` flag,
and both were rendered as a snapshot card. **Nothing ever diffed it** — so the
classic post-compromise persistence step, and the equally classic
nobody-told-me change, were invisible.

Design decisions worth knowing:

| Behaviour | Why |
|---|---|
| First contact baselines **silently** | Enrolling a device is not a privilege grant. Firing here would make the first heartbeat of every host a HIGH alert. |
| Removals do **not** fire | A revoke is not a security signal. An event that cries wolf gets muted — and a muted event catches no real escalation. |
| No recover event | A privilege grant is an **event**, not a condition. An auto-resolving alert would let a real escalation vanish from the inbox the moment the attacker cleaned up. |
| Baseline still tracks removals | Otherwise a re-grant after a revoke would be invisible. |

## Windows Defender AV posture

Defender joins the existing AV pipeline (`_ingest_av` → `av_status.json` →
Needs-Attention → `av_infected` / `av_warning` / `av_clean`) as a **distinct
`defender` engine**. It deliberately does *not* masquerade as `clamav`: an
operator — or a webhook consumer — reading "clamav infected=1" on a Windows box
would be actively misled.

Collected via `Get-MpComputerStatus`: signature age, threat count, last scan
time, and **real-time protection state**.

The genuinely new signal is **`av_realtime_off`**. "AV is installed but switched
off" has no equivalent among the on-demand Linux scanners, and it makes every
other AV number on that host meaningless — a clean scan report from a machine
whose protection is disabled is not good news. Unlike the privilege tripwire it
is a **condition, not an event**, so it:

* fires even on **first contact** — a host that enrols already unprotected is
  exactly the one you need to hear about; and
* **auto-resolves** (`av_realtime_on`) when protection comes back.

`realtime_enabled` is tri-state on purpose: **absent** means "this engine has no
real-time concept" (ClamAV, rkhunter). Treating absent as `False` would have
reported every Linux host in the fleet as unprotected.

## USB device tripwire

A USB device appearing on a host is a physical-access signal →
**`usb_device_added`**.

* Read from **sysfs** (`/sys/bus/usb/devices`), not `lsusb` — `usbutils` is not
  in the containerized agent's image, so `lsusb` would have returned nothing
  there, silently, forever. sysfs *is* bind-mounted.
* Keyed by **VID:PID**, so the same stick moved to another port does not re-fire.
  The kernel's bus/port path would have.
* Hubs and root hubs are skipped — fixed silicon, not somebody plugging something in.
* **Detection only.** Peripheral *enforcement* is endpoint-management territory
  and clashes with this product's audit-first posture.
* Default routing: **Alerts inbox + Recent Activity, but no paging and no health
  impact.** On a rack server this is physical access; on a homelab desktop it is
  a phone charger. An event that pings ntfy at 3am for a phone charger is one you
  mute within a week — which is how you lose the one that mattered.

## Host-wide disk-usage explorer

**"Disk 94% — of *what*?"**

Disk-fill forecasting has always been able to tell you *when* a mount runs out.
Nothing told you what to delete. This closes that loop.

An opt-in agent `du` runs over configurable roots and reports the biggest space
consumers per path, shown in the device drawer **directly under the mounts
table** — where you are already standing when you ask the question — with an
on-demand **Scan disk usage** button.

Bounded three ways, because it runs unattended on somebody's NAS:

* `-x` — never crosses a filesystem boundary (no walking into `/proc`, an NFS
  mount, or a 40 TB media array hung off `/srv`);
* `--max-depth=1` — "/var/lib is 80 G", not every file;
* a per-path timeout **and** an overall wall-clock budget.

`du` is shelled deliberately rather than hand-rolled in Python: it is C-fast and
already solves hardlink double-counting, sparse files and bind mounts — the traps
an `os.walk` + `st_blocks` summer gets subtly wrong.

**Off by default** — it walks the filesystem. Settings → Security.

Endpoints: `GET /api/devices/{id}/disk-usage`, `POST …/disk-usage/scan`.

## Device reliability prediction

A composite per-host **failure likelihood (0–100)** with an explainable factor
breakdown, on **Monitoring → Predictive health**. `GET /api/reliability`.

This is deliberately a **separate number from the risk score**:

* **Risk** answers *"how exposed is this host?"* — CVEs, world-open ports, policy
  violations, EOL.
* **Reliability** answers *"how likely is it to break?"* — dying hardware.

A fully-patched server with a failing disk is **low-risk and low-reliability**,
and an operator needs to see exactly that. Merging them would hide it.

No new collection. Every input was already stored:

| Signal | Source | What it contributes |
|---|---|---|
| SMART verdict | `hardware.json` | the drive says it is failing — believe it |
| Reallocated sectors **growing** | `smart_history.json` (~6 months daily) | the predictive one: a disk with 4 reallocated sectors that has had 4 for a year is fine; one that went 0 → 4 this month is on its way out |
| Pending sectors, wear %, NVMe spare | `hardware.json` | remap pressure and spent endurance |
| ECC errors | posture state / sysinfo | an **uncorrectable** error means the DIMM could *not* fix it — a different universe from a corrected one |
| Reboot churn | `uptime.json` (last 500 transitions) | a box that keeps coming back has been going away |
| Health-score trajectory | `health_history.json` (~6 months daily) | is this host getting *worse*? |
| Thermals, recent OOM | `hardware.json` / sysinfo | heat kills hardware |

Only **high** and **critical** hosts raise a Needs-Attention card. A low bar here
becomes noise, and a noisy predictor is one people learn to ignore — which is
strictly worse than not having a predictor at all. The card is muteable (an
unmuteable Needs-Attention item would depress the host's *and* the fleet's health
score forever, with no operator recourse).

### What is deliberately NOT scored

**Unit flapping.** `services.json` stores systemd's `NRestarts` as a *cumulative*
counter and keeps only the latest value; the flap **delta** that the
`unit_flapping` event fires on is computed in flight and never persisted. So
there is nothing to derive a flap rate from — a long-lived host that restarted a
unit once during a deploy two years ago carries `restarts=1` forever, and scoring
on `restarts > 0` would mark half a healthy fleet as failing while a genuinely
crash-looping unit that reset its counter scored zero.

An honest omission beats a confidently wrong number. Persisting a daily delta
would make this a real factor; until then it is left out on purpose.

## JIT credential checkout

*"Who **can** reveal this credential?"* becomes *"who has **active, justified,
expiring** access to it right now?"* — the useful half of PAM without becoming a
PAM.

Opt-in via `vault_checkout_required` (**default off** — switching it on
immediately requires every admin to check out before revealing anything, which is
a hell of a surprise for an upgrade).

A grant is:

* **scoped to one credential and one person** — Alice checking out `cred_a` on
  `host1` does not open `cred_b`, `host2`, or Bob's access;
* **reasoned** — an unreasoned checkout is just a slower reveal; the reason *is*
  the audit value;
* **bounded** — 15 minutes to 24 hours, defaulting to one hour;
* **self-expiring** — expiry is evaluated at *read* time, not by a pruning sweep,
  so a lapsed grant is dead the moment it lapses. A sweep that had not yet run
  would otherwise silently extend everyone's access.

Who currently holds live access, and why, is visible to admins **and auditors**
(`GET /api/cmdb/vault/checkouts`) — that visibility is the whole point.

### It is a floor, not a ceiling

A credential flagged `break_glass` still requires its **two-person approval**. The
checkout gate deliberately sits *after* the break-glass check, never instead of
it, so a flagged credential needs both.

| | Break-glass | Checkout |
|---|---|---|
| Who | a **second** admin approves | **self**-service |
| Duration | one-shot, 15 min | reusable within an N-hour window |
| Applies to | credentials flagged `break_glass` | all credentials, when enabled |

### What it is *not*

**The server never holds the vault key.** The passphrase is entered in the
browser, the derived key is returned to it and replayed as an `X-RP-Vault-Key`
header on every operation. So a "checkout" **cannot lease key material**
server-side without inverting that design — it is an *authorization* grant that
gates the reveal handler, while the browser continues to supply the key.

## Windows third-party app patching (winget)

Windows Update patches the OS. The CVEs live in Chrome, 7-Zip and VLC.

Pending third-party app updates now flow into the existing patch catalog under
the same `third_party: {manager: {count, names}}` structure the Linux agent uses
for flatpak/snap/pip/npm. Unlike those — which are detection-only — **winget can
remediate**: `winget:<id>` upgrades one package, `winget:` upgrades all.

The command is argv-only (never through a shell) and the package id is
charset-validated against `^[A-Za-z0-9][A-Za-z0-9.+_-]{0,127}$` — anything else is
*refused*, not escaped. Feature-invisible where winget isn't installed (Server
SKUs, older Windows 10 builds).

## Governed AI executor — Phase 0

The category pitch of the moment is the **autonomous AI operator**: a model that
watches your fleet and fixes it. This is the opposite bet.

> **Assume the model is fully prompt-injected by a hostile log line. Now design so
> that the worst it can do is still safe.**

Three constraints do all the work. None of them is a new enforcement chokepoint —
they compose gates that already existed.

### 1. The model returns an ID, never a command

The executor may only **select** from a server-built catalog of artifacts an
operator already wrote: saved scripts and registered playbook fixes. The model's
answer is an *index into that list*, and anything that is not in the list is refused
outright and audited.

So a completely compromised model can, at absolute worst, **pick a different script
you wrote yourself**. It cannot author shell. There is deliberately **no free-form
command path at all** — not an oversight, the design. That single constraint
eliminates most of the threat model, which is why it is the one to keep if you keep
only one.

`GET /api/ai-exec/catalog` returns the whole action space. You should be able to
read every action your AI could ever propose, in one place, and see that it contains
nothing but your own work.

### 2. A human always approves

A proposal is an ordinary entry in the **existing confirmations ledger**. That is the
entire point of reusing it — it inherits, for free:

* the TTL and the pruning,
* the tenant filter,
* the **separation-of-duties** rule (`change_approval_no_self`: the proposer cannot
  approve their own change),
* the audit trail, with `ai_host` / `ai_prompt` attribution already in the schema,
* the approve/reject UI that already exists.

**Phase 0 executes nothing on its own, ever.** A proposal returns `202` — parked, not
done.

### 3. What you approve is what runs

The proposal pins a **SHA-256 of the script body** it was built from. At approval the
body is re-resolved from the live store and re-hashed; if another admin edited the
script in between, execution is **refused** rather than quietly running something the
approver never saw.

The command is *always* re-resolved from the live store at execution time — no command
string is ever stored in the ledger, so there is no place a command could be tampered
into.

### It ships off

`ai_exec` is an optional module with **default `False`**. Off means a **404 on the whole
API prefix at the dispatcher** — an enterprise that wants zero AI-initiated actions can
make that structurally true, rather than trusting a UI preference.

### A gate this surfaced

Building it exposed a real bypass in the *existing* post-approval executor.
`_mcp_execute` — the code that runs an already-approved maker-checker confirmation —
checked quarantine but honoured **neither maintenance mode nor audit mode**. An
approved command could therefore fire at a host in the middle of a controller drain,
or at an agent that would refuse it locally anyway.

Both paths now share **one predicate** (`_command_block_reason`), rendered two ways —
`respond()` for the request path, an `{'ok': False}` dict for the executor. Not a
second chokepoint: the same one.

### What Phase 0 is not

It is not autonomy, and nothing here becomes autonomous by configuration. Selective
autonomy for low-risk actions (read-only diagnostics; specific operator-blessed
playbook fixes) is **Phase 1** and is not built. Closed-loop outcome reporting is
**Phase 2** and is not built. Phase 0 is deliberately the whole shippable increment:
it is already most of the value, and it is strictly *safer* than the existing mitigate
flow, which executes on a typed `RUN` with no second pair of eyes at all.

## Quotes

The mirror image of an invoice.

An **invoice looks backward**: it is derived from hours already logged, and issuing
it locks them. A **quote looks forward**: hand-authored line items — a labour
estimate, hardware, licences — that, once the customer accepts, *become* an invoice.
draft → sent → accepted / declined, on the **Billing** page.

The money maths is shared with invoices (`billing.invoice_totals`) rather than
reimplemented. Two subtly different VAT calculations in one product is a bug waiting
to be found by a customer.

### The rules that matter all protect the customer

A billing module's worst bugs share a victim, so each of these is enforced on the
server and pinned by a test:

**A quote converts exactly once.** The claim (`invoice_id`) is stamped under the
*same lock* that checks it, so a concurrent second request — a double-click, a
retried POST — loses the race and is refused. Billing a customer twice is the worst
bug a billing module can have.

**Only an accepted quote converts.** Invoicing a draft or a declined quote bills
someone for work they never agreed to.

**The invoice snapshots the agreed numbers.** Line items, VAT rate and currency are
copied from the quote, not recomputed from today's config. A rate-card edit or a VAT
change between acceptance and invoicing must not silently re-price a deal the
customer already agreed to.

**Expiry is evaluated at read time** — the same principle as the credential checkout.
A lapsed quote is expired the moment it lapses, not whenever a sweep next runs; a
sweep that hadn't run would otherwise let a customer accept a price that expired last
Tuesday. Accepting an expired quote is refused outright.

**But an accepted quote never expires.** `accepted` / `declined` / `invoiced` are
terminal: a customer who accepted in time must not lose the deal because an admin was
slow to invoice it. The clock stops when they say yes.

**An invoiced quote is frozen** — re-opening it would let someone "decline" work that
has already been billed.

### It rides the billing kill switch

`/api/quotes` is registered in the `_MODULES` billing prefix tuple. A new route under
a gated module that *isn't* in its prefix tuple silently escapes the kill switch — the
nav hides but the API answers. Turning billing off 404s quotes too.

## Regulated-data scan (PII)

*"Where is our regulated data?"* is the GDPR/PCI question that no amount of
monitoring answers. Opt-in (`pii_scan_enabled`, **off by default** — it walks the
filesystem), the agent sweeps configurable roots on a ~24 h cadence and reports
which **files** contain emails, card numbers, national IDs, IBANs or phone
numbers. Results land on the **Compliance** page.

Card matches are **Luhn-checked**. Without that, every 16-digit order id, serial
and timestamp reads as a credit card — and a report that opens with 400 false
cards is a report nobody reads twice.

### The value never leaves the host — not raw, not redacted, not hashed

This is the rule the whole design serves, and the second half is the part that is
easy to get wrong:

1. **A PII scanner that ships PII into its own database is not a control — it is a
   second breach with a nicer UI.**
2. **Hashing does not anonymise low-entropy data.** There are only 10⁹ possible US
   national IDs, and a card number is pinned by its BIN plus the Luhn check. A
   rainbow table over either is minutes of work. A "fingerprint" of a national ID
   is a *reversible copy of the national ID*, wearing a disguise.

This is exactly why the sibling `secret_findings` scan **may** fingerprint its
matches and this one may not: an API key is high-entropy, so its hash reveals
nothing. A national ID's hash reveals the national ID.

So a finding carries only:

| | |
|---|---|
| `path` | which file |
| `kind` | email / credit_card / ssn / iban / phone |
| `count` | how many |
| `lines` | first few line numbers |

Everything an operator needs to go and look. Nothing an attacker who pops the
RemotePower server can use.

**And the server does not merely trust the agent to honour that.** `_ingest_pii_findings`
rebuilds each stored entry from those four known-safe fields — a *whitelist*, not a
blacklist filter — so a tampered, malicious or simply newer agent that posts a
`preview`, a `value` or a `fingerprint` is **structurally incapable** of getting one
into the store. There is a test that plants a real national ID in every one of those
fields and asserts none of it reaches disk.

### A report, not an alert

PII sitting in `/srv/data` is the **expected state of a business**, not an incident.
An event per finding would be pure alert fatigue — and it would train operators to
mute the one category they must never mute. So this fires nothing: it is an
inventory, and it reads as one.

The incident case is already covered: a *leaked credential* still raises
`secret_exposed`.

Not scanned by default: `/etc`. It is full of maintainer emails in config files, and
a report that opens with 400 hits from `/etc` buries the real ones. The defaults look
where an organisation's **data** lives (`/home`, `/srv`, `/var/www`, `/opt`), not
where its config lives.

## EDR coverage

Three new read-only connectors — **Wazuh**, **CrowdStrike Falcon**,
**SentinelOne** — under a new `security` category.

**The connectors are not the feature.** A tile reading *"EDR: healthy"* while
three servers have no agent on them is worse than no tile at all, because it is
*reassuring*. So each connector also reports the set of hosts it **protects**, and
`GET /api/edr/coverage` cross-references that against the fleet RemotePower
actually knows about:

> **The gap is the product.** The page exists to name the machines your endpoint
> protection has never heard of.

Surfaced as a card on the **Risk** page, uncovered hosts first. It stays hidden
until an EDR integration is configured — an empty "0 uncovered" table on a fleet
with no EDR would read as an all-clear.

### Two things it would have been easy to get wrong

**Hostname matching.** Consoles and agents disagree endlessly about case and
domain suffix: `WEB01.corp.example.com` in Falcon is `web01` in the fleet. Match
the raw strings and almost every host reports as uncovered — which is the failure
mode that teaches an operator to distrust the page and scroll past a real gap.
Matching is on the short, lowercased name.

**Stale agents.** The most likely way an EDR rollout fails is not "we forgot a
host" — it is "the agent installed, then stopped reporting". Folding those into a
`covered` count hides exactly the case the page is for, so a stale agent is its
own state, listed apart from a protected one. An agent that isn't reporting is
protecting nothing.

### Why no Microsoft Defender for Endpoint

MDE's OAuth token endpoint lives on a *different host* (`login.microsoftonline.com`)
from its API host. The shared integration client is deliberately **bound to the
instance's base URL and refuses absolute URLs** — that binding is what keeps the
SSRF guard meaningful. Loosening it so one connector could fetch a token from
another host would weaken the guard for all 47. It is left out on purpose; the
guard is worth more than the connector.

## DNS-blocker control (Pi-hole / AdGuard)

The DNS connectors were read-only: they told you the blocker was up and how much
it had blocked. The single most common thing an operator wants to *do* next —
*"something's broken, is it the ad-blocker?"* — was the one thing they had to
leave RemotePower to do.

From the Integrations page you can now read the blocking state and **pause
blocking** on any Pi-hole or AdGuard Home instance.

### Every pause is timed — on purpose

A DNS blocker switched off and forgotten is a **silent, permanent security
regression** — the exact opposite of what this product exists to do. So "disable"
is modelled as a *timed* action:

* the driver **always** sends a timer (30 s – 4 h), clamped in `dns_control.py`
  itself rather than only in the handler, so the bound does not depend on a caller
  remembering to apply it;
* the blocker **re-enables itself** when the timer lapses — the safe state is
  restored by the remote device, not by a sweep here that might never run;
* there is **no disable-forever action**. Not an oversight; a decision.

Pausing is admin-only and audit-logged *with the window*, so "who turned the
ad-blocker off, when, and for how long" is answerable after the fact.

### Shape

Drivers are pure `fn(inst, client, ...)` functions in `server/cgi-bin/dns_control.py`
over the same SSRF-guarded client every read-only connector uses — the same shape
as `hypervisor.py`, which is the precedent for adding write actions to the
integrations layer. `api.py` owns the admin gate and the audit log; the driver
module never decides who may call it.

| | |
|---|---|
| `GET /api/dns-control/blockers` | Controllable blockers (no secrets/urls) |
| `GET /api/dns-control/{id}/blocking` | Is blocking on, and for how long |
| `POST /api/dns-control/{id}/blocking` | `{enabled, seconds}` — admin, audited |

## Fixed

* **Windows and macOS patch execution never worked.**

  Both agents have implemented the bare `upgrade` / `upgrade:<name>` command since
  v6.0.0. The server never emitted it.

  Every server-side upgrade path — the on-demand `handle_upgrade_device`,
  auto-patch policies, and scheduled jobs — computed `exec:<bash script>` with
  **no OS branch anywhere** (in `handle_upgrade_device` the command was even
  hoisted *outside* the per-device loop). Commands are delivered to agents
  verbatim, so a Windows host was handed a bash script, which its agent passed to
  PowerShell.

  The feature was dead end to end: fully wired on the agent, complete in the
  changelog, documented in `features.md` — and never once triggered by the server.
  This is precisely the "feature that can never fire" class the v6.1.2 sweep
  catalogued, and it is why that note says to *drive the real path* rather than
  trust that something looks wired.

  All three paths are now OS-aware (`_device_os_family` / `_upgrade_command_for`).

  **Fixing it surfaced a security bug.** `_command_kind()` did not know about
  `upgrade`, so a bare `upgrade` classified as `'other'` — which is **not** in
  `_APPROVAL_GATED_KINDS`. Routing Windows hosts onto the bare command (the
  correct fix) would therefore have let an auto-patch policy covering a Windows
  host queue a fleet-wide patch run **with the four-eyes approval gate silently
  bypassed**. `_command_kind` now maps `upgrade` / `upgrade:<pkg>` / `winget:<id>`
  onto the gated `upgrade` kind.

* **The fleet-risk cache never hit on a real fleet.** `_fleet_risk_cached` busted
  whenever `devices.json` changed — but *every heartbeat rewrites it*, so on any
  fleet whose hosts collectively beat faster than the 10-second TTL the cache was
  dead on arrival, and the O(fleet) risk computation (per-device CVE joins, date
  parsing) re-ran on **every single poll** of `/api/risk` and `/api/home`. A
  20-host fleet at a 60-second poll interval rewrites `devices.json` every ~3
  seconds — so the cache was useless for exactly the fleets big enough to need it.
  It now fingerprints the operator-controlled inputs only (the same fix the v6.1.2
  sweep applied to the Needs-Attention digest); accepting ≤10s-stale telemetry in
  a rollup *is* the contract of a 10-second cache.

* **AV alerts lost the name of the engine that fired them.** `av_infected` and
  `av_warning` have set a `tool` field since v5.1.0, but it was never added to the
  `_record_alert` payload whitelist — so the stored alert silently dropped it and
  the inbox could not say which scanner found the malware.
