# v2.3.0 release notes

Released: 2026-05-17.

Proxmox VE integration. RemotePower can now list and control the
QEMU virtual machines and LXC containers on a Proxmox node.

## Added

### Proxmox VE integration

RemotePower connects to a single Proxmox VE node and surfaces its
guests in two places:

- **Virtualization page** (new) — lists QEMU virtual machines:
  vmid, name, status, CPU / memory when running, uptime, tags.
  Start and graceful-shutdown actions.
- **Containers page** — Proxmox **LXC** containers appear as a new
  section below the agent-reported Docker/Podman containers, with
  the same start / shutdown actions.

The integration is **server-to-API**: the RemotePower server calls
the Proxmox REST API (`https://<node>:8006/api2/json`) directly.
There is no agent on the Proxmox node.

**Architecture.** The new `proxmox_client.py` module is a
stdlib-only (urllib + ssl) Proxmox API client — no `proxmoxer`, no
`requests`, consistent with the rest of RemotePower. It handles URL
normalisation (bare host, host:port, or full URL all work), the
`PVEAPIToken` auth header, guest listing and normalisation, and the
action calls.

**Configuration** lives under **Settings → Proxmox**: enable
toggle, host, node name, API token ID, API token secret, and a
"Verify TLS" toggle. A "Test connection" button probes the node and
reports back inline. The Virtualization nav entry only appears when
Proxmox is enabled.

**Actions.** The action set is a hard allow-list in
`proxmox_client.py`: `start`, `shutdown`, `stop`, `status`. The UI
exposes only **start** and graceful **shutdown** — `shutdown` is
ACPI / init-signal graceful, not a hard power-off. `stop` (the hard
kill) is in the allow-list so a future "force stop" button has a
path, but it is not exposed in this release. `migrate`, `clone`,
and `delete` are deliberately absent — they can't be invoked even
by a crafted request.

**Security notes — please read.**

- The Proxmox API token secret is stored in the server's
  `config.json`. That file is written with mode `0600` (owner
  read/write only), but the secret is **not encrypted at rest**.
  This was a deliberate scope decision for this release — there is
  no vault integration. Treat the RemotePower server host
  accordingly.
- **Use a scoped API token.** Create the token in Proxmox under
  Datacenter → Permissions → API Tokens and give it only the
  privileges it needs — `VM.Audit` (to list) and `VM.PowerMgmt`
  (to start/stop). Do not use a full-access `root@pam` token.
- The token secret is **masked everywhere it could leak**: the
  config API never returns it (only a `proxmox_token_secret_set`
  boolean), the settings field shows a placeholder when a secret
  is saved, and the client's error messages are written so they
  never echo the token.
- **TLS verification is ON by default.** Proxmox ships a
  self-signed certificate, so a default Proxmox install will fail
  verification — the operator must either install a trusted
  certificate (recommended) or explicitly turn off "Verify TLS"
  for the connection. The off switch is a visible, deliberate
  choice, not a silent insecure default.

This is a single-node integration. The client's node handling is
written so a future multi-node / cluster expansion is possible, but
clusters are not supported in 2.3.0.

## Upgrading from 2.2.7

Server-side only — no agent change, no data migration. After
deploying, go to Settings → Proxmox to configure the connection.
Until it's configured, nothing changes: the Virtualization nav
entry stays hidden and the Containers page looks exactly as before.

## Honest caveats

1. **Browser-untested.** 921 unit tests pass and all four files
   parse, but I have not loaded the Virtualization page, the
   settings pane, or the LXC section in a real browser, and — more
   importantly — **no part of this has been tested against an
   actual Proxmox node.** The unit tests cover the pure logic
   (URL building, normalisation, the action gate, config
   round-trip); they do not and cannot verify that the API request
   shapes match what your Proxmox 8.x actually expects. The first
   thing to do after deploying is hit "Test connection" in
   Settings, then load the Virtualization page, before relying on
   it.

2. **No live polling / caching.** Every visit to the Virtualization
   page (or the Containers page LXC section) makes a fresh,
   synchronous call to the Proxmox API. There's no background
   poll, no cache, no staleness model like the agent containers
   have. If the Proxmox node is slow or unreachable, the page will
   block for up to the 10-second HTTP timeout and then show an
   error. For a single node this is fine; it would need rethinking
   for a cluster or a slow link.

3. **Actions are fire-and-forget.** A Proxmox start/shutdown
   returns a task ID — the operation is asynchronous on Proxmox's
   side. RemotePower sends the action, waits 1.5s, then refreshes
   the list. It does **not** poll the task to confirm success or
   surface task errors. If a shutdown hangs (a VM ignoring ACPI),
   the UI will just show it still running on the next refresh with
   no explanation. Task-status polling is a sensible follow-up.

4. **The token secret is plaintext in `config.json`.** Covered
   above under security notes — flagging it again here because it
   is the single most important thing to be aware of. The 0600
   file mode is the only protection. If that's not acceptable for
   your environment, the integration shouldn't be enabled until a
   vault-backed version exists.

5. **Single node only.** No cluster support. If you point this at
   one node of a cluster it will only ever see that node's guests.

6. **No VM/LXC detail view, no console, no creation.** This
   release is list + start + shutdown. No per-guest detail modal,
   no noVNC console, no create/clone/delete. Those are all
   deliberate scope cuts for a first release.
