# v5.5.0 — "ScaleMatters"

The persistent-tier + enterprise release. A large opt-in **enterprise-hardening**
program, the **keystone** that lifts the CGI/fork-per-request scale ceiling, **hard
multi-tenancy** (with optional Postgres row-level security), plus the helpdesk-signal
and billing polish below. Everything new is opt-in and **default-off — no breaking
changes.**

## Scale & isolation (keystone)

- **Persistent WSGI app server (opt-in, experimental).** Run the same `api.py` under
  **gunicorn** with threaded workers and thread-local request isolation instead of
  fork-per-request CGI — pre-warmed workers for large fleets. CGI stays the default
  and fallback. Unit: `remotepower-wsgi.service`; see [wsgi.md](wsgi.md) and
  [scaling.md](scaling.md).
- **Out-of-band maintenance scheduler (opt-in).** `remotepower-scheduler.service`
  runs the maintenance cadence from one **leader-elected** process (host file-lock +
  Postgres `pg_advisory_lock`), off the request path — HA-safe and ~25× lower request
  latency on a networked Postgres backend. Pair with `RP_EXTERNAL_SCHEDULER=1`.
- **Hard multi-tenancy (opt-in).** `tenancy_enforced` confines tenant admins to their
  own devices; a default-tenant admin sees all. Flipped live from Settings → Security.
- **Postgres row-level security (opt-in, Postgres only).** `tenancy_rls` adds
  DB-enforced isolation on the `devices` table (`FORCE` RLS + a per-request GUC,
  fail-closed) beneath the app-layer scope — column/trigger/policy applied live, no
  migration script.

See [deployment.md](deployment.md) and [scaling.md](scaling.md) for the full
deployment model, and the in-app **Settings → Security / Advanced** toggles.

## AV / rootkit scan warnings now reach the Alerts inbox

Previously, an endpoint AV/rootkit scan that reported **warnings** (rkhunter
`[Warning]` lines, or a stale ClamAV signature database) only surfaced on the
dashboard **"Needs attention"** card — it never created an **Alert**, so it could
not be routed to a webhook/email channel, acknowledged or tracked. A host showing
"rkhunter reported 2 warning(s)" stayed invisible to anyone watching the inbox.

v5.5.0 adds a first-class **`av_warning`** event:

- **Edge-triggered** on the *rising* warning count (like `av_infected`), so a
  steady count doesn't re-fire every heartbeat; a first sighting on a never-seen
  host still alerts.
- **Coalesced per host** — a flood of warnings on one host is one open alert with a
  bumped count, not an inbox full of duplicates.
- **Medium** severity, routed through the existing **`av_posture`** channel kind.
- **Suppressed** for a tool that *also* reports an active infection this beat — the
  higher-severity `av_infected` already alerts, so there's no double-fire.

There is no recover event: a warning stays actionable until an operator reviews the
scan and the count drops on a fresh report. `WEBHOOK_EVENTS` is now **88**.

## Ticket attachments (inbound + outbound)

The ticket system handled **text only** — inbound email attachments were silently
dropped, and there was no way to send a file with a reply.

- **Inbound:** real attachments on customer emails are now extracted and stored
  (the text/plain body is still threaded as before).
- **Outbound:** an **Attach files** control on the reply box lets you send files
  with an email reply — up to **15 MB each** and **10 per message**. They ride the
  SMTP message as standard MIME attachments and are logged on the ticket.
- **View / download:** every attachment on a message shows a **download** button
  and, for images / PDF / plain text, an in-browser **preview**.

Security: attachment blobs are stored on disk under an **opaque random id** (never a
caller-supplied filename), download access is **bound to the owning ticket** (the id
must appear on one of that ticket's messages), and responses are served
`X-Content-Type-Options: nosniff` — as a download by default, inline only for a small
allowlist of safe types.

## Ticket auto-reply (loop-safe)

An optional, one-time **acknowledgement** can now be emailed automatically when a new
ticket is **auto-created from an inbound email** ("Thank you for contacting
support…"). Configure it under **Settings → Advanced → Tickets**.

It is **loop-safe by construction**:

- Stamped `Auto-Submitted: auto-replied` (so the inbound poller skips it) plus the
  `#RP<number>` / `X-RP-Ticket` marker.
- Sent **at most once per ticket** (only on creation, not on every reply).
- **Never** sent to no-reply / do-not-reply / mailer-daemon / postmaster / bounce /
  notification senders.

## "View email thread"

Every ticket now has a **View email thread** button that opens the full
correspondence in a clean, **printable** window (received / sent / internal notes,
with attachment lines). It is built entirely in the browser using DOM + CSSOM, so it
is fully CSP-safe and needs no new server endpoint.

## Billing is now opt-in (Settings → Advanced)

The **Billing** page (worksheet, invoices, rates & fees) is now gated behind a new
**Settings → Advanced** checkbox, exactly like the ticket system — off by default.
The **time-tracking** half stays available regardless: logging billable / internal
hours on tickets and the weekly **Timesheet** are unaffected. The billing API
handlers return `404` while billing is disabled.

## Devices page — ticket marker

When a host has an open ticket, the Devices page now shows the small **ticket glyph**
(the same icon as the Tickets nav) directly **in front of the hostname**, replacing
the previous "ticket" pill. Click it to open that host's tickets.

## Enterprise hardening batch

A first slice of the enterprise gap-analysis landed in this release — identity,
governance and accessibility checkboxes. **Everything is opt-in**: every new knob
defaults off, so existing installs and behaviour are unchanged until an admin
configures it under **Settings → Security**.

- **Password policy** — configurable minimum length, an optional "3 of 4 character
  classes" rule, and an optional **HaveIBeenPwned** breach check (k-anonymity: only the
  first 5 hex of the SHA-1 ever leaves the server, and it fails *open* if HIBP is
  unreachable). Enforced on new users and password changes; existing passwords are never
  invalidated. A `password_changed_at` timestamp is now recorded.
- **SSO-only** — when enabled and an IdP (OIDC/SAML) is configured, local-password
  logins are refused so the org can mandate IdP auth. Flag an emergency admin's user
  record with `local_login: true` as a break-glass exemption **before** turning it on.
- **Idle session timeout** — a session unused for `idle_timeout_minutes` expires even if
  its absolute TTL hasn't elapsed (sliding window off the activity stamp).
- **Config-change auditing** — every Settings save writes a `config_changed` audit entry
  listing the changed key *names* (values are never logged, so secrets can't leak). The
  most security-sensitive surface (MFA policy, IP allowlist, SSO/SCIM/LDAP, session caps)
  previously saved with no audit trace.
- **Webhook `schema_version`** — the generic-webhook and SIEM payload envelopes now carry
  a contract version (`"1"`) so downstream consumers can guard against shape drift.
- **`max_devices` cap** — new enrollments are refused past a configurable fleet limit
  (existing devices and re-enrollment are never blocked).
- **Localization** — dates, numbers and **billing currency** now follow the in-app
  language picker via `Intl` (previously the browser locale, and billing rendered a bare
  `USD 1,234.00`). A Spanish/Arabic/Hindi/Chinese UI gets matching dates and money.
- **Accessibility (WCAG 2.1)** — toast notifications are now an `aria-live` status region
  (announced to screen readers), the active navigation item carries `aria-current="page"`,
  the device drawer gained a keyboard focus-trap + focus-restore + `aria-labelledby`, and
  a skip-to-content link was added.

A second batch in the same release continues into credential-at-rest, the API
contract layer, and control-plane observability:

- **API keys hashed at rest** — API keys are now stored as a **SHA-256 hash**, so a
  datastore read no longer yields a reusable bearer secret. The raw key is shown once
  at creation; an existing plaintext key keeps working and is transparently migrated
  to a hash on first use. (Hashing device tokens and enrollment tokens — which carry
  more migration/UX coupling — is a deliberate later step.)
- **API versioning** — every endpoint is also reachable under **`/api/v1/...`**, a
  permanent alias of the unversioned path. Integrators can pin a versioned base URL
  today; future breaking changes can land under `/api/v2` without disturbing v1.
- **Correlation IDs** — every JSON response carries an **`X-Request-Id`** header
  (an inbound id from a trusted proxy is honoured, otherwise one is minted). A
  `log_json()` structured-logging helper (gated by `RP_LOG_LEVEL`) and the
  slow-handler ring both carry the same id, so a log line, a slow request and an
  audit lookup can be joined.
- **Frontend error reporting** — uncaught browser errors (`window.onerror` and
  unhandled promise rejections) are beaconed to **`/api/client-error`** (throttled to
  ≤1/5 s and ≤20/page, field-capped, IP-rate-limited, capped ring) so client-side
  failures surface to operators instead of dying silently in a console.

- **Branded notification email** — alert, fleet-report/digest and SMTP-test emails now
  carry a branded HTML alternative (honouring the white-label `brand_name` +
  `brand_accent`) alongside the existing plain-text body, which remains the fallback.
  Email HTML uses inline styles by necessity; it never renders in the app, so it's
  unrelated to the app's CSP.

- **List API convention** — list endpoints now share a documented, backward-compatible
  superset: optional `?q=<substr>` substring filter, `?sort=<field>&order=asc|desc`,
  `?limit`/`?offset`, and `?meta=1` to receive a `{items, total, limit, offset, next}`
  envelope instead of a bare slice. With **none** of these query params the response is
  byte-for-byte the previous bare list, so existing clients are unaffected. Applied to
  the audit log, command history, API-keys and client-errors lists.
- **Signed exports** — the compliance **evidence pack** (`GET /api/report/evidence`) now
  embeds an HMAC-SHA256 `signature` over its canonical contents (verify by recomputing
  over the pack minus the `signature` field), and the **audit-log archive** download
  emits an `X-RP-Signature: hmac-sha256=<hex>` header over the file bytes. Both are keyed
  by a new per-install `export_sign.key` (0600, separate from the audit-chain key), so a
  tampered export is detectable. (HMAC is symmetric — it proves the export wasn't altered
  after generation; a WORM forward sink for full third-party non-repudiation is a
  documented follow-up.)

Further batches in the same release completed the rest of the **bounded** enterprise
gap-analysis (DR depth, governance, identity, supply-chain, observability) — still all
opt-in / backward-compatible:

- **Export-key rotation + posture visibility** — an admin can rotate the export-signing
  key (Settings → Security), and the **Security posture** page now grades the new controls
  (password policy, idle timeout, SSO-only, signed exports, encrypted config secrets,
  off-host backups, WORM sink) so the governance surface is auditable at a glance.
- **OpenAPI now covers the whole API** — the spec (`/api/openapi.json`, Swagger UI) is
  driven from the live route table, so every registered endpoint appears (~290 paths, up
  from ~28); the hand-written rich specs are kept and the document advertises the
  versioned **`/api/v1`** base.
- **Postman collection** — `make postman` generates a Postman v2.1 collection from the
  spec (one request per endpoint, foldered by tag, `X-Token` + `{{baseUrl}}` pre-wired).
- **App-self SBOM + SLSA provenance** — `make sbom-self` emits a CycloneDX SBOM of the
  control plane's **own** Python dependencies, and the release workflow attaches SLSA
  build provenance to the published images.
- **Off-host backups + restore-verify (DR)** — the DR backup can be mirrored to an
  off-host destination (`backup.offsite_dir`), and **Test restore**
  (`POST /api/backup/test-restore`) decrypts + decompresses + structure-checks the latest
  archive to prove it is restorable.
- **Per-tier escalation targets** — an escalation tier can page one specific webhook
  destination (by name or id) instead of re-notifying every channel, so tier 1 hits Slack
  while tier 3 pages PagerDuty / a manager.
- **Availability SLO + error budgets** — a configurable `slo_target_percent` drives a
  per-monitor availability / error-budget / burn-rate computation at `GET /api/slo` and as
  Prometheus gauges for Grafana SLO dashboards and burn-rate alerts.
- **Control-plane uptime** — RemotePower records its own observed availability (hourly
  "served a request" buckets) over 24 h / 7 d / 30 d at `GET /api/self-test` and as a
  Prometheus gauge; the denominator starts at the first tracked hour (never counts
  pre-deployment time), and a gap is labelled honestly as downtime *or* a quiet hour.
- **Guided onboarding tour** — a first-run, CSP-safe coach-mark walkthrough (Dashboard /
  Devices / Alerts / Search / Settings); shows once per account and is re-runnable via a
  **Take a tour** button on the docs page.
- **Credential-at-rest completed** — device auth tokens and one-time enrollment tokens
  now join API keys in being stored as **SHA-256 hashes**, not plaintext. Agents are
  unchanged (they keep sending their plaintext token); the server compares hashes, accepts
  a legacy plaintext token, and migrates it on the device's next heartbeat. Together with
  **config-secret encryption** (opt-in `RP_CONFIG_KEY` → AES-256-GCM for the sensitive
  `config.json` fields, transparent at load/save) every bearer secret on disk is now
  hashed or encrypted.
- **External key sourcing (Vault/KMS)** — `RP_CONFIG_KEY` and `RP_BACKUP_PASSPHRASE` can
  be fetched from an external command via a `<NAME>_CMD` env var (e.g.
  `vault kv get -field=key …`) instead of living in the process environment.
- **Append-only (WORM) audit sink** — point `audit_worm_path` at an operator-immutable
  file (`chattr +a` / a WORM mount) and every hash-chained audit entry is also appended
  there — a tamper-resistant copy independent of the live, bounded log.
- **SSO group → role matrix** — `sso_group_roles` maps an OIDC/SAML group to **any**
  builtin or custom role (admin / auditor / a custom finance role / …), not just the prior
  admin-or-viewer; admin wins on a multi-group match, the legacy admin-group still maps to
  admin, a viewer is promoted to their mapped role on login, and an existing admin /
  custom-role user is never auto-demoted.
- **Distributed trace-context** — an inbound W3C `traceparent` is honoured so the
  structured logs (`trace_id`) and outbound webhooks join the same distributed trace.
- **Request-lifecycle isolation (keystone Stage A)** — `_begin_request()` /
  `_end_request()` reset the per-request process-local state at request boundaries. A
  no-op under today's CGI model, but the foundation that makes a future persistent app
  server safe (so one request can never see another's state).

What is **not** in this release, by design: the persistent app-server keystone
(CGI → WSGI) and hard multi-tenancy are architectural changes scoped in the internal
enterprise plan for a deliberate later major (v6.x); a few "nice-to-have" identity items
(per-key device scope, conditional access, object-level ACLs) and the full data-layer
i18n / design-system token work remain on the backlog.
