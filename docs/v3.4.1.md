# RemotePower v3.4.1 — Release Notes

**Status:** unreleased (dev)

## Overview

**Theme: bind it together.** v3.4.0 grew the fleet outward — health probes,
forecasting, compliance, discovery. v3.4.1 connects those surfaces. Three
cohesion features turn a stack of separate pages into one story, and the
command palette stitches the navigation together.

## Timeline (whole-fleet or per-device)

A new **Monitoring → Timeline** page gives you a single chronological history —
for the **whole fleet** or **one host**, chosen with the scope selector (it
defaults to the fleet view). It merges two streams that previously lived apart:

- **Fleet events** — every event that fired for the device (offline/online,
  config drift, new CVEs, service up/down, SMART failures, brute-force,
  new ports, …), with the event's severity.
- **Command runs** — the commands and scripts executed on the host, with their
  exit status. These are *not* fleet events, so a per-host history never showed
  them before this release.

Choose the scope from the selector (Whole fleet, or any device), filter by
category with the chips, and read top-down. In the fleet view each row is tagged
with its device, and clicking that tag opens the device's own timeline. Severity
uses the same pill vocabulary as the rest of the app. You can deep-link straight
to a device's timeline from the command palette or the home health panel.

**API:** both views share one merge core.
- `GET /api/fleet/timeline?limit=<n>&kinds=<csv>&device=<id>&severity=<csv>` —
  all monitored hosts; returns `{items, kinds, devices, total}`.
- `GET /api/devices/<id>/timeline?limit=<n>&kinds=<csv>` — one host; returns
  `{device_id, device_name, items, kinds, total}`.

Items are `{ts, kind, severity, title, detail, device_id, device_name}`, newest
first.

## Fleet health score

The home dashboard gains a **Fleet health** panel: a single 0–100 score and a
letter grade (good / fair / poor / critical), a severity breakdown, and the
lowest-scoring devices — each a click-through into its timeline.

The score is a weighted rollup of the **same Needs Attention signals** the rest
of the app already computes (offline, patches, CVEs, drift, disk, services,
certs, …). Because it shares that one source of truth, the score can never
disagree with the Needs Attention list. Each device starts at 100 and loses
points per attention item by severity; the fleet score is the mean of the
per-device scores. Only monitored devices count.

**API:** `GET /api/fleet/health` — `{score, grade, devices: [...], counts,
total_devices}`. Also embedded in the `/api/home` bundle, so the dashboard panel
costs no extra request (it reuses the cached attention digest).

### Health-score history & alerts

The score is **sampled once per UTC day** (fleet-wide and per device) into a
time series, rendered as a small **trend sparkline** on the home health panel.
`GET /api/fleet/health/history` returns the fleet series; add `?device=<id>` for
one host.

An **opt-in alert threshold** lives at **Settings → Dashboard → Health-score
alerts** (`0` = off). When a device's score drops below it, RemotePower fires a
new edge-triggered **`health_degraded`** event — once per crossing, not every
heartbeat — and a **`health_recovered`** follow-up that auto-resolves the alert
when the score climbs back. Severity scales with the score (critical < 40,
high < 70, else medium). Like every event, it flows through the channel-routing
matrix (Recent Activity / Alerts / webhook); clicking it in the feed opens the
device's timeline.

## Fleet posture Report + scheduled email

A new **Planning → Reports** page produces one report that binds **patches,
CVEs, the health score, and compliance** into a single artifact:

- **On demand:** download as **JSON** or **CSV**.
- **Scheduled:** email the report on a **cron schedule** to a recipient list
  (or the configured notification recipients). Configuring the schedule is
  admin-only; delivery runs on the heartbeat with a once-per-minute lock so a
  busy fleet never double-sends.

**API:** `GET /api/report/fleet?format=json|csv`,
`GET /api/report/schedule`, `PUT /api/report/schedule`
(`{enabled, cron, recipients}`).

## Command palette, extended

`Ctrl/Cmd-K` (or `/`) now also indexes the new **Timeline**, **Reports**,
**Alerts**, **Compliance**, and **Forecast** pages; offers a
**"<device> — timeline"** jump for every host; a one-keystroke **fleet report
download**; and your **saved scripts**.

## Quick wins

Three small joins over data RemotePower already collects:

- **CVE ↔ patch cross-link.** The Patches page shows, per device, how many
  outstanding critical/high CVEs a pending patch would resolve (a badge that
  links to that device's CVE list). The patch report gains `cve_fixable` per
  device plus a `cve_fixable_total`. The signal is honest: the scanner already
  suppresses findings whose installed version is past the fix, so a remaining
  finding with a known `fixed_version` is genuinely "patch this to clear it".
- **Software inventory search.** A search card on the Patches page answers
  "which hosts run package X (older than version Y)?" over the collected package
  inventory. Name match is substring (or exact); the optional version filter is
  ecosystem-aware (dpkg ordering for Debian/Ubuntu, tuple compare elsewhere).
  `GET /api/inventory/search?q=&op=lt|le|eq|ge|gt&version=&exact=`.
- **End-of-life OS detection.** A built-in table of vendor end-of-support dates
  (Ubuntu, Debian, RHEL/CentOS/Rocky/Alma, Fedora) flags hosts past EOL — or
  within 90 days — in **Needs Attention** (so they lower the fleet health score)
  and as a **compliance** control (PCI 6.3.3b, HIPAA, SOC 2 CC7.1b). It reads the
  os-release ID/VERSION_ID the agent already sends at package-scan time, so
  there's no agent change; it falls back to parsing the OS name string.

## Integrations & notification control

- **Richer Prometheus metrics.** `GET /api/metrics` (status-token or session
  auth) now also exports `remotepower_fleet_health_score`,
  `remotepower_device_health_score{device=…}`,
  `remotepower_attention_items{severity=…}`,
  `remotepower_timeline_events_24h{event=…}`, and
  `remotepower_cve_fixable_total` — so the health score and recent activity are
  graphable in Grafana alongside the existing device/monitor metrics.
- **iCal feed.** `GET /api/schedule.ics?token=<status token>` publishes your
  scheduled jobs and maintenance windows as a calendar subscription. One-shot
  jobs and maintenance windows render exactly; recurring jobs map to
  daily/weekly/monthly `RRULE`s. Auth is the same long-lived status token used
  for `/api/status` and `/api/metrics`, so a calendar app that can't send
  headers can still subscribe.
- **Quiet hours.** Settings → Dashboard → Quiet hours holds non-critical
  webhook/email delivery during a daily window (which may cross midnight, e.g.
  22:00–07:00). The event is still recorded — it lands in the Alerts inbox and
  Recent Activity — only the outbound page is suppressed, and anything at or
  above the chosen severity (default Critical) always goes through.

## Upgrade notes

No migration required. The new endpoints are additive, the report schedule is
off until you enable it, and the health score is derived from existing data —
nothing new to collect on the agents.
