# RemotePower v3.4.2 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.4.2's headline feature is the **automation rules engine**: turn the events
RemotePower already fires into actions. "When event X happens on devices
matching Y, run a saved script and/or notify a destination." It composes three
things RemotePower already has — the event registry, channel routing, and saved
scripts — into a small rule model evaluated on every fired event.

## Automation rules (Admin → Automation)

A rule has three parts:

1. **Match** — one or more event types (e.g. `service_down`, `container_stopped`;
   blank = any), optional severities, and an optional device filter (group,
   tags, or a specific device id).
2. **Actions** — **run a saved script** on the event's device (auto-remediation,
   e.g. restart a unit when it goes down) and/or **notify** a specific webhook
   destination (escalate this particular event past its normal channel routing —
   handy for routing one event straight to PagerDuty/Opsgenie).
3. **Cooldown** — minimum seconds between firings of the same rule (default 60),
   so a flapping event can't hammer the action or create a loop.

**Where it runs.** Rules are evaluated inside the single event-dispatch point,
**after** the unmonitored-device guard — so automation never acts on a host
you've explicitly silenced — and **independently of** the notification channel
gates (automation is a separate concern from who gets paged). Running a script
goes through the normal command queue, so **device quarantine still blocks it**
at the dispatch chokepoint.

**Safety.** Mutations are admin-only and audited (`automation_rule_create` /
`_update` / `_delete`, and `rule_run_script` when an action fires). A rule bug
can never break event firing — evaluation is fully wrapped.

**API.**
- `GET /api/automation/rules` — list (any logged-in user).
- `POST /api/automation/rules` — create (admin).
- `PUT /api/automation/rules/<id>` — replace config (admin).
- `DELETE /api/automation/rules/<id>` — remove (admin).

Rule shape: `{id, name, enabled, match:{events[], severities[],
device_match:{device_id, group, tags[]}}, actions:[{type:'run_script',
script_id} | {type:'notify', dest_id}], cooldown_seconds}`.

## Upgrade notes

No migration. There are no rules until you create them, so behaviour is
unchanged until you opt in. Hard-reload once after upgrading to pick up the new
service-worker cache (`remotepower-shell-v3.4.2`).
