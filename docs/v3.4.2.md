# RemotePower v3.4.2 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.4.2's headline feature is the **automation rules engine**: turn the events
RemotePower already fires into actions. "When event X happens on devices
matching Y, run a saved script and/or notify a destination." It composes three
things RemotePower already has — the event registry, channel routing, and saved
scripts — into a small rule model evaluated on every fired event.

## Automation rules (Admin → Automation)

A rule has three parts:

1. **Match** — one or more event types (e.g. `service_down`, `container_stopped`;
   blank = any), optional severities, and an optional device filter (group,
   tags, or a specific device id).
2. **Actions** — **run a saved script** on the event's device (auto-remediation,
   e.g. restart a unit when it goes down) and/or **notify** a specific webhook
   destination (escalate this particular event past its normal channel routing —
   handy for routing one event straight to PagerDuty/Opsgenie).
3. **Cooldown** — minimum seconds between firings of the same rule (default 60),
   so a flapping event can't hammer the action or create a loop.

**Where it runs.** Rules are evaluated inside the single event-dispatch point,
**after** the unmonitored-device guard — so automation never acts on a host
you've explicitly silenced — and **independently of** the notification channel
gates (automation is a separate concern from who gets paged). Running a script
goes through the normal command queue, so **device quarantine still blocks it**
at the dispatch chokepoint.

**Safety.** Mutations are admin-only and audited (`automation_rule_create` /
`_update` / `_delete`, and `rule_run_script` when an action fires). A rule bug
can never break event firing — evaluation is fully wrapped.

**API.**
- `GET /api/automation/rules` — list (any logged-in user).
- `POST /api/automation/rules` — create (admin).
- `PUT /api/automation/rules/<id>` — replace config (admin).
- `DELETE /api/automation/rules/<id>` — remove (admin).

Rule shape: `{id, name, enabled, match:{events[], severities[],
device_match:{device_id, group, tags[]}}, actions:[{type:'run_script',
script_id} | {type:'notify', dest_id}], cooldown_seconds}`.

## Device dependency map

On the **Network Map**, a device can now declare logical **dependencies** —
"this host needs those upstreams to function" (e.g. an app server → its database
and the switch in front of it). This is distinct from the physical
`connected_to` link.

The payoff is **alert suppression**: when an upstream a device depends on is
offline, RemotePower **holds** that device's alerts — they're collateral, not
the root cause. You get the upstream's alert (the thing you can actually fix)
instead of a storm of downstream noise. Suppression is delivery-only (the event
is still recorded in the Alerts inbox and Recent Activity) and **never** holds a
recovery event, so a device coming back always clears.

The map overlays dependency edges as dashed violet lines (red when the upstream
is down). Edit dependencies in **Network Map → Edit links** (a multi-select per
device). New `PUT /api/devices/<id>/depends-on` (admin, audited); the
`/api/network-map` response gains `dep_edges` and per-node `depends_on`.

## Statistical resource anomalies

A **model-free** complement to the v3.4.0 AI anomaly scan. For each monitored
device, RemotePower fits a baseline (mean + standard deviation) over its daily
metric history and flags the **latest** memory / swap / disk reading when it
deviates more than `z` standard deviations (default 2.5) from that host's own
norm — comparing each host only to itself, so a box that always runs hot doesn't
cry wolf. A standard-deviation floor means a sharp jump off a flat baseline still
flags, while tiny blips stay quiet.

Surfaced as a **Resource anomalies** card on the Reports page. New
`GET /api/fleet/anomalies?z=N`, implemented in the pure, unit-tested module
`anomaly_stats.py` (no model, no network).

## Agent integrity attestation

RemotePower already verifies an agent's binary by sha256 when it self-updates.
v3.4.2 adds **continuous attestation**: every agent reports the sha256 of its
own running binary on each heartbeat, and the server compares it to the
canonical copy it serves.

- **verified** — the running hash matches the canonical published build.
- **mismatch** — the agent claims the current version but reports a *different*
  hash. That's tampering, corruption, or a half-finished update — a **critical**
  Needs-Attention item (so it also lowers the fleet health score).
- **unknown** — the agent hasn't reported a hash yet, or it's on an older
  version (the server only holds the canonical hash for the current build).

Surfaced as an **Agent integrity** card on the Reports page (counts + the
non-verified rows). New `GET /api/fleet/agent-integrity`. This is the detection
half of "signed binaries"; cryptographic release signing (detached signatures)
remains a packaging follow-up — the agent↔server attestation here already
catches a tampered or partially-updated agent in the field.

## Cryptographic release signing

The hash attestation above proves an agent matches the copy the **server** holds
— but if the server itself is compromised, an attacker could swap both the
binary and its advertised hash. Release signing closes that gap with asymmetric
crypto: a detached **GPG signature** the attacker can't forge without your
private key.

**Signing (release time, out-of-band).** Run `tools/sign-agent-release.sh -k
<key>` to produce a detached, armored signature next to the served binary
(`remotepower-agent.asc`). Then publish the **public** key + fingerprint to
RemotePower config (`release_pubkey`, `release_key_fingerprint`) so the server
can self-verify and advertise it.

**Verifying (agent, opt-in, fail-closed).** Pin the same public key on each
agent host at `/etc/remotepower/release.pub`. From then on, before installing
any self-update, the agent fetches the signature (`/api/agent/signature`) and
verifies it over the freshly-downloaded binary against the pinned key (and the
expected fingerprint) using an ephemeral `gpg` keyring. A missing, invalid, or
wrong-key signature **aborts the update**. With no key pinned, nothing changes —
the existing sha256 check still applies — so this is safe to roll out gradually.

The server's own view of the published signature (`valid` / `invalid` /
`unsigned` / `unverifiable`) appears as `release_signature` in the
`/api/fleet/agent-integrity` response. No Python crypto dependency — it shells
out to the ubiquitous `gpg` binary, the same way the agent already calls
`smartctl`, `dpkg`, etc.

This is the "signed binaries" half of the v3.4.1 roadmap's *agent integrity /
signed binaries* item; the hash attestation (above) is the detection half.

### Bake & sign in the UI (Admin → Release Signing)

For shops that want CheckMK-style convenience over the CI path, a server-side
mode: **Generate signing key** creates a server-held Ed25519 key, **Sign current
agent** produces the signature, and a toggle controls enforcement advertisement.
The page shows the public key to pin on agents (`/etc/remotepower/release.pub`)
and the live signed/valid status.

It's deliberately honest about the trade-off: the private key lives on the
server, so this protects against tampering of the published files *at rest*
(mirror/CDN), **not** a full server compromise — the page says so, and points to
the CI path for the strongest guarantee.

**Visibility:** an agent that refuses a self-update (missing/invalid/wrong-key
signature) reports the reason on its next heartbeat (`agent_update_rejected`),
and the Release Signing page lists those devices — so a tampered or
mis-keyed rollout surfaces instead of failing silently in the field.

## Fleet management additions

A batch of fleet-scale views inspired by mature management suites — all
read-only aggregations over data RemotePower already collects, plus one new
agent field (upgradable package names).

### Patch catalog (Patches page)

The device table answers "what does host X need?"; the patch catalog answers the
inverse — **"package X is pending on N hosts."** It aggregates
`upgradable_names` (a new, capped agent field) across monitored devices into a
sortable package → count table, so you can plan a fleet-wide rollout by package
rather than walking hosts. `GET /api/patch-catalog`.

### Post-deployment verification

Queuing an upgrade now snapshots the device's pending-update count and forces a
package re-scan. On the next report the Patches table shows whether the upgrade
**took** (pending dropped → ok), **stalled** (unchanged after an hour), or is
still pending. The marker ages out after 7 days.

### Software metering / license compliance (Reports page)

Define named software meters with an allowance; the report counts installs of
each across the fleet's package inventory and flags any that are
**over-deployed**. `GET /api/inventory/metering`.

### Fleet heat map (Home)

A grid of one cell per device, coloured by health score — green→amber→red. It
scales to hundreds of hosts where a table doesn't, and clicking a cell jumps to
the device.

### After-hours activity detection (Settings → Dashboard)

Pick events to watch (e.g. login, new listening port, command run) and a
business-hours window; any selected event firing **outside** those hours is
recorded and surfaced as a Needs-Attention item. Opt-in; nothing fires until you
enable it and choose events/hours.

### Ad-hoc fleet query (Fleet → Query)

Filter devices by ANDed conditions — group, tag, OS substring, online,
pending-update count, agent integrity, CVE count — and save the queries you
reuse (stored client-side). `GET /api/fleet/query`.

### Signed-agent badge (Devices)

Next to each device's version, a **green ✓** when the running binary's hash
matches the canonical signed build, or a **red ⚠** on a mismatch (tamper /
corruption / partial update). Hover shows the status and the short hash — the
per-device complement to the Reports-page integrity attestation above.

## Deployment & compliance additions

A second batch building on the fleet-management views — the orchestration and
compliance layer over data RemotePower already collects.

### Staged / ring rollouts (Planning → Rollouts)

Push one action — an OS package upgrade, or a saved script — to the fleet in
ordered **rings** (canary → pilot → broad). Ring N is dispatched and watched;
**upgrades use the post-deploy verification above** (pending count must drop),
so a ring is only "done" once its devices actually took the change. Then ring
N+1 is released — automatically (`auto_promote`) or when you click **Promote**.
A ring that verifies on *none* of its devices within the verify window **halts**
the rollout (`failed`) instead of pressing on. Pause / resume / cancel any time;
each rollout keeps a history. Scripts can only be confirmed *delivered* (not
verified for success) — upgrades get true verification. `GET`/`POST
/api/rollouts`, `POST /api/rollouts/<id>/<start|pause|resume|cancel|promote>`,
`DELETE /api/rollouts/<id>`.

### Maintenance change-windows

A maintenance window already suppresses alerts during disruptive work; it can now
also **gate command/upgrade execution**. Tick *"Gate command/upgrade execution
to this window"* and any exec/upgrade command for a covered device (by
device/group/global scope) is **held at the dispatch chokepoint** until one of
its gating windows is active — then released. This is opt-in per window and inert
by default, and pairs with staged rollouts so changes only land in an approved
change window. (`poll_interval` is never gated.)

### CIS-style compliance baseline (Compliance page)

A named set of pass/fail **checks** evaluated against each host's reported state:
no pending updates, no pending reboot, no failed units, root disk < 90%, swap <
80%, no critical/high CVEs, agent binary matches the signed build. Each check is
`pass` / `fail` / `n.a.` (no data), and the fleet score is the **severity-
weighted** pass ratio over applicable checks. The card shows the score, a daily
**trend** sparkline, per-check pass/fail/N-A counts with failing-host lists, and
a per-check enable/disable toggle (disabled checks are stored in config and
excluded fleet-wide). No agent change — checks reuse existing telemetry.
`GET /api/compliance/baseline`.

### Software metering: normalization + reclamation

The v3.4.2 metering now takes optional **aliases** so name variants fold onto one
catalog entry — `openssl = 0 | libssl3, openssl-libs`. And it flags
**reclamation candidates**: hosts where the software is installed but *not seen
running* (no matching process or listening-port owner), i.e. likely reclaimable
licenses. Judged only where process/port telemetry exists, so it never guesses.

### Print-friendly PDF report (Reports page)

**Print / Save as PDF** opens a clean, self-contained posture report — health
score, device counts, pending patches, CVEs, compliance frameworks and the CIS
baseline — and triggers the browser's native print dialog (which includes "Save
as PDF"). Zero new dependency.

## Access control: granular RBAC

Until now access was binary — `admin` (full control) or `viewer` (read-only).
v3.4.2 adds **custom roles** on top of those built-ins. A custom role grants a
chosen subset of action permissions and limits them to a device **scope**:

- **Permissions:** `exec` (run commands / batch scripts), `reboot` (reboot /
  shutdown), `upgrade` (OS package upgrade / update). Read access (browsing the
  dashboard) is implicit for any logged-in user.
- **Scope:** `all` devices, or only those in named **groups**, or only those
  carrying named **tags**.

A user assigned a custom role can act only on devices inside its scope — the
command, exec-batch, reboot/shutdown and upgrade endpoints all enforce it at the
single dispatch chokepoint via `require_perm`, and the Devices roster is filtered
to the role's scope. Everything administrative — server config, user/role/API-key
management, saved-script editing, rollouts — remains **admin-only**.

Manage roles under **Users & Roles**: `GET/POST /api/roles`,
`PUT/DELETE /api/roles/<name>`. Assign a role when creating a user (the role
dropdown lists your custom roles).

**Read scoping (RBAC v2).** Scoping now covers *reads*, not just actions. A single
dispatch-level guard blocks a scoped role from any `/api/devices/<id>/…` request
for an out-of-scope device (403), the few per-device endpoints outside that prefix
(`/api/cmdb/<id>`, `/api/acme/<id>/…`, `/api/patch-report/device/<id>`, the
per-device health-history series) carry the same guard, and the fleet aggregates
that emit per-device rows — patch report/catalog, CVE findings, fleet
query/timeline/events/health/SLA/capacity/anomalies, agent integrity, drift,
containers, inventory search/metering, compliance baseline, network map, log
rules, the alerts inbox — filter their rows to the caller's scope. A scoped
operator sees and touches only their groups/tags.

The guard is fail-safe: it only blocks when the path's first segment is a *real*
out-of-scope device, so non-device sub-paths and unknown ids fall through to the
handler unchanged. **Remaining fleet-level surfaces (by design):** the fleet
posture report (`/api/report/fleet`) and the Prometheus endpoint (`/api/metrics`,
which uses a separate monitor token) return aggregate *counts*, not per-device
identification, and are intentionally fleet-wide. Built-in admin/viewer behaviour
is unchanged.

## Linux-RMM additions

A batch deepening the self-hosted, Linux-first story — the direction picked over
chasing a general Windows/MSP RMM.

### OpenSCAP deep compliance scans (Compliance page)

The lightweight CIS baseline scores what every heartbeat already reports; this is
the **auditor-grade** complement. Queue a scan with a profile (`cis`, `stig`,
`pci-dss`, …); the agent runs `oscap xccdf eval` against its SCAP Security Guide
datastream and POSTs a compact result — overall **score**, pass/fail tallies, and
the failing rule ids. oscap runs **on the endpoint**, so there is no new server
dependency; a host without `openscap-scanner` + `scap-security-guide` reports
"not available" with the reason instead of failing. Scans run in a background
thread (oscap takes minutes) and land on the next heartbeat. `POST /api/scap/scan`
(profile + targets, `upgrade` permission), `GET /api/scap` (fleet results).

### Third-party (non-OS) patch detection

The agent now reports available updates from **flatpak, snap, pip and npm** in
addition to the OS package manager. The patch catalog aggregates them by manager
("flatpak: N hosts, these apps"), so the catalog reflects the *whole* update
surface, not just apt/dnf/pacman.

### On-call & escalation (Settings → Notifications)

An unacknowledged alert should get louder. Define an **escalation policy** — tiers
of "after N minutes unacknowledged, re-notify" — and an **on-call rotation** (a
list of contacts, each on call for a configurable number of days). A periodic
tick re-fires any open, unacknowledged critical/high alert that has aged past a
tier it hasn't hit yet, through the *same* channels the original alert used, with
the current on-call person named in the message. Idempotent per (alert, tier).
`GET /api/oncall`.

### Trends (Planning → Trends)

Zero-dependency multi-series SVG line charts over the daily samples already kept:
**fleet health score**, **fleet compliance %**, and a per-device **memory / swap /
disk** history (pick a device). Axes, gridlines and a legend, all hand-rolled SVG
— no charting library. `GET /api/devices/<id>/metrics-history`.

## Operations, onboarding & UX

### Install software from repos (by host or tag/group)

A new **Install software** action (Patches page) installs one or more packages
from the host's own repositories on a single device or across a whole group/tag.
It detects the package manager (apt / dnf / yum / zypper / pacman / apk) at run
time. Package names are the parameter and are strictly validated (letters,
digits, `. _ + : @ / -` only) so they can't inject shell; the action is gated on
the `exec` permission and honours quarantine + change-windows like any queued
command. `POST /api/install` ({device_id|device_ids|tag|group, packages}).

### OpenSCAP scans by tag/group

The OpenSCAP scan form now has a target-type selector (all / group / tag /
device) — so you can scan an entire tag in one go — with the SCAP profile as the
parameter. (The endpoint already accepted tag/group targets; this surfaces it.)

### Settings → Install (getting-started checklist)

A live onboarding checklist that reflects current state: set the admin password,
enroll the first device, configure a notification channel, enable scheduled
backups, turn on 2FA. Each step shows done/todo and links straight to where to do
it. `GET /api/setup-status`.

### Smaller things

- **Signing the agent release re-verifies the admin password** (mirrors disabling
  enforcement) — it's the trust root for every agent.
- **Fleet Query** gained many filters: agent version, package manager,
  has-package, monitored / agentless / quarantined, reboot-required, failed
  units, kernel-outdated, third-party updates, disk % / memory % thresholds, and
  offline-for-N-days.
- **Settings re-categorized**: CVE details cache → Security; Status endpoint →
  Integrations; the duplicate raw-seconds "Session timeout" card consolidated
  into Security's "Session length".

## Security hardening

RBAC v2 added per-device read scoping with a dispatch-level guard, but that guard
only fires for `/api/devices/<id>` paths. A handful of per-device routes are
addressed under other prefixes and weren't covered — now they are:

- **Mitigate** (`POST /api/mitigate/<id>/investigate` and `/fix`) require the
  `exec` permission on an *in-scope* device. These endpoints queue a command on
  the host, so they're an action, not a read: a read-only viewer or an
  out-of-scope operator can no longer kick off a diagnostic or a remediation run
  on a device outside their scope. Previously they only required authentication.
- **ACME action log** (`GET /api/acme/<id>/log/<action>`) is scope-checked, so a
  scoped role can't read another host's certificate-action output.
- **Batch / install job tracker** (`GET /api/exec/batch`, `/api/exec/batch/<id>`)
  filters per-host status, return codes, command output and job labels (which can
  name installed packages) to the caller's in-scope devices, and omits jobs that
  touch no in-scope device. Admin / all-scope callers are unaffected.

Plus the **patch report CSV** (`GET /api/patch-report.csv`) neutralizes
spreadsheet formula injection — a cell beginning with `=`, `+`, `-`, `@`, tab or
CR is prefixed with a single quote so it opens as literal text rather than an
executable formula in Excel / Sheets / LibreOffice.

**OIDC SSRF guard.** The OIDC login flow makes two back-channel HTTP fetches —
the issuer's discovery document and the token endpoint it advertises. The issuer
is admin-set, but the token endpoint comes from the discovery document, so a
hostile or compromised IdP could point it at the cloud-metadata service. Both
fetches now reject non-`http(s)` URLs and any target that resolves to a
link-local / metadata / unspecified address (e.g. `169.254.169.254`). RFC1918 and
loopback remain allowed so internal and dev IdPs keep working — the same policy
the webhook sender uses.

**Install tracker — no more stuck "pending".** Re-issuing an identical software
install while the first one was still in flight could de-duplicate the command
(it's already queued) yet create a second job whose run had effectively already
happened — leaving it "pending" forever because no output arrived after the new
job was created. The tracker now resolves such a job from the most recent run of
that command once it's no longer queued, while a command still waiting to run
stays pending exactly as before.

## Device detail — completeness ("bind it together")

The device drawer's **System Info** tab now shows data the agent already reports
but the detail view used to drop:

- **Top processes** — the host's busiest processes by CPU (PID / name / CPU% /
  mem%), the same data shown fleet-wide on the Processes page.
- **Filesystem type** per mount (ext4 / xfs / zfs / btrfs / nfs …).
- **Reboot required** — an amber indicator and the *reason* (the packages that
  asked for it), right on the device, not just on the home attention feed.
- **Container age** — an Age column in the per-device container list.

Two fixes came with it: the **Load avg** pill was reading the wrong field and was
always blank (now shows the 1-minute load average), and the **Fleet Query**
results table wires its sort headers eagerly and no longer throws on a malformed
response.

## Upgrade notes

No migration. There are no rules, rollouts, gating windows, baselines, custom
roles, escalation policies or scheduled scans in effect until you create/enable
them, so behaviour is unchanged until you opt in. OpenSCAP and third-party patch
detection need the relevant tools on the host (`oscap`, `flatpak`, `snap`, `pip`,
`npm`); where they're absent the feature is simply a clean no-op.
Hard-reload once after upgrading to pick up the new service-worker cache
(`remotepower-shell-v3.4.2`). Agents pick up the new upgradable-package reporting
on their next self-update; until then the patch catalog shows them under "no
detail yet".
