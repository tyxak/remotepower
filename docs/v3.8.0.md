# RemotePower v3.8.0 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.8.0 is a consolidation release: a security pass over the v3.5–v3.7 feature
work, a "bind it together" sweep that wires dropped agent data into the UI, more
AI-investigate coverage, and polish/consistency fixes. No new headline features.

## Security

- **Change approval respects the exec allowlist/denylist.** Previously, enabling
  maker-checker change approval routed an arbitrary command through the
  confirmation queue *without* the per-device `allowed_commands` allowlist or the
  catastrophic-command denylist that the immediate-exec path enforces. Both are
  now applied when the command is submitted and again when it's approved.
- **Ansible runner inventory injection closed.** The inventory host alias is now
  a sanitised token (a device name with a space/newline could otherwise inject
  Ansible host-vars or whole inventory lines), and the SSH password is written to
  a 0600 JSON extra-vars file rather than interpolated into the INI. The runner
  also skips quarantined devices (it reaches hosts over SSH, outside the agent
  command queue where quarantine is otherwise enforced).
- **2FA recovery codes consume atomically** under a file lock — two concurrent
  logins can no longer both redeem the same one-time code.
- **Audit-log forwarding SSRF hardening** — the HTTP forwarder no longer follows
  redirects (a 3xx can't bounce it to a cloud-metadata/loopback address past the
  SSRF guard), and the syslog destination host is SSRF-checked too.
- **SFTP upload** checks the encoded size before base64-decoding, so an oversized
  upload can't force a large in-memory decode.

## Fixes

- The delete buttons on the Sites / Auto-patch / Backups / Ansible admin pages
  coerce their IDs to strings — an opaque token that happened to be all digits
  could otherwise be numerically mangled and target the wrong/no record.
- **RAID member disks render again** in the device drawer's Health & Hardware
  card. The agent emits the member list as a space-separated string; the UI
  assumed an array and silently dropped it (so the "which disk to replace" hint
  was blank).

## Bind it together

- **Boot reason** is now stored and displayed. The agent already reported the
  command that preceded a restart (e.g. `reboot`, `self-update`) on the first
  heartbeat after boot, but the server dropped it. It's now persisted and shown
  in the device drawer's System Info tab — useful provenance for "why did this
  box restart".

## AI investigate (broadened)

The **Investigate** button now covers most of what the dashboard surfaces — nine
new diagnose-and-suggest playbooks join the existing ones (disk, memory, swap,
cpu, patches, cve, drift, service-down, container, reboot, brute-force), for
about 20 kinds total. Each runs a bounded, read-only diagnostic on the host and
feeds a tailored AI prompt:

- **Malware / AV posture** — ClamAV signature-DB age + recent rkhunter warnings.
- **Stale agent version** — running agent + service status.
- **End-of-life OS** — `/etc/os-release`, kernel, and the release-upgrade path.
- **Hardware health** — per-disk SMART health/attributes + disk errors in dmesg.
- **Backup stale / missing** — free space, recent backup files, cron/timers.
- **New SSH authorized key** — authorized_keys, recent logins, accepted-key log.
- **New listening port** — listening sockets with owning process.
- **Agent integrity** — binary hash/listing + service status (tamper triage).
- **Log-pattern alert** — recent error-level journal + failed units.

Security-sensitive playbooks (new key, new port, agent integrity) are
deliberately cautious — they recommend investigation and never propose a blind
destructive command.

## Polish

- The **Audit log forwarding** and **Change approval** settings sections moved
  from Settings → Advanced to **Settings → Security**, where they belong.
- The **Confirmations** page (left sidebar) is renamed from "MCP Confirmations",
  since it now also holds change-approval requests, not just MCP write actions.
- The nginx template's security headers (`Referrer-Policy`,
  `Permissions-Policy`, CSP) were reviewed and confirmed current.

## Upgrade notes

- No schema migration. The agent is bumped to 3.8.0 for lockstep; the only agent
  behaviour relevant here (boot_reason) already shipped — this release just stops
  the server from discarding it.
- The repository's CSP is clean: no inline event handlers, no `eval`, no inline
  `<style>`/`style=` in templates, no external CDNs. (Programmatic
  `element.style` assignments are CSSOM and are not subject to `style-src`.)
