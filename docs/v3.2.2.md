# RemotePower v3.2.2 — Release Notes

**Released:** 2026-05-27

## Overview

v3.2.2 is a hotfix release on top of v3.2.1. No new features. All
changes are bug fixes and configuration corrections surfaced by
real-fleet debugging on a v3.2.1 deployment.

No schema migrations. Drop in the new tarball, reload nginx, hard-reload
the browser once to refresh the service-worker cache
(`remotepower-shell-v3.2.1` → `remotepower-shell-v3.2.2`).

**If devices were flapping due to a low `online_ttl`:** after deploying,
go to **Settings → General** and set `online_ttl` to `300`. The race
condition fix is in this release but the TTL value lives in your config.

## Bug fixes

### Race condition in offline/online detection

`check_offline_webhooks()` runs on every API request to detect
device-offline transitions. It reads `offline_notified` from
`config.json`, checks each device, and writes the updated state back.
Under concurrent CGI load (two or more heartbeats arriving at the same
millisecond), two processes could both read `offline_notified = {}`,
both conclude the device was newly offline, and both fire
`device_offline` — producing duplicate alerts, duplicate fleet events,
and duplicate webhook deliveries.

**Fix:** the read-check-write sequence for `offline_notified` and
`patch_alerted` is now serialised inside `_LockedUpdate(CONFIG_FILE)`.
Webhooks and uptime recording happen after the lock releases, so the
locked section stays tight. Duplicate events are no longer possible.

### Debug log timestamp skew

The unified debug.log mixes server-side entries (heartbeat lines) and
browser-side entries (UI `dbg()` calls). Server-side entries used
`datetime.now()` — local time — while the browser always emits UTC via
`new Date().toISOString()`. On a CEST server the two streams were 2
hours apart, making cross-referencing the timeline impractical.

**Fix:** both server-side write sites now emit UTC via
`datetime.now(timezone.utc)`.

### Misleading `online_ttl` comment

The source comment illustrating `online_ttl` used `{"online_ttl": 180}`
as the example value. In v3.2.1, `MIN_ONLINE_TTL` dropped from 300 to
150. Previously a configured value of 180 was clamped to 300 (harmless);
after the bump it took effect. At a 60-second heartbeat interval, 180 s
means just 3 missed polls trigger an offline alert — causing false
positives on any LAN with brief congestion.

**Fix:** example value changed to 300 (the default) in the comment.

### `MAX_FLEET_EVENTS` too small for active fleets

The fleet event log cap was 200, sized for "quiet fleets" per its own
comment. With the race condition above firing 2–3 duplicate events per
device per offline cycle, and multiple devices flapping every 3 minutes,
the cap rolled over in under an hour — pushing legitimate events (patch
alerts, drift detections) off the Home dashboard activity panel.

**Fix:** raised to 1000.

### nginx `client_max_body_size` not updated when server limit was raised

In v1.7.0 the server-side body limit was raised from 64 KB to 50 MB to
support large package-list uploads (`/api/packages`). The nginx configs
(`server/conf/remotepower.conf` and `docker/nginx-docker.conf`) were
not updated. Systems with approximately 850 or more installed packages
were receiving a silent HTTP 413 on every `/api/packages` POST, leaving
the CVE scanner with a stale or empty package list.

At ~76 bytes of JSON per package entry, 850 packages ≈ 65 KB — just
over the old 64 KB limit.

**Fix:** `client_max_body_size` raised to `2m` in both configs. 2 MB
matches the largest legitimate payload: the per-device log buffer cap
(`MAX_LOG_BUFFER_BYTES = 2 MB`).

### nginx rate-limit comment used wrong rate for LAN fleets

The commented-out example zone used `rate=30r/m` (30 requests per
minute per client IP). On a LAN where all fleet devices share one NAT
address, normal operation already produces ≥17 req/min (5 devices ×
1 heartbeat + browser polling every 5 s). Enabling the limit as-is
would immediately throttle the fleet.

**Fix:** example updated to `rate=10r/s` with an explanatory note about
NAT/LAN deployments.

## UI improvements

### Sortable column headers

The following tables now support click-to-sort column headers.
Shift-click for multi-column sort; sort state persists in UI prefs.

- Alerts inbox
- CMDB asset list
- Configuration Drift
- ACME Certificates
- Webhook log
- MCP Confirmations
- Inbound webhooks
- Scripts
- Listening Ports

### Drift table header corrected

The Drift table thead had 4 columns (Drift, Missing, Last check,
actions) while the tbody rendered 7 cells per row. The Device, Group,
and Files watched columns were missing from the header, causing
misaligned column widths and no sort targets for those fields.

## Test coverage

Tests added for every fix above: `MAX_FLEET_EVENTS` floor assertion,
`client_max_body_size` not-64k assertion, and the full
`test_v322.TestVersionBumps` strict pin suite. `test_v321.py` pins
loosen to regex following the project convention.

## Upgrade

```
make dist          # builds dist/remotepower-3.2.2.tar.gz
```

Deploy the tarball, reload nginx, hard-reload the browser once.
