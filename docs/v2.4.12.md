# v2.4.12 release notes

Released: 2026-05-18.

Three dashboard fixes and an asset cache-busting change that
prevents stale-frontend problems after an in-place upgrade.

## Fixed

### Needs Attention CVE items led to a blank page

Clicking a CVE item in the dashboard's Needs Attention digest
opened a blank page. `_renderHomeAttention` routed the CVE kind to
a page named `cves`, but the page element id is `cve` — so
`showPage('cves')` matched nothing and no page became active.
Clicking the same kind of item from Recent Activity worked,
because that path used the correct name. The digest's page map is
now correct, and a test cross-checks every page it can route to
against the `page-<name>` elements that actually exist.

### "Devices online" tile counted unmonitored devices

The dashboard tile showed totals like `8 / 9` including devices
set to `monitored: false`. Unmonitored devices are deliberately
silenced everywhere else — the attention digest, the fleet roster,
the alert pipeline — so counting them here was inconsistent and
made the fleet look bigger (and a silenced host show as
"offline"). The tile now counts only monitored devices.

## Added

### Virtualization search box

The Virtualization page now has a search box that filters the
QEMU VM list by name or VMID, client-side — no extra Proxmox API
calls. Useful once a node has more than a handful of VMs.

### Asset cache-busting

`index.html` now references `app.js` and `styles.css` with a
version query string — `app.js?v=2.4.12`. The nginx config sets
`expires 1h` on static files, so after an in-place upgrade a
browser could keep running the previous `app.js` against a fresh
`index.html` for up to an hour. That mismatch makes parts of the
UI break in confusing ways — buttons that do nothing — with no
error, because the cached JS is internally valid, just out of step
with the new markup.

With a per-release `?v=`, each release's assets are distinct URLs,
so a browser can never serve a stale one across an upgrade — while
still caching aggressively within a release. A test asserts the
`?v=` matches `SERVER_VERSION`, so `make dist` fails the build if a
future release bumps the version but forgets to update the asset
URLs.

## Tests

`tests/test_v2412.py` (8 tests): the CVE digest routes to a real
page id and every routed page exists; the online tile excludes
unmonitored devices; the Virtualization search box and filter are
present; the asset URLs carry `?v=`, no unversioned reference
remains, and the `?v=` matches `SERVER_VERSION`.

**Total: 1064 tests, all passing.**

## Upgrading from 2.4.11

Drop-in — frontend only, no API/agent/data changes.

One-time note: the **2.4.11 → 2.4.12** upgrade itself still needs a
hard refresh (Ctrl+Shift+R), because the `index.html` currently in
browsers predates the `?v=` change. From 2.4.12 onward the
cache-busting is automatic and no hard refresh is needed.

Deploy the whole extracted tree at once — do not hand-copy
individual files. `index.html`, `app.js` and `styles.css` are a
matched set; mixing versions across them is exactly the failure
this release's cache-busting prevents.

## Honest caveats

1. **Cache-busting protects upgrades made from 2.4.12 onward.** It
   cannot retroactively fix a browser already holding a stale
   2.4.11 asset — hence the one-time hard refresh above.

2. **Browser-untested**, as usual — the fixes are covered by asset
   and unit tests, not a real browser load.
