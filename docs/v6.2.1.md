# v6.2.1 — "In1tMatters"

A single-purpose release: a **critical fix for Linux hosts that boot via
initramfs** (Debian, Ubuntu, and derivatives), plus two safety nets so the
same class of failure can never take a host down silently again.

If your fleet runs the Linux agent installed from `install-client.sh` or the
AUR package, **act on the "Existing installs" steps below before the next
patch-window reboot.**

## The problem

The systemd unit shipped with the Linux agent enabled
`ProtectKernelModules=yes`. Besides denying module loading, that option makes
`/usr/lib/modules` **inaccessible** to the unit — and to every command the
agent executes on your behalf.

The consequence: when a package upgrade run through RemotePower triggered an
initramfs rebuild (any package that fires the `initramfs-tools` trigger —
lvm2, systemd, libc, and many others), `mkinitramfs` could not see the kernel
module tree and built an initrd containing **no kernel modules at all**. The
tooling only prints warnings for this and exits 0, so the upgrade reports
success and nothing looks wrong — until the host's next reboot, when the
initrd cannot find the disk or assemble the root volume and boot stops at the
`(initramfs)` shell. On LVM roots this appears as *"Volume group not found"*.
Because the trigger rebuilds the initrds of **all** installed kernels,
falling back to an older kernel from the GRUB menu does not help.

Hosts whose initramfs rebuilds only ever ran outside the agent (interactive
`apt upgrade`, `unattended-upgrades`) were unaffected — which is exactly what
made this hard to spot: the breakage depended on *which path* performed an
otherwise routine upgrade.

## The fixes

- **The agent unit no longer sets `ProtectKernelModules`.** New installs
  (installer script and AUR package) get the fixed unit automatically.

  **Existing installs** (the agent's self-update never rewrites unit files —
  do this once per host, via fleet exec or by hand):

  ```sh
  sed -i 's/^ProtectKernelModules=yes/# ProtectKernelModules=yes/' \
      /etc/systemd/system/remotepower-agent.service
  systemctl daemon-reload && systemctl restart remotepower-agent
  update-initramfs -u -k all
  lsinitramfs /boot/initrd.img-$(uname -r) | grep -c '\.ko'   # must be > 0
  ```

  Do not reboot a host until the last check passes.

- **The upgrade command now protects itself.** Before touching the package
  manager it verifies that `/lib/modules/$(uname -r)` is accessible whenever
  an initramfs toolchain (`update-initramfs`) is present. If not, it refuses
  to run with a clear error (rc=3) that points at service sandboxing — so any
  similar misconfiguration in the future fails loudly *before* the upgrade,
  not silently at the next boot. Hosts with no initramfs tooling (WSL,
  containers) are unaffected.

- **Scheduled patch-window reboots are now gated twice.** The trailing reboot
  fires only when the upgrade actually succeeded (previously it rebooted
  "success or not"), and only after `lsinitramfs` confirms that every initrd
  in `/boot` contains kernel modules. If either check fails, the reboot is
  aborted (rc=4), the host stays up and reachable, and the failure is visible
  in the command output and `/var/log/remotepower/remotepower_update.log`.
  Hosts without `lsinitramfs` (dracut- or mkinitcpio-based distributions)
  skip the initrd check.

## Recovering a host that already hit this

Boot a live/rescue ISO, then (adjust names to your layout):

```sh
vgchange -ay
mount /dev/mapper/<vg>-<root-lv> /mnt
# mount any separate /var and /boot from /mnt/etc/fstab — /var is required
# (dpkg's database lives there), /boot is where the initrd is written:
mount /dev/mapper/<vg>-<var-lv> /mnt/var    # if /var is separate
mount /dev/<boot-partition> /mnt/boot
for d in dev dev/pts proc sys run; do mount --bind /$d /mnt/$d; done
chroot /mnt /bin/bash
apt-get install --reinstall lvm2 initramfs-tools
update-initramfs -u -k all      # must print "Generating ..." per kernel
update-grub
exit
umount -R /mnt
reboot
```

Then apply the unit fix above so the host cannot re-arm.

## Tests

`tests/test_v621_patch_safety.py` exercises all three defenses functionally
(the real command strings run under `sh` against stubbed binaries): the
refusal when kernel modules are hidden, the aborted reboot on a module-less
initrd, the skipped reboot on a failed upgrade, and that hosts without
`lsinitramfs` are not blocked. A guardrail also fails if a module-hiding
directive ever reappears in the shipped unit.
