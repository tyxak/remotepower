# RemotePower v3.14.0 — Favorites, stale-image badges & fleet thermals

A small, additive release with three operator-facing conveniences. There are
**no breaking changes** and **no new dependencies** — the thermal page and the
stale-image badge both render data the agent already reports.

After upgrading, hard-reload the dashboard once so the new front-end loads
(service-worker cache `remotepower-shell-v3.14.0`).

---

## Per-account sidebar favorites

Star any entry in the sidebar's collapsible groups and a copy is pinned under
the **"Main"** label, so the pages you use most are always one click away — no
hunting through groups.

- **Hover-reveal star.** Hover a nav row and a star appears on its right; click
  to pin. Favorited rows show a filled yellow star and a pinned copy at the top.
  Keyboard accessible (Enter / Space on a focused star).
- **Per-account, syncs across devices (new in v3.14.0).** Favorites are stored
  on your user record and returned by `GET /api/me`; every toggle saves them via
  `POST /api/favorites`. They now follow you to any browser or device you log in
  from. Browser `localStorage` is kept only as a fast-first-paint cache and an
  offline / signed-out fallback.
- Works for real pages *and* the section-jump rows (Listening Ports, Targets,
  Custom Scripts, …).

> Upgrading from a build that stored favorites only in the browser? Your existing
> favorites are adopted automatically on first login and pushed up to the server.

## Per-container stale-image badge

The device drawer's **Containers** table now shows an **update** badge on any
container whose running image differs from the latest digest in the registry —
the same staleness the fleet **Image Updates** page computes.

- The badge uses the **exact same digest join** as the fleet page (a shared
  `_image_stale` primitive on the server), so the drawer and the Image Updates
  page can never disagree.
- Locally-built / loaded images (no reported digest) and images you've
  **accepted** ("ignore this version") never show the badge.
- No agent change — it reuses the registry-digest cache the fleet scan already
  maintains.

## Fleet thermal roll-up — "Hottest hosts"

A new **Thermal** page (under the posture group, next to Storage) answers "which
hosts are running hot?" fleet-wide, instead of having to open each device drawer.

- One row per host showing its **single hottest sensor** — CPU package, chipset,
  or a SMART disk — with the sensor label, type, and a sensor count.
- Sorted **hottest-first**; anything **≥ 75 °C** is flagged amber and **≥ 85 °C**
  red, matching the per-device drawer thresholds.
- Sortable columns and a device/sensor filter, like the other fleet tables.
- Backed by a read-only `GET /api/fleet/thermal` aggregation over the
  `hardware.json` temps (`temps[].current_c`) and SMART (`smart[].temperature_c`)
  the agent already reports. No agent or schema change.

## Upgrade notes

- **Hard-reload once** after deploying so the service worker picks up
  `remotepower-shell-v3.14.0` (this is also why favorites appeared to "reset" on
  a normal refresh before the cache version bumped).
- No database migration. New per-user `favorites` fields are created lazily on
  first save.
