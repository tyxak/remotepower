# RemotePower v3.14.0 — Favorites, stale-image badges & fleet thermals

A small, additive release with three operator-facing conveniences. There are
**no breaking changes** and **no new dependencies** — the thermal page and the
stale-image badge both render data the agent already reports.

After upgrading, hard-reload the dashboard once so the new front-end loads
(service-worker cache `remotepower-shell-v3.14.0`).

---

## Per-account sidebar favorites

Star any entry in the sidebar's collapsible groups and a copy is pinned under
the **"Main"** label, so the pages you use most are always one click away — no
hunting through groups.

- **Hover-reveal star.** Hover a nav row and a star appears on its right; click
  to pin. Favorited rows show a filled yellow star and a pinned copy at the top.
  Keyboard accessible (Enter / Space on a focused star).
- **Per-account, syncs across devices (new in v3.14.0).** Favorites are stored
  on your user record and returned by `GET /api/me`; every toggle saves them via
  `POST /api/favorites`. They now follow you to any browser or device you log in
  from. Browser `localStorage` is kept only as a fast-first-paint cache and an
  offline / signed-out fallback.
- Works for real pages *and* the section-jump rows (Listening Ports, Targets,
  Custom Scripts, …).

> Upgrading from a build that stored favorites only in the browser? Your existing
> favorites are adopted automatically on first login and pushed up to the server.

## Per-container stale-image badge

The device drawer's **Containers** table now shows an **update** badge on any
container whose running image differs from the latest digest in the registry —
the same staleness the fleet **Image Updates** page computes.

- The badge uses the **exact same digest join** as the fleet page (a shared
  `_image_stale` primitive on the server), so the drawer and the Image Updates
  page can never disagree.
- Locally-built / loaded images (no reported digest) and images you've
  **accepted** ("ignore this version") never show the badge.
- No agent change — it reuses the registry-digest cache the fleet scan already
  maintains.

## Fleet thermal roll-up — "Hottest hosts"

A new **Thermal** page (under the posture group, next to Storage) answers "which
hosts are running hot?" fleet-wide, instead of having to open each device drawer.

- One row per host showing its **single hottest sensor** — CPU package, chipset,
  or a SMART disk — with the sensor label, type, and a sensor count.
- Sorted **hottest-first**; anything **≥ 75 °C** is flagged amber and **≥ 85 °C**
  red, matching the per-device drawer thresholds.
- Sortable columns and a device/sensor filter, like the other fleet tables.
- Backed by a read-only `GET /api/fleet/thermal` aggregation over the
  `hardware.json` temps (`temps[].current_c`), SMART (`smart[].temperature_c`)
  and now GPU sensors the agent reports. No agent or schema change.

## Operator quality-of-life batch

A set of smaller, high-value additions — most surface data the agent already
collects.

### CVE prioritization — Known-Exploited (KEV) + EPSS
The **CVEs** page now ranks by real-world risk, not just CVSS. RemotePower joins
two daily feeds — the **CISA KEV** catalog (actively exploited in the wild) and
**FIRST EPSS** scores (probability of exploitation) — onto your findings. A red
**KEV** badge and an **EPSS %** appear on each finding, the device list gains a
**KEV** column, and both the fleet list and the per-device findings sort
KEV-first → criticals → EPSS. Feeds refresh once a day through the SSRF-safe
fetcher and degrade gracefully to plain CVSS if unreachable.

### Active session management
**My Account → Active sessions** lists every browser/device signed in as you
(IP, device, signed-in and last-active times, with "this session" marked).
Revoke any one, or **sign out all other sessions** in a click. Backed by
`GET /api/me/sessions` + `DELETE /api/me/sessions/<id>` — the raw token never
leaves the server (sessions are identified by a one-way id).

### Saved & shareable views (Devices)
Save the current Devices filter set (search + status/SNMP/group/site) as a named
**view** from the toolbar's **Views** menu. Views are stored on your account and
shareable via the URL (`#devices?view=Name`).

### SSD / NVMe endurance
The drawer's **SMART** table gains a **Wear** column (used %, from NVMe
`Percentage Used` / SATA life-remaining attributes) with a projected
end-of-life, flagged amber ≥80 % / red ≥90 %.

### GPU monitoring
Hosts with `nvidia-smi` (or `rocm-smi`) report a **GPUs** card in the drawer —
utilization, memory, temperature and power draw — and GPU temps feed the new
Thermal roll-up.

### Local certificate-file inventory
The agent inventories your own **service** certificates (Let's Encrypt `live`,
nginx/apache TLS, `/etc/ssl/*.crt`) with `openssl x509` and the drawer shows a
**Local certificate files** card, expiry-coloured like the TLS page. It
deliberately does **not** scan the system **CA trust bundle**
(`/etc/ssl/certs`, `/etc/pki/ca-trust`, …) — that's hundreds of vendor CA certs,
not yours.

### Local account audit
A **Local accounts** drawer card from `/etc/passwd` + shadow age + sudo
membership, flagged-first: extra **UID 0**, **empty** / **stale** passwords,
locked accounts. No password hashes ever leave the host.

### Power scheduling
The **Schedule** page adds **Suspend**, and **Wake (Wake-on-LAN)** actions
alongside shut down / reboot — on a one-shot or recurring window, per device.
Scheduling a power-down on a host with no MAC on record warns that a scheduled
wake won't be possible.

## Visibility & prediction batch

A second wave of operator features — most surface data the agent already
collects, plus one new agent probe (UPS).

### Predictive maintenance ("Predictive health")
A new **Predictive health** page flags hardware *before* it fails. Each
slow-cadence heartbeat snapshots per-disk SMART counters (reallocated/pending
sectors, SSD wear, temperature) to a 6-month daily history; a least-squares
trend (the same fit the disk-fill forecast uses) projects an **ETA** when bad
sectors are growing or wear is climbing toward 100%. Combines the reactive SMART
verdict with the predictive trend; healthy disks aren't listed. The page also
lists **frequently-restarting hosts** (≥3 returns-to-online in 7 days, from the
uptime-transition history) with their last boot reason — a flapping/unstable-host
signal beyond disks.

### Power & energy ("Power")
A new agent probe reads **UPS** status via NUT (`upsc`) or apcupsd
(`apcaccess`) — status, battery %, load, runtime, and real power draw. The
device drawer shows a **UPS / power** card, and a new fleet **Power** page lists
every host's UPS state and watts, hosts-on-battery first, with a client-side
electricity price that turns the live total into an estimated daily/monthly
cost. (GPU watts count too.)

### SSH key audit
A new **SSH keys** page (Security) surfaces the `authorized_keys` RemotePower
already baselines for the *new SSH key* alert — every key across the fleet with
its real OpenSSH **SHA256 fingerprint**, weak/legacy key types flagged, and a
reuse count (the same key present on N hosts), weak-first.

### Container log tailing
The device-drawer **Containers** table gains a per-container **logs** button
that fetches the container's recent `docker`/`podman` logs on demand (queued to
the agent, shown in a viewer) — no SSH needed.

### Global omnisearch
The command palette (`/` or Ctrl-K) now also searches **open alerts** and **CVE
findings**, not just pages/devices/scripts — and warms those caches when opened,
so a cold palette still finds them.

### Custom report builder
Beyond the single fixed posture report, **Reports → Custom reports** lets you
build a report from just the **sections** you want (devices, health, attention,
patches, CVEs, SLA, compliance), pick **JSON/CSV**, download it, and optionally
**schedule** each saved report to its own recipients on its own cron.

> Several Tier-2 ideas were already shipped and are unchanged here: bulk
> multi-select device actions, the automation rules engine, the custom-RBAC
> roles UI, host SBOM export, and drift auto-correct (apply/enforce).

### Refinements
- **Container images in the SBOM.** A host's CycloneDX/SPDX SBOM now also lists
  its running container images as components (`pkg:docker/<image>@<digest|tag>`),
  so the supply-chain picture includes what's running, not just host packages.
- **Fleet drift-enforcement policy.** Beyond the per-device apply/enforce flags,
  the **Drift** page can now set an enforcement policy by **tag or group**
  ("apply every poll" or "correct on drift") that applies to every matching host
  with a desired config — the per-device flag still wins.

## Alerts for the new posture signals

The v3.14.0 signals now raise **alerts/webhooks**, not just dashboard views:

- **`disk_predict_fail`** — a disk is *predicted* to fail (SMART trend with an
  ETA), fired ~6-hourly, edge-triggered. Distinct from the reactive
  `smart_failure`.
- **`ups_on_battery`** / **`ups_on_line`** — the host's UPS switched to battery
  (critical) and the auto-resolving return to line power.
- **`cert_file_expiring`** — one of your **service** certificates is within 21
  days of expiry. **Opt-in** (*Settings → Security → "Alert on expiring local
  certificate files"*, default off) because it is noisy on hosts with many
  certs; the server **coalesces to one alert per host** (soonest cert + a count),
  ignores CA-bundle paths even from older agents, and the inbox shows **which**
  cert and **how many days** remain.
- **`rogue_uid0`** — an unexpected UID-0 (root-equivalent) account appeared.

Each routes to the Alerts inbox, the configured channels, and the dashboard
activity feed like every other event. (Windows hosts inherit the existing
`patch_alert` for pending updates.)

## Change approval — now covers the risky actions

The optional maker-checker workflow (*Settings → Security → Change approval*)
previously gated only ad-hoc **Run command**. It now also parks **reboot**,
**shutdown**, package **update**/**upgrade**, agent **uninstall**, and
**container** start/stop/restart for a second admin to approve on the
**Confirmations** page — reusing the same confirmation store and self-approval
guard. Off by default; when off, every action queues immediately as before.

## Customizable dashboard

The Home page gains a **Customize** button. It opens a small panel where you can
**show/hide** each widget and **reorder** them with up/down controls:

- Widgets: **Fleet health**, **Fleet heat map**, **Needs attention + Recent
  activity**, **Fleet roster**, and **Quick links**.
- The layout is **per-account** (stored in `ui_prefs.dashboard` as an ordered
  `[{key, on}]` list, validated server-side against the known widget set) and
  follows you across devices.
- It reuses the existing dashboard cards — there's no new data path, just
  show/hide and reorder. The summary tiles and the "Ask AI about your fleet"
  omnibox always stay on top.
- Forward-compatible: a widget added in a future release shows by default
  without resetting anyone's saved layout.

## Interface language (i18n)

*My Account → Language* switches the UI between **English, 中文 (Mandarin),
हिन्दी (Hindi), Español, and العربية (Arabic)**.

- **Per-account, synced.** The choice is stored on your user record
  (`GET /api/me` returns `lang`; `POST /api/me/lang` sets it, validated against
  the supported set) so it follows you across devices. `localStorage` keeps a
  copy for instant first paint before `/api/me` resolves.
- **Build-free and graceful.** A self-hosted `static/js/i18n.js` (CSP-safe —
  served under `script-src 'self'`) translates the navigation chrome
  **by source text**: it walks clean text containers (sidebar nav, group names,
  page/section titles) and looks up the English text in a per-language table.
  Anything not yet translated simply stays English, so partial coverage degrades
  gracefully and new UI needs no i18n wiring to keep working.
- **RTL.** Selecting Arabic sets the document to `dir="rtl"`. This is a first
  pass focused on the chrome; deeper right-to-left polish of every page is a
  later iteration.

## Metrics push (Prometheus)

**Settings → Integrations → Metrics push** can periodically POST the fleet's
Prometheus exposition (the same one at `/api/metrics`) to a **Pushgateway** or
compatible remote target — for stacks that pull from a gateway instead of
scraping. Off by default; interval-gated; sent through the SSRF-safe fetcher.
Config is `metrics_push {enabled, url, interval, job}`
(`GET/PUT /api/metrics/push/config`).
