# v6.3.0 — "UndoMatters"

The first wave of a UX improvement program: **undo instead of "are you
sure?"**, optimistic alert actions, and fixes for the everyday friction
operators hit most — silent filters, dead-end error states, lost settings
edits, and one-at-a-time ticket handling.

## Undo instead of confirm

Deleting a low-risk, operator-authored object — a contact, a link, a
command-library snippet, a saved device view — no longer interrupts with a
confirmation dialog. The row disappears immediately and a toast offers
**Undo** for a few seconds. Under the hood the delete is *deferred*: the API
call is only sent when the toast expires, and Undo cancels it outright.
Nothing ever leaves the server until the window closes, so an interrupted
delete always fails toward "still there", never silent loss.

Destructive fan-out operations (Docker prune, device removal, VM rollback,
alert purges) deliberately keep their explicit — sometimes typed —
confirmations. Undo is for the deletes you do ten times a day.

## Optimistic alert actions

- Pressing `a` in the keyboard-driven alert inbox acknowledges the selected
  alert **instantly** — the row flips before the server answers — and the
  success toast carries **Undo**, wired to the real
  `POST /api/alerts/<id>/unack`.
- **Resolve** flips the row optimistically and reverts with an error toast if
  the server declines. (It keeps its confirm dialog: there is no un-resolve.)

## Filtered tables say so

Every table managed by the shared table controller now shows
**"N of M shown — filter active"** with a one-click **Clear filter** chip
whenever the filter hides rows. Table filters persist across sessions, which
used to make a table look mysteriously short days later with no visible cause.

## Retry buttons on failed loads

About a dozen "Failed to load — refresh to retry" dead ends (Checks, CVE
views, CMDB credentials, Thermal / Power / Disk health, Firewall, Rollouts,
GPUs, Drift, Proxmox backups, Compliance baseline, Tuning) now render an
inline **Retry** button that re-invokes the loader — no full-page refresh.

## Unsaved-changes guard on Settings

Editing any setting marks the page dirty. Navigating to another page — or
closing the tab — asks before discarding, and a successful save clears the
flag. (The page's own filter/search boxes don't count as edits.)

## Tickets: bulk actions

All four ticket tables gain row checkboxes and a bulk bar with **Resolve
selected** and **Assign to me** — the multi-select pattern's first outing
beyond the Alerts inbox. Bulk operations reuse the same per-ticket API calls
as the single-row flows.

## Wave 2 — quick wins

- **Ctrl/Cmd-Enter** submits the active modal (its primary button); Esc
  already cancels.
- **No more double submits** — any button whose action is still in flight
  dims and ignores clicks until it settles, automatically, app-wide.
- **Click-to-copy** on CVE ids and command-library commands (`data-copy` —
  renderers can opt any monospace value in with one attribute).
- **Rows-per-page (15/50/100) and CSV export** on every paged table — the
  choice persists per table on your user record; the export honors the
  current filter and sort and includes all pages.
- **Recently-viewed devices** at the top of the command palette (last 8,
  per browser).
- **"A new version of RemotePower is available — Reload"** toast when the
  service worker installs an update while an old page is still running.
- **Skeleton loading** on the remaining bare "Loading…" modal bodies and
  cards.

## Toast actions (for integrators)

`toast(msg, type, opts)` accepts an optional third argument:
`{action: {label, fn}, duration, onTimeout}`. Action toasts stay up 6 s and
settle exactly once — either the action fires or `onTimeout` does, never both.
