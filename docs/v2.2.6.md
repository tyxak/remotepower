# v2.2.6 release notes

Released: 2026-05-16.

A correctness + telemetry release. Fixes a security-scanner false
positive that was undermining trust in the CVE page, hardens the
Docker deployment, fixes the mobile modal stacking, and expands what
the agent reports.

## Fixed

### CVE scanner: false positives on already-patched packages

The CVE page was flagging packages that were already patched. The
field example: `lua5.1 5.1.5-9build2` flagged against
`UBUNTU-CVE-2025-49844` with "fixed in 5.1.5-8.1ubuntu0.22.04.1~esm1"
— but `5.1.5-9build2` is *newer* than that ESM fix
(`dpkg --compare-versions` confirms it).

Root cause: the scanner turned **every** vuln OSV returned into a
finding, with no comparison between the installed version and the
fixed version. OSV flags a package whenever its name matches an
advisory's affected range, but that range model doesn't always line
up with how Debian/Ubuntu revision the same package — so the final
distro-aware version check is the scanner's job, and it wasn't being
done.

**Fix**: each finding is now gated by `_already_patched()`. For
Debian/Ubuntu it shells out to `dpkg --compare-versions` (the
authoritative implementation — handles `~`, `+bN`, `ubuntuN`,
`~esmN`, epochs); other ecosystems use a conservative tuple
comparator. If the installed version is >= the fixed version, the
finding is suppressed as a false positive. `fixed_versions` can hold
several comma-joined values — the package counts as patched if
installed >= any one of them.

**Fail-safe**: any uncertainty — no fixed version from OSV, a
comparison error, `dpkg` not present — *keeps* the finding. A false
positive is annoying; a suppressed real CVE is dangerous, so the
failure mode errs toward showing.

The scan result now includes a `suppressed_patched` count so
operators can see the filter is working.

### Docker: random admin password instead of `changeme`

The `docker-compose.yml` shipped `RP_ADMIN_PASS=changeme` as the
first-run default — a real password sitting in plaintext, and one
everyone has. The entrypoint also did nothing useful if
`RP_ADMIN_PASS` was unset.

**Fix**: when `RP_ADMIN_PASS` isn't set and there's no existing
`users.json`, the entrypoint generates a strong random password
(`secrets.token_urlsafe(18)`) and prints it **once** to the
container log inside a banner:

```
╔══════════════════════════════════════════════════════════╗
║  GENERATED ADMIN CREDENTIALS — SAVE THESE NOW             ║
...
```

Grab it with `docker logs <container>`. The env var is unset after
use so the password doesn't linger in the process environment.
Setting `RP_ADMIN_PASS` explicitly still works for anyone who wants
a known password.

### Docker: healthcheck always reported `unhealthy`

The `HEALTHCHECK` ran `curl -sf http://localhost:8080/` — but `curl`
was never installed in the image. The check could never succeed, so
the container was flagged `unhealthy` even while nginx served fine.

**Fix**: the healthcheck now uses Python's `urllib.request` (Python
is always in the image — it's `python:3.12-slim`). No new package.

### Docker: `duplicate MIME type "text/html"` nginx warning

`gzip_types` listed `text/html`, which nginx always gzips
implicitly — the explicit listing triggered a startup warning.
Removed; `image/svg+xml` added in its place (genuinely useful to
compress).

### `remotepower-passwd`: empty-default username bug

The change-password prompt showed `Username [admin]:` but pressing
Enter passed an empty string → "User '' not found". The bracket was
a lie — it wasn't actually a default.

**Fix**: with exactly one user, Enter now defaults to it (the
bracket means what it says). With several users, an empty entry
re-prompts instead of erroring.

### Mobile: modal windows stacked over each other

On phones, opening a modal while the nav drawer was open — or
opening a second modal — produced overlapping windows. The z-index
scale had grown organically and collided: the device-row dropdown
sat at `z-index: 10000`, *above* modal overlays at 200, so an open
dropdown bled through any modal.

**Fix**: the whole z-index scale is normalised into clean tiers
(content 1 → sidebar 90 → header 100 → dropdowns 200 → mobile nav
800/810 → modals 1000 → nested modals 1100 → toasts 2000).
Additionally: opening a modal now closes the mobile nav drawer and
locks body scroll; on mobile, modals render as true full-bleed
sheets so a shorter modal can't leave an older one peeking out.

## Added

### Drift: expanded watch list + dormant handling for missing files

The default watched-files list grew from 8 to 13 — added
`/etc/passwd`, `/etc/group`, `/etc/login.defs`, `/etc/pam.d/common-auth`,
and `/etc/apt/sources.list` (a changed apt source is a supply-chain
red flag worth noticing).

A watched file that isn't present on a host used to count as drift
*forever* — its missing hash never matches the baseline, so the
operator got nagged about a file that simply isn't there. v2.2.6
adds **dormant** handling: after `DRIFT_MISSING_DORMANT_AFTER` (3)
consecutive heartbeats reporting the file absent, it's marked
dormant — fires one `drift_detected` event so the operator knows,
then goes quiet. It stops counting toward the device's drift total.
If the file reappears, dormant clears automatically and normal drift
comparison resumes. The file is never deleted from state, so the
operator can always see it was being watched.

### Agent: host health telemetry

The agent's `get_host_health()` collects a batch of cheap signals
that weren't gathered before, merged into `sysinfo`:

- **reboot-required** — `/run/reboot-required` (Debian/Ubuntu),
  with the triggering package list when available.
- **failed systemd units** — `systemctl --failed`.
- **logged-in users** — distinct usernames with an active session.
- **listening ports** — `ss -tulnH`, proto + port + process (process
  needs root; proto/port always available).
- **last boot** — derived from `/proc/uptime`.

Every probe is best-effort: a failure omits that field, never
breaks the heartbeat. Older agents simply don't report these, and
the UI degrades gracefully.

These surface in the device detail modal: a reboot-required amber
banner, a failed-units red banner, logged-in/booted info pills, and
a collapsible listening-ports table.

### Containers: per-container CPU / memory + health badge

The agent now runs a `docker stats --no-stream` sample alongside
`docker ps`, so each container reports `cpu_percent`, `mem_percent`,
and `mem_usage`. It also parses the health state — `(healthy)` /
`(unhealthy)` / `(health: starting)` — out of the docker status
string into its own `health` field.

The containers page shows a health pill on the container card and a
CPU/MEM resource line (memory colour-coded by pressure). Both are
omitted for kubectl pods and older agents.

## Upgrading from 2.2.5

Server and agent are both drop-in. The agent self-updates; once it's
on 2.2.6 the new host-health fields start appearing in device
detail. The expanded drift watch list applies to devices using the
default list — devices with a per-device `watched_files` override
are unaffected (edit them individually if you want the new files).

Docker: rebuild the image to pick up the healthcheck and entrypoint
fixes. If you currently have `RP_ADMIN_PASS=changeme` in your
compose file, you can delete that line — once `users.json` exists in
the volume the entrypoint skips admin creation entirely.

## Honest caveats

1. **Browser-untested.** 887 unit tests pass and everything parses,
   but I haven't loaded the mobile fixes or the new detail-modal
   sections in a real browser. The z-index normalisation is the
   highest-risk change — it touches stacking across the whole app.

2. **`docker stats` adds latency to the container heartbeat.**
   It's a separate subprocess call with a timeout, run every
   `CONTAINER_CHECK_EVERY` polls. On a host with many containers it
   can take a second or two. It fails safe (stats omitted) but it's
   not free.

3. **`get_host_health` probes need the tools present.** `ss`,
   `who`, `systemctl` — all standard on a normal Linux host, but a
   minimal container-host image might lack `who` or `ss`. Missing
   tool → that field is just omitted, no error.

4. **The dormant threshold is hardcoded at 3.** Not configurable.
   Three heartbeats is roughly 3 minutes at the default poll
   interval — reasonable, but if you poll slowly it's longer.

5. **The CVE comparator's non-Debian fallback is simplistic.** The
   tuple comparator handles common cases but isn't full PEP 440 /
   semver. For PyPI/npm ecosystems it errs toward keeping the
   finding (safe), but a more correct comparator would suppress more
   genuine false positives. Debian/Ubuntu — the main use case — uses
   real `dpkg` and is correct.

6. **No new monitor types.** "More monitor options" from the feature
   request is *not* in this release — new monitor check types
   (process-running, disk-threshold, cert-expiry, custom-command)
   each need check logic and a UI form, and doing them properly is
   its own release. This release covers everything else from the
   request: drift, containers, agent telemetry, plus all the
   correctness fixes.
