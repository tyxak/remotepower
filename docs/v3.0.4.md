# RemotePower v3.0.4 — bug fixes

> Released 2026-05-24, one day after v3.0.3. Eight real production bugs, no
> schema changes, no migrations. Drop in the new tarball, restart nginx,
> hard-reload the browser once for the service worker to pick up the new
> cache name.

This is a maintenance release. Every fix below was hit by an operator in the
hours after v3.0.3 shipped, surfaced by adding more telemetry coverage and
actually exercising the AI features end-to-end. The CHANGELOG entry has the
full forensic detail; this document is the operator-facing summary.

## What was broken

### AI Assistant returned a 500 on every chat

Symptom: clicking Send on the AI Assistant page produced `Failed to load
resource: 500` in the browser console, with no traceback in nginx's error
log.

Cause: the helper that sends HTTP POSTs to the AI provider referenced a
variable that wasn't in its scope. The reference raised `NameError` before
the request ever left the server. The bug was added in v3.0.2 but stayed
latent — nobody at this site had actually run an AI chat against the new
code path until today, so the exception had never fired in production.

Fix: pass the relevant config flag explicitly into the function. The path
that talks to Ollama / LocalAI / OpenAI / Anthropic now works again. A
regression test (`TestAiProviderHttpPostJson`) inspects the function body
with AST to guarantee no unbound `cfg` reference can sneak back in.

### Monitor page row showed "OK" while Needs Attention flagged a warning

Symptom: a device on the Monitor page rendered an "OK" badge in the Alert
column even though Needs Attention was showing `swap: 44% used (warning ≥
20%)` for the same host. The two views contradicted each other on the same
page load.

Cause: `/api/devices` returned a hand-curated list of fields and forgot
`metric_state` — the dict that records which thresholds are breached. The
JS row aggregator looked up `d.metric_state`, found nothing, and concluded
nothing was wrong.

Fix: `metric_state` is now part of the device list response. The Monitor
row aggregator and Needs Attention agree.

### No 🩺 Investigate button on memory / swap / CPU alerts

Symptom: a swap warning fired and showed up in Needs Attention, but the
🩺 button next to it didn't render. Disk alerts had the button; the three
metric-pressure kinds didn't.

Cause: the AI prompt keys for memory / swap / CPU mitigation existed
since v3.0.1, but the playbook dictionary that wires the button to a
diagnostic command never gained entries for those three kinds.

Fix: three new playbooks. Each runs a read-only diagnostic on the agent
that gathers the standard forensic snapshot for that resource — `free`,
top consumers, OOM events, PSI pressure files, sysctl knobs — and feeds
it into the AI prompt. None of them can alter system state; the test
suite enforces that.

The client side also had a hardcoded duplicate list of which kinds
support mitigation. That list has been brought in sync with the server
and a regression test now compares the two lists on every test run.
Long-term plan: derive the client list from a `/api/mitigate/kinds`
endpoint so the dual source of truth goes away entirely.

### "Save settings" button in the device drawer returned 404

Symptom: opening a device's drawer, changing a setting, and clicking "Save
settings" toasted **Not found** with no apparent reason. The button
appeared to be wired but did nothing.

Cause: the client posts the full settings bundle to `POST
/api/devices/<id>` as one atomic save. That endpoint had never been
implemented; the request fell through to the dispatcher's catch-all 404.
Per-field PATCH endpoints existed for some of the fields, but the drawer
wasn't using them.

Fix: a new bulk-save handler validates every field with the same rules as
the per-field endpoints, writes once atomically, and audits the save. The
existing per-field PATCH endpoints continue to work and are unchanged —
the bulk handler is purely additive.

Two storage-name divergences are handled inside the bulk handler so the
operator doesn't have to know about them:

- The client sends `watched_services`; the server stores it as
  `services_watched` (historical naming).
- The client sends `cmd_allowlist`; the server stores it as
  `allowed_commands` (the canonical field that the command-execution
  enforcement code actually reads).

### "Re-run AI" on a mitigation playbook returned an empty summary

Symptom: a diagnostic captured fine ("Done rc=0 — captured 2.9 KB,
triggering AI analysis…"), but the AI step returned an empty summary,
empty suggested fix, and no error. The "Re-run AI" button reproduced
the empty result reliably.

Cause: the helper that bridges the mitigation page to the AI provider
passed arguments in the wrong order. The provider received the system
prompt where it expected a messages array; `messages.extend(string)`
iterated character by character; the provider rejected the malformed
payload; the helper happily returned an empty string.

Fix: build a proper messages list, pass the system prompt as the
`system` argument, unpack the per-prompt overrides into the right
kwargs. Related: provider errors now return `502` with the actual error
message instead of `200` with an empty body. The provider call's
exception traceback also lands in stderr (visible in `journalctl -u
fcgiwrap`) so future AI failures are diagnosable.

### Mobile / PWA sidebar drawer wouldn't collapse

Symptom: on mobile and the installed PWA, tapping the hamburger opened
the side menu but nothing closed it again — neither tapping the
hamburger a second time nor tapping outside the drawer worked. The
only escape was picking a nav item.

Cause: two stacked things. The tap-outside-to-close path required the
click target to be exactly `document.body`, but on real browsers the
target was usually the underlying `<div id="app">` (which fills the
viewport) — the strict equality never matched. The burger-to-close
path also failed because the burger button sits at z-index 100 while
the scrim sits at z-index 800 once the drawer is open, so the
burger's `onclick` never fired — taps at that location were
intercepted by the scrim.

Fix: rewrote the close handler around `closest()` checks instead of
strict-equality target matching. When the drawer is open, any tap
that's not inside `.sidebar` and not on the burger button closes the
drawer on mobile.

**Iter 2:** with that change, tap-outside-to-close worked but tapping
where the burger visually is still didn't close the drawer — the
scrim covering the burger meant the body-level handler had to catch
the bubbled click, which proved unreliable on mobile Chrome and
installed PWAs in practice. Standard mobile-drawer fix added: a
dedicated ✕ button inside the sidebar header, visible only on mobile
(`@media (max-width: 720px)`), positioned absolute top-right. Calls
the same `toggleMobileNav()` the burger does — so there's still one
toggle path, no logic duplication. Always discoverable, always works,
no z-order trickery. Regression tests assert the button exists, calls
`toggleMobileNav()`, has an `aria-label`, hides on desktop, and shows
on mobile.

### "✨ AI Prioritise Updates" button felt unresponsive — and the
deeper issue: no upgrade listing in patch history

Symptom: click the ✨ icon on a device row in the Patches page,
nothing visible happens, eventually a small toast appears. Operators
reported "I clicked it, but nothing happened." The earlier iter-1
fix improved the toast wording to direct the operator to "Force
re-scan packages" — but Force re-scan didn't actually do what was
needed.

Cause: the agent's `get_patch_info()` runs the per-distro listing
command (`pacman -Qu`, `apt-get --simulate upgrade`, etc.) to count
upgradable packages, but discards the output and only keeps
`len(splitlines())`. `patch_history` is populated exclusively from
`CMD_OUTPUT_FILE`, which only stores output from operator-triggered
exec commands. So the listing the AI button needs is never recorded
unless an operator manually exec'd the right command — and Force
re-scan, despite its name, only triggers a fresh count, not a fresh
listing.

Fix: two things. (1) The button visibly disables and switches to ⏳
during the API call so the click registers visually. (2) When the
patch history doesn't yet contain a listing, the handler reads
`pkg_manager` from the patch-report response, picks the matching
listing command, and offers to queue it on the device via
`POST /api/exec` after a confirmation prompt:

- `apt` → `apt list --upgradable`
- `dnf` → `dnf check-update`
- `pacman` → `pacman -Qu`

The result arrives in `CMD_OUTPUT_FILE` on the next heartbeat
(~60 s) and shows up in `patch_history`. One click → wait ~60 s →
click ✨ again, AI engages. No more digging through menus or
guessing at command names.

## Upgrade notes

- Drop in the new tarball over the existing install, run
  `systemctl reload nginx`.
- The service worker cache name bumped to `remotepower-shell-v3.0.4`.
  Existing browser installs will evict the v3.0.3 shell on first reload.
  If a setting button or icon looks stale, do one hard reload (Ctrl+Shift+R
  / Cmd+Shift+R) to pick up the new app.js cache-busted URL.
- The favicon stays where it was; this release does not relocate any
  static file paths.

## What's next

The next minor release (v3.1.0) is in development. Preliminary infrastructure
shipped quietly with this v3.0.4 release — none of it is user-facing yet.
The visible v3.1.0 surface will be MCP write tools for AI hosts (Claude
Desktop, Cursor, etc.) and a stricter Content-Security-Policy posture.
