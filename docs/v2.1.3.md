# v2.1.3 release notes

Released: 2026-05-14.

Two things: an About-page fix for installs running ahead of any
GitHub release, and a first cut of optional AI integration.

## Fixed

### About page showed "Latest release 2.0.0 ✓ up to date" on a 2.1.2 box

Two compounding bugs in `handle_version_check()`:

1. `local = cfg.get('server_version', SERVER_VERSION)` was reading a
   stale value out of `config.json` — installers had stamped
   `server_version: 2.0.0` once and subsequent upgrades never updated
   it. So `local` reported 2.0.0 regardless of what was actually
   running.
2. Even with `local` corrected, `latest` came from GitHub's tagged
   releases. If you're running a dev build (or sitting in the gap
   between cutting a release and publishing it on GitHub), GitHub's
   "latest" is genuinely older than your local version — so the UI
   correctly said "up to date" but displayed the GitHub version as the
   "Latest release", which read as confusingly wrong.

Fix:

- `local = SERVER_VERSION` — read from the module constant, single
  source of truth.
- `latest = max(github_tag, local)` — if you're running ahead of the
  published release, the "latest" you should consider running is the
  version you have.

Three new tests in `tests/test_v213.py::TestVersionCheck` cover the
running-ahead, GitHub-ahead, and stale-config-key cases.

## Added — AI assistant

Optional integration with a few LLM providers, opt-in per-deployment.
This is **disabled by default** and stays that way until an admin
turns it on in Settings → AI assistant. Nothing leaves the building
unless an operator explicitly enables a cloud provider.

### Providers

Single endpoint, two adapters covering five providers:

- **Anthropic (Claude)** — `/v1/messages`, dedicated adapter
- **OpenAI (ChatGPT)**, **DeepSeek**, **Ollama**, **LocalAI** — all
  share the OpenAI-compatible `/v1/chat/completions` adapter

Pure stdlib (`urllib.request`); no pip-installed packages added. Sync
request/response only — no streaming in this release.

### Settings (Settings → AI assistant)

- Provider selector + model + optional base URL override
- API key (masked on read with the last 4 chars visible; `__clear__`
  to wipe; blank to keep the existing value)
- Privacy toggles — what gets sent to the provider:
  - Send hostnames (FQDNs) — *off by default*
  - Send IP addresses — *off by default*
  - Send journal / log content — *off by default*
  - Send command output — *on by default*
- Always-redacted (regardless of toggle): bearer tokens, AWS access
  keys, long hex strings
- Per-response token cap (default 4000), per-user daily request cap
  (default 100; 0 means unlimited)
- "Test connection" button for a one-shot smoke test against the
  configured provider

### ✨ buttons

Inline buttons that funnel through a single reusable modal
(`openAIModal()`), so every surface gets the same context-summary
header, token-usage footer, and Copy / Insert / Close controls.

| Where | Label | Sends |
|---|---|---|
| Device detail → Command output panel | **✨ Explain** | command + output + device name |
| Device detail → Journal panel | **✨ Find the problem** | error/warning lines + 2-line context window around each, fallback to last 50 lines |
| Scripts → Edit modal | **✨ Generate from prompt** | natural-language description; result drops back into the textarea via the modal's "Insert into editor" button |
| Scripts → Edit modal | **✨ Explain** | the script body |
| Scripts → Edit modal | **✨ Audit for risks** | the script body |
| CVE Findings → per-finding row | **✨ Triage** | CVE id + package + version + device + summary |
| Device dropdown (⋯ menu) | **✨ Investigate** | sysinfo + last 30 journal lines + last 10 commands |
| Settings → Notifications → Webhook log row | **✨ Explain** | event type + payload detail |

The Generate button is the only one that writes back into a form
field — and even there, the generated script goes through the same
dry-run + dangerous-pattern detection as a human-written one before
it can be saved. There's no special "AI-trusted" path.

### Endpoints

- `GET  /api/ai/config` — current config, api_key masked
- `POST /api/ai/config` — admin-only, validated, audit-logged
- `POST /api/ai/chat`   — body `{messages: [...], system: 'key'|'literal', context: '...'}`,
  audit-logged with token counts + elapsed_ms; rate-limited per user per day
- `POST /api/ai/test`   — admin-only, round-trips a "say OK" against
  the configured provider

System-prompt keys (so the client doesn't have to ship the prompt
text) — full list in `ai_provider.SYSTEM_PROMPTS`:

```
explain_output       find_problem      explain_script
audit_script         generate_script   triage_cve
investigate_device   explain_alert
```

Pass any of these as `system` and the server expands it; pass any
other string and it's used as a literal system prompt (bounded to 16
KB).

### Security / privacy notes

- **Cloud providers leave your network.** That's why Ollama and
  LocalAI are in the picker and recommended for self-hosted-purist
  deployments. Pick one of those and the data stays in the building.
- **API keys** live in `config.json` (mode 0600, owned by the CGI
  user). The GET endpoint masks them; they're never sent back to the
  browser after the initial set. For stronger storage, future
  versions can swap in `cmdb_vault`.
- **Redaction happens at the adapter layer**, not just in the UI.
  Flip the privacy toggles off and there's no way for a sloppy
  caller to leak hostnames or IPs — the bytes never reach the HTTP
  request.
- **Per-user rate limit** prevents runaway loops from generating a
  memorable cloud-provider invoice. The default of 100 requests/user/
  day is conservative; bump it once you trust the workflows.
- **Audit log entry on every call**: provider, model, token count,
  elapsed time, context label, success/failure. The full prompt and
  response are NOT logged — that's by design (logs grow forever, and
  they'd reproduce the privacy redaction problem).

### Not in this release

Deferred so this didn't sprawl:

- **Tool calls / agent mode.** Having the model query
  `/api/devices` etc. is interesting but requires a separate
  Settings toggle and a read-only action allowlist. Queued.
- **Streaming responses.** Visible benefit small; complications
  with CGI buffering and nginx large. Sync request/response is
  fine for the 1–8 second latencies we see.
- **Saved prompts / standalone AI page.** The inline buttons cover
  90% of the value. A dedicated chat page is a follow-up.
- **`cmdb_vault` integration for the API key.** Mode-0600
  `config.json` is good enough for a self-hosted box; vault makes
  sense once the operator wants per-user key scoping.

toggle-driven), config validation, About-page version logic

## Upgrading

Forward-compatible. No schema changes. The `ai_usage.json` file is
created on first AI request. The `ai_provider.py` module ships
alongside `api.py`; both are picked up by the standard installer.

If you're upgrading from 2.1.2, this is a feature release — the
AI buttons are visible regardless of whether the feature is
configured. They show a "AI is disabled in settings" toast when
clicked before anything's been set up. Once an admin configures a
provider in Settings → AI assistant, every authenticated user can
use the ✨ buttons (with per-user rate limits applying).

If you're upgrading from 2.0 or earlier, see the 2.1.0 / 2.1.1 /
2.1.2 release notes for the heartbeat / offline-detection fixes
that landed in those releases.
