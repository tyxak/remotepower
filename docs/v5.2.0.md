# v5.2.0 — "AccessMatters"

**WG Access** — a built-in, light WireGuard road-warrior VPN. Reach the dashboard
and your fleet over an encrypted tunnel instead of exposing services on the public
internet. WireGuard is silent to anyone without a valid key, so this *reduces* your
attack surface rather than adding to it.

## Highlights

### WG Access (Admin → WG Access)

A two-level model:

- **Tunnels** — each tunnel is a wireguard-go (userspace, no kernel module)
  interface on the RemotePower server host, with its own UDP port, address pool,
  endpoint and optional pushed DNS. A tunnel carries the *policy*:
  - an **"Allow internet (full tunnel)"** toggle for road-warrior egress;
  - a **reach scope** (none / all / site / group / tag) that decides which fleet
    devices the tunnel's clients can reach — reusing RemotePower's RBAC scope
    vocabulary, enforced by per-tunnel firewall rules on the hub;
  - an optional **time-to-live** (minutes / hours / days / weeks / years) after
    which the tunnel is automatically torn down and deleted.
- **Clients** — each client is a WireGuard peer attached to a tunnel and inherits
  the tunnel's policy. Create a client and you get a **QR code** (scan it straight
  into the WireGuard mobile app) plus a downloadable `.conf`. The client's private
  key is generated **in your browser** and never reaches the server. Clients can
  carry their own TTL for temporary / contractor access.

**Stats** are surfaced at both levels: an RP-host rollup per tunnel (interface
status, connected/total clients, pool usage, aggregate transfer) and per-client
live stats (handshake age, transfer, endpoint). Client connect / disconnect /
handshake-stale transitions raise first-class events that flow through the alert
inbox, the activity feed and webhooks (`vpn_client_connected`,
`vpn_client_disconnected`, `vpn_handshake_stale`). Tunnel expiry is recorded in the
audit log.

**Requirements:** the hub uses `wireguard-go` plus a small root-owned helper
(`remotepower-wg-apply`) invoked through a scoped sudoers rule — the CGI itself
stays unprivileged. Until both are installed the page shows an "unavailable"
notice; everything else in RemotePower is unaffected.

## Security notes

- Client private keys are generated in-browser (X25519) and never transmitted; the
  per-tunnel hub key is generated and held root-only by the helper.
- Reach is enforced on the hub with nftables rules derived from the tunnel scope —
  client-side AllowedIPs are treated as advisory.
- All tunnel/client create / edit / delete actions are admin-only and audit-logged.
- The privileged helper accepts only a structured JSON spec and rebuilds the
  wg/nft config itself (argv-only, no shell) — no untrusted string is ever
  interpolated into a command.

## Upgrade

No breaking changes. New optional dependency for the VPN feature only:
`wireguard-go` + the `remotepower-wg-apply` helper and its scoped sudoers drop-in
(installed by the server packages / installer). The rest of the upgrade is the
standard version bump.
