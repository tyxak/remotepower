# v5.4.1 — "RackMatters" (follow-up)

A small follow-up patch on the v5.4.0 RackMatters line: a missing alert signal,
two helpdesk improvements, a billing opt-in, and a Devices-page polish. No breaking
changes; no schema migration.

## AV / rootkit scan warnings now reach the Alerts inbox

Previously, an endpoint AV/rootkit scan that reported **warnings** (rkhunter
`[Warning]` lines, or a stale ClamAV signature database) only surfaced on the
dashboard **"Needs attention"** card — it never created an **Alert**, so it could
not be routed to a webhook/email channel, acknowledged or tracked. A host showing
"rkhunter reported 2 warning(s)" stayed invisible to anyone watching the inbox.

v5.4.1 adds a first-class **`av_warning`** event:

- **Edge-triggered** on the *rising* warning count (like `av_infected`), so a
  steady count doesn't re-fire every heartbeat; a first sighting on a never-seen
  host still alerts.
- **Coalesced per host** — a flood of warnings on one host is one open alert with a
  bumped count, not an inbox full of duplicates.
- **Medium** severity, routed through the existing **`av_posture`** channel kind.
- **Suppressed** for a tool that *also* reports an active infection this beat — the
  higher-severity `av_infected` already alerts, so there's no double-fire.

There is no recover event: a warning stays actionable until an operator reviews the
scan and the count drops on a fresh report. `WEBHOOK_EVENTS` is now **88**.

## Ticket attachments (inbound + outbound)

The ticket system handled **text only** — inbound email attachments were silently
dropped, and there was no way to send a file with a reply.

- **Inbound:** real attachments on customer emails are now extracted and stored
  (the text/plain body is still threaded as before).
- **Outbound:** an **Attach files** control on the reply box lets you send files
  with an email reply — up to **15 MB each** and **10 per message**. They ride the
  SMTP message as standard MIME attachments and are logged on the ticket.
- **View / download:** every attachment on a message shows a **download** button
  and, for images / PDF / plain text, an in-browser **preview**.

Security: attachment blobs are stored on disk under an **opaque random id** (never a
caller-supplied filename), download access is **bound to the owning ticket** (the id
must appear on one of that ticket's messages), and responses are served
`X-Content-Type-Options: nosniff` — as a download by default, inline only for a small
allowlist of safe types.

## Ticket auto-reply (loop-safe)

An optional, one-time **acknowledgement** can now be emailed automatically when a new
ticket is **auto-created from an inbound email** ("Thank you for contacting
support…"). Configure it under **Settings → Advanced → Tickets**.

It is **loop-safe by construction**:

- Stamped `Auto-Submitted: auto-replied` (so the inbound poller skips it) plus the
  `#RP<number>` / `X-RP-Ticket` marker.
- Sent **at most once per ticket** (only on creation, not on every reply).
- **Never** sent to no-reply / do-not-reply / mailer-daemon / postmaster / bounce /
  notification senders.

## "View email thread"

Every ticket now has a **View email thread** button that opens the full
correspondence in a clean, **printable** window (received / sent / internal notes,
with attachment lines). It is built entirely in the browser using DOM + CSSOM, so it
is fully CSP-safe and needs no new server endpoint.

## Billing is now opt-in (Settings → Advanced)

The **Billing** page (worksheet, invoices, rates & fees) is now gated behind a new
**Settings → Advanced** checkbox, exactly like the ticket system — off by default.
The **time-tracking** half stays available regardless: logging billable / internal
hours on tickets and the weekly **Timesheet** are unaffected. The billing API
handlers return `404` while billing is disabled.

## Devices page — ticket marker

When a host has an open ticket, the Devices page now shows the small **ticket glyph**
(the same icon as the Tickets nav) directly **in front of the hostname**, replacing
the previous "ticket" pill. Click it to open that host's tickets.

## Enterprise hardening batch

A first slice of the enterprise gap-analysis landed in this release — identity,
governance and accessibility checkboxes. **Everything is opt-in**: every new knob
defaults off, so existing installs and behaviour are unchanged until an admin
configures it under **Settings → Security**.

- **Password policy** — configurable minimum length, an optional "3 of 4 character
  classes" rule, and an optional **HaveIBeenPwned** breach check (k-anonymity: only the
  first 5 hex of the SHA-1 ever leaves the server, and it fails *open* if HIBP is
  unreachable). Enforced on new users and password changes; existing passwords are never
  invalidated. A `password_changed_at` timestamp is now recorded.
- **SSO-only** — when enabled and an IdP (OIDC/SAML) is configured, local-password
  logins are refused so the org can mandate IdP auth. Flag an emergency admin's user
  record with `local_login: true` as a break-glass exemption **before** turning it on.
- **Idle session timeout** — a session unused for `idle_timeout_minutes` expires even if
  its absolute TTL hasn't elapsed (sliding window off the activity stamp).
- **Config-change auditing** — every Settings save writes a `config_changed` audit entry
  listing the changed key *names* (values are never logged, so secrets can't leak). The
  most security-sensitive surface (MFA policy, IP allowlist, SSO/SCIM/LDAP, session caps)
  previously saved with no audit trace.
- **Webhook `schema_version`** — the generic-webhook and SIEM payload envelopes now carry
  a contract version (`"1"`) so downstream consumers can guard against shape drift.
- **`max_devices` cap** — new enrollments are refused past a configurable fleet limit
  (existing devices and re-enrollment are never blocked).
- **Localization** — dates, numbers and **billing currency** now follow the in-app
  language picker via `Intl` (previously the browser locale, and billing rendered a bare
  `USD 1,234.00`). A Spanish/Arabic/Hindi/Chinese UI gets matching dates and money.
- **Accessibility (WCAG 2.1)** — toast notifications are now an `aria-live` status region
  (announced to screen readers), the active navigation item carries `aria-current="page"`,
  the device drawer gained a keyboard focus-trap + focus-restore + `aria-labelledby`, and
  a skip-to-content link was added.

A second batch in the same release continues into credential-at-rest, the API
contract layer, and control-plane observability:

- **API keys hashed at rest** — API keys are now stored as a **SHA-256 hash**, so a
  datastore read no longer yields a reusable bearer secret. The raw key is shown once
  at creation; an existing plaintext key keeps working and is transparently migrated
  to a hash on first use. (Hashing device tokens and enrollment tokens — which carry
  more migration/UX coupling — is a deliberate later step.)
- **API versioning** — every endpoint is also reachable under **`/api/v1/...`**, a
  permanent alias of the unversioned path. Integrators can pin a versioned base URL
  today; future breaking changes can land under `/api/v2` without disturbing v1.
- **Correlation IDs** — every JSON response carries an **`X-Request-Id`** header
  (an inbound id from a trusted proxy is honoured, otherwise one is minted). A
  `log_json()` structured-logging helper (gated by `RP_LOG_LEVEL`) and the
  slow-handler ring both carry the same id, so a log line, a slow request and an
  audit lookup can be joined.
- **Frontend error reporting** — uncaught browser errors (`window.onerror` and
  unhandled promise rejections) are beaconed to **`/api/client-error`** (throttled to
  ≤1/5 s and ≤20/page, field-capped, IP-rate-limited, capped ring) so client-side
  failures surface to operators instead of dying silently in a console.

The remaining themes (config-secret encryption, signed audit exports, the API
pagination convention, the persistent-app-server keystone, hard multi-tenancy, …) are
scoped in the internal enterprise gap-analysis and plan, to be taken in deliberate
later phases.
