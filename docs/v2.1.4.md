# v2.1.4 release notes

Released: 2026-05-14 (same-day follow-up to 2.1.3).

A point release fixing the most painful real-world issue with the
2.1.3 AI launch and adding the standalone chat page that was queued
as "Not in this release" 24 hours ago.

## Fixed

### JSON.parse error on every ✨ button against slow local models

**Symptom (reported on a 2.1.3 box pointing at Ollama smallthinker:**
"Test connection" succeeded with `✓ smallthinker:latest responded
(30+8 tokens)`, but every actual ✨ button returned:

> Network error: SyntaxError: JSON.parse: unexpected character at line 1 column 1 of the JSON data

**Root cause.** Two failure modes compounding:

1. The Test Connection button issues a 1-message request with
   `max_tokens=8` — fast, finishes well under 60s. Every other ✨
   button defaults to `max_tokens=4000`. On a thinking model like
   smallthinker / qwq / deepseek-r1, generating 4000 tokens routinely
   takes 60–180 seconds. The Python urllib timeout was already 60s
   (raised from default), but **nginx's `fastcgi_read_timeout`
   defaults to 60s and closes the upstream connection first**,
   returning a 504 Gateway Timeout HTML page to the browser.
2. The JS `api()` helper unconditionally called `r.json()`, which
   throws `SyntaxError: JSON.parse: unexpected character at line 1`
   when the body is HTML. The error bubbled up as "Network error"
   with no useful diagnostic.

**Fix.**

- **`HTTP_TIMEOUT_S`** in `ai_provider.py` bumped from 60 to 300 (5
  min) — local thinking models routinely need that.
- **Per-button `max_tokens`** on every inline ✨ trigger, scaled to
  the typical response length:

  | Button | max_tokens |
  |---|---|
  | Explain alert | 800 |
  | Triage CVE | 1000 |
  | Explain command output | 1500 |
  | Find the problem (journal) | 1500 |
  | Explain script | 1500 |
  | Investigate device | 2000 |
  | Audit script | 2000 |
  | Generate script | 4000 |

  The server respects the configured global cap regardless; this just
  asks for less when less is enough.
- **New `aiApi()` helper** specifically for AI endpoints. Reads the
  raw response text first, attempts JSON parse, and on failure
  synthesises a structured error showing the HTTP status, a snippet
  of the actual response body, and a *contextual hint* — including
  the specific nginx config block to fix a `fastcgi_read_timeout`
  problem when the response looks timeout-shaped.
- **Live "(Xs elapsed)" ticker** in both the ✨ modal and the AI
  page so a slow request doesn't look frozen.

**Operator action required for nginx.** If you're using a slow local
model, add this to your nginx server block — the timeout fix on the
Python side helps but cannot override nginx's tighter default:

```nginx
location /api/ai/ {
    include fastcgi_params;
    fastcgi_pass unix:/var/run/fcgiwrap.socket;
    fastcgi_read_timeout 300s;
    fastcgi_send_timeout 300s;
    fastcgi_param SCRIPT_FILENAME /var/www/remotepower/cgi-bin/api.py;
    fastcgi_param PATH_INFO $uri;
}
```

This block needs to appear **before** any catch-all `location ^~ /api/`
block in the same server stanza — nginx routes on first match.

## Added — AI Assistant page

Sidebar entry under Planning. Stand-alone chat UI, separate from the
inline ✨ buttons. Three things in one page:

### Status header

When the provider supports introspection (Ollama / LocalAI), the
header shows:

- Provider name + base URL
- Reachability (● Reachable / ● Unreachable / ● Error)
- Server version (Ollama only)
- Currently-loaded models with VRAM use + expiry

Refresh button re-fetches without reloading the page.

### Per-conversation model picker

A dropdown showing every model the provider lists. For Ollama, that's
`GET /api/tags` — names + size + parameter count. For LocalAI / OpenAI
/ DeepSeek, `GET /v1/models`. For Anthropic, a hardcoded list (no
introspection endpoint exists).

The picker overrides the global default *for this conversation only*.
Settings → AI assistant still controls the global default. Useful
when you've got multiple Ollama models loaded and want to compare.

### Chat

Free-form chat. Multi-turn — the conversation history persists in
`localStorage` (last 40 messages, capped to keep the request size
bounded). The full conversation is sent with each request so the
model has context.

- **Ctrl/⌘+Enter** sends; plain Enter inserts a newline (standard
  chat UX).
- **Clear** wipes the local history. The server-side audit log of
  individual requests is untouched — by design.
- **Conversation is local-only.** Not synced server-side, not visible
  to other users, gone if you clear browser storage. The audit log
  records that requests happened, with token counts; it doesn't
  record content. This was deliberate and remains so.

System prompt: a new `free_form` key in `ai_provider.SYSTEM_PROMPTS`,
short and operator-flavoured ("be concise, accurate, avoid filler;
default to short direct answers").

## Added — provider introspection endpoints

- `GET /api/ai/models` — `{ok, models: [{name, size_bytes?,
  modified?, family?, param_size?}, ...]}` for the configured
  provider. Ollama gives rich details; cloud providers give a name
  list (live where possible, hardcoded fallback otherwise).
- `GET /api/ai/stats`  — `{ok, provider, base_url, version?,
  loaded_models?, reachable, local}`. Ollama gets a `/api/version`
  + `/api/ps` round-trip; LocalAI just checks `/v1/models`
  reachability; cloud providers report `reachable: bool(api_key)`.

Both require auth, both honour the AI-disabled state, neither leaks
the API key.

## Internal

- `ai_provider.py`: new `_http_get_json`, `_ollama_root` (strips a
  trailing `/v1` so operators can paste either URL form),
  `list_models`, `provider_stats`, `CLOUD_MODELS` fallback table.
- `ai_provider.chat()` gains a `model` kwarg for per-request
  overrides (shallow-copies `cfg` so the override doesn't mutate the
  caller's dict).
- `handle_ai_chat()` accepts `model` (1–200 chars) and `max_tokens`
  (capped to the configured global limit) from the request body.
  Anything else is ignored — defence against a client sending a
  50-KB string as the model name.

## Upgrading from 2.1.3

Drop-in. No schema changes. `ai_usage.json` from 2.1.3 is honoured.

If you actually hit the JSON.parse bug in 2.1.3 against a slow local
model, you must do the nginx config edit above in addition to
upgrading the server files — the Python side of the fix helps but
nginx is the bottleneck.

## Upgrading from 2.1.2 or older

Read the [2.1.3 release notes](v2.1.3.md) first — that's where the
AI assistant landed. This release fixes that release's most painful
real-world bug and adds the standalone chat page.
