# v2.1.1 release notes

Released: 2026-05-13.

Bugfix release on top of 2.1.0. The headline fix is a regression I
introduced in 2.1.0 itself: the heartbeat handler was using the new
non-blocking save path for `last_seen`, which under flock contention
silently dropped the write — so devices drifted past TTL and got marked
offline even when their agents were heartbeating fine. If you saw the
"flock fluctuation" bug seemingly come back after upgrading to 2.1.0,
this is what it was.

## Bug fixes

### Offline regression from 2.1.0

The 2.1.0 release moved the heartbeat's six chained saves to a new
`_save_nb` helper that returns HTTP 202 instead of stalling on lock
contention. Goal was to stop slow saves from causing the request to
exceed the agent's HTTP timeout. Problem: the **first** save in the
chain is `DEVICES_FILE`, which is the one that persists `last_seen`.
If that save hit `LockBusy`, we 202'd before `last_seen` ever made it
to disk. The agent treated 202 as success and moved on; the server
still thought the device was last seen however-long-ago. Repeat enough
times under sustained contention and the device crosses the offline
TTL.

Fix: the `DEVICES_FILE` save in the heartbeat is back to blocking. The
rest of the saves (cmd_output, containers, config, cmds, update_logs)
keep their non-blocking semantics — losing any one of those for a
single heartbeat is recoverable, but losing `last_seen` isn't. The
2.1.0 `save()` redesign already moved `fsync` outside the lock so the
blocking call is now microseconds, not the tens-of-ms it used to be in
2.0 — the original flock-fluctuation rationale still holds.

### Diagnostics that are actually visible

The bigger problem was that the **field reports said "nothing in nginx
logs"** when the bug fired. That's not a misconfiguration — it was
the truth. The 2.0 code had two relevant blind spots:

1. `check_offline_webhooks()` only logged when a webhook actually fired,
   and only inside `fire_webhook()`. An operator who hadn't configured
   webhooks (or who had `device_offline` toggled off) got a silent
   flip on the dashboard with no server-side artifact.
2. `main()` wrapped every per-request maintenance task in
   `try: fn() except Exception: pass`. Any exception inside any of
   them was swallowed, including the ones an operator most needs to
   see when something stops working.

Both fixed:

- `check_offline_webhooks()` now writes
  `[remotepower] OFFLINE dev=... name='...' last_seen=... delta=...s ttl=...s poll_interval=...s`
  to stderr on every transition, regardless of webhook config. Same
  for ONLINE. These land in nginx's error log via fcgiwrap by default.
- The bare except-pass blocks in `main()` are replaced with `_safe(fn,
  label)` which logs the exception class, message, and full traceback
  to stderr before continuing. The CGI response itself still completes
  regardless — a maintenance task failing shouldn't fail the request.
- `handle_heartbeat()` logs every heartbeat at the per-write level:
  `[remotepower] heartbeat dev=... name='...' last_seen=... pid=...`.
  That's noisy if you want to grep, but trivial to redirect; the alternative
  is being unable to answer "is the agent even reaching the server?".

### Default online TTL raised to 5 minutes

`DEFAULT_ONLINE_TTL` is now 300 seconds (5 missed polls at the default
60s interval). The 2.0 default was 180s (3 missed polls). Field reports
of "device went offline" turning out to be brief network blips the
agent recovered from on its own; 5 missed polls is the new bar.
`MIN_ONLINE_TTL` is now 150 — anything below ~2.5 poll intervals causes
flapping. Operators who want the tighter old behaviour can override via
`POST /api/config {"online_ttl": 180}` or the Settings → Webhooks page.

### `log_alert` webhook now includes the matched line

Pre-2.1.1 the message read:

```
pmg01.tvipper.com/postfix.service: pattern "warning|error|critical|FATAL" matched 1 times
```

Useless on its own — the operator wants to know WHICH line tripped the
rule. The webhook payload always had `sample` (first three matching
lines, captured server-side); `_webhook_message` just wasn't using it.

Now:

```
pmg01.tvipper.com/postfix.service: pattern "warning|error|critical|FATAL" matched 1 time(s)
→ Nov 13 12:00:01 pmg01 postfix/smtpd[1234]: warning: unknown[10.0.0.5]: SASL LOGIN authentication failed
```

If there were more than one match, a `(+ 2 more matching lines)` footer
is appended. The line is truncated at 200 chars to fit Discord embeds
and similar.

## Features

### Per-container actions on the Containers page

The Containers page now has Start / Stop / Restart / Logs buttons on
every container the agent reports. Same patterns as the v2.1.0 compose
dropdown:

- Agent dispatches `container:<runtime>:<action>:<id>` via the existing
  exec channel. New helper `_run_container_action()` parses the verb,
  validates the runtime (docker | podman; kubectl pods aren't actionable
  through this path), checks the action against an allowlist
  (`start, stop, restart, pause, unpause, logs`), validates the container
  ID against `[a-zA-Z0-9][a-zA-Z0-9_.-]{0,127}`, and invokes via argv —
  never `shell=True`.
- Server endpoint `POST /api/devices/<id>/containers/action` enforces
  the same allowlists and additionally verifies that `container_id` is
  one the agent reported in its last heartbeat. Defence-in-depth:
  even a compromised admin token can't ask a device to act on a
  container the device hasn't told us about.
- UI buttons are conditional: Start appears only when the container is
  not currently running; Stop appears only when it is. Restart and Logs
  always available.
- Kubernetes pods (runtime='kubectl' in the agent's listing) don't get
  action buttons. Acting on pods needs kubectl context the agent
  doesn't generally have.

The route is `POST /api/devices/<id>/containers/action` with body:

```json
{
  "runtime":      "docker",
  "action":       "restart",
  "container_id": "7875f69a7c4c"
}
```

Returns `200 {"ok": true, "queued": "container:docker:restart:7875f69a7c4c"}`.
Output appears in the device's standard Command Output panel on the
next heartbeat (~60 s).

## Upgrading

Forward-compatible with 2.1.0 and earlier. No schema changes; the
new files (`scripts.json`, `batch_jobs.json`, `log_watch.json`) are
created on first use if missing.

If you're on 2.1.0 and saw the offline regression: just upgrade. The
fix is in `handle_heartbeat()` and takes effect immediately on the next
request.

If you want to verify the diagnostic logging actually works on your
deploy, tail nginx's error log and force a heartbeat:

```bash
sudo tail -f /var/log/nginx/error.log &
sudo systemctl restart remotepower-agent     # on the agent
# expect to see: [remotepower] heartbeat dev=... last_seen=... pid=...
```

## Upgrading the public demo site

The demo at `demoremote.tvipper.com` needs the seed data refreshed to
show v2.1 features. Procedure on the demo host:

```bash
# 1. Pull the new tarball
wget https://github.com/tyxak/remotepower/releases/download/v2.1.1/remotepower-2.1.1.tar.gz
tar -xzf remotepower-2.1.1.tar.gz

# 2. Drop the new code in place
sudo bash remotepower-2.1.1/install-server.sh --upgrade-only

# 3. Re-seed (overwrites the data dir — that's the intent for the demo)
sudo -u www-data python3 remotepower-2.1.1/packaging/seed-demo-data.py --apply

# 4. Verify
curl -s https://demoremote.tvipper.com/api/version
# → {"server_version": "2.1.1", ...}
```

The `RP_READ_ONLY=1` flag in the demo's fcgiwrap config is what keeps
visitors from breaking things; the seed script writes once and is then
held immutable.
