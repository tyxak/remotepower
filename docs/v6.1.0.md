# v6.1.0 — "Runt1meMatters"

The enterprise-productization release: the single-node "enterprise" topology
is now the default install, and the server runs entirely on gunicorn + Flask
— the CGI (`fcgiwrap`) and SCGI prefork tiers are retired. Upgrading is a
straight code swap; `install-server.sh`/`docker-compose.yml` on a **fresh**
install now provision Postgres + the app server + the scheduler + a scanner
satellite with no flags. No breaking API changes.

## Single-node enterprise defaults

- **`install-server.sh`** now provisions **PostgreSQL**, the **out-of-band
  maintenance scheduler**, and a co-located **scanner satellite** (Security →
  Pentest) by default, alongside the app server — a single command stands up
  the full topology. `--no-postgres` / `--no-scheduler` / `--no-scanner` opt
  back down to a lighter install.
- **`docker-compose.yml`** gained a `postgres` service and a `scanner`
  service (built from a new `Dockerfile.scanner`, running `nuclei` directly —
  no Docker-socket mount needed), both on by default; `docker compose up -d`
  now stands up the same topology as the bare-metal installer.
- New **`packaging/scanner-setup.sh`** installs a scanner satellite as a
  systemd service, mirroring the existing relay-satellite installer.
- Bootstrap data (the initial admin user, the scanner's token) is minted into
  the file backend, then migrated into Postgres automatically as part of the
  same install/boot — no manual `Settings → Advanced → Storage backend` step
  needed for a fresh install.

## The app server is now gunicorn + Flask, full stop

- `server/cgi-bin/wsgi.py` is a real Flask app (`application = Flask(...)`),
  served by gunicorn — the **only** server. The CGI entry point (`api_cgi.py`)
  and the SCGI prefork worker (`api_worker.py` / `remotepower-api.service`)
  are deleted. Every shipped nginx config now `proxy_pass`es directly to
  gunicorn — no more runtime nginx-block rewriting at install time.
- Request handling is unchanged: the same dispatch table, auth/CSRF/read-only
  enforcement, and ~33 maintenance sweeps run exactly as before — only the
  transport shell changed. Every existing handler, including the raw
  file/export download handlers, works unmodified.
- Fixed a latent bug this surfaced: the `/install` one-line agent installer
  read the server hostname and enrollment token from the process environment
  instead of the per-request context — harmless under the old fork-per-request
  CGI model, but meant the generated installer silently pointed at `localhost`
  with no token under any persistent server. Now correct.
- `packaging/remotepower-server-update.sh` (guided self-update) now restarts
  `remotepower-wsgi` by default instead of the retired SCGI unit.

## Under the hood

- `packaging/install-demo.sh`'s read-only public demo now runs its own
  dedicated `remotepower-wsgi-demo` gunicorn process (own port, own
  `RP_DATA_DIR`/`RP_READ_ONLY`) instead of relying on per-request CGI
  parameter overrides, which a persistent server can't take.
- `Makefile`'s `app-server-wsgi`/`app-server-cgi`/`app-server-status` targets
  and `packaging/remotepower-app-server.sh` are removed — there is no longer
  a tier to switch between.
- Full documentation refresh: `docs/wsgi.md`, `docs/scaling.md`,
  `docs/architecture.md`, and the nginx/install/deployment guides all
  describe the gunicorn/Flask-only architecture.
- A full SAST + structured code-review pass on the new default topology and
  the Flask cutover found and fixed six pre-release issues — client-IP
  resolution broken by the new proxy hop, a Postgres migration data-loss risk
  on independent volume resets, a default Postgres password shipped
  enabled-by-default, a Flask method-routing gap, and a response-capture
  proxy that could be silently defeated by stdout reassignment. See the
  published [security review](security-review-6.1.0.md).
