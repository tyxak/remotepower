# v5.7.0 — "F4ct0rMatters"

A refactor-and-fix release: the server monolith is broken into clean modules,
the New-UI theming/accent bugs operators reported are fixed, and the whole app
starts and responds faster. Everything new is **opt-in and default-off** — no
breaking changes.

## Fixed — New UI (light mode & theming)

Five issues in the industrial "New UI", all reported by **@AndiBSE**:

- **Devices can be deleted again (#17).** The drawer's *Remove device* button
  called an undefined function and silently did nothing; it now confirms by name
  and issues `DELETE /api/devices/<id>` for every device type (agent,
  direct-deploy, agentless/netscan).
- **Profile menu is readable in light mode (#18).** The account dropdown had
  relied on an undefined CSS variable and rendered dark-on-dark in light mode.
- **The accent picker works in the New UI (#19).** The industrial skin was
  overriding the picked accent; it now applies everywhere while keeping the logo
  blue as the default.
- **Themes work in the New UI.** Tokyo / Dracula / Nord / Gruvbox / … now repaint
  the industrial skin (its card gradient derives from the active palette), and
  named light themes paint their own light palette. Pick a theme *and* an accent
  and both apply.
- **Accent reaches chamfered buttons in light mode** (e.g. *Enable
  notifications*) — the chamfer border used to stay blue in light mode.

The underlying cause was one class of CSS specificity/definition bug in the
industrial skin; the accent and palette are now cleanly separated axes.

## Fixed — mobile / narrow viewports

No page scrolls horizontally at phone width anymore. Header controls and nav
rows are proper touch targets, the devices grid reflows to stacked rows on
phones, and the CMDB scoped-credentials row wraps instead of clipping on a
docked-sidebar tablet.

## Added

- **Ticket lifecycle events.** `ticket_opened` and `ticket_resolved` fire from
  every creation path (operator, inbound email, automation rule) to the activity
  feed and webhooks; resolving a ticket auto-closes its open SLA-breach alert.
- **Config-secret encryption — full coverage** (opt-in `RP_CONFIG_KEY`). Set the
  key and *every* secret-bearing config value at any depth is AES-GCM encrypted
  at rest — ACME DNS credentials, webhook tokens and token-bearing URLs, AI API
  keys, integration secrets — decrypted transparently on use. No key set leaves
  `config.json` byte-identical to before.
- **Postgres row-level security — multi-table** (opt-in multi-tenancy). Tenant
  isolation now covers `entity`, `listrow` and `metric_samples` in addition to
  the device roster, with non-device rows shared across tenants to match the
  application layer. Default off.

## Faster

- **First paint & navigation.** Fonts load in parallel with preloads (the
  serialized `@import` chain is gone) and the service worker uses navigation
  preload. Eight heavy pages and the large Settings page ship as inert
  `<template>`s cloned on first visit, cutting the boot DOM roughly a third.
  Always-on background animations were removed.
- **Heartbeat under Postgres.** Eight more device-keyed stores moved to per-row
  storage, so an agent report updates one row rather than rewriting the whole
  fleet's blob.

## Cleaner (no behaviour change)

- **The server is modular now.** `api.py` dropped from ~57k to ~52k lines:
  notification builders, the per-host checks engine, and the tickets,
  provisioning, backups and CMDB subsystems moved into sibling modules. Every
  fleet/webhook event derives from a single registry, the request router is a
  declarative table, and the self-locking recorders are now nesting-safe. The
  browser bundle shed ~2,500 more lines into per-page modules.
- **Housekeeping.** Dead code removed; a queued-command view no longer echoes
  ACME DNS secrets to admins; the test suite runs ~20% faster.

## Settings

The Settings tabs are grouped into four labelled sections — **Setup**,
**Monitoring**, **Connections**, **System** — and a search box filters them.
Same tabs, easier to find.
