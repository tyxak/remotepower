# v2.2.1 release notes

Released: 2026-05-15.

A **design polish release** — no new feature surfaces, just nine
focused improvements to how the existing ones look and feel. Plus
one new sub-feature (drift diff visualisation) that completes the
v2.2.0 drift detection story.

## What changed

### 1. Distro logos next to device names

Every device now shows its distro's brand mark inline: Ubuntu
orange, Debian red swirl, Arch / CachyOS blue chevron, Fedora blue
circle, RHEL family red, openSUSE green, Alpine blue mountains,
NixOS lambdas, Raspbian pink. Branded SVGs, ~14×14, inline — no
external requests. Fallback for unknown distros is a generic
terminal-block icon in muted accent.

Visible on the Devices page (cards and minimal table), the Drift
page, and the Home dashboard. The existing `osIcon()` API is
preserved, so any caller using the old function automatically gets
the new branded logos.

### 2. Sparkline mini-charts

Disk and memory percentage on every device card now shows a 52×14
SVG sparkline next to the number. Colour auto-picks from current
value: green when comfortable, amber when rising, red when near
capacity. Empty data → empty box (keeps layout stable).

Client-side ring buffer in `window._metricsHistory` keeps the last
24 readings per metric per device, so the line builds up over time
as you stay on the page.

### 3. Status colour palette refinement

Refined palette in CSS variables: `--green-soft`, `--green-edge`,
`--amber-soft`, `--amber-edge`, `--red-soft`, `--red-edge`,
`--accent-soft`, `--accent-edge`. Existing `--green` / `--amber` /
`--red` preserved so nothing breaks.

New `.status-pill` component with `ok / warn / critical / neutral /
info` variants. Pulse animation on critical state only (deliberate
— pulsing every yellow dot would be a disaster at fleet scale).
`prefers-reduced-motion` respected throughout.

### 4. Skeleton loaders

Replaced centred "Loading…" spinners with shimmer-animated skeleton
rows / cards. Eleven tables and six JS-injected loading states
converted. New helpers: `renderSkeletonRows()` and
`renderSkeletonCards()`. Perceived performance is noticeably better
— you see the *shape* of what's coming instead of a generic spinner.

### 5. Home dashboard

New default landing page (replacing Devices as the first page you
see). Four big-number tiles: **Devices online**, **Pending
updates**, **Drift events**, **CVE findings**. Each tile colours
its background by state (green/amber/red).

Two side-by-side panels:

- **Needs attention** — offline devices, devices with significant
  patch backlog, drifted devices. Each item is one click to drill
  in to the relevant detail.
- **Recent activity** — the last 8 webhook events with appropriate
  status pills.

Plus a **fleet roster with 7-day status stripe per device** —
GitHub-contribution-graph style. Currently shows current state only;
historical stripe needs an uptime-history endpoint that's deferred
to a future release.

### 6. ✨ identity extended

Sparkle is now the consistent AI signature:

- Every ✨ button across the app gets a `.ai-btn` class plus
  `.available` (cloud provider) or `.local` (Ollama / LocalAI /
  LM Studio) tint. Subtle animated glow on hover when AI is
  reachable.
- AI "thinking" state shows three sparkles cycling in opacity —
  not a generic spinner.
- AI-generated content (modal responses, runbooks, page-chat) gets
  a thin gradient accent on its left edge, so you always know at
  a glance what came from the model vs what came from your data.
- `MutationObserver` auto-applies the styling to newly-rendered
  buttons, throttled at 400ms.

### 7. Typography upgrade

Inter (UI) + JetBrains Mono (technical identifiers) loaded via
bunny.net — privacy-friendly Google Fonts mirror, no cookies, no
tracking, GDPR-friendly. Falls back gracefully to the existing
system font stack if bunny.net is unreachable (air-gapped
deployments, restrictive firewalls).

`--font-mono` CSS variable applied to: code, pre, kbd, command
output, journal output, device IDs, CVE IDs, file paths, hashes,
the diff viewer. Inter font-feature-settings `cv02 cv03 cv04 cv11
ss01` enabled for the better "1", "a", "g" glyphs.

### 8. Per-row hover affordances

Minimal devices table rows reveal a `Detail · Logs · Run` strip on
hover. Saves a click vs. opening the row's dropdown for the most
common actions. CSS opacity transition (`prefers-reduced-motion`
respected). Hidden entirely on mobile (no hover on touch — row tap
opens detail directly).

### 9. Mobile dashboard view

Phone-sized layout for <720 px viewports:

- Sidebar collapses behind a burger button; slides in with a dim
  overlay.
- Dashboard tile grid stacks to one column.
- Low-priority columns (Hostname, Group, Version) hidden on
  devices table.
- Hover affordances disabled (no hover on touch).
- Tap targets bumped to ≥36 px.
- Header strips to just the logo + burger (logo text and live
  indicator hidden).
- Modals go nearly full-screen.

Carrying the fleet in your pocket during on-call now works
without horizontal scrolling.

### 10. Drift diff visualisation

The diff view that completes the v2.2.0 drift detection story. New:

- **Backend endpoint** `POST /api/devices/<id>/drift/fetch_content`
  — takes a list of watched paths, queues `exec:cat <path>` for
  each. Refuses paths on the denylist (`/etc/shadow`,
  `/etc/gshadow`, and their rotated `-` siblings) regardless of
  operator role.
- **Backend endpoint** `GET /api/devices/<id>/drift/content?path=...`
  — returns stored captures for a path (last 2 chronologically).
- **Mirror hook** in the heartbeat output-ingest path detects
  `exec:cat <watched_path>` outputs and mirrors them into
  `drift_contents.json`. Only mirrors paths currently being
  watched on the device that produced them — defence against an
  arbitrary file-read primitive.
- **Storage** `drift_contents.json` with last 2 captures per
  `device_id:path`, capped at 256 KB per capture.
- **UI** "Show diff" button on each drifted file in the per-device
  drift modal. Opens a sub-modal that polls for the cat output
  (every 5s, up to 90s) and renders a unified diff between the
  two most recent captures.
- **Pure JS diff** — LCS-based, hunked with 3 lines of context,
  syntax-coloured via `.diff-line.add/del/hunk` CSS, line numbers,
  collapsed unchanged regions.

### 11. Logo → Home

The header logo now navigates to the Home dashboard (was: Devices
page in v2.2.0).

## Upgrading from 2.2.0

Drop-in for the server. The new `data/drift_contents.json` file is
created on the first content-fetch request — no migration needed.

For the agent: no v2.2.1 agent changes were necessary. The drift
content fetch uses the existing `exec:cat` mechanism — any agent
that supports `exec:` (which is every agent from v1.0+) works
with this feature.

## Honest caveats

1. **Browser-untested.** All 824 unit tests pass and JS / HTML
   parse cleanly, but I haven't clicked through any of this in a
   real browser. The Home dashboard, drift diff modal, mobile
   layout, hover affordances, and ✨ glow animation are where I'd
   look first if anything misbehaves.

2. **Sparklines start blank.** The metrics history ring buffer
   builds up client-side as you sit on the Devices page. First
   open → no sparkline. Stay 10 minutes → you have a meaningful
   line. Server-side metrics history would solve this; deferred.

3. **The 7-day status stripe is currently single-day.** All cells
   except today show "unknown" because there's no uptime-history
   endpoint to read historical state from. The visual scaffolding
   is in place; wiring it up needs that endpoint, which is its
   own piece of work.

4. **bunny.net dependency.** Inter and JetBrains Mono load from
   `https://fonts.bunny.net`. Air-gapped deployments will fall
   back to the system font stack (acceptable) but won't get the
   refined glyph design. If this matters, self-host the woff2
   files — bunny.net publishes the URLs and they're
   redistributable.

5. **Drift diff needs two captures.** First capture is the
   baseline. Real diff only appears after the second fetch
   (typically after another config change). The current alternative
   — storing baseline content on every device — would mean
   storing the contents of `/etc/sudoers`, `/etc/ssh/sshd_config`,
   etc. on the server, which the v2.2.0 design deliberately
   avoided for privacy reasons. This trade-off may be worth
   revisiting if "I want a baseline diff" is a common ask.

6. **Mobile is functional, not designed.** The <720 px layout is
   good enough for "on-call check from a phone" but isn't a
   ground-up mobile-first design. If you spend a lot of time
   operating from mobile, expect rough edges.

7. **The ✨ MutationObserver is a sledgehammer.** It re-scans the
   entire document body on any DOM mutation, throttled to once per
   400 ms. Not measurable on real-world pages but in principle
   could be wasteful on very large DOMs. If profiler shows it as
   a bottleneck, switch to per-rendered-fragment stamping.

8. **Drift content denylist is hardcoded.** `/etc/shadow`,
   `/etc/gshadow`, and their rotated `-` siblings. If your fleet
   has *other* sensitive paths under watching (e.g.
   `/etc/wireguard/<host>.conf` with private keys), add them to
   `DRIFT_CONTENT_DENYLIST` in `api.py` and rebuild. The hash
   tracking still works; only content retrieval is refused.
