# RemotePower v3.11.0 — Release Notes

**Status:** unreleased (dev)

## Overview

v3.11.0 is a **fleet-posture batch**: seven features that turn data the agent
already collects (or can collect cheaply) into first-class security and
operational signals. There are no new daemons and no new dependencies — every
feature rides the existing heartbeat path and the flat-JSON store.

Hard-reload once after upgrading (service-worker cache
`remotepower-shell-v3.11.0`).

## Features

### 1. Attack-surface / Exposure monitor
The agent has always enumerated listening sockets, but it discarded each
socket's **bind address** — the single most security-relevant attribute. It
now keeps the address and derives an exposure **scope**:

- `local` — loopback (127.0.0.0/8, ::1), never reachable off-host
- `lan` — RFC1918 / link-local / ULA, reachable on the local network
- `world` — wildcard (`0.0.0.0`, `::`) or any global address

A new **`port_exposed_world`** event fires edge-triggered the first time a
service becomes bound to a world-reachable address (distinct from the existing
`new_port_detected`, which fires on any new port regardless of scope). The new
fleet **Exposure** view lists every listening socket with a World/LAN/Local
filter and highlights world-reachable services.

### 2. Fleet Software Policy
A policy is a list of rules evaluated against the installed-package inventory
every host already pushes (~every 6h):

- **banned** — package must not be installed (e.g. `telnetd`)
- **required** — package must be installed (e.g. `fail2ban`)
- **min_version** — if installed, version must be ≥ the configured minimum

Each rule may be scoped to one or more device tags; an unscoped rule applies
fleet-wide. Violations are persisted per device and **`software_policy_violation`**
fires edge-triggered (once per new violation, not every push). New policy
editor and violations table; `POST /api/software-policy` re-evaluates the whole
fleet.

### 3. Storage / RAID health
A new agent probe reports redundant-storage state for **ZFS**, **mdadm** and
**btrfs** — pool/array state, capacity, and (ZFS) last-scrub string. Events:

- **`storage_degraded`** — a pool/array entered a degraded/faulted/offline state
- **`storage_recovered`** — it returned to healthy (auto-resolves the alert)
- **`scrub_overdue`** — a ZFS pool whose last scrub is older than the
  configured threshold (`scrub_overdue_days`, default 35), when the scrub
  timestamp is parseable

New fleet **Storage** view, sorted degraded-first.

### 4. Access watch
Recent successful logins and their source IPs are collected (`last -i`).
**`login_new_source`** fires on a first-seen source address for a host.
Brute-force bursts keep using the existing `brute_force_detected`.

### 5. Host firewall drift
A stable fingerprint of the active **ufw / nftables / iptables** ruleset rides
the heartbeat (iptables packet/byte counters are zeroed first so the hash only
moves on a real rule change). **`firewall_changed`** fires when the fingerprint
diverges from the captured baseline.

### 6. Scheduled-job failure lens
Systemd timers are inventoried (`systemctl list-timers`). **`timer_failed`**
fires when a timer's backing job (`activates`) enters a failed state — catching
the classic "the backup timer silently died" case.

### 7. Scheduled posture digest
An opt-in **daily or weekly email** summarising offline hosts, pending updates,
reboots required, critical/high CVEs, software-policy violations and degraded
storage — assembled from current state and sent over the existing SMTP path.
Configure cadence + recipients in Settings; "Send test now" delivers it
immediately.

## Event wiring

All eight new events (`port_exposed_world`, `software_policy_violation`,
`storage_degraded`, `scrub_overdue`, `storage_recovered`, `login_new_source`,
`firewall_changed`, `timer_failed`) are registered across every registry —
`WEBHOOK_EVENTS`, `_ALERT_RULES` (severity), `CHANNEL_KINDS` (routing matrix),
`_webhook_title`, the dashboard `FLEET_EVENTS` allowlist and the
`_homeActivityAttrs` click-through router — and are covered by the event-set
guardrail tests (`test_v184`, `test_v223`, `test_v225`).

## Detection model

Every detection is **edge-triggered** with per-device state in
`posture_state.json` (and `port_baseline.json` for exposure), so a steady-state
condition does not re-alert on each heartbeat. First contact seeds the baseline
silently.

## New API endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET    | `/api/exposure` | fleet listening-socket inventory (`?scope=world`) |
| GET/POST | `/api/software-policy` | read / replace the policy ruleset |
| GET    | `/api/software-policy/violations` | current fleet violations |
| GET    | `/api/storage` | fleet ZFS/mdadm/btrfs pool health |
| POST   | `/api/posture-digest/test` | send the digest now |

## Upgrade notes

- No schema migration; new state files are created on demand.
- The exposure/storage views populate after each managed host's next
  heartbeat carries the new fields (agents must be on v3.11.0).
