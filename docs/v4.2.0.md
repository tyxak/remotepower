# RemotePower v4.2.0 — "5ecur1tyM4tter5"

## The story behind this release

v4.1.0 was about *seeing* the fleet. v4.2.0 is about **defending** it — and
about meeting the access-control bar that real organisations are held to. The
theme is **security and integrations**: harder sign-in, evidence you can trust,
guardrails on the account model, and — the headline — the ability to actively
**find** the holes on the hosts and sites you own, from inside RemotePower.
Hence the codename, **5ecur1tyM4tter5**.

No breaking changes. Every new control is **off until you turn it on**, and the
two features that need extra software (passkeys, SAML) degrade quietly to
"unavailable" when that software isn't installed — nothing about your current
sign-in changes until you opt in.

## Authorized vulnerability scanning — the "Pentest" page

RemotePower can now **scan the hosts and websites you own** for vulnerabilities,
using the same battle-tested open-source tools a security team would reach for —
orchestrated, scheduled and collected in one place. It's white-hat only:
**you can only scan what you've proven you own.**

- **Targets are authorization-gated.** Enrolled hosts are fair game (the target
  IP is derived server-side from the device record, never typed in). To scan a
  **website you don't have an agent on**, you first prove ownership with an
  ACME-style check — a **DNS TXT record** or a **`.well-known` file** — exactly
  like issuing a TLS certificate. Private/loopback ranges are refused.
- **Tools.** A **passive** profile (safe to run any time) drives **nuclei**,
  **nikto** and **nmap**; an **active** profile adds **OWASP ZAP** and
  **wapiti**. There's also an on-host **lynis** audit that runs through the
  agent for system-hardening findings.
- **Scanner satellites.** The toolchain runs on a **scanner satellite** — a
  hardened relay node — not on your production hosts, so there's no footprint on
  the machines being scanned and web scanners get a realistic external vantage.
  Pin a scan to a specific satellite (e.g. one per network segment).
- **The active tier is gated.** Intrusive scans require an explicit
  **attestation** that you're authorized, and for enrolled hosts an active
  **maintenance window** (or a recorded override) — both written to the audit
  log.
- **Scheduled scans** run on a cron-style cadence; recurring findings can notify
  a channel. Per-scan **intensity** (quick / full) and a **vhost** field for
  name-based virtual hosts round it out. Findings land on the **Pentest** page
  (under Monitoring), severity-ranked and scrollable.

## Passkeys — phishing-resistant, passwordless sign-in

Sign in with a **passkey** — a security key, your phone, or this device's
biometrics — built on the **WebAuthn** standard. Add one under **My Account →
Passkeys**, then use **"Sign in with a passkey"** on the login page. Passkeys
can't be phished or replayed, a cloned-authenticator (sign-count regression) is
refused, and **a passkey satisfies the MFA-required policy** below, so you can go
fully passwordless. (Requires the optional `webauthn` library on the server;
without it the feature simply stays hidden.)

## SAML 2.0 single sign-on

Sign in through your enterprise identity provider — **Okta, Azure AD / Entra,
OneLogin, Ping, ADFS** and any SAML 2.0 IdP. Configure it under **Settings →
Authentication → SAML**: paste the IdP entity ID, SSO URL and signing
certificate, map a username/groups attribute, and an **admin group** for
role-on-first-login. RemotePower publishes its SP metadata for the IdP to
consume, verifies the signed assertion (with replay protection), and provisions
the user on first login — running alongside your existing OIDC, LDAP and local
accounts. (Requires the optional `pysaml2` library and the `xmlsec1` binary;
absent either, SAML stays disabled.)

## Tamper-evident audit log

The audit log is now a **hash-chain**: every entry is cryptographically linked to
the one before it, so a silently deleted or edited entry is detectable. A
**Verify integrity** button on the Audit page walks the chain and reports the
first break, if any. Clearing the log is no longer a quiet wipe — it requires an
**admin password re-prompt** and writes an immutable **pre-wipe archive** first,
so evidence can't simply vanish.

## Account-security guardrails

- **MFA enforcement policy.** Require MFA (**TOTP or a passkey**) for chosen
  roles — admins are prompted to enrol before they can do anything else, just
  like the forced password change.
- **Concurrent-session cap.** Limit how many sessions a single user can hold;
  the oldest is evicted past the cap.
- **API-key default expiry.** Set a default expiry window so new API keys don't
  live forever by default.
- **Security-posture self-check.** A graded checklist on the **Audit** page
  scores your hardening — MFA enforced, admins with MFA, session cap, key
  expiry and more — so you can see at a glance what's left to turn on.
