# v2.2.4 release notes

Released: 2026-05-15.

Two real-world bugs surfaced by the first deployment of the Home
dashboard:

1. **The recent fleet events panel was empty even after devices went
   offline.** Root cause turned out to be a deeper design issue,
   not just a wrong endpoint path.
2. **Unmonitored devices showed up in "Needs attention".** Operators
   set `monitored: false` to silence a host — the dashboard
   shouldn't bring them back up.

## Fixed

### Fleet events panel was a delivery log, not an event log

The activity panel in 2.2.3 read from `/api/webhook/log`. That file
records **delivery attempts** — every time a webhook was sent to a
configured URL, or an email was dispatched, or a delivery failed.
If a `device_offline` event fired but:

- no webhook URL was configured, AND
- email wasn't enabled for that event,

…then nothing got logged anywhere. The event vanished into the void.
A typical fresh install with only SMTP configured (and the default
email-events-off behaviour) would see exactly this: events firing,
nothing showing on the dashboard.

The fix is a new dedicated **fleet event log** that records every
fleet event the moment it fires, regardless of whether any
destination is configured downstream.

**New file**: `data/fleet_events.json` with
`{events: [{ts, event, payload}, …]}`. Capped at `MAX_FLEET_EVENTS =
200` entries with oldest evicted.

**New helper** `_record_fleet_event(event, payload)` called from
the **top** of `fire_webhook` — before the per-event toggle, CVE
severity filter, and maintenance-window gates. Those gates still
suppress *deliveries* (you'd still get no Discord ping during a
maintenance window) but the event itself stays in the activity log.

**Payload summarised** to just the discriminator keys
(`device_id, device_name, name, host, path, unit, metric, cve_id,
severity, critical, high, upgradable, pattern, level`). String
values capped at 256 chars. The full payload — references arrays,
description bodies, etc. — is dropped because the dashboard only
needs enough to render one feed item.

**`test` events not recorded.** Operator-triggered SMTP / webhook
tests don't belong in the fleet event log. (They still land in
`webhook_log.json` for the Settings → Webhook log view.)

**New endpoint** `GET /api/fleet/events?limit=N` (default 50, max
200). Returns newest-first. **Auth: `require_auth`** — viewer-role
operators can read this, unlike `/webhook/log` which is admin-only
(delivery URLs and detailed errors are admin-sensitive; "what
happened on the fleet" is fine for viewers).

**Dashboard updated**: `loadHome()` now calls `/fleet/events?limit=50`
instead of `/webhook/log`. Renderer adjusted for the new payload
shape (`{ts, event, payload}` instead of `{ts, event, status, detail}`).
Empty-state copy updated to reflect the new behaviour:

> *No recent fleet events*
> *Fleet events (device offline, drift detected, CVE found, etc.)
> show up here as they fire — regardless of whether any webhook or
> email destination is configured.*

### Unmonitored devices excluded from "Needs attention"

In 2.2.3 the attention panel surfaced **all** offline devices,
including those with `monitored: false`. Operators explicitly set
this flag to silence alerts on decommissioned hosts, dev boxes, or
hosts being rebuilt — they shouldn't drive the dashboard either.

**Fix**: `_renderHomeAttention` filters `devs.filter(d => d.monitored
!== false)` at the top, then reuses the filtered list for offline
detection, patch backlog, and drift cross-reference. Same predicate
the existing alert pipeline already applies.

The drift section also gets a cross-reference: drift overview
includes all devices that have reported drift; we intersect with the
monitored set so unmonitored hosts with drift don't show up either.

Empty-state copy updated: *"No offline monitored devices, no
significant patch backlog, no drift detected."*

## Upgrading from 2.2.3

Drop-in. The `data/fleet_events.json` file is created on the first
event firing — no migration needed. On a fresh upgrade, the
activity panel will be empty until the next event fires. The
"Needs attention" panel benefits immediately.

## Honest caveats

1. **No backfill.** Existing-but-pre-2.2.4 events aren't in the new
   log. The dashboard sees them as soon as new events fire. The
   alternative (deriving synthetic events from current device state)
   was ruled out — it would conflate "this device has been offline
   for 19 days" with "this device just went offline now", which is
   misleading.

2. **`test` filter is in two places.** `_record_fleet_event` refuses
   to write `test` events, and the JS allowlist filters them again
   on the client. Belt-and-braces — if a future API consumer ever
   submits a `test` event directly via some other path, the
   client-side filter catches it; and the server-side filter keeps
   the on-disk file clean.

3. **Payload summary key list is hardcoded.** A future event with
   a new discriminator key (say, `interface_name` for a new
   network-monitor event) won't appear in the activity feed detail
   field until the SUMMARY_KEYS tuple in `_record_fleet_event` is
   updated. The event itself still records correctly — just no
   detail text alongside it.

4. **`/fleet/events` returns the whole limit at once.** No pagination,
   no since-cursor. With `MAX_FLEET_EVENTS = 200` and ~200 bytes
   per entry, the response is ≤40 KB — fine for the dashboard but
   not built for an event-streaming UI. If you want a real
   event-stream view, a future release should add SSE / WebSocket
   support and a paginated read.
