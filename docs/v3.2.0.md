# RemotePower v3.2.0 — Release Notes

**Released:** 2026-05-26

## Overview

v3.2.0 is a feature release with four major additions:

1. **Alert inbox (B1)** — every fired webhook event lands in a mutable
   ledger you can acknowledge and resolve.
2. **Inbound webhooks (B2)** — receive alerts from external systems
   (Grafana, Alertmanager, Authentik, n8n, Home Assistant) and route
   them into the same inbox.
3. **MCP write tools (A1)** — Stage 4 of the MCP roadmap. The four
   initial write tools (`reboot_device`, `run_saved_script`,
   `force_package_scan`, `force_acme_rescan`) are now callable from
   MCP clients, gated by `MCP_ACTION_ALLOWLIST` and the per-device
   `require_confirmation` flag. Destructive tools queue a pending
   confirmation; an admin approves it in the dashboard before anything
   runs.
4. **OIDC SSO (B3)** — sign in via Authelia, Authentik, Keycloak,
   Pocket-ID, Google, etc. alongside existing local + LDAP auth.

No schema migrations required. New optional state files:
`alerts.json`, `inbound_webhooks.json`, `pending_confirmations.json`,
`oidc_states.json`.

## Upgrade

```
make dist          # builds dist/remotepower-3.2.0.tar.gz
```

Deploy the tarball, reload nginx, then hard-reload the browser once to
evict the service-worker cache (`remotepower-shell-v3.1.0` →
`remotepower-shell-v3.2.0`).

## What's New

### B1 — Alert inbox

Every actionable webhook event (`device_offline`, `cve_found`,
`service_down`, `tls_expiring`, `custom_script_fail`, etc.) now writes
a row to a new mutable ledger at `alerts.json`. Sidebar gets an
**Alerts** entry (top-level, between Home and Links) with a count
badge showing the number of open alerts.

The Alerts page has four views:

| View | Shows |
|------|-------|
| Open | Alerts with no ack and no resolve |
| Acknowledged | Ack but not resolved |
| Resolved | Closed alerts |
| All | Everything |

Each row has Ack / Unack / Resolve buttons. Bulk-resolve a selection
with the checkbox column. Recover events (`device_online`,
`service_recover`, `custom_script_recover`) auto-resolve their matching
open alert.

Severity is auto-derived: `cve_found` reads `critical`/`high` counts,
`tls_expiring` reads `days`, etc. See `_alert_severity()` in `api.py`.

**Endpoints:**
- `GET /api/alerts?status=open|ack|resolved|all&limit=N`
- `GET /api/alerts/summary` (for the sidebar badge)
- `POST /api/alerts/<id>/ack`
- `POST /api/alerts/<id>/unack`
- `POST /api/alerts/<id>/resolve`
- `POST /api/alerts/bulk-resolve` (admin)

### B2 — Inbound webhooks

Settings → **Integrations** → Inbound webhooks. Create a named token
with optional pinning to a device. The token is shown ONCE; copy the
full receive URL on creation.

**Receive URL:** `https://your-server/api/webhook/in/<token>`

**Payload shape:**
```json
{
  "severity": "critical|high|medium|low",
  "title":    "CPU 95% on web01",
  "body":     "Optional long description",
  "source":   "grafana",
  "device":   "web01",
  "device_id": "d-xxxxx",
  "links":    [{"url": "...", "label": "..."}]
}
```

Only `severity` and `title` are required. Body is JSON. The endpoint
needs no auth header — the token in the URL IS the auth.

Inbound alerts land in the same Alerts inbox (B1) with `source: "inbound"`.

Tested shapes:
- **Grafana**: matches via "Generic JSON" contact point.
- **Alertmanager**: webhook receiver.
- **Authentik / Authelia**: notification rules → webhook.
- **Home Assistant**: `rest_command` integration.
- **n8n**: HTTP Request node.

Inbound webhooks do **not** fan out to outbound destinations by default
(the source already delivered to whatever it cares about).

### A1 — MCP write tools (Stage 4)

`MCP_ACTION_ALLOWLIST` is now populated with four write tools:

| Tool | Destructive | Notes |
|------|-------------|-------|
| `reboot_device` | Yes | Honours `require_confirmation` |
| `run_saved_script` | Yes | Must reference an existing scripts.json entry — MCP cannot supply arbitrary bash |
| `force_package_scan` | No | One-shot fresh inventory; immediate |
| `force_acme_rescan` | No | One-shot ACME refresh; immediate |

Destructive tools against a device with `require_confirmation=true`
(the default for new devices) return **202** with a `confirmation_id`.
An admin approves or rejects the action in the dashboard at
**Admin → MCP Confirmations**, which displays the originating AI host
(from `X-MCP-Client` header) and the natural-language prompt (from
`X-MCP-Prompt`) that triggered the action.

The MCP server (`mcp/remotepower-mcp.py`) bumps to **v3.2.0** and
exposes the four write tools alongside the existing read tools.
Authentication now sends both `X-Token` and `Authorization: Bearer`
headers (the server accepts either).

**New endpoints:**
- `POST /api/mcp/reboot_device`         (mcp role)
- `POST /api/mcp/run_saved_script`      (mcp role)
- `POST /api/mcp/force_package_scan`    (mcp role)
- `POST /api/mcp/force_acme_rescan`     (mcp role)
- `GET  /api/confirmations`             (admin)
- `POST /api/confirmations/<id>/approve` (admin)
- `POST /api/confirmations/<id>/reject`  (admin)

Pending confirmations expire after 1 hour.

### B3 — OIDC SSO

OpenID Connect sign-in via the standard confidential-client authorization
code flow. Configure in **Settings → Integrations → OIDC**:

- **Issuer URL** — your IdP's base URL. Server fetches
  `<issuer>/.well-known/openid-configuration` automatically.
- **Client ID + Secret** — from the IdP application registration.
- **Scopes** — defaults to `openid profile email groups`.
- **Admin group** — optional. Members of this group land as `admin`;
  everyone else lands as `viewer`.

**Redirect URI to register with your IdP:**
`https://<your-server>/api/auth/oidc/callback`

The login page picks up `oidc_enabled` from `/api/public-info` and
shows a **Sign in with SSO** button automatically once OIDC is on.

First sign-in auto-provisions the local user with role based on
group membership. Existing OIDC users on subsequent logins get
promoted viewer → admin if their group membership says so (never
auto-demoted — operator may have manually changed role for cause).

Sessions minted via OIDC carry an `oidc: true` flag in `tokens.json`
for visibility. They use the same TTL as regular sessions.

**Security notes:**
- The id_token signature is **not** verified — we rely on the
  back-channel HTTPS POST + client_secret authentication to the
  token endpoint (the standard RFC 6749 confidential-client posture).
- State + nonce are both checked. State storage is server-side
  in `oidc_states.json` with a 10-minute TTL.
- Tested against Authelia, Authentik, and Keycloak shapes.

**New endpoints:**
- `GET /api/auth/oidc/start`     (no auth — initiates the flow)
- `GET /api/auth/oidc/callback`  (no auth — completes the flow)

## Other changes

- `get_token_from_request()` now also accepts `Authorization: Bearer`
  in addition to `X-Token` (previously only `/api/metrics` did).
- `/api/public-info` adds `oidc_enabled` (boolean).
- New audit-log actions: `alert_ack`, `alert_unack`, `alert_resolve`,
  `alert_bulk_resolve`, `inbound_webhook`, `inbound_webhook_create`,
  `inbound_webhook_revoke`, `inbound_webhook_toggle`,
  `mcp_reboot_device`, `mcp_run_saved_script`,
  `mcp_force_package_scan`, `mcp_force_acme_rescan`,
  `mcp_confirmation_requested`, `mcp_confirmation_rejected`,
  `oidc_auto_provision`, `oidc_role_promoted`, `login_oidc`.

## Migration notes

No data migrations required. New files are created on first write:

| File | Purpose |
|------|---------|
| `alerts.json` | Alert inbox ledger |
| `inbound_webhooks.json` | Inbound webhook tokens |
| `pending_confirmations.json` | MCP confirmation queue |
| `oidc_states.json` | In-flight OIDC login state |

All four are backed up alongside the existing files by `tar` of
`/var/lib/remotepower/`.
