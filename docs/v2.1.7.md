# v2.1.7 release notes

Released: 2026-05-14.

Two big items: **per-device AI-generated runbooks** and **Level-1 RAG
context** that makes every AI call aware of the project and the
fleet. Plus a few polish bits for the README and documentation page.

## Added

### AI-generated device runbooks

New **✨ Generate runbook** action in the device dropdown (Inspect
group). Produces a structured Markdown document about the host,
saved per-device and regenerable any time.

The AI gets a snapshot of the device's current state:

- `sysinfo` (uptime, OS, package counts, platform)
- Recent journal lines (last 40)
- Watched systemd services + their state
- Containers (name, image, state, status)
- Last 15 command outputs
- CVE findings (non-ignored, top 20)
- Patch status
- Notes, group, tags, IP, MAC

Plus the project + fleet context (see below), so the runbook says
"this is your mail server pmg01, part of the mail group" rather
than "this is a Linux box."

Output is a Markdown document with 7 named sections — **Purpose**,
**Stack**, **Services**, **Exposure**, **Scheduled work**,
**Recent activity**, **Health & risks**, **Operating notes** — kept
under 1500 words by system prompt.

**Storage**: `runbooks.json`, one entry per device ID. Cheap
(~3-10 KB per runbook). Survives restarts, included in the backup
ZIP.

**UI surfaces**:

1. ⋯ menu → **✨ Generate runbook** opens a modal, fires the request,
   shows a live elapsed-time ticker while the model thinks (30–120 s
   typical for cloud, 60–180 s for slow local models). Result renders
   as Markdown with Copy + Regenerate buttons.
2. **Device detail modal** (System info) now has a "Runbook" section
   at the bottom. Shows the saved runbook inline if one exists,
   otherwise a "✨ Generate runbook" button. Open the full view,
   regenerate, or delete.

**Endpoints**:

```
GET    /api/devices/<id>/runbook            — fetch stored runbook (or exists:false)
POST   /api/devices/<id>/runbook/generate   — generate fresh + save
DELETE /api/devices/<id>/runbook            — remove stored runbook
```

Rate-limited under the same per-user-per-day cap as `/api/ai/chat`.
Audit-logged with `device=`, `tokens_in=`, `tokens_out=`, `elapsed_ms=`.

**Cost honest reckoning**:

- Each generate sends ~5–30 KB of snapshot to the model.
- On Anthropic Sonnet 4.5: ~$0.03–$0.06 per runbook (~10K input
  tokens + ~1K output at $3/$15 per million).
- On Ollama smallthinker: 60–180 s of inference, free.
- For a 9-device fleet, a full re-generation is ~$0.50 / 30 min on
  Anthropic; ~10 min total on a busy 8B Ollama box.

There is **no batch "regenerate runbooks for all devices" button**
deliberately — that's a cost-sensitive operation and should be a
considered click per device, not a one-tap fleet-wide command. If
you want that, it's a one-liner in your terminal with `xargs` and a
session token, and you'll know what you're doing.

### Level-1 RAG — project + fleet context awareness

Every AI request now prepends two blocks of context to the system
prompt:

- **Project context** (~1 KB, hand-curated): what RemotePower is,
  the agent/heartbeat model, the JSON storage layer, the script
  library + dangerous-pattern detector, CMDB conventions, CVE
  scanner, TLS monitor. Tells the model where it is.
- **Fleet snapshot** (~50-200 bytes per device, generated on every
  call from `devices.json`): one line per device with name, OS,
  status, group, tags, optional one-line note. Online devices
  first so the model attends to live state under context length
  pressure.

For a 9-device fleet this adds ~2.5 KB of preamble per request.
Cheap on cloud (~$0.0001 extra per call); ~100-200 ms of context
processing on local Ollama.

**Why this matters**: without it, the AI gives generic Linux advice.
With it, the AI says "you have 3 web servers in the `web` group and
this one matches their conventions — likely safe to restart" or
"this is the only mail server in the fleet, so apply the patch
during the maintenance window not now."

**Configurable** in Settings → AI assistant → **Context
awareness**. Two checkboxes:

- **Include project context** — non-sensitive, default on
- **Include fleet snapshot** — contains hostnames + group names by
  design (a redacted fleet snapshot is useless). Default on.

If you're on a cloud provider and don't want hostnames egressing,
turn fleet context off; project context stays on and the model
still knows what RemotePower is.

**No embeddings, no vector store, no Chroma/FAISS dependency.** This
is "Level 1 RAG" — for ~5000 lines of docs and ~10 devices, dumping
a 2 KB preamble is cheaper, simpler, and just as effective as a
real RAG pipeline. Level 2 (keyword retrieval over docs/) and
Level 3 (embedding similarity) are queued behind concrete failure
modes of this simpler approach.

**Module**: `server/cgi-bin/ai_context.py` (~180 lines, pure
stdlib). Functions: `build_project_context()`,
`build_fleet_context(devices)`,
`build_combined_system_prompt(base, devices, include_project,
include_fleet)`. Used by `handle_ai_chat` and
`handle_runbook_generate`.

## Upgrading from 2.1.6

Drop-in. New file `server/cgi-bin/ai_context.py` ships alongside
the other CGI modules — picked up by the standard installer.
`runbooks.json` is created on first generate.

If you already had v2.1.6 AI working, runbooks will work
immediately after upgrade — no nginx config change needed (same
endpoint family as `/api/ai/chat`).

## Honest caveats

1. **The runbook is only as good as the snapshot.** If the agent
   hasn't checked in recently, or sysinfo is missing fields, the
   runbook reflects that. Adding active discovery (e.g. agent runs
   `ss -tlnp; crontab -l; ...` before generation) is a future
   Phase-2 feature — for now we work with what the heartbeat
   already brings.

2. **The fleet context defaults on for everyone.** If you have
   sensitive hostnames and a cloud provider configured, **turn it
   off in Settings → AI assistant → Context awareness before
   anything ✨ goes out**. We default on because for the typical
   deployment (local Ollama / self-hosted) it's safe and useful,
   and a default-off would mean most users never benefit.

3. **No regression test for the markdown rendering of runbooks**
   specifically — `renderMarkdown()` is verified by the v2.1.5 work
   but the runbook content paths through it aren't unit-tested.
   Visually verified in dev.

4. **No "batch generate runbooks" button**, by design. See the cost
   reckoning above.
