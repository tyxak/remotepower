# v2.1.2 release notes

Released: 2026-05-14.

Critical bugfix. The v2.1.0 `save()` redesign — moving the tmp-file
write outside the lock — was correct in isolation but **broke an
unspoken contract** with callers that did read-modify-write. The
heartbeat handler was the worst-affected because it has the highest
volume of concurrent writers (one per device on the fleet).

If you saw the offline bug "still persistent" after upgrading to
2.1.1, this is the actual fix. v2.1.1's improved logging is what
made the bug diagnosable.

## The bug

The user's nginx error log made the cause obvious once we knew where
to look:

```
19:39:39 [error] heartbeat dev=pmg01 last_seen=1778693978 ...
19:40:39 [error] OFFLINE dev=pmg01 last_seen=1778693858 delta=181s ttl=180s ...
```

At 19:39:39, `pmg01`'s `last_seen` was set to `1778693978`. One minute
later, the offline check read `last_seen=1778693858` — **120 seconds
older than what we'd just written**. The newer value had been
overwritten by an older one.

This wasn't a delayed-write or fsync issue. It was a classic
lost-update race introduced by v2.1.0:

1. v2.0 `save()` held the flock from before the tmp-file write through
   after the rename. The lock window was large but the read-modify-
   write pattern across multiple files (load → mutate → save) was
   *still* not atomic — only the rename was.
2. v2.1.0 narrowed the lock window dramatically by moving the tmp
   write + fsync outside the lock. This was the right optimisation
   for save() in isolation (much shorter critical section, no more
   "flock fluctuation"). But the caller's RMW pattern now had a
   wide-open race window:

```
CGI-A: load → devices = {pmg01: T-180, web01: T-180}
CGI-B: load → devices = {pmg01: T-180, web01: T-180}   ← stale read between
                                                          CGI-A's load and save
CGI-A: pmg01.last_seen = T; write tmp.A; lock; rename
       → disk: {pmg01: T, web01: T-180}
CGI-B: web01.last_seen = T; write tmp.B; lock; rename
       → disk: {pmg01: T-180, web01: T}   ← CGI-A's update LOST
```

Every concurrent heartbeat from a different device clobbered every
other concurrent heartbeat's `last_seen` update. The bug was
*deterministic* given concurrent heartbeats, but only *visible* when
the lost update happened to be the one keeping a device under the
offline TTL.

## The fix

Holding the lock for longer (the v2.0 approach) doesn't fix this —
it just narrows the window. The correct fix is to make the
**read-modify-write atomic**: the load, the mutation, and the save
all happen under the same lock acquisition.

New primitive in `api.py`:

```python
with _locked_update(path) as data:
    data[key] = value
    # auto-saved on clean exit, under the lock we acquired at enter
```

- `__enter__` acquires the flock (blocking by default), loads the file,
  yields the loaded dict.
- `__exit__` saves the (possibly mutated) data via `_save_held()` —
  a save() variant that skips the lock acquisition because the caller
  already holds it. Then releases the lock.
- If the with-block raises (including SystemExit from `respond()`),
  the save is **skipped** — no partial write on disk. The lock is
  always released.

`handle_heartbeat()` is rewritten around this pattern: the entire
load → sanitise → mutate → save block runs under one lock acquisition.
Concurrent heartbeats now serialise correctly: each one sees the
results of the previous one's update and merges its own changes on top.

The `compose_projects` update, which previously did a *separate* save
after the main one, is now part of the same atomic transaction.

## What didn't change

- **save() itself** still uses the v2.1.0 fast-path: tmp write outside
  the lock, lock held only for the rename. The optimisation is correct
  for **single-shot saves where the caller doesn't read first**. The
  fix above is for **callers that do RMW**, who must now use
  `_locked_update` explicitly.
- **`_save_nb` (non-blocking save)** is still used for the *other*
  files the heartbeat writes (cmd_output, containers, config, cmds).
  These are independent of devices.json — each has its own flock, so
  no race between them. Losing one of these on contention is
  recoverable; losing `last_seen` was not.

## What's not yet fixed

Other RMW sites in `api.py` (admin actions: setting notes, group,
tags, poll interval; the offline-notified bookkeeping; the patch-alert
tracker) all use the unsafe pattern. They're much lower frequency
(human-driven, not poll-driven) so the race is far less likely to
fire, but it does exist. Migrating these to `_locked_update` is on
the roadmap; doing them all in one release would have produced an
unreviewable diff.

If you're seeing weird behaviour on a busy admin workflow (two admins
editing the same device's notes simultaneously, for example), the
same race could be involved. Open a bug.

## Testing

13 new tests in `tests/test_v212.py`, including:

- A threaded test (`test_concurrent_atomic_updates_preserve_all`) that
  spawns 20 threads, each updating a different device's `last_seen`,
  and asserts every update is visible afterwards.
- A demonstration test (`test_concurrent_racy_updates_demonstrate_bug`)
  using the **old** pattern to verify the race is reproducible. On a
  modern dev box this shows ~10 lost updates out of 50 threads.
- Field-preservation tests confirming concurrent heartbeats don't
  clobber admin-set fields like `notes` or `group`.

Total suite is now **660 tests, all passing.**

```
$ make test
…

OK (skipped=1)
```

## Upgrading

Forward-compatible with all 2.x releases. No schema changes.

If you're upgrading from 2.1.0 or 2.1.1, this is a **must-deploy**
release — the offline bug is real and visible on any fleet with more
than ~5 devices polling at the same interval. The fix is purely
internal to `handle_heartbeat()` and the new `_locked_update`
primitive; agents need no changes.

Standard upgrade path:

```bash
wget https://github.com/tyxak/remotepower/releases/download/v2.1.2/remotepower-2.1.2.tar.gz
tar -xzf remotepower-2.1.2.tar.gz
sudo bash remotepower-2.1.2/install-server.sh --upgrade-only
```

Verify with `make test` on the unpacked tree, and watch nginx's error
log for the new heartbeat lines. The TTL drift bug should disappear
within one poll interval (~60s).
