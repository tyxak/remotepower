# v2.4.11 release notes

Released: 2026-05-18.

Two CVE-ignore fixes.

## Fixed

### CVE "Ignore" locked the interface after a few uses

The CVE ignore flow used two native browser dialogs back to back —
`prompt()` for the reason, then `confirm()` for the scope. Browsers
deliberately throttle pages that open repeated dialogs: after a
handful, the browser stops showing them and `prompt()` simply
returns `null`. `ignoreCVE()` treated `null` as "cancelled" and
bailed silently — so every further click of Ignore did nothing,
sent no request (hence nothing in the nginx log), and the UI
looked locked. Doing several ignores in a row — exactly what a
fleet-wide sweep looks like — is what triggered it.

The ignore flow is now a normal in-page modal: a reason field and
a "this device / all devices" scope choice, with Ignore and Cancel
buttons. Browsers never throttle in-page UI, so it cannot lock.
The modal also makes the scope choice explicit rather than hiding
"fleet-wide" behind an OK/Cancel on a confirm dialog.

### Ignored CVEs still appeared in the Needs Attention digest

`_compute_attention()` counted every CVE finding on a device and
never consulted the ignore list, so a vuln an operator had
explicitly accepted as a risk still showed up under Needs
Attention on the dashboard. It now applies the ignore list
(global and per-device scope) and excludes ignored findings — if
every finding on a device is ignored, that device no longer
raises a CVE attention item at all.

While fixing this, `cve_scanner.apply_ignore_list()` was hardened
to tolerate a finding with no `vuln_id` field rather than raising
on it.

## Tests

`tests/test_v2411.py` (8 tests): the attention digest shows an
un-ignored CVE, excludes a device-scoped ignore, excludes a global
ignore, and keeps the rest when only some findings are ignored;
the ignore flow contains no `prompt()`/`confirm()` calls, the
confirm path posts to `/cve/ignore`, and the in-page modal markup
(reason field, both scope options, wired confirm button) is
present.

**Total: 1056 tests, all passing.**

## Upgrading from 2.4.10

Drop-in. Frontend plus a server-side digest fix — no data or agent
changes. Existing entries in `cve_ignore.json` keep working
unchanged.

## Honest caveats

1. **Existing ignores are unaffected — including their scope.** If
   you ignored CVEs "per device" earlier because the fleet-wide
   path appeared broken, those entries are still device-scoped.
   Re-ignore them with the "all devices" option if you want them
   fleet-wide, or edit `cve_ignore.json` directly (set `scope` to
   `global`).

2. **Browser-untested**, as usual — the new modal is covered by
   asset and unit tests, not a real browser load.
