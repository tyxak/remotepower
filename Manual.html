RemotePower: A No-Nonsense Manual

Written by someone who actually read the code

So you've got a bunch of Linux boxes (and maybe a Windows machine you're forced to keep around) and you want to reboot them, run commands, and check their vitals without opening a dozen SSH terminals. RemotePower does that. It's a small, self-hosted web dashboard that talks to a lightweight agent on each client. The agent phones home every minute to ask "got any work for me?" If you queued a reboot, shutdown, or a custom command, the agent picks it up, runs it, and reports back. No inbound firewall holes. No message brokers. Just a CGI script and some JSON files held together with determination.

This manual is based entirely on reading the source code—no marketing fluff, no README regurgitation. I'll show you how to get it running, how to use it effectively, and where the sharp edges are.

1. What you're actually installing

Three components:

  1. Server: A single Python CGI script (server.py) that lives behind Nginx. It serves the dashboard HTML and handles every API call. It stores all its data as flat JSON files in /var/lib/remotepower/.
  2. Agent: A single Python script (agent.py) that runs on every client, usually as a systemd service (or NSSM on Windows). It polls the server, executes commands, collects metrics. It runs as root because it eventually needs to call shutdown and reboot.
  3. Monitor (optional): ICMP/TCP/HTTP probes the server runs against external targets — not the same as agent service monitoring. Pre-v1.11.8 these only ran when somebody loaded the Monitor page (a real bug — see CHANGELOG). From v1.11.8 onwards a periodic runner inside main() sweeps all configured monitors at most once every monitor_interval seconds (default 300s). The runner is triggered piggyback on every incoming CGI request — agent heartbeats every 60s keep it ticking even when no users are browsing.

The whole thing is about 2500 lines of Python, no framework, no database—just cgi, json, fcntl for file locking, and raw sockets for Wake‑on‑LAN.

2. Hardware and software requirements

  • Server: Any Linux with Python 3.8+, Nginx, fcgiwrap, and systemd. (A Raspberry Pi works fine.)
  • Client: Python 3.8+. Linux or Windows. The agent needs root to run commands.
  • Your server needs a static IP or hostname reachable from clients. HTTPS strongly recommended (the install script sets up a self‑signed certificate; you can replace it).

3. Installation – the real steps

I'm going to walk through what the install scripts actually do, so you can fix things when they go sideways.

3.1 Server

Clone the repo:

git clone https://github.com/tyxak/remotepower
cd remotepower

Run the server installer as root:

sudo bash install-server.sh

Here's what it does, in order:

  • Detects your distro (apt, dnf, pacman) and installs packages: nginx, fcgiwrap, python3, python3-pip, python3-venv (or virtualenv), and bcrypt via pip.
  • Creates a system user remotepower if it doesn't exist, otherwise uses www-data. The data directory /var/lib/remotepower/ ends up owned by that user with mode 700.
  • Copies server.py to /usr/lib/cgi-bin/remotepower/api (the file literally becomes the endpoint). It's executable, with a shebang pointing to python3.
  • Writes an Nginx config to /etc/nginx/sites-available/remotepower (and symlinks it). This config:
    • Serves static dashboard files from /var/www/remotepower/.
    • Passes anything under /api/ to fcgiwrap via a Unix socket.
    • Sets fastcgi_read_timeout to 130s (needed for long‑poll exec).
  • Enables and starts Nginx and fcgiwrap.
  • Generates a random admin password, stores it in /var/lib/remotepower/users.json as a bcrypt hash with a "must_change": true flag. The dashboard will force you to change it on first login.
  • Prints the server's IP address and tells you to open https://that-ip/ in a browser.

If you prefer Docker, there's a docker-compose.yml that builds a container running Python's built‑in HTTP server on port 8080. You need to put a reverse proxy in front for SSL. Data lives in a Docker volume.

3.2 Client (Linux)

On each machine you want to manage, run:

sudo bash install-client.sh

That script:

  • Copies agent.py to /usr/local/bin/remotepower-agent and makes it executable.
  • Creates /etc/remotepower/ (the agent stores its config there later).
  • Installs a systemd unit file remotepower-agent.service that executes the agent as root with the --daemon flag.
  • Enables and starts the service, but the agent won't work yet because it hasn't enrolled. The script then prompts you to run enrollment manually.

After the install script finishes, enroll the client:

sudo remotepower-agent enroll

You'll be asked for the server URL (e.g., https://my.server.lan) and a 6‑digit PIN. Get the PIN from the dashboard: log in as admin, click the + Enroll device button. The PIN is single‑use and expires in 10 minutes.

Once enrolled, the agent saves the device token to /etc/remotepower/config.json (permissions 600) and starts polling. Within 60 seconds the dashboard shows the device as online.

3.3 Client (Windows)

Run PowerShell as Administrator and execute:

powershell -ExecutionPolicy Bypass -File install-client.ps1

It installs agent.py to C:\Program Files\RemotePower\, installs NSSM if needed, and creates a Windows service called RemotePowerAgent. Then enroll similarly by running python agent.py enroll from that directory.

4. Using the dashboard

Open https://your-server/ and log in with admin credentials. The dashboard is a single-page vanilla‑JS application. No frameworks, just a pile of fetch() calls and DOM manipulation. Every 60 seconds it asks the server for device statuses.

Each device is shown as a card with hostname, online/offline indicator, group, tags, and a row of tiny buttons. Hover over them to see tooltips. The main ones:

  • Reboot – queues a reboot. The agent picks it up on its next heartbeat.
  • Shutdown – same, but shutdown.
  • >_ Execute – opens a modal where you can type a shell command. The output appears in the history a bit later.
  • WoL – sends a Wake‑on‑LAN magic packet. Works only if the server and client are in the same broadcast domain, or if you've set up directed broadcast on your router. The code uses the client's last known IP, falling back to broadcast if configured.
  • Update agent – tells the agent to download and replace itself from the server.
  • Metrics – if the client has psutil installed, you'll see sparklines of CPU/RAM/disk.

The navigation menu (gear icon) gives access to:

  • Settings – global config, webhooks, monitors, maintenance windows.
  • API Keys – create named keys that never expire. Use them for scripts.
  • Schedule – define one‑shot or cron‑based jobs to reboot/shutdown devices.
  • Command Library – save your favorite shell commands for quick reuse.
  • CMDB – attach notes, tags, asset info, and even encrypted credentials (AES‑GCM vault) to each device.

Groups and tags are just strings you assign. You can filter the dashboard by them, and batch‑command an entire group (e.g., reboot all "lab" machines at once).

4.1 Filter, sort, and density (v1.11.5, expanded in v1.11.6 and v1.11.7)

Every fleet table — Devices, Services, CVE Findings, Containers, Monitor, TLS, Patches, Audit Log, Command History, Schedule, Maintenance — has a substring filter input above the table. Type any text, the table updates as you go. As of v1.11.6 the admin tables (Users, API Keys, Command Library) also have filter inputs.

Click any column header to sort by that column. The first click sorts ascending, the second descending, the third clears the sort. Hold Shift and click a second column header to sort by that as a secondary key — useful for things like "sort by group, then by status within each group". A small superscript number shows the priority order when you have a multi-column sort.

The Devices grid additionally has a four-button density toggle in the toolbar:

  • Minimal (v1.11.6, rebuilt as a real <table> in v1.11.7) — one device per row in an aligned table layout. Columns: Status, Name, Hostname, Group, OS, IP, Version, Last seen, Actions. Headers click-to-sort like every other table. Responsive breakpoints drop low-priority columns rather than letting them overflow: hostname goes at ≤ 1280px, version at ≤ 1080px, group at ≤ 920px, IP at ≤ 760px, OS at ≤ 620px. Built for fleets where you want to scan 50+ devices at a glance.
  • Compact — denser cards with smaller padding and tighter grid gap.
  • Comfortable — the default card layout.
  • Spacious — larger padding and bigger fonts for very large displays.

The dropdown menu on each device (the ⋯ button) is identical across all four density modes — same handlers, same items. As of v1.11.7 the menu includes a per-device Upgrade packages option that runs apt / dnf / pacman upgrade on that single device without needing to select the checkbox first. Output appears in the device's "Update history" panel ~30–120s after the run completes.

All of this — density, filter strings, sort orders — is stored per user, server-side, under the ui_prefs key in users.json. Log into the same account on a different browser, your filters and sort orders come with you. The server-side schema is sanitised: anything you don't recognise gets dropped silently, individual filter strings cap at 256 chars, sort lists cap at 5 columns, total per-user prefs cap at 16 KB. The /api/ui-prefs endpoint accepts GET (read), POST (replace whole document), and DELETE (wipe) — useful if you ever need to reset somebody's UI to defaults from a script.

5. How the heartbeat and command queue actually work

This part is important if you want to understand delays.

The agent wakes up every 60 seconds (configurable via environment variable RP_POLL_INTERVAL). It sends a POST /api/heartbeat with its device ID, auth token, and system info (hostname, IPs, MAC, metrics). The server records the heartbeat timestamp and immediately checks commands.json for any pending command with that device ID.

If a command is pending, the server returns it in the heartbeat response and marks it "in‑flight". The agent receives the command, executes it, and immediately sends a follow‑up heartbeat with just the result (a minimal payload: device id, token, cmd_output). For commands that take a while — apt upgrade, dnf upgrade, custom shell scripts — the follow‑up arrives whenever the command finishes, not at the next 60s tick. Result is typically visible in the dashboard within a couple of seconds of the command completing.

Before v1.11.7 the agent assigned the result to the heartbeat payload it had already sent, then reset the payload on the next loop iteration — the result silently disappeared. If you ran upgrades on v1.10.0–v1.11.6 and the "Update history" panel stayed empty, that's why. The fix in v1.11.7 sends the dedicated follow-up heartbeat. If the follow-up POST itself fails (network blip, server restart at the wrong moment), the result is stashed to /var/lib/remotepower-pending-cmd.json and replayed on the next successful heartbeat.

Shutdown/reboot commands are slightly special: the agent runs the OS shutdown command (shutdown -h +0 on Linux, shutdown /s /t 0 on Windows) and immediately reports success on the next heartbeat—before the machine actually powers off. That's why you sometimes see "executed" but the device stays online for another few seconds. You can set a shutdown delay in server settings (which prepends a + argument or changes the /t value).

If a command fails or the agent never reports back, the command remains "in‑flight" forever in commands.json. The dashboard shows a spinner. There's no timeout or automatic cancellation; you have to manually clear it from the history (the dashboard has a "discard" button that deletes the command entry).

6. The API – endpoints that matter

All API calls go to https://your-server/api/<endpoint>. Authentication is via the X-Token header (session token from login, or API key). Admin role required for destructive actions.

6.1 List devices

curl -k https://your-server/api/devices \
  -H "X-Token: YOUR_API_KEY"

Returns an array of devices with device_id, hostname, online (bool), group, tags, last_seen_epoch, and metrics if available.

6.2 Reboot a single device or a whole group

# single device
curl -k -X POST https://your-server/api/reboot \
  -H "X-Token: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"device_id": "abc123"}'

# by group
curl -k -X POST https://your-server/api/reboot \
  -H "X-Token: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"group": "lab"}'

Response: {"status": "queued", "commands": 3}.

6.3 Execute a command and wait for output

curl -k -X POST https://your-server/api/exec/wait \
  -H "X-Token: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"device_id": "mybox", "cmd": "df -h /", "timeout": 90}'

This keeps the HTTP connection open for up to timeout seconds. As soon as the agent returns the command output, you get it in the response. If the client is offline or the agent doesn't check in, the request hangs until the timeout, then returns an error. Nginx's fastcgi_read_timeout must be greater than your timeout.

6.4 Queue a regular command (without waiting)

curl -k -X POST https://your-server/api/exec \
  -H "X-Token: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"device_id": "mybox", "cmd": "apt-get update && apt-get -y upgrade"}'

Returns a UUID. Check the result later via /api/history.

6.5 Send Wake‑on‑LAN

curl -k -X POST https://your-server/api/wol \
  -H "X-Token: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"device_id": "mybox"}'

The server sends a magic packet from your server, not from the dashboard browser.

6.6 Get a quick digest (for email or scripts)

curl -k https://your-server/api/digest \
  -H "X-Token: YOUR_API_KEY"

Returns JSON: total, online, offline, pending_patches, recent_commands.

6.7 Prometheus metrics

curl -k https://your-server/api/metrics

No auth needed (configurable in settings). Returns text/plain metrics for scraping.

7. Real‑life examples

Morning ritual: Check that everything came back after a power outage.

curl -s https://my.server/api/digest -H "X-Token: abc123" | jq '.'
# If offline > 0, panic.

Patching a bunch of servers without leaving your coffee: Push a full upgrade command to your "homelab" group.

curl -k -X POST https://my.server/api/exec \
  -H "X-Token: myadminapikey" \
  -H "Content-Type: application/json" \
  -d '{"group": "homelab", "cmd": "apt-get update && apt-get -y upgrade && [ -f /var/run/reboot-required ] && reboot || true"}'

(The && reboot trick works because the agent runs as root; the command will trigger a reboot if the file exists.)

Scheduling a weekly reboot: Use a cron expression to bounce a flaky media server every Sunday at 3 AM.

curl -k -X POST https://my.server/api/schedule \
  -H "X-Token: myadminapikey" \
  -H "Content-Type: application/json" \
  -d '{"cron": "0 3 * * 0", "action": "reboot", "device_id": "media-server", "label": "weekly bounce"}'

Enabling metrics: SSH into client, install psutil, restart agent.

pip3 install psutil
sudo systemctl restart remotepower-agent

After a minute, sparklines appear on the dashboard.

Adding a new device to an existing group via API:

curl -k -X POST https://my.server/api/groups \
  -H "X-Token: ..." \
  -H "Content-Type: application/json" \
  -d '{"device_id": "newbox", "group": "homelab"}'

8. Troubleshooting with the source code in mind

SymptomWhat the code says
Dashboard shows device offline but agent is runningThe server marks a device offline if last_heartbeat is older than 120 seconds. Check journalctl -u remotepower-agent -f. Likely network issue or token mismatch. The agent logs HTTP errors, including 403 if token invalid.
Enrollment says "invalid PIN"PIN is stored in enroll.json with a created_epoch. Server rejects if current time > created + 600 seconds, or if PIN already used.
Reboot command never executesThe agent only runs commands received in a heartbeat response. Make sure the device ID in commands.json matches exactly. Use the dashboard to issue the command. Check agent logs: it prints "received command: reboot".
Long‑poll exec always times outNginx fastcgi_read_timeout must be ≥ your timeout parameter. The install script sets it to 130s. If you changed it, fix it. Also, the agent must pick up the command within that window.
Wake‑on‑LAN does nothingServer code uses the device's last known IP from heartbeat, constructs a broadcast address by replacing the last octet with 255 (or uses the wol_broadcast config if set). If the server and target are on different subnets, you need directed broadcast enabled on your router, or use a WoL relay. Check the Nginx access log for the /api/wol call and any errors.
Agent won't start after installationThe systemd service expects the binary at /usr/local/bin/remotepower-agent. Verify it exists and is executable. Run sudo remotepower-agent run in foreground to see tracebacks.
Dashboard cannot change admin passwordThe first login forces a password change because users.json has "must_change": true. After that, POST /api/change-password updates the bcrypt hash.
Metrics not showingAgent only collects metrics if psutil is importable. Install it, restart agent. Metrics are sent every heartbeat if available. The server stores up to 1440 snapshots.
Server uses a lot of CPUNot from the code—it's CGI, so a process spawns per request. High traffic could stress the server; add caching or use a reverse proxy. The monitor script is separate and runs periodically via cron.

9. A few quirks you should know about

  • No database, just JSON files locked with fcntl.flock. If two simultaneous requests try to write, one waits. It's robust enough for a handful of devices, but don't expect enterprise throughput.
  • The agent update downloads the new script over whatever URL the server gives. The install script sets up HTTPS, but if you misconfigure Nginx, the agent might download in plain text. The SHA‑256 check still protects integrity, but the agent's code is visible on the wire. Not a big deal on a trusted LAN.
  • The agent runs as root all the time. There's no privilege dropping. Custom commands execute as root. Set a per‑device command allowlist in the CMDB if you have other people using the dashboard.
  • The long‑poll exec hack is clever but fragile: the CGI script writes to longpoll.json, the heartbeat handler reads it and sets a threading.Event, which the waiting CGI thread picks up. If the file gets corrupted or the timing is off, the long poll hangs until Nginx kills it.
  • WoL might not work if the IP changes while the device is asleep. The server remembers the last IP, but if DHCP reassigns it, the magic packet goes into the void. Assign static reservations.
  • The dashboard has no logout button. Seriously, there's no logout in the code. Close the browser or delete the session cookie manually.
  • API keys never expire. Rotate them manually. They're stored alongside a bcrypt hash for verification, but the plain key is returned on creation and never shown again, so it's okay.

10. Backups and disaster recovery

Everything lives in /var/lib/remotepower/. Back it up:

sudo tar czf remotepower-backup-$(date +%F).tar.gz /var/lib/remotepower/

The dashboard has an Export backup button that zips the same files (minus session tokens). You can import by extracting into a fresh installation on another server.

If you lose everything, reinstall the server, then re‑enroll each client. Clients keep their tokens, but the server won't know them until they re‑enroll. Use sudo remotepower-agent re-enroll to keep device history.

11. Summary

RemotePower is a straightforward, file‑based remote management tool that does exactly what it says on the tin. It's not pretty inside—one giant CGI script, no framework, some threading hacks—but it works reliably for home labs and small deployments. The agent's polling design means zero firewall headaches, and the API is simple enough to integrate with your cron jobs or monitoring stack.

Read the source if something breaks; the logic is linear and well‑commented. And if you're still stuck, re‑enroll the client—it fixes 90% of problems.

Now go manage your machines from the comfort of your browser, and remember: with great power comes great "oops, I just rebooted production."
